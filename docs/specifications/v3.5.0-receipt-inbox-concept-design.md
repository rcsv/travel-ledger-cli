## 1. 背景

v3.4.0 では **Itinerary-level Planned vs Actual difference** を `trip export-md` に導入した。確定済み Estimate と Expense を前提とした、予定と実績の比較が Itinerary 単位で見えるようになった。

ただし、実際の家族旅行では、すべての支払いが即座に Itinerary に紐づけられるとは限らない。

例:

* 初日に 1,700 円のレシートがある
* 日付と金額は分かる
* レシート画像は残っている
* しかし、何の支払いだったかすぐには思い出せない
* 後で調べたら、親が買ったお土産だった

このような支払いは、発生時点では **Expense として確定できない**。しかし、後で確認するための **証拠としては残しておきたい**。

Receipt Inbox は、この「未整理の支払い証拠」を受け止める概念である。

---

## 2. 問題意識

現行モデルでは Expense は Itinerary にぶら下がる。

```text
Trip
  └─ Day
      └─ Itinerary
          └─ Expense
```

この構造は、**使途が分かっている支出** には適している。

一方で、以下のような支払いには扱いづらい。

* どの Itinerary に紐づくか分からない
* 何の支払いか分からない
* レシート画像だけ先に残したい
* 後で調べてから Expense 化したい
* 家族の誰かが支払ったものを後から確認したい

「とりあえずレシートを残す」行為を、Expense やダミー Itinerary に押し込むと、**Expense の意味が曖昧** になり、**Itinerary が未整理支出の置き場** になってしまう。

---

## 3. Receipt Inbox の目的

Receipt Inbox は、**Itinerary に紐づける前の支払い証拠** を一時保存するための概念である。

目的は旅行中の予算監視ではなく、**旅行後の振り返り・確認・整理** を支援すること。

主な用途:

* 金額だけ分かる支払いを残す
* 日付だけ分かる支払いを残す
* 「これなんだっけ？」というメモを残す
* 後で Itinerary / Expense に紐づける
* 不要・対象外と判断したものを無視済みにする

レシート画像の保存も **概念的には** 有用だが、画像ファイルの扱いは Caglla CLI の現在の責務を超えるため **§7 で deferred** とする（metadata-only を先に検討する）。

---

## 4. Expense との違い

Receipt と Expense は **同じものではない**。

| Concept | Responsibility |
|---|---|
| **Estimate** | Planned amount before the trip |
| **Receipt** | Unreviewed payment evidence |
| **Expense** | Confirmed actual spending |

- **Expense** は Planned vs Actual の **Actual** として扱える。
- **Receipt** は、まだ Actual として **確定していない**。

そのため、Receipt Inbox に入っている段階では、通常の Planned vs Actual difference には **直接含めない**。

ただし、post-trip review では **補助情報** として表示できる余地がある。

例:

```text
Confirmed Actual: JPY 18,300
Unreviewed Receipts: JPY 1,700
Potential Actual: JPY 20,000
```

`Potential Actual` は参考表示であり、Difference の正本ではない。Difference は **確定済み Expense** を前提とする（v3.3.0 / v3.4.0 方針を継続）。

---

## 5. Receipt の状態

候補:

```text
unreviewed
linked
converted
ignored
```

| 状態 | 意味 |
|---|---|
| `unreviewed` | まだ確認していない |
| `linked` | Itinerary または Expense に紐づいた |
| `converted` | Receipt を元に Expense を作成済み |
| `ignored` | 旅行費用として扱わない、または重複・不要と判断した |

`linked` と `converted` の使い分けは実装計画フェーズで確定する。本 concept では「未整理 → 整理済み」の遷移を表現できることのみ要求する。

---

## 6. 想定フィールド

将来の Entity Design 向け **初期候補**（v3.5.0 時点では **実装しない**）:

```text
receipt_id
trip_id
day_id optional
itinerary_id optional
linked_expense_id optional
amount optional
currency optional
occurred_date optional
memo optional
image_path optional   ← future candidate; see §7 (deferred)
status
created_at
updated_at
```

重要なのは、**最初からすべてを必須にしない** こと。

Receipt Inbox の価値は、**分からないものを分からないまま保存できる** ことにある。

実装する場合の **第一候補** は metadata-only。最低限、以下だけでも登録できる余地を残す。

```text
trip_id
amount
currency
memo
```

または、

```text
trip_id
memo
```

`image_path` は将来候補としてフィールド一覧に残すが、**先行実装しない**（§7）。

## 7. Receipt image handling

Receipt images are useful as payment evidence, but **image handling is intentionally deferred**.

Caglla CLI does not currently define how receipt image files should be stored, exported, imported, displayed, archived, or migrated. A partial `image_path` implementation would make the model look simple at first, but it would also introduce unresolved responsibilities around file location, portability, missing files, and future attachment/photo handling.

For this reason, Receipt Inbox remains a **concept-level design** in v3.5.0. Receipt image support, generic attachment/photo handling, OCR, and automatic receipt parsing are **future work**.

If Receipt Inbox is implemented in the future, **metadata-only receipts** should be considered before introducing image file handling.

### 日本語での意図

レシート画像は便利だが、画像を扱い始めると Caglla CLI / CUI の現在の責務を超える。議論の中で整理が必要になった未決事項の例:

* 画像ファイルをどこに置くのか
* 相対パスか絶対パスか
* export / import で画像をどう扱うのか
* 画像ファイルの移動・欠損をどう扱うのか
* ZIP / archive 化するのか
* CUI で画像をどう確認するのか
* OCR や自動解析に踏み込むのか
* Attachment / Photo 汎用モデルとどう分けるのか

Receipt Inbox は有用な概念として残すが、**画像対応を含む実装は急がない**。

### Partial implementation を避ける（現時点）

以下は **意図的に避ける**:

```text
- receipts table の追加
- image_path field の先行実装
- receipt image の export / import
- receipt image archive
- OCR
- automatic receipt parsing
- CUI image viewer 的な機能
- generic Attachment / Photo model への拡張
```

Receipt image は **Attachment / Photo / media handling strategy** と合わせて将来検討する。中途半端に `image_path` だけを先行実装すると、後で export/import や Attachment / Photo 設計と衝突する可能性がある。

## 8. CLI / CUI コマンド案

あくまで **concept-level** の案。実装計画・サブコマンド名・フラグは別フェーズで確定する。

**metadata-only** の例:

```bash
caglla receipt add --trip okinawa --day 1 --amount 1700 --currency JPY --memo "これなんだっけ？"
```

将来の image 対応案（**deferred** — §7）:

```bash
caglla receipt add --trip okinawa --day 1 --amount 1700 --currency JPY --image ./receipts/day1-1700.jpg --memo "これなんだっけ？"
```

```bash
caglla receipt list --trip okinawa
caglla receipt list --trip okinawa --unreviewed
```

```bash
caglla receipt show 3
```

```bash
caglla receipt link 3 --itinerary 12 --memo "お土産"
```

```bash
caglla receipt convert 3 --itinerary 12 --title "お土産" --category shopping
```

```bash
caglla receipt ignore 3 --memo "旅行費用ではない"
```

`convert` は Receipt を元に **新規 Expense を作成** し、Receipt を `converted` に遷移させる想定。`link` は既存 Expense / Itinerary への関連付けのみ（詳細は Entity Design で確定）。

---

## 9. 設計上の注意点

### Expense の意味を広げすぎない

Expense は **「確定済み実績支出」** として維持する。Receipt はその前段階の **未整理証拠** である。Receipt を Expense に昇格させるまで、Actual 集計に含めない。

### Itinerary をゴミ箱化しない

「未整理レシート」という **ダミー Itinerary** を作る案は避ける。Itinerary は **旅行行動の単位** であり、未整理支出の置き場ではない。

### Planned vs Actual に即時反映しない

Receipt は未確定のため、通常の Actual Expense と同列に扱わない。ただし、未整理金額として **補助表示** する価値はある（§4 の `Unreviewed Receipts` / `Potential Actual`）。

### Balance / Settlement とは分ける

Receipt Inbox は、誰が払ったか・誰が負担するか・誰に返すかを扱う機能ではない。それは **Balance / Settlement** の領域であり、別レイヤーとして defer する（[shared-expense-model.md](shared-expense-model.md) 方針を継続）。

### Attachment / Photo 汎用化に広げすぎない

Receipt image は **支払い証拠** であり、旅行写真や添付ファイル一般とは目的が異なる。v6 Travel Journal の Photo / Attachment 汎用化とは **別トラック**。Receipt image 自体も **§7 で deferred** — generic Attachment / Photo / media handling strategy と合わせて将来検討する。

---

## 10. 位置づけ

Receipt Inbox は、**旅行中の予算統制機能ではない**。旅行後に「この支払いは何だったのか」を整理するための **post-trip review 支援機能** である。

| 観点 | v3.4.0 Itinerary Difference | Receipt Inbox |
|---|---|---|
| 前提 | 確定済み Estimate + Expense | 未整理の支払い証拠 |
| タイミング | 旅行中〜旅行後（確定分） | 主に旅行後の振り返り |
| Planned vs Actual | Difference の Actual に含む | **含めない**（補助表示のみ） |

v3.4.0 の Itinerary-level Planned vs Actual difference は、確定済み Expense を前提とした比較機能だった。Receipt Inbox は、その **前段階** にある未整理支出を受け止める概念である。

---

## 11. Non-goals

本 concept design および v3.5.0 の **現時点では実装しない** 範囲:

```text
- Receipt Inbox の実装（future work — metadata-only から検討）
- receipts table / DB migration
- image_path field の先行実装
- receipt image export / import / archive
- image binary storage
- CUI image viewer 的な機能
- OCR
- automatic receipt parsing
- generic Attachment / Photo model
- export schema 変更
- import schema 変更
- Day-level Planned vs Actual implementation
- Balance / Settlement
- Participant-based payment sharing
- doctor / advisor 連携（別候補として defer 可能）
```

Day-level Planned vs Actual difference、Balance / Settlement、Attachment / Photo 汎用化、**Receipt image handling** は、本書の対象外として **引き続き defer**。

---

## 12. 結論

```text
Estimate    = planned amount
Receipt     = unreviewed payment evidence
Expense     = confirmed actual spending
Difference  = planned-vs-actual delta based on confirmed Expenses
```

Receipt Inbox は、家族旅行における現実的な支出整理に合っている。旅行中にすべての Expense を正確に登録するのではなく、**とりあえず未整理の証拠を残し、旅行後に見返し、分かったものを Expense に昇格させる** 流れを支える **概念として採用済み**。

**Receipt Inbox 実装は future work。** 実装する場合も **metadata-only を先** にし、Receipt image 対応はさらに後段とする（§7）。

次フェーズ（未着手）: Entity Design → Implementation Plan → Post-Implementation Review。doctor / advisor との連携は Receipt 実装後または並行で v3.5.x 以降に検討。
