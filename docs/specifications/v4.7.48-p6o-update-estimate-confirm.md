## 1. Confirm transaction

```text
with_transaction("update_estimate confirm")
  1. Fragment 再 parse（intent == update_estimate）
  2. blocking 条件再確認（parse / required_decisions 相当）
  3. Itinerary target 再 resolve
  4. estimate_id から Estimate 再取得
  5. 親子関係再確認（itinerary_id 一致）
  6. proposed value 再計算
  7. gate preview agreement 検証
  8. stale baseline / TOCTOU 検証（expected_* + field_changes.before）
  9. exactly one UPDATE（WHERE id + itinerary_id）
 10. 更新後 row 再取得・verification
 11. updated_estimate_id 設定
 12. commit
```

失敗時は **rollback**、DB 不変。

## 4. UPDATE 契約


| affected rows | 扱い |

## 7. Confirm safety tests（v4.7.48）

正式リリース時に以下を unit test で検証済み。

### Candidate / preview mismatch

* stale gate report（例: dry-run proposed amount `1250`）と変更済み confirm candidate（例: `1300`）を拒否
* `updated_estimate_id` なし
* 対象 Estimate / `updated_at` / Expense 不変
* **CLI からの直接再現は不可**（単一 `fragment apply --confirm` 呼び出しでは gate と transaction が同一 JSON を共有）。production fault injection なしの `apply.rs` unit test で検証（`delete_itinerary` TOCTOU テストと同型）

### Scoped UPDATE 0 rows

* `update_estimate_row_scoped` へ存在しない Estimate ID または誤 `itinerary_id` → error
* exactly-one UPDATE 契約維持
* commit 可能な success を返さない

### Post-update verification mismatch

* `verify_updated_estimate_matches_proposed` が proposed と再取得 row の不一致（例: amount）を error として検出

### Test layering

dry-run と confirm は同一 parser / resolver を共有。全 blocking case を confirm 専用 CLI test へ重複させない。confirm テストの重点: transaction、preview agreement、TOCTOU、scoped UPDATE、post-update verification、rollback。
