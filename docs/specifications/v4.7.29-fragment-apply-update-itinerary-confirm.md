[v4.7.28](v4.7.28-fragment-apply-update-itinerary-dry-run.md) の `update_itinerary` dry-run に続き、**P-6i Phase B** として `fragment apply --confirm` による Itinerary 1行 UPDATE を実装します。

---

## 1. 処理フロー

```text
fragment validate
  → target itinerary 解決
  → update_itinerary preview（v4.7.28 同型 gate）
  → confirm scope 検証
  → TOCTOU 再検証（expected_<field> + preview.before）
  → itinerary_items UPDATE 1件
  → itinerary show / trip export で確認
```

---

## 2. supported scope

```text
intent:   update_itinerary
target:   itinerary のみ
DB:       itinerary_items UPDATE 1件のみ
```

### allowed MVP fields（v4.7.28 維持）

`title`, `note`, `location`, `category`, `start_time` / `time`, `duration_minutes` / `duration`, `travel_minutes` / `travel_time_minutes` / `travel`

`day`, `day_id`, `sort_order`, reorder, delete, done/undone, child entity updates は **unsupported**。

---

## 3. gate report

| フィールド | 意味 |
|---|---|
| `updated_itinerary_id` | confirm 成功時の更新対象 Itinerary ID |
| `preview.itinerary_field_changes[]` | dry-run 同型 `{ field, before, after }` |

`inserted_*_id` は設定しません。report `schema_version` は **2** のまま。

---

## 4. TOCTOU

confirm write 直前に:

1. 現行 DB 値を再読込み
2. `expected_<field>` baseline を再検証
3. `preview.itinerary_field_changes[].before` と現行 DB を比較

不一致時は **UPDATE 前に blocking** — DB 不変。

---

## 5. 安全条件

以下を満たさない場合は DB 更新しません:

- Fragment validation valid
- Trip 存在
- target itinerary 一意解決（unresolved / ambiguous は blocking）
- intent `update_itinerary`
- allowed MVP field のみ有効
- preview 生成成功
- blocking errors なし
- required decisions 空（`adoption_hints.conflicts` 含む）
- no-op / invalid category / invalid time / negative numeric → blocking
- TOCTOU / baseline mismatch → blocking
- `--confirm` 明示

---
