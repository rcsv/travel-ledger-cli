# v4.7.37 — P-6l cross-day itinerary move planning (documentation-only)

v4.7.37 は **planning-only** として、Itinerary を **Day を跨いで移動**する structural operation を定義する。
本書は **実装を含まない**（Rust / tests / fixtures / Cargo version / schema v8 / DB migration / golden は変更しない）。

---

## 0. Status（結論サマリ）

| 観点 | 判定 |
|---|---|
| intent | **`move_itinerary`**（canonical） |
| scope | **Trip 内 cross-day move（Day → Day）**、Itinerary 1 件のみ |
| same-day | **reject**（`reorder_itinerary` を使用） |
| ordering | Sequence-first（`sort_order` 主導）を維持 |
| selectors | **ID-based order arrays を推奨**（GUI drag-and-drop 契約の第一候補） |
| confirm | transaction + TOCTOU + row count verification 必須 |
| mutation boundary | `itinerary_items.day_id` + `sort_order` + `updated_at`（影響行のみ） |

---

## 1. Definition（`move_itinerary` の意味）

`move_itinerary` は、**1 つの itinerary item を source Day から destination Day へ移動**し、destination Day の ordering に **挿入**する structural operation である。

- **対象**: Day 所属（`day_id`）と ordering（`sort_order`）
- **非対象**: title / start_time / category / location / note / child entities
- **原則**: Sequence-first ordering（`sort_order` が primary sequence）

GUI 操作（drag & drop）の DB 意味論:

- source Day から **除外**
- destination Day に **挿入**
- 2 つの Day の ordering が同時に変わり得る

---

## 2. Why not extend `reorder_itinerary`

本 intent は `reorder_itinerary` に統合しない。

```text
reorder_itinerary = same-day sort_order change（Day 内のみ）
move_itinerary    = cross-day day_id + ordering change（Day 間）
```

理由:

- `reorder_itinerary` は **単一 Day の ordering**のみを扱う（surface が小さい）
- cross-day move は **2 Day**（source + destination）を同時に扱い、TOCTOU surface が大きい
- GUI 見た目は近くても、DB semantics が異なる

---

## 3. Canonical intent name

### 3.1 採用: `move_itinerary`

v4.7.37 planning では intent 名を **`move_itinerary`** とする。

- **短く自然**で、将来の move variant（例えば move+reorder の拡張）も包含しやすい
- 「Day 所属の変更」を **payload の必須フィールド**と validation で明示し、曖昧さを排除する

`move_itinerary_to_day` は冗長であり、Day 所属変更は `from_day` / `to_day` によって十分に表現できるため採用しない。

---

## 4. Scope（v4.7.37 planning が定義する対象）

将来実装（v4.7.38+ 想定）の scope:

- **1 itinerary item** のみ
- **同一 Trip 内**の move のみ（cross-trip 禁止）
- source Day と destination Day は **異なる**
- destination Day の挿入位置は **explicit order** で指定
- validation は **baseline mismatch を blocking** とする（expected order 必須）
- child entities（Expense / Estimate / Reservation / Note entity / Venue / Place 等）は変更しない

---

## 5. Non-goals（本 planning の明示的な除外）

- v4.7.37 での実装（dry-run / confirm ともに含めない）
- `reorder_itinerary` の仕様変更
- 複数 itinerary の同時移動
- Day の移動、Trip 間移動
- child entities の移動・削除・cascade
- Venue / Place model work
- schema v8 changes / DB migration / export schema changes
- JSON golden changes、unrelated refactoring

## 6. Proposed fragment payload（案）

GUI-friendly / safety-first を優先し、**expected/after を source と destination の両方**で明示する。

```json
{
  "target": { "target_type": "itinerary", "day_reference": 1, "itinerary_reference": 102 },
  "fragment": {
    "intent": "move_itinerary",
    "candidate_content": {
      "from_day": 1,
      "to_day": 2,
      "expected_source_order": [101, 102],
      "expected_destination_order": [201, 202],
      "after_source_order": [101],
      "after_destination_order": [201, 102, 202]
    }
  }
}
```

### 6.1 Selector contract

- **preferred**: itinerary_id の配列（GUI drag-and-drop の自然な契約）
- **allowed (CLI convenience)**: title selector / sort_order selector
- いずれも apply 時に **itinerary_id へ一意に解決**し、曖昧な selector は reject
- 数値 selector は `itinerary_id` 優先、無ければ `sort_order`（両方一致は曖昧として reject）

### 6.2 same-day handling

`from_day == to_day` は **reject**。

```text
same-day ordering is handled by reorder_itinerary
```

自動変換はしない（silent behavior change を避ける）。

---

## 7. Required validation model（将来 dry-run / confirm 共通）

### 7.1 Shared validation（必須）

- target itinerary が存在
- source Day / destination Day が存在
- source/destination が **同一 Trip に属する**
- source Day と destination Day が **異なる**
- target itinerary が **現行 DB で source Day に属する**
- `expected_source_order` / `expected_destination_order` / `after_source_order` / `after_destination_order` が必須
  - source Day の `after_source_order` は、source が 1 件のみだった場合 **空配列を許可**
- 各配列内に重複なし
- selector が一意に解決される（ambiguous reject）
- `expected_source_order` が現行 DB の source Day の順序と一致
- `expected_destination_order` が現行 DB の destination Day の順序と一致
- `after_source_order` == `expected_source_order` から moved itinerary を **ちょうど 1 つ**除いた結果
- `after_destination_order` == `expected_destination_order` に moved itinerary を **ちょうど 1 回**挿入した結果
- moved itinerary は `after_destination_order` に 1 回だけ出現し、`after_source_order` には出現しない
- unrelated itinerary の追加/削除は禁止（集合差分が moved itinerary 以外にあれば reject）
- cross-trip move は reject

## 8. Future dry-run preview shape（案）

dry-run は「どれが移動し、source/destination の ordering がどう変わるか」を明示する。

```json
{
  "preview": {
    "action": "move_itinerary",
    "move_preview": {
      "itinerary_id": 102,
      "title": "Itinerary 1-2",
      "from_day_number": 1,
      "to_day_number": 2,
      "source_order_changes": [
        { "itinerary_id": 101, "title": "Itinerary 1-1", "before_sort_order": 1000, "after_sort_order": 1000 }
      ],
      "destination_order_changes": [
        { "itinerary_id": 201, "title": "Itinerary 2-1", "before_sort_order": 1000, "after_sort_order": 1000 },
        { "itinerary_id": 102, "title": "Itinerary 1-2", "before_day_number": 1, "after_day_number": 2, "before_sort_order": 2000, "after_sort_order": 2000 },
        { "itinerary_id": 202, "title": "Itinerary 2-2", "before_sort_order": 2000, "after_sort_order": 3000 }
      ]
    }
  }
}
```

注記:

- moved itinerary は destination 変化側で `before_day_number` / `after_day_number` を明示
- source/destination は **別配列**で表示し、影響範囲が直感的に分かるようにする

## 9. Future confirm behavior（方針のみ / 未実装）

confirm は transaction + TOCTOU revalidation が必須。

transaction 内で:

1. source Day itineraries を再取得
2. destination Day itineraries を再取得
3. selector を再解決（すべて itinerary_id）
4. source baseline を再検証（expected_source_order）
5. destination baseline を再検証（expected_destination_order）
6. moved itinerary が source Day に属することを再検証
7. source/destination が同一 Trip であることを再検証
8. source after ordering の sort_order を計算
9. destination after ordering の sort_order を計算
10. moved itinerary を更新:
    - `day_id`
    - `sort_order`
    - `updated_at`
11. 影響する source/destination の rows を更新:
    - `sort_order`
    - `updated_at`
12. row count verification
13. 成功時のみ commit

失敗時は rollback。

## 11. Sort order strategy（難所と推奨案）

### 11.1 難所

same-day reorder は「件数が不変」なので既存 `sort_order` slot をそのまま再割当できる。
一方 cross-day move は destination の件数が **+1** になり、**新しい slot** が必要になり得る。

### 11.2 v4.7.38+ 推奨案（初期実装方針）

**方針: “最小更新 + deterministic slot 生成”**

- **source Day**:
  - moved itinerary を除外した後、残存 items に対して **既存 source slots（昇順）**を割当
- **destination Day**:
  - destination の既存 slots を保持しつつ、挿入位置に必要な **追加 1 slot** を生成してから、after ordering に割当
  - 追加 slot の生成は以下の順で試行（deterministic）:
    1. 挿入前後の sort_order の **midpoint**（隙間があれば）
    2. 先頭/末尾への挿入なら `SORT_ORDER_STEP` で前後に追加
    3. どれも不可能なら、destination Day のみ **normalize（再採番）**を別フェーズで検討（初期は blocking でもよい）

この方針は「既存の並びが概ね安定している」ケースで update 範囲を小さくできる。
