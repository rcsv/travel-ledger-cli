## 1. 目的

```text
fragment validate
  → fragment apply --dry-run --trip <id>（推奨）
  → fragment apply --confirm --trip <id>
  → 既存 Day に itinerary を1件追加（拡張フィールド付き）
  → trip export / itinerary list / day show で確認
```

v4.7.19 では `title` / `note` / `location` のみ反映していました。v4.7.21 では dry-run preview と confirm 後 DB の差を減らし、より実用的な itinerary として保存できるようにします。

## 2. confirm 反映フィールド

### 2.1 v4.7.19 から維持

| Fragment ソース | DB / export フィールド | 備考 |
|---|---|---|
| `fragment.notes` | `note` | 従来通り |
| `candidate_content.title` | `title` | **必須** |
| `candidate_content.location` | `location` | 未指定時 `None` |

### 2.2 v4.7.21 で追加

| Fragment ソース | DB / export フィールド | 検証 |
|---|---|---|
| `candidate_content.category` | `category` | 既存 `ItineraryCategory` enum / `parse_itinerary_category` — 不正値は **blocking error** |
| `candidate_content.start_time` | `start_time` | `HH:MM` — 不正値は **blocking error** |
| `candidate_content.duration_minutes` | `duration_minutes` | 非負整数 — 不正値は **blocking error** |
| `candidate_content.travel_minutes` または `travel_time_minutes` | `travel_minutes` | 非負整数；両方指定で矛盾時 **blocking error** |

**category 有効値:** `flight`, `hotel`, `restaurant`, `activity`, `transport`, `shopping`, `beach`, `museum`

### 2.3 未指定フィールド

未指定は従来通り安全な default / `None` です（`category`, `start_time`, `duration_minutes`, `travel_minutes` はすべて optional）。

## 3. sort_order と ordering hint

### 3.1 confirm 方針（安全優先）

```text
confirm: Day 末尾 append（sort_order: None → add_itinerary_item の sequence-first 解決）
dry-run preview: 既存 Day の max(sort_order) + SORT_ORDER_STEP（1000）
```

- **reorder 実装なし** — `--after` / `--before` は未対応
- Fragment 側の明示的 ordering hint は **confirm では無視**し、常に Day 末尾に append

### 3.2 dry-run warning（ordering hint）

以下が Fragment に含まれる場合、gate report の `warnings` に `ordering_hint:` プレフィックス付きで記録します（**blocking ではない**）:

| キー | warning 内容 |
|---|---|
| `candidate_content.sort_order` | confirm では無視、Day 末尾 append |
| `candidate_content.placement_hint` | preview のみ — confirm は末尾 append |
| `candidate_content.after` / `before` | v4.7.21 未対応 — 末尾 append |
| `fragment.placement_hints` | v4.7.21 未反映 — 末尾 append |

本格的な `--after` / reorder は **後続マイルストーンへ defer**。

### 3.3 dry-run preview と confirm の sort_order

dry-run preview は `SORT_ORDER_STEP`（1000）による sparse ordering で末尾位置を **シミュレート**します。confirm は `add_itinerary_item(..., sort_order: None)` により同一 Day の sequence-first 規約で末尾に割り当てます。通常は preview の `sort_order` と confirm 後 export の `sort_order` が一致します。

## 4. unsupported field

`candidate_content` に supported 以外のキーがある場合、**warning**（`unsupported_field:` プレフィックス）を出します。DB は更新しません（confirm 時も gate 通過後に supported フィールドのみ反映）。

supported キー一覧:

```text
title, location, category, start_time, duration_minutes,
travel_minutes, travel_time_minutes,
sort_order, placement_hint, after, before  （後4つは ordering hint — confirm 未反映）
```

---

## 5. dry-run / confirm 原則

```text
fragment apply --dry-run:
  read-only DB access
  in-memory simulation（共有パーサー）
  preview output（--output 指定時）
  Trip domain data は変更しない

fragment apply --confirm:
  gate 通過後に itinerary insert 1件
  supported フィールドのみ DB 反映

fragment apply --dry-run --confirm:
  error
```

---

## 6. 安全条件（confirm）

以下を満たさない場合は DB 更新しません:

- Fragment file validation が valid
- `--trip <id>` の Trip が存在
- target Day が一意に解決できる
- intent が `add` / `add_itinerary`
- apply preview が生成できる
- blocking errors がない
- required decisions が unresolved のまま残っていない
- `--confirm` が明示されている
- 反映対象フィールドが supported scope 内（不正 category / time / 整数は blocking）
