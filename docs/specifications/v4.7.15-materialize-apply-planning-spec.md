## 2. `materialize` と `apply` の責務

### 2.1 定義（確定方針）

| 操作 | 入力 | 出力 | DB |
|---|---|---|---|
| **`proposal materialize`** | Trip Proposal Envelope + 人間パラメータ | **新規** schema v8 Trip | 初期は JSON file；後段で DB insert |
| **`fragment apply`** | Proposal Fragment + target Trip + 人間パラメータ | **既存** schema v8 Trip の更新 | 同上 |

```text
materialize:
  「旅行全体の未採用案」→ 新しい Trip 候補を生成

apply:
  「既存 Trip への部分提案」→ 既存 Trip / Day / Itinerary への変更

混同しない:
  fragment materialize（小さい Trip 化）— 却下
  proposal apply（Envelope を既存 Trip に上書き）— 却下
  trip import で Proposal を取り込む — 却下
```

### 2.2 なぜ分離するか

1. v4.7.3 — Fragment は **小さい Trip ではない**
2. gate 入力が異なる — Envelope は日付・全体構成 / Fragment は target・intent・conflict
3. 出力が異なる — 新規 Trip ID vs 既存 Trip 更新
4. リスクが異なる — apply は既存データ破壊リスクが高い
5. CLI / validator が既に `proposal` / `fragment` で分離済み

### 2.3 採用しない方針

| 方針 | 却下理由 |
|---|---|
| 単一 `proposal adopt` で Envelope / Fragment 両対応 | 境界曖昧；エラーメッセージが貧弱 |
| Envelope を既存 Trip に merge する `apply` | Envelope の意味（全体案）と矛盾 |
| Fragment から新規 Trip を `materialize` | Fragment は部分提案；新 Trip は Envelope 経路 |
| validation PASS = 自動 DB 反映 | AI 無批判取り込み；v4.7.4 で却下済み |
| gate なし `trip import` 経路 | schema v8 専用 import の意味を壊す |

## 3. Adoption gate フロー（P-5 具体化）

[v4.7.4 §5.4](v4.7.4-materialize-gate-concept-validation-rules.md) を P-6 実装向けに展開。

### 3.1 共通 gate パイプライン

```text
1. Load proposal file（Envelope or Fragment）
2. Kind-specific file validation（既存 validate コマンド再利用）
3. Collect required decisions（hints + gate ルール）
4. Apply human parameters（CLI flags / 将来 interactive）
5. Gate blocking validation
6. Gate warnings 列挙
7. Preview / dry-run（DB 触らない）
8. Human explicit confirm
9. Commit（JSON file または DB）— P-6 段階で選択
10. Post-commit: trip validate-export on output Trip
11. Lifecycle: pending → adopted（+ trip_id 参照）
```

**ステップ 7〜8 が v4.7.15 で必須とする安全境界。**

### 3.2 Envelope materialize フロー

```text
入力:
  envelope.json
  人間パラメータ: start_date, end_date, 採用する candidate_days / itineraries 等

Gate blocking（例）:
  実日付未確定
  title / destination 欠如
  採用 itinerary が空

Gate warning（例）:
  stale_source, valid_until 超過, materialize_hints 非空

Dry-run 出力:
  schema v8 Trip JSON（stdout または --output）

Confirm 後（P-6b+）:
  trip import 相当の DB insert（専用経路 — trip import コマンドとは別）
```

### 3.3 Fragment apply フロー

```text
入力:
  fragment.json
  target_trip_id（または --trip）
  人間パラメータ: 解決済み day / itinerary, placement 等

Gate blocking（例）:
  target Trip 不在
  target Day / Itinerary 削除済み
  intent と candidate_content の不整合

Gate warning（例）:
  conflicts, unresolved_target_hints, adoption_hints

Dry-run 出力:
  更新後 schema v8 Trip JSON（既存 Trip export + apply 差分の preview）

Confirm 後（P-6c+）:
  既存 Trip への DB 更新（itinerary add 等 — 専用経路）
```

## 4. dry-run / preview / diff — 必須か

### 4.1 v4.7.15 判断: **DB 書き込み前は必須**

| 段階 | dry-run | preview | diff | 必須度 |
|---|---|---|---|---|
| P-6a file-only materialize | `--dry-run` → JSON | human-readable summary | optional `trip diff` vs empty | **必須** |
| P-6b DB materialize | 同上を先に実行 | 同上 | 推奨 | **必須（DB 前）** |
| P-6c fragment apply | 更新後 Trip JSON preview | target / intent 要約 | **既存 Trip export vs preview** | **必須** |

```text
理由:
  Trip DB 変更は不可逆に近い
  既存 trip diff コマンドを gate 後 preview と接続できる
  file-only 段階でも gate 設計を検証できる
  AI 生成物は必ず人間が diff を見る文化を作る
```

### 4.2 推奨 CLI 形状（P-6 候補・未実装）

```bash
# Envelope — dry-run only（P-6a 最小）
caglla proposal materialize envelope.json --dry-run --output candidate-trip.json
caglla trip validate-export candidate-trip.json

# Envelope — DB commit（P-6b、要 --confirm）
caglla proposal materialize envelope.json \
  --start 2026-04-26 --end 2026-04-29 \
  --dry-run --output /tmp/preview.json
caglla trip diff /dev/null /tmp/preview.json   # 新規の場合は新 Trip 全体が diff
caglla proposal materialize envelope.json ... --confirm

# Fragment — preview against existing trip export
caglla trip export 1 --output current.json
caglla fragment apply fragment.json --trip 1 --dry-run --output preview.json
caglla trip diff current.json preview.json
caglla fragment apply fragment.json --trip 1 --confirm
```

`--dry-run` と `--confirm` は **排他ではない** — 典型フローは dry-run → 人間確認 → confirm 付き再実行。

### 4.3 採用しない方針

| 方針 | 却下理由 |
|---|---|
| dry-run なしで DB 直接更新 | 安全境界違反 |
| preview のみで confirm 不要 | 誤操作リスク |
| diff を gate 内部に隠す | 既存 `trip diff` との再利用・テスト性が下がる |

## 5. blocking / warnings / required decisions

### 5.1 三層の関係

```text
required decisions:
  人間がまだ決めていない項目 — 未解決なら gate 通過不可（blocking）

blocking errors:
  正式化の契約違反 — 採用停止

warnings:
  古さ・曖昧さ・衝突可能性 — 表示のみ、confirm 時に目視
```

**validation（file）と gate（adoption）は別レイヤー:**

| レイヤー | タイミング | 例 |
|---|---|---|
| file validation | validate コマンド | JSON 構造、intent 範囲 |
| adoption gate | materialize / apply 直前 | 実日付確定、target 存在、採用 itinerary 非空 |

file validation PASS でも gate blocking あり得る。

### 5.2 required decisions の表現（P-6 候補）

Envelope / Fragment の hints を拡張せず、**gate report** で統一:

```json
{
  "required_decisions": [
    {
      "id": "trip_dates",
      "label": "Confirm start_date and end_date",
      "status": "pending",
      "source": "materialize_hints"
    },
    {
      "id": "target_day",
      "label": "Resolve fragment target to day 2",
      "status": "resolved",
      "resolved_value": "day:2"
    }
  ]
}
```

| field | 意味 |
|---|---|
| `id` | 安定識別子（CLI `--set-decision` 候補） |
| `status` | `pending` \| `resolved` |
| `source` | `materialize_hints` / `adoption_hints` / `gate_rule` |

v4.7.15 では JSON schema **確定しない** — gate report の概念モデルのみ。

### 5.3 warning acknowledgment

P-6 初期: **confirm フラグが acknowledgment 代わり**（warnings を stdout に出したうえで人間が再実行）。

将来: `--acknowledge-warning stale_source` 等の明示列挙を検討。v4.7.15 では **必須にしない**。

## 6. ユーザー確認なしの Trip DB 変更禁止

### 6.1 安全境界（P-6 全体で守る）

```text
禁止:
  validate PASS だけで DB 更新
  AI assistant がバックグラウンドで materialize / apply
  デフォルトで DB に書く CLI（opt-out 式）

必須:
  --dry-run 可能
  DB 書き込みには --confirm（または別コマンド confirm ステップ）
  出力 Trip は trip validate-export で検証可能
```

### 6.2 AI / GUI / CLI の共通フロー

```text
AI 生成 → file 保存
       → proposal/fragment validate
       → show / inspect（人間レビュー）
       → materialize/apply --dry-run
       → diff / preview
       → 人間 confirm
       → DB commit（P-6b+）
       → trip validate-export（採用後 Trip）
```

GUI 将来: inbox で pending 表示 → preview 画面 → confirm ボタン。DB 書き込みは confirm ハンドラのみ。

## 7. lifecycle（pending / adopted / rejected / expired）との関係

[v4.7.13 §5](v4.7.13-proposal-storage-strategy-planning.md) を gate と接続。

| State | materialize / apply との関係 |
|---|---|
| **pending** | dry-run / preview 可能；confirm 前 |
| **adopted** | gate 通過・出力 Trip 生成済み；`adopted_trip_id` 記録 |
| **rejected** | gate 却下；DB 変更なし |
| **expired** | warning のみ；blocking にしない |

```text
遷移:
  pending + dry-run OK + confirm → adopted
  pending + reject 操作 → rejected（将来 CLI）
  pending + time → expired（soft、判断は人間）
```

file-only 段階: lifecycle は **manifest または手動メモ** でよい。DB storage なしでも gate 設計は成立。

## 8. file-only で planning / P-6a が成立するか

### 8.1 判断: **成立する**

| 操作 | file-only 可否 |
|---|---|
| validate / show / inspect | **済** |
| gate report 生成 | **可能** — JSON report 出力 |
| materialize dry-run → Trip JSON | **可能** — DB 不要 |
| `trip validate-export` on output | **可能** — 既存コマンド |
| `trip diff` preview | **可能** — 既存コマンド |
| DB trip insert / fragment 反映 | **P-6b 以降** |

```text
P-6 分割案:
  P-6a  Envelope materialize --dry-run only（JSON 出力 + gate report）
  P-6b  Envelope materialize --confirm → DB（専用 writer、trip import とは別）
  P-6c  Fragment apply --dry-run + --confirm（既存 Trip 前提）
```

DB storage 前に **P-6a だけ** で gate 思想と出力 Trip の品質を検証できる。

### 8.2 DB storage 前に許容する操作範囲

```text
許容:
  proposal/fragment file を path 指定
  dry-run 出力を任意 path に書く
  既存 trip export を diff 左辺に使う
  adopted 状態を manifest に手書き（optional）

許容しない（storage なしでは困難）:
  proposal list / inbox 一覧
  複数 pending の filter
  採用履歴の永続監査
```

```text

  + 出力 JSON が trip validate-export PASS
  + DB 触らない
  + --confirm なし（dry-run のみ）
```

## 11. Future consideration — AI-assisted Fragment authoring

将来の caglla.travel / GUI では、作成中の Trip / Day / Itinerary をもとに、旅程の不足部分を **Proposal Fragment** で補う操作が考えられる。v4.7.15 では **記録のみ** — 実装 scope 外。

### 11.1 責務の分離

```text
Gap / issue detection:
  Caglla-side doctor / advisor / rule-based analyzer の責務

Fragment generation:
  AI authoring / external provider の責務

Validation:
  fragment validate の責務

Adoption:
  materialize / apply gate の責務
```

**生成 AI が単独で「歯抜けを探す」前提にしない。** まず Caglla 側が doctor / advisor 的に不足・矛盾・曖昧さを **構造化 finding** として検出し、その finding を入力として AI が Proposal Fragment 候補を生成する。

### 11.2 Finding と Fragment の関係

Caglla 側が検出しうる finding の例:

- missing meals
- unresolved transportation
- overloaded days
- missing backup plans
- expired reservations
- ambiguous targets

finding は用途によって分類しうる:

- **人間向けのみ** — 説明・注意喚起
- **AI authoring 向け** — Fragment 生成の入力
- **両方** — 人間レビューと AI 候補生成の両方

将来の doctor / advisor 出力には、次のような **構造化分類** が必要になる可能性がある（v4.7.15 では schema 未確定）:

```text
severity, audience, target, category,
human_message, ai_action_hint, required_decision,
fragment_generatable（この finding から Fragment を生成してよいか）
```

### 11.3 安全境界（継承）

AI authoring は structured findings を入力に Proposal Fragment を生成してよい。ただし:

1. 生成物は **`fragment validate` を通過** すること
2. Trip データ変更は **`fragment apply` gate（dry-run → confirm）** を通過すること
3. finding 検出と Fragment 生成を **同一プロンプトに混ぜない** — 検出は Caglla、生成は provider

本節は P-6 / GUI 設計時の接続メモであり、doctor / advisor schema 変更・AI generation 実装・Fragment generation command は **v4.7.15 scope 外**。
