**固定ルール（v4.7.0〜v4.7.2 から継承）:**

```text
schema v8 Trip          = 採用済みの正式データ（実日付を持つ）
Trip Proposal Envelope = 旅行全体の未採用案（schema 外）
Proposal Fragment      = 既存 Trip / Day / Itinerary への部分提案（schema 外）
Materialize / Apply    = 人間が採用したときだけ正式 Trip へ反映する gate
```

**重要:** Fragment を「小さい Trip」として扱わない。既存 Travel Ledger Trip に対する **候補パーツ** である。

## 1. Proposal Fragment とは何か

### 1.1 定義

**Proposal Fragment** は、**既に存在する schema v8 Trip**（またはその Day / Itinerary）に対して、「これを追加・差し替え・補足したらどうか」と提案される **部分候補** です。

```text
Proposal Fragment:
  既存 Trip との関係が本質
  target（差し込み先）を持つ — 未確定可
  intent（何をしたい提案か）を持つ
  未採用・不完全可
  schema v8 の外側
  trip export 対象外
```

AI、旅行業者、ブログ、手入力など多様な source があり得る。採用されるまでは、正式な Trip / Day / Itinerary / Reservation / Expense / Note **ではない**。

### 1.2 Fragment が Trip ではない理由

schema v8 の各エンティティは **採用済み** の旅行記録を表す。Fragment は採用前の候補であり:

- 既存計画との **conflict** がありうる
- target が **未確定** でありうる
- 予約・金額などが **未確定** でありうる
- 情報が **古くなりうる**

これを itinerary や note に無理に押し込むと、正式データと候補が混ざり、export / validate-export / Travel Book の意味が曖昧になる。

```text
誤ったモデル:
  AI の「おすすめレストラン」を placeholder Itinerary として DB に入れる
  → 採用済み予定と候補案が区別できない

正しいモデル:
  Proposal Fragment（外側）で保持
  → 人間が target / conflict を確認して採用
  → schema v8 Trip の正式要素へ反映
```

### 1.3 Proposal Fragment の責務

| 責務 | 説明 |
|---|---|
| **識別** | Fragment を一意に参照（`fragment_id` 等） |
| **target 指定** | 既存 Trip / Day / Itinerary への差し込み先（未確定可） |
| **intent 表現** | 追加・補足・代替候補などの意図 |
| **候補内容** | 提案されるパーツ本体（不完全可） |
| **adoption hints** | 採用前の決定事項・conflict・warning |
| **schema v8 との分離** | export JSON の一部ではないことを明示 |

Fragment は **保存形式の実装詳細ではない**。v4.7.3 では概念のみ確定する。

## 2. Trip Proposal Envelope との違い

| 観点 | Trip Proposal Envelope | Proposal Fragment |
|---|---|---|
| スコープ | **旅行全体**の案 | **既存 Trip の一部**への提案 |
| 既存 Trip | なくても成立 | **関係が本質** |
| 日付 | 全体として未定可（`date_policy`） | Trip の日程に対する整合・conflict が重要 |
| 主な概念 | `date_policy`, 候補 Day 列 | **target**, **intent**, **placement**, **conflict** |
| 採用結果 | **新しい** schema v8 Trip | **既存** schema v8 Trip へ反映 |
| 典型 source | AI 旅行プラン全体 | レストラン提案・オプションツアー・予約補足 |

```text
Trip Proposal Envelope:
  「沖縄3泊の旅行案」（まだ Trip がない）

Proposal Fragment:
  「沖縄 Trip の Day 2 午後にこの観光を追加しませんか」
  （既存 Trip がある）
```

両方とも schema v8 の **外側**。混同しない。

## 3. schema v8 Trip との境界

### 3.1 採用前後

```text
採用前:
  Proposal Fragment
  schema v8 の外側
  trip export / validate-export 対象外

採用後:
  人間の確認を経て
  schema v8 Trip / Day / Itinerary / Reservation / Expense / Note 等へ反映
```

### 3.2 比較表

| 観点 | Proposal Fragment | schema v8 Trip 内の要素 |
|---|---|---|
| 状態 | 未採用 | 採用済み |
| export | 含まれない | `trip export` に含まれる |
| conflict チェック | 採用前に warning 対象 | 正式データとして整合が要る |
| 不完全性 | 許容 | export 契約に準拠 |

### 3.3 変換の gate

Fragment → 正式 Trip 要素への反映は **Adoption / Apply / Materialize**（v4.7.4 で詳細化）のみ。

```text
Proposal Fragment ──adoption gate──► schema v8 Trip の正式要素
```

`trip import` は既存の schema v8 JSON 用であり、Fragment の代替ではない。

## 4. 概念構造

以下は **概念整理**。フィールド名・JSON shape の確定は v4.7.3 scope 外。

```text
Proposal Fragment
  ├─ metadata
  │   ├─ fragment_id
  │   ├─ created_at
  │   ├─ valid_until
  │   ├─ no_expiration
  │   ├─ source
  │   └─ provider
  │
  ├─ target
  │   ├─ target_type
  │   ├─ trip reference
  │   ├─ day reference
  │   ├─ itinerary reference
  │   └─ unresolved target hints
  │
  ├─ fragment
  │   ├─ intent
  │   ├─ candidate content
  │   ├─ placement hints
  │   └─ notes
  │
  └─ adoption hints
      ├─ required decisions
      ├─ conflicts
      ├─ missing fields
      └─ warnings
```

### 4.1 metadata

[v4.7.2](v4.7.2-trip-proposal-envelope-concept-spec.md) の Expiry 方針を継承（§7 参照）。

| 概念 | 用途 |
|---|---|
| `fragment_id` | ローカル参照用識別子 |
| `created_at` | 生成・取り込み時刻 |
| `valid_until` / `no_expiration` | soft expiry |
| `source` / `provider` | 出所（ai / provider / manual 等） |

### 4.2 fragment（本体）— 最小情報

| 概念 | 必須度 | 説明 |
|---|---|---|
| `intent` | 推奨 | 何をしたい提案か（§5） |
| `candidate content` | 推奨 | 提案される内容（行程・予約候補・メモ等） |
| `placement hints` | 任意 | 時間帯・順序・Day 内位置のヒント |
| `notes` | 任意 | 自由記述・前提 |

**必須にしないもの:** 正式な itinerary ID、確定予約番号、確定金額（採用前）。

## 5. target（差し込み先）

Fragment は **既存のどこに関する提案か** を持つ。未確定を許容する。

### 5.1 target 概念

```text
trip-level target:
  Trip 全体に対する提案
  例: 旅行全体への注意事項、全体方針、候補 Day の追加

day-level target:
  特定 Day に対する提案
  例: 2日目の午後に観光候補を追加

itinerary-level target:
  特定 Itinerary に対する提案
  例: 既存の食事予定に予約情報候補を補足

unresolved target:
  どこに入れるべきか未確定
  例: おすすめレストランだが何日目か未定
```

### 5.2 未確定 target を許容する理由

AI や外部 provider は、ローカル Trip の Day / Itinerary ID や正確な日付を知らないことが多い。Fragment は **提案 inbox** として受け取り、採用時に人間が target を確定する。

```text
取り込み時: unresolved target でも可
採用時:     人間が Day / Itinerary を指定（v4.7.4）
```

### 5.3 target_type（概念）

| 値（概念） | 意味 |
|---|---|
| `trip` | Trip 全体 |
| `day` | 特定 Day |
| `itinerary` | 特定 Itinerary |
| `unresolved` | 差し込み先未確定 |

正式 enum の確定は v4.7.4 以降。

## 6. intent（提案の意図）

Fragment は単なるメモではなく **何をしたい提案か** を持つと整理しやすい。v4.7.3 では概念候補のみ（正式 enum は確定しない）。

```text
add:
  新しい候補を追加する提案
  例: Day 2 に新しい観光候補

enrich:
  既存要素に情報を補足する提案
  例: 既存レストラン Itinerary に予約電話・URL を追加

replace_candidate:
  既存案の代替候補を提示
  例: 別ホテル・別ルートの候補

reorder_hint:
  並び順や時間帯の見直し候補
  例: 午前と午後の行程入れ替え提案

warning:
  既存計画への注意喚起
  例: 閉店・工事・天候リスク
```

将来の validation / GUI review / materialize で参照する **語彙** として整理する。

## 7. Expiry（有効期限）

[v4.7.2](v4.7.2-trip-proposal-envelope-concept-spec.md) と同一方針。

```text
valid_until がある:
  それを soft expiry として使う

valid_until がない:
  created_at + 1 year を既定扱い（warning 用）

no_expiration: true:
  期限なしとして扱う

期限切れ:
  warning
  import 禁止・破棄にはしない
```

Fragment でも期限切れは **stale_source** 系 warning の一因であり、自動削除しない。

## 8. Conflict / warning（採用前）

Fragment は既存 Trip に差し込む前提のため、Trip Proposal より **conflict** が重要。v4.7.3 では概念のみ（実装は v4.7.4 以降）。

| 概念 | 意味 |
|---|---|
| `target_missing` | 指定 Day / Itinerary が存在しない |
| `date_mismatch` | 候補日が Trip 日程と合わない |
| `time_overlap` | 既存 Itinerary と時間帯が重なる |
| `duplicate_candidate` | 似た予定・予約・メモが既に存在しうる |
| `sort_order_conflict` | 差し込み位置が曖昧 |
| `reservation_ambiguity` | 予約として採用できるほど確定していない |
| `expense_ambiguity` | 金額・通貨・支払対象が未確定 |
| `stale_source` | 提案元情報が古い可能性（expiry 含む） |

これらは **採用阻止のハードブロックではない** ことを基本とする。人間が確認したうえで採用・却下・修正する。v4.7.4 の materialize gate で validation ルールを詳細化。

### 8.1 adoption hints との関係

```text
adoption hints.conflicts  → 上記 conflict 概念の列挙
adoption hints.warnings   → 人間向けの要約メッセージ
adoption hints.required_decisions → 採用前に決めること（target 確定、どの候補を採るか等）
```

## 9. Adoption / Apply / Materialize との境界

v4.7.3 では command 実装・詳細 validation に踏み込まない。境界のみ。

```text
Proposal Fragment:
  部分提案
  未採用
  target 未確定可
  不完全可

Adoption / Apply / Materialize:
  人間が採用する gate
  target を確定する
  conflict を確認する
  schema v8 Trip の正式要素に反映する

schema v8 Trip:
  採用済み
  Travel Ledger の正式 Trip
```

| 段階 | 保持 | 人間の役割 |
|---|---|---|
| 取り込み後 | Fragment（外側） | 一覧・比較 |
| 検討中 | Fragment | target 確定・conflict 確認 |
| 採用時 | adoption gate | 明示的「採用」 |
| 採用後 | schema v8 Trip 内 | 通常編集・export |

**Trip Proposal の materialize** は新 Trip 生成。**Fragment の adoption** は既存 Trip への merge — v4.7.4 で両方の gate を統合的に整理する。

## 10. 入力 source の想定

| Source | 典型 Fragment | 備考 |
|---|---|---|
| **AI** | レストラン・観光・移動の差し込み案 | target 未確定が多い |
| **Provider** | オプションツアー・アップグレード提案 | 商業条件は warning |
| **Manual** | 同行者からの「ここ寄りたい」 | source: manual |
| **Import** | ブログ・メールからの転記 | stale_source 注意 |

Envelope と同様、取り込み前の正規化は将来の `fragment import` の責務。

---

## 11. Fragment の粒度

Fragment は **1つの提案意図** に絞るのが望ましい（概念上のガイド）。

```text
良い粒度:
  「Day 2 午後の観光候補を1件追加」
  「Itinerary #12 に予約電話を補足」

避けたい粒度:
  旅行全体の再提案を1 Fragment に押し込む
  → それは Trip Proposal Envelope の領域
```

複数の無関係な候補は **複数 Fragment** に分けるか、Trip Proposal Envelope を検討する。
