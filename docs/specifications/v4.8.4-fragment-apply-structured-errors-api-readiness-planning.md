## 1. 背景 — P2 横断テーマの統合

P-6n / P-6o / P-6p の各 post-release review で繰り返し **P2（defer）** とされた項目:

| ID | テーマ | 現状 |
|---|---|---|
| P2-1 | structured errors | `errors: Vec<String>` のみ |
| P2-2 | strict idempotency / retry token | 再 confirm は intent ごとに blocking（not found 等） |
| P2-3 | version / ETag optimistic concurrency | `expected_*` は value-level baseline のみ |
| P2-4 | GUI / API error code / field path | 文字列マッチングに依存 |
| P2-5 | public Proposal Fragment schema versioning | additive 運用は文書化のみ、report schema version 固定 |
| P2-6 | `required_decisions` container 再確認 | field-level blocking は transaction 内再検証済み；opaque container は gate report 参照 |

v4.8.4 では **P2-1 / P2-4 / P2-5 の設計基盤** を主題とし、P2-2 / P2-3 は **境界定義のみ**（実装は別計画）。

## 2. 現状整理 — error / blocking surface

### 2.1 Report envelope（`FragmentApplyDryRunReport`）

```text
schema_version: 2（固定）
valid / fragment_valid / trip_export_valid
errors: Vec<String>        — blocking
warnings: Vec<String>      — non-blocking
required_decisions: Vec<String>  — 人間判断待ち（非空なら valid=false）
resolved_target / preview / success-only IDs
```

`--json` は上記 struct を JSON 直列化。human-readable output は `print_fragment_apply_report` が **同じ report** をテキスト整形（文言は独立に進化しうるが、現状は errors をそのまま列挙）。

### 2.2 Gate pipeline（dry-run / confirm 共有）

`fragment_apply_gate_json` の段階:

| Stage | 発生箇所 | 典型 blocking | report への反映 |
|---|---|---|---|
| A. Fragment file validation | `analyze_proposal_fragment_json` | envelope / fragment shape 不正 | `errors` + `fragment_valid=false` |
| B. JSON envelope parse | gate 入口 | top-level / target / fragment 欠落 | `errors` |
| C. `required_decisions` 収集 | `collect_required_decisions` | unresolved target hints | `required_decisions`（後段で `errors` にも昇格） |
| D. Trip context | `get_trip` | trip 不存在 | `errors` |
| E. Target resolver | `resolve_apply_target` | missing type、unresolved、ambiguous、not found | `errors` |
| F. Preview simulation | `simulate_apply_preview` → intent handler | parser / ownership / baseline / unsupported | `errors`（一部 intent は preview 返却後 invalid） |
| G. Export validation | `analyze_trip_export_json` on preview | preview export 不正 | `errors` + `trip_export_valid=false` |
| H. Confirm scope | `validate_confirm_scope` | intent/target/preview 必須フィールド不足 | `errors`（confirm のみ） |
| I. Required decisions gate | gate 末尾 | `required_decisions` 非空 | `errors` + `valid=false`（preview は返却されうる） |

confirm 成功 path は gate `valid=true` 後、`execute_confirm_*` が transaction 内で再検証。失敗は `anyhow` → `error.to_string()` → `report.errors`。

### 2.3 Error path 分類（横断）

| カテゴリ | 例（現行 message パターン） | dry-run | confirm |
|---|---|---|---|
| **parser validation** | `candidate_content.estimate_id は正の整数`、mutation field 拒否、explicit null | ✓ gate | ✓ transaction 再 parse |
| **target resolver** | `target が unresolved`、`曖昧`、`itinerary_reference ... 見つかりません` | ✓ | ✓ 再 resolve |
| **ownership check** | `Estimate {id} は Itinerary {id} 配下ではありません` | ✓ | ✓ |
| **required_decisions** | `required decisions が未解決` | ✓ | ✓（gate report 参照） |
| **mutation field rejection** | `candidate_content.amount は delete_estimate では指定できません` | ✓ | ✓ |
| **baseline mismatch** | `baseline mismatch: expected_amount` | ✓ | ✓ transaction |
| **preview mismatch** | `gate preview と recomputed ... が一致しません` | n/a | ✓ |
| **TOCTOU drift** | `TOCTOU mismatch: title の before` | n/a | ✓ |
| **scoped write rows** | `削除が N 行に影響` / `更新できませんでした` | n/a | ✓ rollback |
| **post-write verification** | `削除後も存在` / proposed mismatch | n/a | ✓ |
| **unsupported intent/target** | `add_expense は itinerary target のみ` | ✓ | ✓ scope check |
| **transaction failure** | `DB 更新しません` 系の汎用文言 | n/a | ✓ rollback |

### 2.4 代表 intent 別の追加 surface

調査対象として `add_expense` / `update_estimate` / `delete_estimate` を正本例にする。

#### `add_expense`（P-6g — Actual Money 追加）

- parser: amount / currency / date / title / note；unsupported field → warning
- target: itinerary only
- confirm: `parse_add_expense_fields` + `expense_add::add_expense`（scoped INSERT 系）
- TOCTOU / preview agreement: **なし**（add 系は insert preview パターン）
- 典型 blocking: invalid amount/currency、itinerary unresolved

#### `update_estimate`（P-6o — Planned Money 部分更新）

- parser: partial fields、currency-only 拒否、clear flags、unknown-only blocking
- ownership + `estimate_id`
- baseline: `expected_*` + `estimate_field_changes[].before`
- confirm: preview agreement + TOCTOU + scoped UPDATE exactly-one + post-verify
- 典型 blocking: `更新 field が 0 件`、stale baseline、gate preview drift

#### `delete_estimate`（P-6p — Planned Money 削除）

- parser: `estimate_id` only + mutation rejection
- ownership + optional `expected_*`
- confirm: `estimate_delete_preview` agreement + TOCTOU + scoped DELETE + post-not-found verify
- 典型 blocking: not found、already deleted、preview mismatch、row count ≠ 1

**共通:** 3 intent とも `resolve_apply_target` / `required_decisions` / unsupported target を **同一 gate** で処理。intent 固有の差分は parser と confirm transaction のみ。

### 2.5 Warnings と errors の境界

| 種別 | 例 | valid への影響 |
|---|---|---|
| warning | `unsupported_field:`、`ordering_hint:`、destructive undo 非保証 | 単独では blocking しない |
| error | parser / resolver / ownership / baseline / preview / transaction | `valid=false` |
| required_decision | adoption_hints / unresolved target | 非空で blocking（専用配列 + errors 昇格） |

## 3. Structured error contract 案

### 3.1 設計原則

1. **Stable `code`** — API / GUI / automation の branching 正本
2. **Unstable `message`** — human-readable；ロケール・文言改善可
3. **Additive JSON** — 既存 `errors: Vec<String>` を維持（後方互換）
4. **Dry-run / confirm 共有** — 同一 registry；`phase` で区別
5. **Field path** — proposal authoring 修正候補の粒度
6. **Retry / idempotency とは分離** — error code は「なぜ失敗したか」；retry token は「安全に再試行できるか」

### 3.2 提案 DTO（planning 案）

```json
{
  "code": "APPLY_ESTIMATE_PARENT_MISMATCH",
  "kind": "blocking",
  "message": "Estimate 3 は Itinerary 12 配下ではありません",
  "field_path": "fragment.candidate_content.estimate_id",
  "intent": "delete_estimate",
  "target_type": "itinerary",
  "entity_type": "estimate",
  "entity_id": 3,
  "expected": { "itinerary_id": 12 },
  "actual": { "itinerary_id": 99 },
  "phase": "confirm_transaction",
  "retryable": false,
  "blocking": true,
  "help": "target itinerary と estimate の親子関係を確認してください"
}
```

| フィールド | Required | 意味 |
|---|---|---|
| `code` | **yes** | stable registry ID（`SCREAMING_SNAKE`） |
| `kind` | yes | `blocking` \| `warning` \| `decision_required` |
| `message` | yes | human-readable（unstable） |
| `field_path` | no | JSON pointer 風（例: `fragment.candidate_content.amount`） |
| `intent` | no | 発生時 intent |
| `target_type` | no | trip / day / itinerary |
| `entity_type` | no | estimate / expense / itinerary / … |
| `entity_id` | no | DB PK when known |
| `expected` / `actual` | no | baseline / TOCTOU / preview mismatch 材料 |
| `phase` | no | `fragment_validate` \| `gate` \| `confirm_scope` \| `confirm_transaction` |
| `retryable` | no | strict retry policy **導入前**は参考値のみ |
| `blocking` | no | `valid=false` に寄与するか（`kind` と冗長化回避要検討） |
| `help` / `hint` | no | authoring / GUI 向け短文 |

### 3.3 Code registry 案（初期カテゴリ）

prefix 規約: `APPLY_`（Fragment apply 横断）

| Code | カテゴリ | 現行 message 相当 |
|---|---|---|
| `APPLY_FRAGMENT_INVALID` | A | fragment validate 失敗 |
| `APPLY_ENVELOPE_INVALID` | B | JSON / target / fragment 欠落 |
| `APPLY_TRIP_NOT_FOUND` | D | trip 不存在 |
| `APPLY_TARGET_UNRESOLVED` | E | unresolved target |
| `APPLY_TARGET_AMBIGUOUS` | E | itinerary/day ambiguous |
| `APPLY_TARGET_NOT_FOUND` | E | day / itinerary reference not found |
| `APPLY_REQUIRED_DECISIONS_PENDING` | C/I | required_decisions 非空 |
| `APPLY_INTENT_UNSUPPORTED` | F/H | unsupported intent / target combo |
| `APPLY_CANDIDATE_PARSE_ERROR` | F | parser validation 一般 |
| `APPLY_CANDIDATE_MUTATION_FIELD` | F | mutation field 拒否 |
| `APPLY_CANDIDATE_EXPLICIT_NULL` | F | explicit null on supported key |
| `APPLY_ENTITY_NOT_FOUND` | F | estimate / itinerary not found |
| `APPLY_ENTITY_PARENT_MISMATCH` | F | ownership mismatch |
| `APPLY_BASELINE_MISMATCH` | F | `expected_*` mismatch |
| `APPLY_PREVIEW_MISMATCH` | confirm | gate vs recomputed preview |
| `APPLY_TOCTOU_DRIFT` | confirm | snapshot drift |
| `APPLY_SCOPED_WRITE_ZERO_ROWS` | confirm | affected rows 0 |
| `APPLY_SCOPED_WRITE_MULTI_ROWS` | confirm | affected rows 2+ |
| `APPLY_POST_VERIFY_FAILED` | confirm | post-delete / post-update verify |
| `APPLY_PREVIEW_EXPORT_INVALID` | G | trip_export_valid failure |
| `APPLY_CONFIRM_SCOPE_INVALID` | H | validate_confirm_scope |

intent 固有 code（例）:

| Code | intent |
|---|---|
| `APPLY_ESTIMATE_UPDATE_NO_FIELDS` | update_estimate |
| `APPLY_ESTIMATE_UPDATE_CURRENCY_ONLY` | update_estimate |
| `APPLY_EXPENSE_INVALID_AMOUNT` | add_expense |

registry は **単一ファイル / 単一文書** で管理し、P-6 全 intent へ段階的にマッピングする。

### 3.4 `required_decisions` との関係

| 概念 | 役割 | structured 化案 |
|---|---|---|
| `required_decisions[]` | 人間が adoption / target を確定すべき **opaque hints** | `kind: decision_required` + `code: APPLY_REQUIRED_DECISIONS_PENDING` + `details[]` |
| field-level blocking | parser / ownership 等の **機械判定** | `kind: blocking` + 具体 `code` |

**planning 判断:** `required_decisions` を errors に昇格する現行挙動は維持。structured layer では **decision と validation error を別 kind** で表現し、GUI が「修正 vs 人間判断」を分岐できるようにする。

## 4. JSON report との関係

### 4.1 Backward compatibility

| 方針 | 内容 |
|---|---|
| 既存 `errors: Vec<String>` | **維持** — 各 structured error の `message` と同期（または従来互換の要約） |
| 新規 `structured_errors: Vec<ApplyStructuredError>` | **optional additive** — `schema_version` は 2 のまま据え置き可 |
| `warnings` / `required_decisions` | 当面維持；structured 側に mirror 可能 |
| CLI 既定 output | 変更なし — human text は unstable |
| `--json` | structured_errors を追加出力；旧 client は未知フィールドを無視 |

### 4.2 責務分離

```text
errors[]              — 後方互換の blocking message 一覧（unstable text）
structured_errors[] — stable code + metadata（API 正本）
warnings[]            — non-blocking（ordering_hint、unsupported_field 等）
required_decisions[]  — adoption / unresolved 人間判断（opaque）
valid                 — 総合判定（blocking + pending decisions）
preview               — 成功候補の simulation 結果（error 時も返却されうる）
```

`blocking_reasons[]` は **新設しない**（planning 案）— `structured_errors` の `kind=blocking` で足りる。既存 client が `errors` のみ見ている前提を壊さない。

### 4.3 Dry-run / confirm code 共有

同一 `code` を gate と transaction で共有する。

| 例 | dry-run phase | confirm phase |
|---|---|---|
| `APPLY_ENTITY_PARENT_MISMATCH` | `gate`（simulate） | `confirm_transaction`（revalidate） |
| `APPLY_PREVIEW_MISMATCH` | — | `confirm_transaction` only |

`phase` フィールドで区別。API client は code で集計、phase で UI 表示を調整可能。

## 5. API / GUI readiness

### 5.1 GUI が必要とする粒度

| 能力 | 必要な structured フィールド |
|---|---|
| Field-level highlighting | `field_path` + `code` |
| Inline fix suggestion | `help` + `expected` / `actual` |
| Target picker assist | `APPLY_TARGET_*` + `target_type` |
| Destructive confirm guard | `kind` + intent-specific preview codes |
| Decision workflow | `required_decisions` / `kind=decision_required` 分離 |

### 5.2 API client が必要とする粒度

| 能力 | 必要な structured フィールド |
|---|---|
| Programmatic branching | stable `code` |
| Retry policy（将来） | `retryable` + 別 idempotency 設計 |
| Conflict reporting | `expected` / `actual` + `APPLY_TOCTOU_DRIFT` |
| Observability | `phase` + `intent` + `entity_type` / `entity_id` |

### 5.3 Proposal authoring 側

| 能力 | 必要な structured フィールド |
|---|---|
| 修正候補の自動提示 | `field_path` + `help` |
| LLM 向け安定識別子 | `code`（message は参考） |
| Schema バージョン（将来） | report / fragment の `schema_version` 方針（§6） |

### 5.4 Public vs internal（planning 判断）

| 層 | v4.8.4 時点 | 将来 public 固定候補 |
|---|---|---|

## 6. Retry / idempotency / optimistic concurrency との境界

| テーマ | structured errors との関係 | v4.8.4 判定 |
|---|---|---|
| **structured errors** | 失敗 **理由** の安定識別 | 本 planning の主題 |
| **retry token** | 同一操作の **安全な再試行** 許可 | 実装しない；`retryable` は将来用ヒントのみ |
| **strict idempotency** | 成功済み再 apply の意味論 | 実装しない；`APPLY_ENTITY_NOT_FOUND` 等で表現 |
| **ETag / version column** | 並行更新検出の **別軸** | `expected_*` / TOCTOU は現行；version token は別計画 |
| **transaction rollback** | DB 不変の **内部機構** | error code では「ユーザー向け undo」と混同しない（v4.8.3 P3-7 継承） |
