## 1. Existing facts（現行実装）

### 1.1 Estimate entity

`estimates` は Itinerary の子で、主要列は次のとおり。

```text
id, itinerary_id, title, amount, currency, note, sort_order,
created_at, updated_at
```

- `id`: DB primary key
- `itinerary_id`: NOT NULL の親 ID
- Trip / Day 直下の Estimate は存在しない
- `amount`: INTEGER minor units
- Estimate は Planned Money、Expense は Actual Money

### 1.2 Existing CLI delete

現行 `estimate delete <id>` は `get_estimate(id)` 後に以下を実行する。

```sql
DELETE FROM estimates WHERE id = ?
```

現行 helper `estimate::delete_estimate` は:

- `id` のみを受け取る
- 親 Itinerary を WHERE に含めない
- affected row count の exactly-one 検査をしない
- retry token / undo journal を持たない

これはローカル CLI の既存契約として維持するが、Proposal Fragment confirm の安全条件には不足する。P-6p 実装では scoped helper の新設または同等処理を使い、既存 CLI behavior は変更しない。

### 1.3 Existing Fragment assets

そのまま流用できるもの:

- common itinerary target resolver（ID-first、cross-trip / unresolved / ambiguous blocking）
- `FragmentApplyDryRunReport` / preview envelope
- `with_transaction` confirm pattern
- `delete_itinerary` の destructive warning / rollback pattern
- `update_estimate` の `estimate_id`、親子検証、preview agreement、TOCTOU pattern
- success-only result ID の慣行

流用しないもの:

- `delete_itinerary` の blocking child count（Estimate 自身の削除には子 cascade がない）
- `update_estimate` の field patch / amount parser（削除 candidate は値を更新しない）
- CLI の unscoped `delete_estimate(conn, id)` を安全検証なしで直接呼ぶこと

## 2. Intent / target / identity

### 2.1 Canonical intent

```text
intent: delete_estimate
```

alias intent は設けない。

### 2.2 Parent target

```text
target_type = itinerary
```

| ケース | 判定 |
|---|---|
| itinerary target が一意解決 | 続行 |
| Trip / Day target | **blocking** |
| unresolved target | **blocking** |
| ambiguous target | **blocking** |
| selected Trip 外の target | **blocking** |

親 resolver は既存共通 resolver を再利用する。数値 `itinerary_reference` は JSON number の場合のみ numeric selector とし、既存 ID-first 契約を維持する。string `"2"` は title selector のまま。

### 2.3 Estimate identity（required decision D1）

#### 案 A — `estimate_id`（採用・推奨）

```json
{
  "fragment": {
    "intent": "delete_estimate",
    "candidate_content": {
      "estimate_id": 3
    }
  }
}
```

採用理由:

- DB primary key で一意
- add / update Estimate 契約と整合
- title、amount、sort_order の重複を許容したまま安全に識別できる
- exactly-one DELETE と対応する
- GUI / API が一覧取得結果の ID をそのまま使用できる

#### 案 B — title / sort_order / amount selector（不採用）

いずれも同一 Itinerary 内で一意とは限らず、破壊操作の identity として弱い。配列 index や複合 heuristic も採用しない。

### 2.4 Parent ownership

解決後、必ず次を確認する。

```text
get_estimate(estimate_id).itinerary_id == resolved_target_itinerary_id
```

不一致は **blocking**。存在する Estimate であっても別 Itinerary 配下なら削除候補にしない。

## 3. Candidate content contract

`candidate_content` は JSON object 必須。

| Field | Required | JSON type | 意味 |
|---|---:|---|---|
| `estimate_id` | **yes** | integer number | 削除対象 Estimate の DB ID |
| `expected_amount` | no | integer number | optional author-supplied baseline（minor units） |
| `expected_currency` | no | string | optional baseline |
| `expected_title` | no | string | optional baseline |
| `expected_note` | no | string | optional baseline |
| `expected_sort_order` | no | integer number | optional baseline |
| `expected_updated_at` | no | string | optional coarse version baseline |

規則:

- `estimate_id` の string / float / null / non-positive value は blocking
- optional `expected_*` の明示的 JSON `null` は blocking（NULL を表す baseline は本 planning では公開入力に含めない）
- unknown field は既存慣行どおり warning 候補。ただし誤削除に関係する identity alias は追加しない
- amount / currency / title / note / sort_order を「削除時に変更する」field として受け付けない

`expected_*` は author が対象を再確認する追加条件であり、confirm の TOCTOU 防止を optional 入力だけに依存させない。dry-run が生成する完全 snapshot を confirm の必須 baseline とする。

## 4. Missing / already deleted / idempotency

### 4.1 Required decision D2 — not found

| 案 | 判定 |
|---|---|
| 既に存在しないなら成功 | 不採用 |
| **blocking error** | **採用** |

理由:

- Fragment が誤った ID を指定した場合と、既に削除された場合を安全に区別できない
- success 扱いでは `deleted_estimate_id` が「今回削除した ID」ではなくなる
- stale preview / competing delete を隠す
- destructive intent は対象の存在と内容を利用者に preview してから削除する責務がある

したがって、dry-run 前から不存在、gate 後に消失、再 confirm のいずれも **blocking / DB unchanged**。

### 4.2 Required decision D3 — strict idempotency

**v4.8.0 planning では strict idempotency を契約に含めない。**

同一 Fragment の再 confirm:

```text
first confirm  → success, deleted_estimate_id
second confirm → blocking (not found)
```

成功済み retry を success として再現するには、idempotency key、operation receipt、apply journal 等の永続状態が必要である。今回それらを追加すると DB schema / public API / apply 横断設計になるため持ち越す。

代替案:

- idempotency key + persisted result
- apply journal から同一 operation の success result を再返却
- API 層で retry token を管理

いずれも P-6p dry-run / confirm の初期実装には含めない。

## 5. Dry-run contract（将来実装）

### 5.1 Flow

```text
Fragment parse / validate
  → intent == delete_estimate
  → parent Itinerary を一意 resolve
  → estimate_id parse
  → Estimate 取得 + parent ownership 検証
  → optional expected_* 検証
  → immutable delete snapshot 生成
  → in-memory export から対象 Estimate 1 件を除去
  → gate report / preview export
  → DB unchanged
```

### 5.2 Preview shape（推奨）

`FragmentApplyPreviewSummary` に additive field を追加する案:

```json
{
  "action": "delete_estimate",
  "estimates_before": 4,
  "estimates_after": 3,
  "estimate_delete_preview": {
    "target_itinerary_id": 42,
    "target_itinerary_title": "Aquarium",
    "target_estimate_id": 3,
    "title": "Tickets",
    "amount": 14000,
    "currency": "JPY",
    "note": null,
    "sort_order": 0,
    "updated_at": "2026-07-11 08:00:00"
  }
}
```

必須 agreement / baseline fields:

- `target_itinerary_id`
- `target_estimate_id`
- `title`
- `amount`
- `currency`
- `note`
- `sort_order`
- `updated_at`

informative only:

- `target_itinerary_title`（identity / agreement には使わない）
- Trip / Day の人間可読表示

`created_at` は削除対象説明に載せてもよいが、初期 agreement には不要。全 user-editable fields と `updated_at` の一致を必須にする。

### 5.3 Preview behavior

- DB read-only
- preview export では `days[].itineraries[].estimates[]` から対象 1 件のみ除去
- sibling Estimate の順序・`sort_order` は変更しない
- Estimate count は exactly one 減少
- Expense / Reservation / Note / Itinerary は不変
- invalid / blocking report では削除後 export preview を抑制
- hard delete であり success 後 undo を保証しない旨を warning で表示

## 6. Preview agreement / TOCTOU

### 6.1 Required decision D4 — baseline strength

| 案 | 判定 |
|---|---|
| `estimate_id` の存在だけ再確認 | 不採用 — 内容変更を見逃す |
| optional `expected_*` だけ比較 | 不採用 — 未指定時に無防備 |
| **dry-run snapshot 全体 + optional `expected_*`** | **採用** |

confirm は以下を一致させる。

| 層 | 一致条件 |
|---|---|
| intent | 再 parse 後も `delete_estimate` |
| parent identity | resolved Itinerary ID == preview target Itinerary ID |
| child identity | candidate `estimate_id` == preview target Estimate ID |
| ownership | current `itinerary_id` == resolved Itinerary ID |
| author baseline | 指定された全 `expected_*` == current row |
| gate baseline | preview の全 delete snapshot field == current row |
| preview export | 除去対象 ID が candidate / snapshot と一致 |

### 6.2 Concurrent changes

| gate 後の変化 | confirm |
|---|---|
| Estimate が削除された | blocking、DELETE しない |
| Estimate の amount 等が更新された | snapshot mismatch、blocking |
| `updated_at` のみ変化 | snapshot mismatch、blocking |
| Estimate が別 Itinerary に移動 | ownership mismatch、blocking |
| parent Itinerary が削除された | target resolve / ownership failure、blocking |
| parent title のみ変化 | 許可（ID が identity、title は informative） |
| candidate JSON が変化 | preview agreement mismatch、blocking |

version / ETag 専用列は今回追加しない。`updated_at` を含む snapshot は初期の安全条件だが、公開 API 用の正式な version token と同一視しない。

## 7. Confirm transaction contract（将来実装）

### 7.1 Transaction flow

```text
with_transaction("delete_estimate confirm")
  1. Fragment / intent / candidate を再 parse
  2. authoritative field-level blocking conditions を再検証
  3. adoption_hints.required_decisions container を再確認
  4. parent Itinerary を再 resolve
  5. estimate_id で current Estimate を再取得
  6. parent ownership を再確認
  7. optional expected_* を再確認
  8. gate delete snapshot と current row を完全比較
  9. candidate / preview identity agreement を確認
 10. DELETE ... WHERE id = ? AND itinerary_id = ?
 11. affected rows == 1 を確認
 12. get_estimate(id) が not found であることを post-delete verification
 13. sibling Estimates / Expense が不変であることを必要範囲で検証
 14. commit
 15. deleted_estimate_id を success report に設定
```

失敗時は **rollback、DB unchanged**。

### 7.2 Required decision D5 — scoped exactly-one DELETE

採用 SQL contract:

```sql
DELETE FROM estimates
WHERE id = ? AND itinerary_id = ?
```

| affected rows | contract |
|---:|---|
| 0 | **error + rollback** |
| 1 | success candidate、post-delete verification へ |
| 2+ | **contract violation + rollback** |

PK により 2+ rows は通常起きないが、affected row count を明示検査する。現行 CLI helper はこの契約を満たさないため、Fragment 専用の scoped delete helper または共通安全 helper を実装段階で追加する。

### 7.3 Success result

`FragmentApplyDryRunReport` への additive field 候補:

```text
deleted_estimate_id: integer
```

- confirm 成功時のみ設定
- dry-run / failure / rollback 時は absent
- `deleted_itinerary_id`、`updated_estimate_id`、`inserted_estimate_id` は設定しない
- success 後に削除済み row 全体を report に重複保存しない（preview snapshot を参照）

## 8. Estimate / Expense boundary

`delete_estimate` が許可する mutation:

```text
DELETE exactly one estimates row
```

禁止:

- Expense の作成・更新・削除
- Estimate → Expense conversion
- Itinerary / Trip / Day の更新・削除
- sibling Estimate の並び替え・再採番
- Note / Reservation / Receipt / Participant への mutation
- amount 差額や Actual Money の自動調整
- cascade delete

Estimate 削除によって集計表示上の Planned Money 合計が変わるのは派生結果であり、Expense 行や Actual Money は変えない。

---

## 9. `adoption_hints.required_decisions`

### 9.1 Required decision D6 — relation to blocking

- `adoption_hints.required_decisions` 非空は gate で blocking
- confirm transaction 内でも container 自体を再確認することを **P-6p の推奨契約**とする
- ただし data-safety の正本は container の文言ではなく、transaction 内で再 parse / 再検証する authoritative conditions

`required_decisions` に委ねず直接 validation error とするもの:

- `estimate_id` 欠落 / 型不正 / 不存在
- parent target unresolved / ambiguous / cross-trip
- parent ownership mismatch
- expected / snapshot / preview mismatch
- unsupported target / candidate mutation field

人間の選択で安全に解消できる曖昧性が存在する場合のみ required decision を生成する。初期契約では Estimate selector を ID のみに限定するため、子 Estimate の ambiguous target は通常発生しない。

## 10. Structured errors / public schema

### 10.1 Required decision D7 — structured errors

初期 dry-run / confirm 実装では既存 `errors: Vec<String>` / `warnings` / `required_decisions` envelope と整合させる。P-6p 単独で cross-cutting error model を導入しない。

将来の structured error 候補:

- `code`: `estimate_not_found`, `estimate_parent_mismatch`, `stale_delete_preview`, `delete_row_count_mismatch`
- `path`: `fragment.candidate_content.estimate_id` 等
- retryability / conflict category

分類・code registry は Proposal Fragment 全 intent 横断で設計する。

### 10.2 Required decision D8 — public Proposal Fragment schema versioning

`delete_estimate` intent、`estimate_delete_preview`、`deleted_estimate_id` は additive な候補だが、今回の planning release では public schema / runtime validation を変更しない。

dry-run 実装前に確認:

- intent allowlist 追加の compatibility
- report additive field の versioning policy
- public examples / AI authoring guide を同時更新する範囲
- unknown intent を扱う旧 client の behavior

schema version field 自体の導入は cross-cutting P2 として別計画に残す。

## 12. Findings（planning）

### P0 — data loss / boundary violation

- planning 時点の既知 P0 はなし
- 実装時、**wrong Estimate delete、Expense delete、unscoped / unchecked DELETE、preview mismatch 後の DELETE** は P0 として扱う

### P1 — dry-run / confirm 前に必須

| ID | 内容 | 完了条件 |
|---|---|---|
| P1-1 | canonical identity / ownership | `estimate_id` + parent Itinerary 検証 |
| P1-2 | destructive preview | full row snapshot + export 除去 preview |
| P1-3 | TOCTOU | snapshot / expected / candidate agreement |
| P1-4 | scoped delete | `id AND itinerary_id` + exactly-one |
| P1-5 | rollback | 0 / 2+ rows、verification failure で rollback |
| P1-6 | boundary | sibling Estimate / Expense 不変 |
| P1-7 | missing semantics | already deleted を blocking とする |

### P2 — API / cross-cutting

- strict idempotency / retry token / apply journal
- structured errors、machine-readable error code、field path
- version / ETag 相当の optimistic concurrency token
- public Proposal Fragment schema versioning
- destructive operation の recovery / undo（soft delete、journal、backup hint）

### P3 — optional / test layering

- parser / resolver を dry-run と confirm で共有
- preview mismatch / row-count mismatch は production fault injection を増やさず unit test で検証
- `target_itinerary_title` は informative only
- full snapshot DTO と update preview DTO の重複整理は必要最小限に留める

## 13. Future implementation test plan

- candidate / preview identity mismatch は blocking

- sibling Estimates 不変
- Expense / Reservation / Note / Itinerary 不変
- second confirm は not-found blocking

### 13.4 Regression

- `add_estimate` / `update_estimate` 契約不変
- regular `estimate delete` CLI behavior 不変
- `delete_itinerary` の Estimate-child blocking 不変
- export schema v8 / import / Markdown / JSON golden 不変
