  - **推奨**: 将来 `code + message + path` などの structured error を検討（段階的導入）

## 1. planning / dry-run / confirm の契約整合（A）

### 1.1 planning（v4.7.41）→ dry-run（v4.7.42）

- **intent/target**: `intent: add_estimate` + `target_type: itinerary` を維持
- **semantic boundary**: Planned Money / Actual Money の分離を維持
- **amount**: string decimal text / integer minor units を共通契約として採用
- **note/title alias**: `title → description → label`、`note → memo → fragment.notes` を dry-run が実装

### 1.2 dry-run（v4.7.42）→ confirm（v4.7.43）

- **field parser**: dry-run と confirm が `parse_add_estimate_fields` を共有（candidate の解釈差が出にくい）
- **target resolver**: `resolve_apply_target` を共通使用し、numeric selector は itinerary ID-first を維持
- **report**: confirm は `inserted_estimate_id` を追加（dry-run では absent を維持）

## 2. Estimate / Expense 境界（B）

固定境界:

```text
Estimate = Planned Money
Expense  = Actual Money
```

確認結果:

- `add_estimate --confirm` の write は `estimates` のみ（`add_estimate_minor_units` 経由）
- `Expense` / `Reservation` / `Note` / `Itinerary` への mutation は混入していない
- 自動 conversion（Estimate→Expense）は導入されていない

---

## 3. minor-unit helper の責務（C）

### 3.1 二重変換の回避（critical）

`amount: 1250`（USD）を `"1250"` として再パースすると `125000` になるリスクがある。

確認結果:

- Fragment confirm は `ParsedAddEstimateFields.amount`（minor units）を **そのまま** `add_estimate_minor_units` に渡す
- 回帰テストで `1250` が `1250` のまま保存されることを検証している

### 3.2 helper の設計評価

- `add_estimate_minor_units` は **narrow**（単一行 INSERT + defensive amount check + currency normalization + optional text normalization）
- 将来の `update_estimate` / `delete_estimate` 等にも使い回しやすい粒度

## 4. target ID-first resolution（D）

契約:

```text
numeric itinerary_reference:
  1) itinerary ID
  2) sort_order fallback
  3) ambiguity reject
```

確認結果:

- dry-run / confirm ともに `resolve_apply_target` を使用し、同一ルールで解決する
- 「ID と sort_order が別 itinerary を指す」ケースは ambiguous として reject する

---

## 5. transaction / TOCTOU safety（E）

confirm の write は:

```text
with_transaction("add_estimate confirm")
```

確認結果:

- transaction 内で Fragment JSON を再 parse し、`intent == add_estimate` を確認
- transaction 内で target を再解決し、gate preview（`estimate_preview`）との一致を要求
- INSERT 後に `get_estimate` で stored row を検証し、失敗時は rollback

観測:

- preview agreement は candidate fields と `target_itinerary_id` に限定される（title 変更は許容）

---

## 6. delete_itinerary の Estimate blocking（F）

確認結果:

- `delete_itinerary` の child blocking に **estimates が含まれる**（dry-run / confirm の両方）
- `count_blocking_children_for_itinerary_db` が `estimates` を数え、confirm の TOCTOU 再検証でも blocking を維持する
- `add_estimate` 実装による regression は見当たらない（既存テストでも confirm blocking を確認）

## 7. GUI / API readiness（G）

### 今すぐ必要（次の intent 実装前に検討したい）

- **target identity**: `target_itinerary_id` を正本として扱う方針（title 変更は許容）
- **inserted ID**: `inserted_estimate_id` を success-only で返す契約（dry-run / invalid / rollback では absent）
- **amount representation**: minor units を内部正本にし、string decimal は入力表現として限定する

### 将来必要（API/GUI を見据えた改善候補）

- structured error（error codes / paths）
- idempotency（retry を前提とした confirm 仕様）
- report schema versioning と additive fields の運用ルール明文化

## 8. 次 Proposal Fragment intent の推薦（第一候補）

**第一候補: `update_estimate`（Itinerary target）**

理由（比較観点）:

- **利用価値**: Planned Money の調整（amount/currency/title/note/sort_order）は現実に発生しやすい
- **既存 entity/service 整合**: Estimate は既に CRUD が存在し、Planned Money 境界を維持できる
- **dry-run/confirm 契約再利用**: `parse_expense_amount_field` / currency 正規化 / target resolver / TOCTOU 方式を再利用できる
- **mutation の単純さ**: `UPDATE exactly one estimates row` に絞れる（列も限定可能）
- **transaction safety**: 既存 confirm パターン（transaction + preview agreement + stored-row verification）を踏襲しやすい
- **test 準備**: 既存 fixture/seed patterns を流用可能
- **GUI/API 連携**: 「提案→確認→確定」フローに自然に接続できる

非推薦（今回の第一候補にしない理由）例:

- `update_expense` は shared-expense 等の責務が重く、P-6 系列の「最小 mutation」から外れやすい
- `add_receipt` は intent/target の再設計が必要で、P-6n 直後の連続実装としてはスコープが大きい

次の作業としては以下を第一候補とする（本書は planning / 実装には着手しない）:

