# v4.6.30 — Expense write path boundary review

v4.6.26〜v4.6.27 で read-only expense path の enriched context ownership を整理した後、**write path**（add / update / delete）の責務境界を documentation-only で棚卸しする review です。

| Item | Status |
|---|---|
| Release | v4.6.30（documentation-only boundary review） |
| Scope | write handler / `expense.rs` mutation / adapter / itinerary 親子 / v4.6.x 整合 |
| 実装 | **なし** |

関連:

| Version | 内容 |
|---|---|
| v4.6.25〜v4.6.27 | expense output DTO migration（read-only） |
| v4.6.22 | read-only service boundary 完了 — write は Tier 3+ defer |
| **本書** | **v4.6.30** — write path boundary review |
| v4.6.31（候補） | expense write path migration plan |

---

## Status

| 観点 | 判定 |
|---|---|
| write path service 化 | **未着手**（v4.6.x 意図どおり） |
| 責務の所在 | **handler + `expense.rs` モジュール** — repository 層なし |
| read/write 非対称 | **あり** — list/show は service、add/update/delete は未 service 化 |
| adapter 残置 | **妥当** — add/update 後 `print_expense_detail` |
| itinerary 親子 | **一貫** — add は `--itinerary`、永続化は `itinerary_id` 列（DB FK なし・app validation） |
| 短期 | **現状維持** — 無理な service 化はしない |
| JSON golden | write は JSON 出力なし — **影響なし** |

**結論（推奨）:**

```text
expense write path は command handler が薄いオーケストレーションを行い、
入力正規化・participant 解決・永続化・トランザクションは expense.rs に集約されている。
read-only とは非対称だが、v4.6.x の「write は Tier 3+ defer」方針と整合する。
DTO 変換（ExpenseJson）と表示整形は read path で service + CLI mapper に移行済み。
write 成功後表示だけが print_expense_detail adapter 経由で enrich を再実行している。
短期は現状維持。次は migration plan（v4.6.31 候補）で write service 化の Phase を定義する。
v4.6.30 では実装変更しない。
```

## 1. Current state

### read-only vs write（非対称）

| Command | 層 | 備考 |
|---|---|---|
| `expense list` | `services::expense_list` → CLI mapper | enriched |
| `expense show` | `services::expense_show` → CLI mapper | enriched |
| `expense add` | handler → `expense.rs` | **未 service 化** |
| `expense update` | handler → `expense.rs` | **未 service 化** |
| `expense delete` | handler → `expense.rs` | **未 service 化** |

Note / Reservation も同型: **read-only のみ service 化、write は module 直結**。

### write path フロー（確認済み）

#### Add

```text
ExpenseAction::Add
  handler:
    parse_expense_shared_options_for_add(conn, itinerary, ...)  → ExpenseSharedOptions
    add_expense(conn, itinerary, amount, currency, ..., &shared) → expense_id
    println!("Expense を追加しました ...")
    get_expense(conn, id)
    print_expense_detail(conn, &expense)   ← adapter（enrich + human）
```

#### Update

```text
ExpenseAction::Update
  handler:
    get_expense(conn, id)                    ← itinerary_id 取得用
    parse_expense_shared_options_for_update(conn, expense.itinerary_id, ...)
    update_expense(conn, id, ..., &shared)
    println!("Expense を更新しました ...")
    get_expense(conn, id)
    print_expense_detail(conn, &expense)
```

#### Delete

```text
ExpenseAction::Delete
  handler:
    get_expense(conn, id)                     ← 削除前スナップショット
    delete_expense(conn, id)
    println!("Expense を削除しました ...")
    println amount only                      ← enrich なし
```

### `expense.rs` 内の write 関連関数

| 関数 | 責務 |
|---|---|
| `parse_expense_shared_options_for_add` | CLI 構造化オプション → participant ID 解決、trip participant 列挙 |
| `parse_expense_shared_options_for_update` | 同上 + clear フラグ |
| `add_expense` | 検証、itinerary 存在、amount/currency/date、INSERT、beneficiaries |
| `update_expense` | 部分更新、beneficiary 置換/clear、トランザクション |
| `delete_expense` | beneficiary 削除 + expense 行削除 |
| `import_expense_v3` | trip import 経路（別 write path） |
| `delete_expenses_for_itinerary` / `_for_trip` | cascade 用（itinerary/trip delete から呼ばれる） |

**repository 層は存在しない** — SQL は `expense.rs` 内に直書き。

## 2. Boundary problem — 責務の混在観点

### 層別インベントリ

| 責務 | 所在 | 評価 |
|---|---|---|
| CLI 引数（clap） | `main.rs` / command 定義 | ○ handler 妥当 |
| 構造化オプション解釈 | `parse_expense_shared_options_*` | △ handler と expense.rs の境界上 |
| 入力正規化（amount/currency/date） | `add_expense` / `update_expense` | ○ mutation 近傍 |
| participant / trip 整合検証 | parse_* + add/update 両方 | △ 重複気味だが意図的防御 |
| 永続化・トランザクション | `add_expense` / `update_expense` / `delete_expense` | ○ 単一モジュール |
| 成功メッセージ | handler `println!` | ○ v4.6.11 方針（human は CLI） |
| 成功後 detail 表示 | handler → `print_expense_detail` | △ read enriched path と非対称 |
| JSON 出力 | **write ではなし** | — |

### handler に寄りすぎているか

| 観点 | 判定 |
|---|---|
| mutation ロジック | **handler にはない** — `add_expense` 等に集約 |
| handler の行数 | add/update は parse → mutate → display の **薄い配線** |
| reservation / note write と比較 | **同型** — 業界標準的な CLI 薄 handler |
| 問題になりうる点 | update で `get_expense` が **2 回**（handler + `update_expense` 内） |

**結論:** handler は **オーケストレーション過多ではない**。責務の主塊は `expense.rs`。

### DTO / 表示 / 永続化の分離

| 経路 | DTO | 永続化 | 表示 |
|---|---|---|---|
| read list/show | `ExpenseJson`（CLI mapper） | service → `Expense` + enrich | `_from_enriched` |
| write add/update | なし（domain `Expense` のみ） | `expense.rs` | `print_expense_detail` → enrich 再実行 |
| write delete | なし | `expense.rs` | amount 1 行のみ |
| trip import | `ExportExpenseV3` | `import_expense_v3` | なし |

read path で確立した **「service enriched → CLI DTO / human」** は write 成功後表示に **未適用**（adapter 経由で結果は一致）。

## 3. Itinerary 配下 entity としての一貫性

### 親子関係

```text
Trip
  └── Day / ItineraryItem
        └── Expense（itinerary_id — application-level 参照、DB FK なし）
        └── Reservation（itinerary_id — application-level 参照、DB FK なし）
        └── Note（trip / day / itinerary の owner 多態、DB FK なし）
```

> **整合性の所在（v4.6.31 訂正）:** `expenses` / `reservations` / `estimates` の `itinerary_id` は
> schema 上 `INTEGER NOT NULL` のみで、**DB-level FOREIGN KEY constraint は未定義**。
> itinerary との参照整合は **application-level validation**（`get_itinerary_item` 等）に依存する。
> 詳細は [v4.6.31 §15](v4.6.31-expense-write-path-migration-plan.md) および
> [v4.6.1 FK/orphan review](v4.6.1-sqlite-fk-orphan-data-hardening-review.md) を参照。

### write path での itinerary 前提

| 観点 | 挙動 |
|---|---|
| `expense add` | **`--itinerary` 必須**（clap） |
| `add_expense` | `get_itinerary_item(conn, itinerary_id)?` |
| participant 解決 | `trip_id_for_itinerary` → trip スコープの participant |
| structured options | itinerary に紐づく trip に participant がいることを要求 |
| `expense update` | expense 行の `itinerary_id` を基準に participant 検証 |
| cascade | `delete_expenses_for_itinerary` — itinerary delete 時 |

**判定:** Expense は **Itinerary 配下 entity** として write path でも一貫。Reservation add と同型（`--itinerary` 起点）。

### list との整合

| list スコープ | write スコープ |
|---|---|
| `--trip` or `--itinerary` | add は **itinerary のみ**（trip 直下 add なし） |

trip 単位の expense は **集計・一覧** であり、新規 expense は常に itinerary に紐づく — ドメイン上妥当。

## 5. すぐ実装 vs 後続 plan に回す

### v4.6.30 でやらない（defer）

| 項目 | 理由 |
|---|---|
| `expense_add` / `update` / `delete` service 化 | Tier 3+ — plan 先行 |
| repository 層抽出 | スコープ大、benefit は write 一括時 |
| `parse_expense_shared_options_*` の handler 移動 | 振る舞い変更リスク、benefit 小 |
| adapter 削除 | write + tests 依存 |
| import path リファクタ | trip import は別論点 |

### 後続 migration plan で扱う（v4.6.31 候補）

| 項目 | 内容 |
|---|---|
| write service result 型 | `{ expense_id }` / `{ expense: ExpenseEnrichedPart }` 等 |
| success display 統一 | add/update 後 `show_expense` service 再利用 |
| parse_* の所在 | command input layer vs service input DTO |
| import_expense_v3 との共通化 | shared mutation core |
| Phase 定義 | read migration（v4.6.25）と同型の gate 表 |

### 低リスクだが v4.6.30 では入れない（任意・plan 後）

| 項目 | gate |
|---|---|
| add/update 後 `show_expense` + `print_expense_detail_from_enriched` | human output **不変** |
| handler の二重 `get_expense` 整理 | 挙動不変 |

**v4.6.30 は review のみ。上記は plan に記載。**

## 7. Write path migration 候補（将来・本 release では未採用）

### Option W-A — 現状維持

handler + `expense.rs` — **短期推奨（v4.6.30）**

### Option W-B — thin write services

```text
services::expense_add::add_expense(...) → ExpenseAddServiceResult { id, expense: ExpenseEnrichedPart }
services::expense_update::update_expense(...)
services::expense_delete::delete_expense(...) → ExpenseDeleteServiceResult { deleted_snapshot }
```

| 観点 | 評価 |
|---|---|
| CLI互換 | handler が service を呼ぶだけなら維持可能 |
| JSON | 影響なし |
| human | enriched result で `_from_enriched` に統一可能 |
| GUI/Tauri | mutation API の土台になる |
| リスク | **中** — transaction / error message 一致が gate |

### Option W-C — command input DTO 層

clap → `ExpenseAddInput` / `ExpenseUpdateInput` → service

parse_* を input DTO 構築に寄せる。**W-B とセット推奨**。

### Option W-D — repository 抽出

`ExpenseRepository` に SQL 集約 — **v4.7.x または write service 安定後**

### 推奨採用順序（概念）

```text
W-A（維持）→ W-B thin write services → W-C input DTO → W-D repository（任意）
```

## 9. 他 entity write path との横並び

| Entity | write handler パターン | read service |
|---|---|---|
| Note | `add_note` / `update_note` / `delete_note` + `print_note_detail` | list/show service 済み |
| Reservation | `add_reservation` / … + `print_reservation_detail` | list/show service 済み |
| Expense | `add_expense` / … + `print_expense_detail`（enrich adapter） | list/show service 済み（enriched） |

Expense だけ成功後表示が **DB enrich adapter** — read migration の影響を受けた差分。**設計負債というより意図的な段階的移行の残り**。
