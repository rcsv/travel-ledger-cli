## 2. materialize と apply の責務差（再確認）

| 観点 | `proposal materialize` | `fragment apply` |
|---|---|---|
| 入力 | Trip Proposal Envelope | Proposal Fragment |
| 対象 Trip | **新規作成** | **既存**（`--trip <id>`） |
| 出力イメージ | 丸ごと Trip 候補 | 部分変更候補（preview Trip JSON） |
| dry-run DB | file-only（DB open なし） | read-only DB access |
| confirm DB | 新規 Trip 全体 insert | supported scope 内の部分 write のみ |
| 混同禁止 | Fragment materialize / proposal apply / trip import で Proposal 採用 | 小さい Trip 化 |

v4.7.15 の分離方針は v4.7.16〜v4.7.19 の実装で **コード・CLI・docs とも維持** されている。

## 3. コマンド体系（現状）

### 3.1 Proposal Envelope

```bash
caglla proposal validate <file> [--json]
caglla proposal show <file>
caglla proposal inspect <file>
caglla proposal materialize <file> --dry-run [--output trip.json] [--start] [--end] [--json]
caglla proposal materialize <file> --confirm [--start] [--end] [--json]
```

- validate / show / inspect: **file-only**
- materialize `--dry-run`: **file-only dispatch**（`main.rs` で DB open 前）
- materialize `--confirm`: **DB open 後** — `import_trip_from_json_with_summary`

### 3.2 Proposal Fragment

```bash
caglla fragment validate <file> [--json]
caglla fragment apply <file> --dry-run --trip <id> [--output preview.json] [--json]
caglla fragment apply <file> --confirm --trip <id> [--json]
```

- validate: **file-only**
- apply（dry-run / confirm とも）: **DB open 必須** — 既存 Trip 読み取りのため

## 4. dry-run / confirm の原則

### 4.1 共通原則

```text
dry-run:
  preview / simulation
  no Trip domain data side effects

confirm:
  explicit user intent
  DB write allowed only within supported scope
```

| 操作 | dry-run | confirm |
|---|---|---|
| フラグ必須 | `--dry-run` または `--confirm` のいずれか | 同上 |
| 併用 | **`--dry-run --confirm` → error** | 同上 |
| フラグなし | **error** | 同上 |

### 4.2 文言差（既知・許容）

| 系統 | dry-run 副作用の説明 |
|---|---|
| proposal materialize | `dry-run means no side effects` |
| fragment apply | `dry-run means no Trip domain data side effects` |

**意味:** materialize dry-run は DB を開かない。fragment apply dry-run は **read-only DB access** を持つが、Trip / Day / Itinerary は変更しない。v4.7.18 以降 docs で明記済み。

### 4.3 file-only / read-only DB / DB write

| パス | DB open | Trip domain data |
|---|---|---|
| proposal validate / show / inspect | なし | — |
| fragment validate | なし | — |
| proposal materialize `--dry-run` | なし | 変更なし |
| proposal materialize `--confirm` | あり | **新規 Trip 全体** insert |
| fragment apply `--dry-run` | あり（read） | 変更なし |
| fragment apply `--confirm` | あり（read + write） | **itinerary INSERT 1件**（supported scope 内） |

## 5. Gate report / 出力形式

### 5.1 Gate report schema

| 系統 | schema version | 主フィールド |
|---|---|---|
| proposal validate | 1 | `ProposalValidationReport` |
| fragment validate | 1 | `FragmentValidationReport` |
| proposal materialize | **2** | `dry_run`, `confirm`, `envelope_valid`, `trip_export_valid`, `trip_id?`, `output?` |
| fragment apply | **2** | `dry_run`, `confirm`, `fragment_valid`, `trip_export_valid`, `resolved_target?`, `preview?`, `inserted_itinerary_id?` |

`--json` は **gate report JSON** を stdout に出す。Trip JSON 候補 / apply preview そのものではない。

### 5.2 Human output

| 操作 | 成功時 human 出力 |
|---|---|
| materialize dry-run | gate report + 任意 `--output` 書き込み通知 |
| materialize confirm | gate report + import summary + `trip list` 向け trip_id |
| apply dry-run | gate report + 任意 preview JSON（`--output` または stdout 続き） |
| apply confirm | gate report + inserted itinerary 概要 |

### 5.3 `--output` と downstream

**推奨フロー（materialize）:**

```bash
caglla proposal materialize envelope.json --dry-run --output candidate.json
caglla trip validate-export candidate.json
# optional: trip diff ...
caglla proposal materialize envelope.json --confirm
```

**推奨フロー（apply）:**

```bash
caglla trip export 1 --output current.json
caglla fragment apply fragment.json --dry-run --trip 1 --output preview.json
caglla trip validate-export preview.json
caglla trip diff current.json preview.json   # optional
caglla fragment apply fragment.json --confirm --trip 1
```

| ルール | 内容 |
|---|---|
| 後続 CLI 向け | **`--output` に書く** — stdout 混在を避ける |
| `--json` | gate report のみ — preview / candidate は含まない |
| apply confirm | **`--output` なし** — DB 更新が主成果 |

## 6. confirm 前の gate（安全条件）

### 6.1 proposal materialize `--confirm`

1. Envelope validation valid
2. 日付解決（proposal / `--start` `--end` / confirmed ranges）
3. schema v8 Trip JSON 生成
4. `trip validate-export` 自己検証
5. blocking errors なし
6. 明示 `--confirm`

DB 書き込み: `import_trip_from_json_with_summary` — **新規 Trip**。

### 6.2 fragment apply `--confirm`

1. Fragment validation valid
2. `--trip` の Trip 存在
3. target 解決（dry-run 同型の simulation まで）
4. apply preview 生成 + `trip validate-export` 自己検証
5. **confirm scope** — Day target + `intent: add` / `add_itinerary` のみ
6. `required_decisions` 空
7. ambiguous / unresolved target なし
8. blocking errors なし
9. 明示 `--confirm`

DB 書き込み: `add_itinerary_item` — **INSERT 1件のみ**。

### 6.3 failure path と DB 不変

| 失敗条件 | materialize confirm | apply confirm | テスト |
|---|---|---|---|
| validation 失敗 | DB 不変 | DB 不変 | unit / integration |
| gate / preview 失敗 | DB 不変 | DB 不変 | ✓ |
| unsupported intent/target | N/A | DB 不変 | `apply-confirm-enrich-*`, itinerary-target |
| required_decisions 非空 | N/A | DB 不変 | `apply-confirm-required-decisions-*` |
| unresolved target | N/A | DB 不変 | `warn-fragment-unresolved-target` |
| `--dry-run --confirm` | error（DB 不変） | error（DB 不変） | CLI tests |

**評価:** confirm 系は gate valid 後のみ DB 触る構造。failure path での DB 不変は **テストで固定済み**。

## 7. fragment apply `--confirm` の現在 scope（明記）

```text
target_type:     day のみ
intent:          add → add_itinerary のみ
DB operation:    itinerary_items への新規 INSERT 1件のみ
```

| 項目 | dry-run preview | confirm 実 DB |
|---|---|---|
| title | ✓ | ✓ |
| note（fragment.notes） | ✓ | ✓ |
| location（candidate_content） | ✓ | ✓ |
| start_time / duration / travel | preview のみ（export 形） | **未反映** — 常に None |
| sort_order | preview で自動算出 | **未指定** — day 末尾 append |
| category | 未 | 未 |
| trip / itinerary target | preview 可能 | **confirm blocking** |
| enrich / warning / reorder_hint | preview のみ | **confirm blocking** |

Trip / Day / 既存 itinerary の **update / delete** は実装されていない。

## 9. update / delete / reorder 前の未解決課題

| 課題 | 内容 | P-6e+ への示唆 |
|---|---|---|
| **target 一意性** | itinerary title 重複時 ambiguous | update/delete 前に reference 規約の強化が必要 |
| **競合検出** | 現状 preview のみ — 実 DB と Fragment 前提の差分未検証 | update 系は **before/after diff 必須** |
| **required_decisions** | 非空で confirm blocking — 良い | update でも人間解決ゲートを維持 |
| **sort_order 方針** | confirm add は day 末尾 append | reorder は sparse ordering との整合設計が必要 |
| **部分フィールド反映** | add は title/note/location のみ | 拡張時はフィールド単位の scope 表が必要 |
| **rollback** | なし | delete/reorder より先に undo 方針を決めるか、明示 defer 継続 |
| **Envelope 側の部分更新** | 意図的に非対象 | materialize は新規 Trip のみを維持 |
| **Note / Expense / Reservation** | Fragment apply 未対応 | 既存 write command との責務境界レビューが先 |

## 10. P-6e+ 拡張候補 — 推奨優先順位

実装順の **推奨**（安全度・既存基盤との整合を優先）:

| 順位 | 候補 | 内容 | 安全条件（概要） |
|---|---|---|---|
| **1** | **add_itinerary 拡張** | category / location / duration / travel_time / start_time / sort_order（`--after` 相当）の confirm 反映 | dry-run preview と confirm 出力の一致；sort_order 規約文書化 |
| **2** | **add_note** | Trip / Day / Itinerary への note 追加 | target 解決 + gate；既存 `note add` との整合 |
| **3** | **add_expense / add_reservation** | Itinerary 配下のみ | expense/reservation write path 境界（v4.6.30 review）との整合必須 |
| **4** | **update_itinerary** | 既存 itinerary のフィールド更新 | **競合検出** + trip export diff + required_decisions；confirm 前に現行 DB スナップショット比較 |
| **5** | **delete / reorder** | 構造破壊リスク大 | 最後尾；dry-run で削除/並び preview + 明示 confirm + 二段階 gate を検討 |

```text
P-6e  推奨: add_itinerary フィールド拡張（confirm scope 拡大・破壊的変更なし）
P-6f  候補: add_note
P-6g  候補: add_expense / add_reservation（Itinerary 配下）
P-6h  候補: update_itinerary（diff / conflict 必須）
P-6i  defer: delete / reorder
```

各 phase は **dry-run 先行 → confirm 追加** の P-6a〜d パターンを踏襲することを推奨。

## 11. Future consideration — m17n / i18n readiness

Full CLI localization is **out of scope for v4.7.x**.

ただし、将来の GUI / caglla.travel では、gate report、validation error、warnings、required decisions、doctor / advisor finding を UI が扱うことになる。

今後の設計では、人間向け英文メッセージだけに依存せず、安定した **machine-readable code** と **human-facing message** を分ける方向を意識する。

これにより、現時点の CLI 出力は読みやすいまま維持しつつ、将来の GUI / caglla.travel surface では prose を parse せずに localization / 表示制御できる。

**v4.7.20 では実装しない:** i18n / localization、翻訳ファイル、message catalog、gate report schema 変更、doctor / advisor schema 変更。
