
## 2. owner validation 契約

### 2.1 責務の所在（確定）

| 関数 / 層 | 責務 | N-3 後の呼び出し元 |
|---|---|---|
| `resolve_note_owner_for_add` | CLI フラグ `(trip, day, itinerary)` → `ResolvedNoteOwner` + **owner 存在確認** | **`note_add` service のみ**（list は既存 `note_list`） |
| `resolve_note_owner_for_list` | add と同一（委譲） | `note_list` service（**変更なし**） |
| `add_note` | body 検証、**owner 再確認**、INSERT | **`note_add` service のみ** |
| `update_note` | 部分更新、body 検証、`get_note` | **`note_update` service のみ** |
| `delete_note` | 存在確認 + DELETE | **`note_delete` service のみ** |
| handler | owner 解決 **呼ばない** | — |

### 2.2 `resolve_note_owner_for_add` の責務（維持）

```text
1. CLI フラグの組み合わせを排他検証（trip / trip+day / itinerary のいずれか1つ）
2. Trip → get_trip
3. Day → find_day_id_by_trip_and_day_number（trip 所属の day_id に解決）
4. Itinerary → get_itinerary_item
5. ResolvedNoteOwner を返す
```

**service 化後も関数本体は `note.rs` に残す。** `note_list` と add で同一ロジックを共有し続ける（list は `resolve_note_owner_for_list` → add 委譲）。

### 2.3 `add_note` 内 defensive double-check — **維持**

| 観点 | 判定 |
|---|---|
| resolve 直後の再確認 | **維持** |
| 理由 | `add_note` の直接呼び出し（tests、import、将来 GUI）に対する防御 |
| service 経路 | resolve → add_note で trip/day/itinerary を **2 回**確認 — **許容**（read list との一貫性より mutation 安全性を優先） |
| N-3 で削除するか | **しない** — plan 上明示的に維持 |

### 2.4 update / delete の owner 扱い

| Action | owner validation |
|---|---|
| update | **不要** — note ID のみ。owner は変更不可 |
| delete | **不要** — note ID のみ。削除前 `get_note` で title 取得 |

owner 多態の複雑さは **add に集中**。update/delete は Expense と同程度に単純。

### 2.5 DB integrity（継続）

```text
notes.owner_type + owner_id — DB-level FK なし
存在確認は application-level（get_trip / find_day_* / get_itinerary_item）
cascade は delete_notes_for_trip/day/itinerary（trip/day/itinerary delete から）
```

FK 導入は migration runner track — **本 plan 対象外**。

## 3. write service result 型の決定

### 3.1 候補比較（add / update）

| 候補 | 内容 | 判定 |
|---|---|---|
| `note_id` のみ | `{ id: i64 }` | **不採用** — handler が再 fetch が必要 |
| **`Note`** | domain 行 | **採用** — read `note_show` と同型、JSON/human 直接利用可 |
| `NoteJson` 新設 | wire DTO | **不採用** — `Note` が既に serde 済み |
| enriched part 新設 | `NoteEnrichedPart` | **不採用** — payer/beneficiary 等がなく enriched 不要（v4.6.35 確定） |

### 3.2 確定型（概念・未実装）

```rust
// services/note_add.rs
pub struct NoteAddParams {
    pub trip: Option<i64>,
    pub day: Option<i64>,
    pub itinerary: Option<i64>,
    pub title: Option<String>,
    pub body: String,
}
pub struct NoteAddServiceResult {
    pub id: i64,
    pub note: Note,
}

// services/note_update.rs
pub struct NoteUpdateParams {
    pub id: i64,
    pub title: Option<String>,
    pub body: Option<String>,
}
pub struct NoteUpdateServiceResult {
    pub note: Note,
}

// services/note_delete.rs
pub struct NoteDeleteParams {
    pub id: i64,
}
pub struct NoteDeleteServiceResult {
    pub id: i64,
    pub title: Option<String>,
}
```

**`id` を add result に含める理由:** 成功メッセージ `Note を追加しました (ID: {id})` が handler に残るため。`note.id` と同値だが、Expense W-1 と同様に handler 可読性のため明示。

**delete の `title`:** 現行 human `Title: {title|-}` と一致。`Option<String>` — body は不要。

### 3.3 enriched layer 不要の再確認

| 観点 | Note | Expense |
|---|---|---|
| participant 解決 | なし | あり → `ExpenseEnrichedPart` |
| JSON 出力 | `Note` 直接 serialize | `ExpenseJson` mapper |
| human 表示 | `print_note_detail(&Note)` | `print_expense_detail_from_enriched` |
| service result | **`Note` で十分** | **`ExpenseEnrichedPart` 必須** |

**Note write service は Expense より result 型が薄い** — 実装・テストともに軽量見込み。

## 5. service API と params 設計

### 5.1 `NoteAddParams`（handler → service）

clap `NoteAction::Add` の mirror。`String` / `Option` — **serde 不要**。

```text
trip: Option<i64>
day: Option<i64>
itinerary: Option<i64>
title: Option<String>
body: String
```

service 内 orchestration:

```text
resolve_note_owner_for_add(conn, trip, day, itinerary)
  → add_note(conn, owner, title, body)
  → get_note(conn, id)
  → NoteAddServiceResult { id, note }
```

**handler から `resolve_note_owner_for_add` / `add_note` / `get_note` を除去。**

### 5.2 `NoteUpdateParams`

```text
id: i64
title: Option<String>
body: Option<String>
```

service 内:

```text
update_note(conn, id, title, body)   // 内部 get_note + validation
get_note(conn, id)
  → NoteUpdateServiceResult { note }
```

**handler から `update_note` / `get_note` を除去** — 二重 fetch は service 内に集約（出力不変）。

### 5.3 `NoteDeleteParams` / `note_delete`

```text
pub fn delete_note(conn: &Connection, params: NoteDeleteParams) -> Result<NoteDeleteServiceResult>
```

内部:

```text
get_note(conn, id)           // snapshot（title）
delete_note(conn, id)        // 内部でも get_note — 許容（現行と同型）
return { id, title: note.title }
```

**handler から `get_note` / `delete_note` を除去。**

### 5.4 `add_note` / `update_note` / `delete_note` の配置

**Phase N-2〜N-5:** `note.rs` に残置。service は orchestration のみ。

```text
services/note_add::add_note
  → note::resolve_note_owner_for_add
  → note::add_note
  → note::get_note
```

将来 repository 化（N-D / v4.7.x）で SQL を移動する可能性はあるが、**本 plan のスコープ外**。

## 6. handler 最終形（Phase N-3 目標）

```text
NoteAction::Add { trip, day, itinerary, title, body } => {
    let result = services::note_add::add_note(&conn, NoteAddParams {
        trip, day, itinerary, title, body,
    })?;
    println!("Note を追加しました (ID: {})", result.id);
    note::print_note_detail(&result.note);
}

NoteAction::Update { id, title, body } => {
    let result = services::note_update::update_note(&conn, NoteUpdateParams {
        id, title, body,
    })?;
    println!("Note を更新しました (ID: {})", result.note.id);
    note::print_note_detail(&result.note);
}

NoteAction::Delete { id } => {
    let result = services::note_delete::delete_note(&conn, NoteDeleteParams { id })?;
    println!("Note を削除しました (ID: {})", result.id);
    println!("  Title: {}", result.title.as_deref().unwrap_or("-"));
}
```

### handler に残す責務（確定）

```text
clap 分解済み引数 → Params struct 組み立て
成功 / 削除メッセージ println!
print_note_detail（add/update）— read show と同一関数
delete title 表示
```

### handler から除去する責務

```text
resolve_note_owner_for_add
add_note / update_note / delete_note 直接呼び出し
get_note（add/update/delete すべて）
```
