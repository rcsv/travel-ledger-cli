## 1. Why `add_estimate` is the next candidate

[v4.7.40 review](v4.7.40-p6m-itinerary-ordering-move-post-release-review.md) は P0 / P1 なし、blocking cleanup 不要とし、**次の Proposal Fragment intent** への継続を推奨した。

Fragment apply が既にサポートする intent:

```text
add_itinerary
add_note
add_expense
add_reservation
update_itinerary
delete_itinerary
reorder_itinerary
move_itinerary
```

**Estimate** は Itinerary 配下の既存子エンティティだが、Proposal Fragment apply path が **未整備**。

`add_estimate` は:

- 新 DB エンティティや schema 変更を要しない
- 既存 `estimate add` / `src/estimate.rs::add_estimate` と同型の Planned Money 境界を踏襲する
- AI / 旅行業者 / ブログ / 手動 Fragment が **既存 Itinerary に計画費用を提案**できる Planned Money ギャップを埋める

## 2. Entity placement（維持）

```text
Trip
  └─ Day
       └─ Itinerary
            ├─ Estimate     ← Planned Money（本 intent）
            ├─ Expense      ← Actual Money
            ├─ Reservation
            └─ Note
```

| 属性 | 方針 |
|---|---|
| 親 | **Itinerary のみ** |
| カーディナリティ | **0..N** per Itinerary |
| 1 行の意味 | **1 つの計画費用明細** |
| Trip / Day 直下 | **unsupported** |
| payer / beneficiary | **非対象** |
| unit price × quantity | **非対象** |
| FX / tax | **非対象** |

---

## 3. Semantic boundary（必須）

```text
add_estimate must never create an Expense row.
add_expense must never be used for planned or approximate costs.
```

| 文脈 | 正しい intent |
|---|---|
| 「入場料はだいたい JPY 10,000 くらい」 | **`add_estimate`** |
| 「入場料に JPY 9,800 を支払った」 | **`add_expense`** |

追加ルール:

- **Estimate → Expense の自動変換はしない**
- 旅行日が過ぎても Estimate は **Planned Money のまま** — Actual Money へ推論しない
- `candidate_content.note` は **Estimate note のみ** — Note エンティティや `itinerary.note` に変換しない

## 5. Target contract

### 5.1 Target type

```text
target_type = itinerary
```

| target_type | 判定 |
|---|---|
| `itinerary` | **supported** |
| `trip` / `day` | **unsupported** — blocking |
| `unresolved` | **blocking** |
| ambiguous itinerary | **blocking** または required decision |

### 5.2 Resolution rules

- target は **既存 Itinerary 1 件**に一意解決すること
- **`--trip <id>`** は CLI 上の正本 Trip
- 解決された Itinerary は **選択 Trip に属すること**（cross-trip は blocking）
- **preferred selector（GUI / API）:** itinerary **ID**
- **CLI convenience:** `day_reference` + `itinerary_reference`（title / sort_order）— 既存 common target resolver を再利用
- 数値 reference は repository 慣行どおり **`itinerary_id` 優先**、次に `sort_order`（両方一致は ambiguous として reject）
- missing / unresolved / ambiguous target は apply 時に **blocking**

### 5.3 Example envelope

```json
{
  "target": {
    "target_type": "itinerary",
    "day_reference": 1,
    "itinerary_reference": 42
  },
  "fragment": {
    "intent": "add_estimate",
    "candidate_content": {
      "amount": "14000",
      "currency": "JPY",
      "title": "Aquarium tickets",
      "note": "Estimated total for five people",
      "sort_order": 0
    }
  }
}
```

`itinerary_reference: 42` は **itinerary ID** を推奨する例。CLI 手書きでは sort_order / title も許容。

## 6. `candidate_content` field policy

`candidate_content` は **JSON object** 必須。

### 6.1 Field table

| Field | Required | JSON type | Meaning |
|---|---|---|---|
| `amount` | **yes** | **string** または **number** | 計画金額（§7） |
| `currency` | **yes** | **string** | 通貨コード（§7） |
| `title` | no | **string** | 見積行タイトル |
| `note` | no | **string** | Estimate 専用メモ |
| `sort_order` | no | **number**（整数） | 同一 Itinerary 内の並び；省略時 `0` |

### 6.2 JSON type confirmation（現行 Fragment 慣行）

`add_expense` の `parse_expense_amount_field` / `parse_add_expense_fields` と **同型**を採用する（v4.7.42 実装時）。

**`amount`:**

| 入力形 | 処理 |
|---|---|
| JSON **string**（例 `"14000"`, `"12.50"`） | `parse_amount_for_currency(text, currency)` — CLI 互換の小数テキスト |
| JSON **number**（例 `14000`） | **整数 minor units** として解釈（`add_expense` 同型） |
| 負数 | **reject** |
| 小数桁過多 | **reject**（通貨精度に基づく） |
| 欠落 / 非数値 | **blocking** |

**`currency`:**

- 非空 **string** 必須
- `validate_currency_code` で正規化（大文字 3 文字）
- 未知 3 文字コード（`XXX` 等）は Expense / Estimate CLI と同様 **許容**

**`title` / `note`:**

- 省略可（キー不在または `null`）
- 空文字列は `normalize_optional_text` により **`null` 相当**（既存 Estimate CLI 同型）
- AI 生成を理由に title 必須化 **しない**

**`sort_order`:**

- 省略時 **`0`**（`add_estimate` CLI 同型）
- JSON **integer number** のみ受け付け
- Estimate は Itinerary 本体の sparse ordering 規則を **適用しない**
- 本 phase で Estimate reorder / normalize は **導入しない**

### 6.3 Optional alias keys（v4.7.42 実装時の推奨）

`add_expense` との AI Fragment 互換のため、実装時に以下の **optional alias** を許容してよい（本 planning では必須としない）:

| Primary | Alias（add_expense 同型） |
|---|---|
| `title` | `description`, `label` |
| `note` | `memo`, `fragment.notes` |

未対応キーは **warning**（`unsupported_field`）とし、blocking にしない（P-6g パターン）。

## 7. Amount / currency validation policy

**Proposal 専用の amount パーサは作らない。** 既存共有契約をそのまま再利用する。

| ルール | 方針 |
|---|---|
| DB 保存 | **最小通貨単位 INTEGER**（`estimates.amount`） |
| 入力 | CLI 互換 decimal text または integer minor units |
| 負数 | **reject** |
| 通貨正規化 | Expense / Estimate CLI と同一 |
| 未知通貨 | 既存挙動を維持 |
| FX 換算 | **なし** |
| 税計算 | **なし** |
| 単価 × 数量 | **なし** |
| payer / beneficiaries | **なし** |

実装時の write path: `src/estimate.rs::add_estimate`（または将来の thin service 化）を **直接呼び出し** — `services::expense_add` とは分離を維持。

## 8. Future dry-run behavior（v4.7.42 想定）

### 8.1 Command

```text
fragment apply <file> --dry-run --trip <id>
```

### 8.2 Flow

```text
fragment validate
  → target itinerary 解決
  → add_estimate field validation
  → structured preview
  → gate report / --output preview Trip JSON
  → trip validate-export PASS
  → DB 非更新
```

### 8.3 Dry-run 原則

- DB **read-only**
- **`estimates` INSERT なし**
- invalid report では既存 Fragment apply 慣行どおり **preview export を抑制**
- `required_decisions` 非空 → `valid: false`、DB 不変

### 8.4 Preview export 配置

| target | export 配置 |
|---|---|
| `itinerary` | `days[].itineraries[].estimates[]` に 1 件 append |

### 8.5 Gate report / preview shape（推奨）

既存 `FragmentApplyDryRunReport` / `FragmentApplyPreviewSummary` への **追加**として、`add_expense` / `add_reservation` と平行な命名を用いる。別 envelope は作らない。

**Report レベル（既存）:**

- `resolved_target` — target 解決サマリ（将来 `itinerary_id` 露出を推奨 — [v4.7.40 GUI 指針](v4.7.40-p6m-itinerary-ordering-move-post-release-review.md)）

**Preview summary 追加候補:**

| フィールド | 意味 |
|---|---|
| `action` | `add_estimate` |
| `estimates_before` / `estimates_after` | Trip 全体の Estimate 件数 |
| `estimate_preview` | 追加予定 Estimate サマリ |

**`estimate_preview` 推奨 DTO（`FragmentApplyEstimatePreview`）:**

```json
{
  "preview": {
    "action": "add_estimate",
    "estimates_before": 3,
    "estimates_after": 4,
    "estimate_preview": {
      "target_itinerary_id": 42,
      "target_itinerary_title": "Okinawa Churaumi Aquarium",
      "amount": 14000,
      "currency": "JPY",
      "title": "Admission tickets",
      "note": "Estimated total for five people",
      "sort_order": 0
    }
  }
}
```

| フィールド | 型 | 備考 |
|---|---|---|
| `target_itinerary_id` | integer | GUI 向け — v4.7.42 で `resolved_target` 拡張または preview 内に明示 |
| `target_itinerary_title` | string | 人間可読確認用 |
| `amount` | integer | **minor units**（preview 内は DB 同型整数） |
| `currency` | string | 正規化後コード |
| `title` / `note` | string or null | optional |
| `sort_order` | integer | 解決後値 |

`expense_preview` と同型の **amount は preview 内 integer**、`candidate_content.amount` は string 入力も許容 — `add_expense` と同じ二段表現。
