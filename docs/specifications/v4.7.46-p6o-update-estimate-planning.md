## 1. Why `update_estimate`

[v4.7.44 review](v4.7.44-p6n-planned-money-post-release-review.md) は **第一候補: `update_estimate`** を推奨した。

Fragment apply が **未整備** の Planned Money 更新:

```text
add_estimate     — v4.7.42 dry-run / v4.7.43 confirm（実装済み）
update_estimate  — 本 planning
delete_estimate  — 将来候補（本 phase 対象外）
```

旅行計画では見積金額・タイトル・メモの **調整** が頻繁に発生する。通常 CLI `estimate update` は存在するが、Proposal Fragment path がない。

## 4. 更新対象の識別

### 4.1 Itinerary target（親）

```text
target_type: itinerary
```

| ルール | 方針 |
|---|---|
| selector | 既存 **ID-first** resolver を再利用（[v4.7.45 §2](v4.7.45-estimate-documentation-and-cli-usage-review.md)） |
| Trip / Day target | **unsupported** — blocking |
| cross-trip | **blocking** |

**既存契約と異なる案は採用しない。**

### 4.2 Estimate identity（子）

#### 案 A — `estimate_id` のみ（**推奨・第一候補**）

```json
"candidate_content": {
  "estimate_id": 3,
  "amount": "15000"
}
```

| 長所 | 説明 |
|---|---|
| identity 明確 | DB PK を直接指定 |
| GUI/API | 一覧・詳細画面から安定参照 |
| 曖昧性なし | title / sort_order / amount 一致 selector を不要 |
| exactly one UPDATE | 1 ID → 0 or 1 行 |

#### 案 B — Itinerary 配下の別 selector

| 候補 | 判定 |
|---|---|
| title selector | **非推奨** — 同一 Itinerary に title 重複がありうる |
| sort_order | **非推奨** — Estimate の sort_order は Itinerary 内 ordering であり itinerary selector と混同しやすい |
| 配列 index | **採用しない** |
| amount / currency 一致 | **採用しない** — 非一意 |

**planning 正本: `estimate_id`（integer, required）**

#### 親子関係の必須検証

解決後に必ず確認:

```text
get_estimate(estimate_id).itinerary_id == resolved_target_itinerary_id
```

不一致 → **blocking**（「Estimate が指定 Itinerary 配下ではない」）。

加えて confirm write 時は（§10）:

```text
UPDATE estimates SET ... WHERE id = ? AND itinerary_id = ?
```

を推奨 — 親子境界を DB write でも防御。

## 5. Fragment payload 契約

### 5.1 Envelope 例

```json
{
  "target": {
    "target_type": "itinerary",
    "day_reference": 1,
    "itinerary_reference": 42
  },
  "fragment": {
    "intent": "update_estimate",
    "candidate_content": {
      "estimate_id": 3,
      "amount": "15000",
      "expected_amount": "14000",
      "expected_currency": "JPY"
    }
  },
  "adoption_hints": {
    "required_decisions": []
  }
}
```

### 5.2 `candidate_content` フィールド

| Field | Required | 意味 |
|---|---|---|
| `estimate_id` | **yes** | 更新対象 Estimate の DB ID |
| `amount` | no | 更新する金額（string 小数 or number minor units） |
| `currency` | no | 通貨コード — **単独指定は拒否** |
| `title` | no | 見積タイトル |
| `note` | no | Estimate メモ |
| `sort_order` | no | 並び順 |
| `clear_title` | no | `true` で title を NULL に（`title` と併用不可、§5.3.2） |
| `clear_note` | no | `true` で note を NULL に（`note` と併用不可、§5.3.2） |
| `expected_amount` | no | TOCTOU baseline（minor units 整数の文字列表現を推奨） |
| `expected_currency` | no | TOCTOU baseline |
| `expected_title` | no | TOCTOU baseline（NULL は `"-"` 等の表示規則で統一） |
| `expected_note` | no | 同上 |
| `expected_sort_order` | no | 同上 |

**存在しない DB 列は追加しない**（payer / beneficiary / unit_price 等は非対象）。

上表のいずれの field も、明示的な JSON `null` が渡された場合は **blocking**（§5.3.1）。

### 5.3 field omission / explicit null / clear flag（確定契約）

**基本規則:**

```text
field omission        → 変更なし（部分更新）
explicit JSON null    → blocking（validation error）
nullable field の削除  → clear flag のみで表現
```

明示的な JSON `null` は **暗黙の clear / 省略として扱わない**。

#### 5.3.1 explicit JSON `null`

以下の **すべての** field に対し、明示的な `null` は **blocking**（validation error）とする。`null` を削除操作として使用しない。

```text
estimate_id
amount
currency
title
note
sort_order
expected_amount / expected_currency / expected_title / expected_note / expected_sort_order
clear_title
clear_note
```

| 入力 | 扱い |
|---|---|
| キー **省略** | **変更なし** |
| キー = **explicit `null`** | **blocking** |

#### 5.3.2 clear flag（nullable field の削除）

nullable field（`title` / `note`）を NULL にするには **明示的な clear flag のみ** を使う。

```text
clear_title: true → title を NULL へ更新
clear_note:  true → note を NULL へ更新
flag 省略 または false → clear しない
```

以下の同時指定は **blocking**（value + clear conflict）:

```text
title + clear_title: true
note  + clear_note: true
```

未対応（本 phase で新設しない）:

```text
clear_amount
clear_currency
clear_sort_order
```

`amount` / `currency` / `sort_order` は NOT NULL 列のため clear 概念を持たない。

#### 5.3.3 空文字列 `""`（実挙動 — コード確認済み）

通常 CLI `estimate update` は `""` を **NULL に正規化**する。これは推測ではなく以下の実装で確認した。

| 確認対象 | ファイル / 関数 | 挙動 |
|---|---|---|
| 正規化 helper | `src/estimate.rs::normalize_optional_text`（L39–51） | `trim()` 後に空なら `None`（= NULL） |
| update path | `src/estimate.rs::update_estimate`（L306–316） | `clear_*` が false かつ `title`/`note` が `Some(value)` の場合 `normalize_optional_text(Some(value))` を通す → `""` は `None` |
| add path | `src/estimate.rs::add_estimate_minor_units`（L127–128） | 同 helper で `title`/`note` を正規化 |

したがって Fragment `update_estimate` の `title: ""` / `note: ""` も **NULL 正規化**を正本とする（既存 CLI と挙動一致）。

| 入力 | DB 結果 |
|---|---|
| `"title": ""` | title = NULL |
| `"note": ""` | note = NULL |

**注意:** 空文字と `clear_*` は結果的に同じ NULL になるが、利用者の意図表現としては **clear flag を推奨**する（GUI/API で「消す」意図が明確）。

#### 5.3.4 確認したテスト（実在）

| テスト | ファイル | 検証内容 |
|---|---|---|
| `cli_estimate_update_clear_title_and_note` | `tests/estimate_cli.rs`（L144–181） | `--clear-title` / `--clear-note` で title/note が JSON `null` になる |
| `test_estimate_crud_roundtrip` 相当（`clear_title: true`） | `src/estimate.rs`（L615–635 のユニットテスト） | `UpdateEstimateParams { clear_title: true }` で title が None |
| `test_update_estimate_title_clear_conflict` | `src/estimate.rs`（L671–685） | `title` + `clear_title: true` 同時指定が **エラー** |
| `test_update_estimate_no_fields_rejects` | `src/estimate.rs`（L662–668） | 更新 field 0 件が **エラー** |
| `test_update_estimate_currency_without_amount_rejects` | `src/estimate.rs`（L687–701） | currency 単独更新が **エラー** |
| `cli_fragment_apply_add_estimate_empty_optional_text_normalizes_to_null` | `tests/fragment_apply_cli.rs`（L2997〜） | add_estimate Fragment path で空 optional text が NULL 正規化 |

**補足:** 空文字列 `""` → NULL の挙動は `normalize_optional_text` のコードで確認したが、`update_estimate` に対する `""` 専用の直接テストは現状 **存在しない**（`--clear-*` 経由のテストのみ）。v4.7.47 dry-run 実装時に、Fragment path 用の空文字正規化テストを追加する（§18）。

#### 5.3.5 unknown field

| 入力 | 扱い |
|---|---|
| unknown field | **warning**（`unsupported_field`）— blocking にしない（P-6g パターン） |

### 5.4 更新可能フィールドの下限

**少なくとも 1 つ** の更新意図が必要:

```text
amount / currency（ペア） / title / note / sort_order / clear_title / clear_note
```

0 件 → **blocking**（CLI 同型）。

## 6. 部分更新 vs 全置換

| 方式 | 判定 |
|---|---|
| **部分更新** | **推奨** — 既存 `estimate update` CLI と同型 |
| 全置換 | **不採用** — GUI 再送コスト・将来 field 追加の互換性が弱い |

理由:

- CLI は指定フィールドのみ更新
- Fragment apply の `update_itinerary` も部分更新
- omission と「現状維持」の区別が GUI 編集と相性良い

注意（実装時）:

- preview agreement は **candidate に存在する更新 field** と **proposed values** を対象
- omitted field は preview `field_changes` に含めない

## 7. amount / currency 更新契約

### 7.1 minor units（P-6n 維持）

```json
{ "amount": 1250, "currency": "USD" }
```

| ルール | 内容 |
|---|---|
| JSON **number** | **minor units 整数** — confirm で再変換 **しない** |
| JSON **string** | `parse_amount_for_currency(text, currency_for_parse)` |
| DB 保存 | 常に INTEGER minor units |

### 7.2 amount のみ更新

| 入力 | 解釈 |
|---|---|
| `"amount": "15000"`（string） | 既存 DB `currency` で parse |
| `"amount": 15000`（number） | minor units としてそのまま — 既存 currency 維持 |
| dry-run / confirm | **同一 parser** — `parse_update_estimate_fields` を共有 |

### 7.3 currency のみ更新

**拒否**（CLI 同型 — 案 B 採用）。

```text
currency を変更する場合は amount も指定してください
```

理由:

- `1250 JPY` → currency だけ `USD` にすると **1250 USD**（意味が変わる）
- 利用者誤解リスクが高い
- 既存 CLI が既に拒否

### 7.4 amount + currency 同時更新

| 順序 | 方針 |
|---|---|
| 1 | `currency` を `validate_currency_code` |
| 2 | `amount` を **新 currency** で parse（string）または minor units（number） |
| 3 | confirm transaction 内で再 parse → preview proposed と一致必須 |
| 二重変換 | numeric amount を string 化して再 parse **しない**（P-6n P0 教訓） |

### 7.5 推奨まとめ

```text
amount-only:     許可
currency-only:   拒否
amount+currency: 許可（新 currency で parse）
numeric amount:  minor units 直渡し（confirm 再変換なし）
```

## 8. no-op update

| ケース | 推奨 |
|---|---|
| **更新 field 0 件** | **blocking**（CLI 同型） |
| **値が現行と同一**（field は指定あり） | **valid success + UPDATE 実行**（CLI 同型） |

根拠:

- `estimate update` は同一値でも `updated_at` を更新する
- GUI「保存」ボタンで idempotent に成功させやすい
- `update_itinerary` Fragment は値同一を dry-run で拒否するが、**Estimate は CLI 優先**と明記

| 観点 | 同一値 success の意味 |
|---|---|
| exactly one UPDATE | SQL は 1 行対象 — SQLite は同一値でも `updated_at` 変更で 1 row affected になりうる |
| idempotency | 同一 candidate 再 confirm は **updated_at のみ変化**しうる — 完全 idempotent ではない |
| preview | `field_changes` が空でも **更新意図があれば** success 可 — または `field_changes` に before==after を載せない（実装時に統一） |

**実装時の推奨（P2）:** dry-run で `field_changes` を出し、before==after の field は省略。confirm は proposed snapshot で UPDATE。

## 9. dry-run 契約（v4.7.47 向け）

### 9.1 フロー

```text
fragment validate
  → itinerary target 解決
  → estimate_id 解決 + 親子関係確認
  → 現行 Estimate 読取
  → candidate 部分更新の validation
  → expected_* baseline 検査（任意キー）
  → proposed value 生成
  → estimate_field_changes[] 生成
  → gate report / --output preview Trip JSON（patched estimate）
  → trip validate-export PASS
  → DB 非更新
```

### 9.2 preview report（候補）

`FragmentApplyPreviewSummary` 追加:

| フィールド | 意味 |
|---|---|
| `action` | `"update_estimate"` |
| `estimate_update_preview` | UPDATE サマリ DTO |

`estimate_update_preview` 推奨 DTO:

| フィールド | 意味 |
|---|---|
| `target_itinerary_id` | 解決済み Itinerary ID |
| `target_itinerary_title` | 人間可読（informative） |
| `target_estimate_id` | 更新対象 Estimate ID |
| `estimate_field_changes` | `{ field, before, after }[]` — `update_itinerary` 同型 |

`inserted_estimate_id` / `updated_estimate_id` は dry-run では **absent**。

### 9.3 blocking

- unsupported target（trip / day）
- unresolved / ambiguous itinerary
- `estimate_id` 欠落 / 不存在
- **いずれかの field への explicit JSON `null`**（§5.3.1）
- Estimate が別 Itinerary 配下
- 更新 field 0 件
- currency のみ
- invalid amount / currency
- `clear_title` + `title` / `clear_note` + `note` 衝突（§5.3.2）
- `expected_*` baseline mismatch
- `required_decisions` 非空

### 9.4 機密性

Estimate title / note に個人情報が入りうる — report は **必要最小限**。export preview への反映は `update_itinerary` と同様 Trip JSON patch。

## 10. confirm 契約（v4.7.48 向け）

### 10.1 transaction フロー

```text
with_transaction("update_estimate confirm")
  1. Fragment 再 parse（intent == update_estimate）
  2. adoption_hints.required_decisions が空であることを再確認（P-6n P2-1 推奨）
  3. itinerary target 再 resolve
  4. estimate_id 再 resolve + 親子関係再確認
  5. 現行 Estimate 再取得
  6. candidate 再 validation
  7. expected_* + preview field_changes[].before と現行 DB 一致（TOCTOU）
  8. preview proposed values と candidate 一致（preview agreement）
  9. exactly one UPDATE（WHERE id AND itinerary_id）
 10. 更新後 row 再取得 + verification
 11. commit
 12. updated_estimate_id を report に設定
```

失敗時: **rollback、DB unchanged**。

### 10.2 success result

| フィールド | 意味 |
|---|---|
| `updated_estimate_id` | confirm 成功時のみ — 更新した Estimate ID |
| `preview.estimate_update_preview` | dry-run 同型を維持可 |

`inserted_estimate_id` は **使用しない**。

過度な肥大化は避ける — `previous_values` 全体の duplicate は不要（`field_changes` で足りる）。

### 10.3 preview agreement

| 層 | 一致対象 |
|---|---|
| **identity** | `target_itinerary_id`, `target_estimate_id` |
| **candidate** | 再 parse した更新 field → proposed values |
| **current-value（TOCTOU）** | preview `before` / `expected_*` ↔ 現行 DB |
| **proposed-value** | preview `after` ↔ UPDATE 直前の computed state |

Itinerary title 変更は identity に含めない（`add_estimate` / `update_itinerary` 同型）。

---

## 11. exactly one UPDATE

| ケース | 方針 |
|---|---|
| 0 rows updated | **エラー + rollback** — 対象削除・親不一致・競合 |
| 1 row updated | **成功** |
| 2+ rows | PK により **起きない** — 念のため verification で検出 |

SQLite「値同一で changes()==0」:

- CLI は `updated_at` 変更のため **1 row affected** になりやすい
- それでも 0 の場合は **blocking**（削除・ID 不一致を疑う）

**WHERE 句推奨:**

```sql
UPDATE estimates SET ... WHERE id = ? AND itinerary_id = ?
```

UPDATE 後 `get_estimate` で全フィールド verification。

---

## 12. TOCTOU / lost-update

### 12.1 想定される競合

| 競合 | 検知 |
|---|---|
| Itinerary 削除 | target resolve 失敗 |
| Estimate 削除 | `get_estimate` 失敗 / 0 rows updated |
| Estimate が別内容に更新済み | **`expected_*` または preview `before` mismatch** |
| Itinerary title 変更 | **許容** — identity は ID |
| Fragment candidate 変更 | preview agreement mismatch |
| 別 Itinerary へ移動 | **現 schema では Estimate の itinerary_id 変更 API なし** — 起きない |

### 12.2 lost-update 防止（推奨）

**`update_itinerary` と同型の楽観的排他:**

- optional `expected_amount`, `expected_currency`, `expected_title`, `expected_note`, `expected_sort_order`
- dry-run が現行 DB から `before` を記録
- confirm で `before` と現行 DB を再比較

`updated_at` を version として必須化は **今回しない**（schema 変更なし、既存 CLI も未使用）。

**last-write-wins のみ**は **不採用** — Fragment apply の update 系と整合しない。

## 13. required_decisions

| 状況 | 扱い |
|---|---|
| 更新 field 0 件 | validation error |
| currency のみ | validation error |
| amount / currency 曖昧 | validation error（currency-only 拒否で緩和） |
| no-op（値同一） | **success**（§8） |
| Estimate 未解決 | blocking |
| 別 Itinerary 配下 | blocking |
| unsupported target | blocking |
| ambiguous itinerary | blocking |
| stale preview / TOCTOU mismatch | blocking |
| unknown field | warning |
| **explicit JSON `null`（任意 field）** | **blocking**（確定 — §5.3.1） |
| **value + clear flag 衝突** | **blocking**（確定 — §5.3.2） |
| 空文字 `""`（title / note） | NULL 正規化（§5.3.3、blocking ではない） |

**confirm transaction 内で `required_decisions` 再確認を推奨**（v4.7.44 P2-1 — v4.7.48 実装時）。

## 14. idempotency

| シナリオ | 現行設計での保証 |
|---|---|
| 同一 candidate 再 confirm | **完全 idempotent ではない** — `updated_at` が変わりうる |
| 値同一再送 | success だが DB write あり（§8） |
| stale preview 再実行 | TOCTOU で **拒否** |
| idempotency key | **対象外**（将来） |

`update_estimate` は `add_estimate` より idempotent になりやすいが、**updated_at 更新**により retry 同一性は保証しない。

## 15. Estimate / Expense 境界

`update_estimate` は以下を **行わない**:

- Expense の作成・更新・削除
- Estimate の新規作成・削除
- Estimate-to-Expense conversion
- 実績金額への自動反映
- 差額の永続化
- payer / beneficiary / 支払状態

**Planned Money のみ** — `estimates` 行の属性更新に限定。
