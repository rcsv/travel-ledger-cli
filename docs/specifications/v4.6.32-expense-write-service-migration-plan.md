## Status

| 観点 | 判定 |
|---|---|
| 基本方針 | v4.6.31 Option W-B を **実装契約レベルで固定** |
| write service result（add/update） | **`ExpenseEnrichedPart`** — read `expense_show` と同型 |
| write service result（delete） | **`{ id, amount, currency }`** — enrich なし |
| display DTO | **`ExpenseJson` は service 外**（CLI のみ） |
| `parse_*` | service 内 orchestration、`expense.rs` に関数残置 |
| `import_expense_v3` | **別経路維持** — 共通化は W-7 以降 |
| `last_insert_rowid` → `RETURNING` | **Phase W-H**（任意 hardening、W-3 後） |
| repository / DB FK | **本 plan 対象外** |
| 次実装 | **v4.6.33** — W-2+W-3 同一リリース推奨 |

**結論（推奨）:**

```text
Expense write service は expense_add / expense_update / expense_delete の 3 module。
add/update の ServiceResult は ExpenseEnrichedPart を返し、read expense_show と対称にする。
delete は削除前スナップショットのみ。ExpenseJson / display DTO は CLI 層に留める。
mutation 実装は初期段階で expense.rs の add_expense 等を service から呼ぶ。
handler は成功 println! と print_expense_detail_from_enriched（add/update）のみ。
W-2 のみ先行リリースは避け、W-2+W-3 を v4.6.33 で一括実装する。
last_insert_rowid hardening と DB FK は本 plan に含めない。
```

## 2. write service result 型の決定

### 2.1 候補比較（add / update）

| 候補 | 内容 | 判定 |
|---|---|---|
| `expense_id` のみ | `{ id: i64 }` | **不採用** — handler が再 fetch + enrich が必要（現行と同じ無駄） |
| `Expense` のみ | domain 行のみ | **不採用** — beneficiary / payer name なし。human 表示に不十分 |
| **`ExpenseEnrichedPart`** | read show と同型 | **採用** — enrich 済み、CLI mapper 再利用可 |
| `ExpenseJson` | CLI wire DTO | **不採用** — service が output DTO を返さない（v4.6.25 原則） |
| display 専用 DTO 新設 | `ExpenseWriteDisplayPart` 等 | **不採用** — `ExpenseEnrichedPart` で十分、重複型を増やさない |

### 2.2 確定型（概念・未実装）

```rust
// services/expense_add.rs
pub struct ExpenseAddParams { /* clap mirror — §4 */ }
pub struct ExpenseAddServiceResult {
    pub id: i64,
    pub expense: ExpenseEnrichedPart,
}

// services/expense_update.rs
pub struct ExpenseUpdateParams { /* clap mirror */ }
pub struct ExpenseUpdateServiceResult {
    pub expense: ExpenseEnrichedPart,
}

// services/expense_delete.rs
pub struct ExpenseDeleteServiceResult {
    pub id: i64,
    pub amount: i64,
    pub currency: String,
}
```

**`id` を add result に含める理由:** 成功メッセージ `Expense を追加しました (ID: {id})` が handler に残るため。`expense.expense.id` と同値だが、handler の可読性のため明示フィールドを維持。

### 2.3 delete が enrich を含まない理由

v4.6.31 §2 と同様。削除後 human は amount のみ。`ExpenseEnrichedPart` は over-fetch。

### 2.4 display DTO との分離

```text
service  → ExpenseEnrichedPart（enriched context、wire ではない）
CLI add/update 後 → print_expense_detail_from_enriched（内部で enriched_expense_to_json 使用）
CLI --json（将来 write が JSON を出す場合）→ 現時点 N/A。出すなら CLI が mapper 経由
```

read path（v4.6.26）と **同一の enriched → display 分離**。

## 3. read / write service の対称性

### 3.1 揃えるもの（必須）

| 観点 | read | write（目標） |
|---|---|---|
| 粒度 | 1 command = 1 service module | **同じ** |
| terminal I/O | なし | **なし** |
| enriched context | `ExpenseEnrichedPart` | add/update も **同型** |
| `ExpenseJson` in service | なし | **なし** |
| `&Connection` in service API | あり | **あり** |

### 3.2 揃えなくてよいもの（許容）

| 観点 | read | write |
|---|---|---|
| 入力 | `trip`/`itinerary` or `id` | **params struct**（clap mirror） |
| delete result | N/A | **snapshot のみ**（enrich なし） |
| 副作用 | なし | **mutation** |
| list vs single | `expense_list` は Vec | write は単件 |

**対称性の目標:** 「service が structured enriched result を返し、CLI が表示/DTO を担当」— mutation の有無は非対称でよい。

### 3.3 モジュール対応表（実装後）

| Command | Read service | Write service |
|---|---|---|
| list | `expense_list` | — |
| show | `expense_show` | — |
| add | — | **`expense_add`** |
| update | — | **`expense_update`** |
| delete | — | **`expense_delete`** |

## 4. service API と params 設計

### 4.1 `ExpenseAddParams`（handler → service）

clap `ExpenseAction::Add` の mirror。`String` / `Option` / `Vec` — **serde 不要**。

```text
itinerary: i64
amount: String
currency: String
title: Option<String>
note: Option<String>
paid_by_name: Option<String>
paid_by_participant: Option<String>   // 未解決の CLI 入力
beneficiary: Vec<String>
shared_with: Option<String>
expense_date: Option<String>
```

service 内で `parse_expense_shared_options_for_add` → `add_expense` → `enrich_expense`。

### 4.2 `ExpenseUpdateParams`

```text
id: i64
title, amount, currency, paid_by_name, paid_by_participant: Option<String>
beneficiary: Vec<String>
shared_with: Option<String>
clear_paid_by: bool
clear_beneficiaries: bool
expense_date, note: Option<String>
```

service 内で `get_expense(id)` → `parse_expense_shared_options_for_update(itinerary_id)` → `update_expense` → `enrich_expense`。

**handler から `get_expense` を除去** — itinerary_id 解決は service 内。

### 4.3 `expense_delete`

```text
pub fn delete_expense(conn: &Connection, id: i64) -> Result<ExpenseDeleteServiceResult>
```

内部: `get_expense`（snapshot）→ `delete_expense` → return `{ id, amount, currency }`。

### 4.4 `parse_expense_shared_options_*` の置き場所

| 層 | 責務 |
|---|---|
| **Phase W-2+W-3** | 関数は `expense.rs` に残し、**service からのみ**呼ぶ（handler から削除） |
| Phase W-5（任意） | private helper 化または `services/expense_shared.rs` へ移動 |
| handler | **呼ばない** |

parse は「CLI 文字列 → `ExpenseSharedOptions`」変換。**service input layer** の一部として扱うが、実装は既存関数再利用。

### 4.5 `add_expense` / `update_expense` / `delete_expense` の配置

**Phase W-2〜W-5:** `expense.rs` に残置。service は orchestration のみ。

```text
services/expense_add::add_expense
  → parse_expense_shared_options_for_add
  → expense::add_expense
  → expense::get_expense
  → expense::enrich_expense
```

将来 repository 化（W-D / v4.7.x）で SQL を移動する可能性はあるが、**本 plan のスコープ外**。

## 5. handler 最終形（Phase W-3 目標）

```text
ExpenseAction::Add { ... } => {
    let result = services::expense_add::add_expense(&conn, ExpenseAddParams { ... })?;
    println!("Expense を追加しました (ID: {})", result.id);
    expense::print_expense_detail_from_enriched(&result.expense)?;
}

ExpenseAction::Update { ... } => {
    let result = services::expense_update::update_expense(&conn, ExpenseUpdateParams { ... })?;
    println!("Expense を更新しました (ID: {})", result.expense.expense.id);
    expense::print_expense_detail_from_enriched(&result.expense)?;
}

ExpenseAction::Delete { id } => {
    let result = services::expense_delete::delete_expense(&conn, id)?;
    println!("Expense を削除しました (ID: {})", result.id);
    println!("  Amount: {}", expense::format_amount_display(result.amount, &result.currency));
}
```

### handler に残す責務（確定）

```text
clap 分解済み引数 → Params struct 組み立て
成功 / 削除メッセージ println!
print_expense_detail_from_enriched（add/update）
delete amount 表示
```

### handler から除去する責務

```text
parse_expense_shared_options_*
add_expense / update_expense / delete_expense 直接呼び出し
get_expense（add/update/delete すべて）
print_expense_detail（adapter）
```

## 6. `print_expense_detail` adapter の寿命

| Phase | write path | adapter |
|---|---|---|
| W-0〜W-1 | handler → `print_expense_detail` | **使用中** |
| **W-3** | handler → `print_expense_detail_from_enriched` | write から **adapter 不使用** |
| W-5 | — | `print_expense_detail` は tests / `expense_to_json` 経由のみ |
| W-6+ | — | tests 整理後に adapter 削除検討 |

**W-3 が adapter 除去の実質的な境界。** W-5 は dead_code / test 整理。

enrich の実行場所:

| 現行 | W-3 後 |
|---|---|
| adapter 内 `enrich_expense` | **service 内** `enrich_expense` → result に含める |

human 出力は `_from_enriched` のみ — **同一 mapper ロジック**で gate。

## 7. `import_expense_v3` と通常 write path

### 現状

| 経路 | 入口 | 永続化 |
|---|---|---|
| CLI `expense add` | handler → `add_expense` | 同一関数 |
| trip import | `import_expense_v3` | **別関数**（participant_ref 解決、独自 INSERT） |

### 方針（本 plan）

```text
Phase W-2〜W-5: import_expense_v3 は触らない
共通化は W-7 以降（任意）— 「shared mutation core」抽出
write service 化と import リファクタを同一 PR にしない
```

**別経路として維持。** import は export schema 契約が絡み、通常 add よりリスクが高い。

## 8. `last_insert_rowid()` → `INSERT ... RETURNING id`

### 現状（`expense.rs`）

```text
add_expense:        tx.execute(INSERT ...) → tx.last_insert_rowid()
import_expense_v3:  同型
```

同期 CLI・単一スレッド・トランザクション内では **実害リスクは低い**（外部レビュー follow-up 結論）。

### 方針

| 項目 | 判定 |
|---|---|
| W-2+W-3 に含めるか | **含めない** — service boundary 変更と混在させない |
| 専用 Phase | **W-H**（hardening、任意） |
| タイミング | **W-3 安定後** — v4.6.34 候補 or 更低優先 |
| 範囲 | 最初は `add_expense` のみ。`import_expense_v3` は別 commit 可 |
| gate | 返却 `id` 不変、`expense_cli` / service tests PASS |

```sql
-- 将来（概念）
INSERT INTO expenses (...) VALUES (...) RETURNING id
```

**repository 抽出（W-D）より先にやってもよいが、write service 実装より後。**

## 11. JSON / golden / human gate

### 不変 gate（W-2〜W-H 共通）


