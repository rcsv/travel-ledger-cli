## 1. Summary

v4.2.0 の章立てと v4.2.2 の polish 後、**Summary と Reservation の責務分離は設計どおり機能している**。Trip / Day summary は旅程本文の前に置かれ、予約詳細は `## Reservations` に集約され、Daily schedule が 58 件でも予約ブロックで膨らまない。

一方、**実旅行のしおりとして読むと**、次の改善余地が残る。

1. **Reservation 見出し行** — `**Day N / {itinerary}** — {provider}` が、便名・駐車場名などでまだ二重に見える（v4.2.2 は `Provider:` 行のみ整理）
2. **種別グループ見出し** — `### Flight` が往路・復路で **2 回** 出る（種別切替の副作用）
3. **Period 表示** — Hotel の `2026-04-26T16:40 — 2026-04-29T10:00` が ISO 生値で、しおり向けではない
4. **Confirmation の視認性** — 駐車場のみ番号あり。便・ホテルは空だが、ある場合のラベル・配置は未最適化
5. **フィールドラベル** — `Confirmation:` / `Remark:` / `Period:` が英語固定で、本文日本語と混在
6. **種別ごとのテンプレ差** — Flight / Hotel / Parking / Rental Car で読者が欲しい情報の優先度が異なるが、現状は一律ブロック

**結論:** データモデル変更は不要。`format_reservations_markdown` と Summary 周辺の **表示ルール・並び・種別テンプレ** を v4.3.0 で明文化・実装するのが適切。章立て全体の再設計は **Do not do**。

**推奨:** 本書をレビュー承認してから実装に入る（変更は中規模。一括 PR より段階実装可）。

## 3. Responsibility boundaries（維持）

以下の役割分担は **v4.3.0 でも変更しない**。

```text
Summary     = Trip / Day の人間向け要約（Abstract）
Reservation = 予約・確認に必要な構造化情報（booking confirmation）
Itinerary   = スケジュール上の行動（時刻・場所・カテゴリ）
Note        = 補足メモ（方針・注意・調整余地）
Remark      = Reservation 付随の短い運用メモ（予約文脈に留める）
```

| 混同しやすい境界 | ルール |
|---|---|
| Summary vs Note | Summary は overview / day 冒頭の要約。長文の方針・調整メモは `## Notes` |
| Summary vs Itinerary | Summary は「その日の読み味」；詳細時刻・場所は Itinerary |
| Reservation vs Itinerary `note` | 予約番号・チェックイン期間は Reservation；乗り場の行き方は Itinerary `note` または Note |
| Reservation vs Planned cost | 金額は Estimate / Planned cost；予約確認は Reservations |

**Do not merge:** Summary に confirmation を埋め込まない、Reservation 章に Day summary を複製しない、Itinerary 全件に予約詳細を inline 展開しない。

## 4. Current state — Okinawa fixture

### 4.1 Trip summary（Trip overview）

**観測:**

```markdown
## Trip overview

高齢者2名を含む5名で、沖縄本島北部・瀬底島を拠点に過ごす4日間の家族旅行。…

- Days: 4
- Itineraries: 58
- Checklist: 2 / 4 completed
```

| 観点 | 評価 |
|---|---|
| 配置 | Cover の次、`## Trip overview` 直下の本文 — v4.1.0 どおり **自然** |
| ラベル | `Summary:` 見出しなし。旅行しおりとして **適切**（要約がそのまま読める） |
| 直後の stats | 運用メトリクスが箇条書きで続く。要約と混ざるが **許容**（要約＝散文、stats＝事実） |
| Participant 表 | 沖縄 sample は未登録のため非表示 — **Keep** |

**分類:** Trip summary の **配置と無ラベル表示は Keep**。v4.3.0 で変更不要。

### 4.2 Day summary（Daily schedule）

**観測:** 各 `### Day N — date` の直後に 1 段落の Day summary、その後 `####` Itinerary 列。

例（Day 1）:

```text
中部国際空港から那覇へ移動し、レンタカーを受け取って首里城を観光。夕方に瀬底島へ入り、…
```

| 観点 | 評価 |
|---|---|
| 位置 | 日見出しの直後 — **スキャン向きで有用** |
| 内容 | その日の「流れ」を itinerary 58 件を読まずに把握できる |
| Day Note との関係 | 同じ日の `## Notes` に Day note あり（例: 朝の休憩）。Summary＝要約、Note＝運用注意 — **責務分離 OK** |
| 重複 | Summary と最初の数 itinerary で題材が重なるのは **意図どおり**（abstract vs schedule） |

**分類:** Day summary の **配置は Keep**。v4.3.0 では体裁の微調整のみ検討可（下記 §6.1）。

### 4.3 Reservations chapter（現行）

**観測（沖縄 5 件）:**

| # | Type | 見出し行 | フィールド |
|---|---|---|---|
| R1 | parking | `**Day 1 / P1 G Parking** — セントレア P1 G Parking` | Confirmation, Remark |
| R2 | flight | `**Day 1 / NU045 …** — NU045 NGO ⇒ OKA` | （本文なし） |
| R3 | rental_car | `**Day 1 / Toyota Alphard…** — Ks Rent A Car` | Provider, Remark |
| R4 | hotel | `**Day 1 / チェックイン** — ヒルトン瀬底` | Provider, Period |
| R5 | flight | `**Day 4 / NU046 …** — NU046 OKA ⇒ NGO` | （本文なし） |

**並び:** DB 取得は `ORDER BY day, sort_order`（旅程順）だが、Markdown 出力は **種別が変わるたびに `### {Type}` を挿入**。その結果 **Flight 見出しが 2 回**（Hotel の後に Day 4 便が再度 `### Flight`）。

**グループ方式の比較:**

| 方式 | 旅行しおりとして | 沖縄 sample |
|---|---|---|
| **A. 種別グループ**（現状） | 種別ごとに探しやすいが、旅程順が崩れ、同一種別見出しが分裂 | Flight ×2 |
| **B. 旅程順フラット** | 時系列で読める。種別は各行ラベル | 往路→レンタカー→ホテル→復路の流れに近い |
| **C. 日別グループ** | 「今日の予約」が分かりやすい。複数日にまたがる Hotel は Day 1 に集約 | Day 4 復路便は Day 4 ブロック |
| **D. 種別グループ＋旅程順ソート** | 全 Flight をまとめるが、時系列は種別内のみ | 往路・復路が連続するが Day 1/4 が離れる |

**推奨（v4.3.0）:** **B — 旅程順フラット** を第一候補。

- しおりは **時系列で読む** 利用が多い
- 沖縄のように件数が少ない Trip では種別索引より「いつ何の予約か」が重要
- 種別は各行先頭またはサブラベルで示す（`[Flight]` 等）。章内 `###` は **予約件ごとではなく付けない**（または Day グループのみ — 下記 §6.2）

件数が多い Trip で種別索引が必要になった場合は **v4.3.x 以降** で TOC 的補助を検討（本リリース非対象）。

## 5. Findings by review point

### 5.1 Trip summary の配置・体裁

| 項目 | 分類 | 提案 |
|---|---|---|
| overview 先頭の散文 summary | **Keep** | 変更なし |
| `Summary:` ラベル追加 | **Do not do** | 章名で十分 |
| summary と stats の間に区切り線 | **Defer** | 現状で可読。必要なら空行のみ |
| Participants 表の位置（summary と stats の間） | **Keep** | v4.1.0 どおり |

### 5.2 Day summary の有用性

| 項目 | 分類 | 提案 |
|---|---|---|
| Day 見出し直後の 1 段落 | **Keep** | 沖縄で有用と確認 |
| 空 summary の日 | **Keep** | 段落省略（現状どおり） |
| Day summary の最大長・折り返し | **Keep** | データそのまま。truncate はしない |
| Day summary を Reservations へ複製 | **Do not do** | 責務重複 |

### 5.3 Reservations のグループ化

| 項目 | 分類 | 提案 |
|---|---|---|
| 種別 `###` の挿入（現状） | **Change → v4.3.0** | 廃止し **旅程順フラット** へ |
| 同一種別見出しの重複（Flight ×2） | **Change** | 上記で解消 |
| 日別 `### Day N` グループ | **Defer** | 件数増加時に再評価。沖縄 5 件ではフラットで十分 |
| Reservations 章の位置（Daily schedule 後） | **Keep** | v4.1.0 |

### 5.4 Confirmation の視認性

| 項目 | 分類 | 提案 |
|---|---|---|
| `Confirmation: {code}` 行 | **Keep** 構造 | ラベルをしおり向けに（§6.3） |
| 値があるときの強調 | **Change → v4.3.0** | 行を予約ブロックの **上部** に（見出し直後） |
| 値がないときの行 | **Keep** | **出さない**（現状どおり） |
| public profile でのマスク | **Defer** | v4.1.0 §3.5 将来 |

### 5.5 種別ごとの表示フォーマット

現状は全種別同一テンプレ。提案は §6.4。

| 種別 | 読者が欲しい情報（優先順） |
|---|---|
| **flight** | 便名・区間・（あれば）予約番号 |
| **hotel** | 宿名・チェックイン〜アウト（人間可読日時）・（あれば）確認番号 |
| **parking** | 場所・確認番号・利用日の補足 remark |
| **rental_car** | 会社名・車種（itinerary 側）・remark（ETC 等） |

### 5.6 Remark の見せ方

| 項目 | 分類 | 提案 |
|---|---|---|
| DB 値の透過表示 | **Keep** | 自動書き換え・除去はしない |
| `Remark:` 英語ラベル | **Change → v4.3.0** | `備考:` 等、しおり向けラベル（§6.3） |
| remark と Checklist 重複（ETC） | **Keep** | Reservation＝予約文脈、Checklist＝持ち物。両方あってよい |

### 5.7 見出し行と title / provider の重複

v4.2.2 で `Provider:` **行** は冗長時省略済み。残る課題は **太字見出し行末尾の `— {provider}`**。

| パターン | 例 | 問題 |
|---|---|---|
| provider ≈ itinerary title | Flight: `NU045…` — `NU045 NGO ⇒ OKA` | 末尾が冗長 |
| title 汎用・provider が本体 | Hotel: `チェックイン` — `ヒルトン瀬底` | **有用** — 残す |
| title と provider が別情報 | Rental: `Toyota Alphard…` — `Ks Rent A Car` | **有用** — 残す |
| parking | `P1 G Parking` — `セントレア P1 G Parking` | やや重複だが駐車場名として許容 |

**提案（v4.3.0）:** 見出し行を次のルールで組み立てる。

```text
primary = "Day {n} / {itinerary_title}"
subtitle = provider が itinerary_title と冗長でないときのみ " — {provider}"
見出し = "**{primary}{subtitle}**"
```

冗長判定は v4.2.2 の `reservation_provider_line_redundant` を **見出し行にも適用**。

`Provider:` 行は、subtitle で既に出した場合は **省略**（現状ロジックの拡張）。

## 6. Proposed display rules（v4.3.0 正本案）

### 6.1 Summary

| ルール | 内容 |
|---|---|
| S1 | Trip `summary` は `## Trip overview` 直下の **最初の本文**（ラベルなし） |
| S2 | Day `summary` は `### Day N — date` 直後の **最初の本文**（ラベルなし） |
| S3 | summary が空なら **段落ごと省略**（プレースホルダなし） |
| S4 | Summary 本文を Notes / Reservations / Itinerary に **複製しない** |
| S5 | stats 箇条書き・Participants 表は summary の **後**（現状維持） |

### 6.2 Reservations — 章構造と並び

| ルール | 内容 |
|---|---|
| R1 | `## Reservations` は Daily schedule の **後**（v4.1.0 維持） |
| R2 | 予約は **旅程順**（`day ASC, sort_order ASC`）— DB 取得順をそのまま出力 |
| R3 | 種別による `### Parking` / `### Flight` 見出しは **使わない** |
| R4 | 予約ブロック間は **空行 1 行** で区切る |
| R5 | 0 件なら章ごと **省略** |

### 6.3 Reservations — 共通フィールド

| フィールド | 表示条件 | 表示形式（案） |
|---|---|---|
| 見出し | 常に | `**Day {n} / {itinerary_title}**` + 非冗長時のみ ` — {provider}` |
| 種別 | 常に | 見出し直後の 1 行: `種別: {日本語表示}`（`format_reservation_type_display` の i18n 化は **Defer** 可。まずは英語表示 Keep も可） |
| 確認番号 | `confirmation_code` あり | `確認番号: {code}` — **見出しの次の行** |
| 期間 | `start_at` / `end_at` あり | `期間: {formatted}` — §6.5 |
| URL | `reservation_site_url` あり | `URL: {url}` |
| 備考 | `remark` あり | `備考: {remark}` |

**ラベル言語:** 沖縄 sample は日本語本文のため、Travel Book では **Confirmation / Remark / Period → 確認番号 / 備考 / 期間** を v4.3.0 の既定とする。章見出し `## Reservations` の英語化は **Defer**（v4.2.1 R12）。

### 6.4 Reservations — 種別ごとの推奨（テンプレの優先度）

種別専用の **別ブロック形式** にはせず、共通ルール R1–R5 + フィールド出し分けで足りる。種別は **どのフィールドを強調するか** のみ定義する。

| 種別 | 見出し | 必須に近いフィールド | 省略しやすいフィールド |
|---|---|---|---|
| flight | 便名・時刻は **itinerary_title** を正とする | 確認番号（あれば） | provider（便名と同一なら見出し subtitle も省略） |
| hotel | subtitle に **provider（宿名）** | 期間（チェックイン〜アウト） | 確認番号（なければ出さない） |
| parking | subtitle に **provider（駐車場名）** | 確認番号 | remark |
| rental_car | subtitle に **provider（会社名）** | 備考（ETC 等） | 期間（なければ出さない） |

### 6.5 Period の人間可読フォーマット

| ルール | 内容 |
|---|---|
| P1 | ISO 8601 生値をそのまま出さない |
| P2 | 同一日の start/end: `2026-04-26 16:40 — 10:00` のように **日付は一度、時刻はローカル表示** |
| P3 | 日をまたぐ: `2026-04-26 16:40 — 2026-04-29 10:00` |
| P4 | 日付のみ / 時刻のみの片方入力は **ある方だけ** 表示 |
| P5 | パース不能時は **生値フォールバック**（データ損失よりマシ） |

実装は `format_period` の Travel Book 専用ラッパ、または markdown.rs 内の表示用関数で足りる（DB / export schema 変更なし）。

### 6.6 Daily schedule 内の Reservation ハイライト（v4.1.0 任意項目）

| 項目 | 分類 | 提案 |
|---|---|---|
| Itinerary に紐づく予約を 1 行で示す | **Defer → v4.3.1 以降** | v4.3.0 は Reservations 章の改善に集中 |
| 例: `予約: 確認番号 EXAMPLE-PARKING-001` | | Daily schedule 58 件との合成コスト・重複リスク |

**理由:** 沖縄では Reservations 章が Daily の直後にあり、5 件なら往復参照で足りる。ハイライトは **別マイルストーン** とする。
