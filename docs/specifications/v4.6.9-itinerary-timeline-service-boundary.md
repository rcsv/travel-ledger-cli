# Itinerary timeline service boundary

v4.6.4〜v4.6.8 の read-only service boundary に続き、**`itinerary timeline` 単独**で service 層を導入する。

| Item | Status |
|---|---|
| Target | v4.6.9 — itinerary timeline service boundary |
| Target command | **`itinerary timeline` のみ** |
| Scope | service 層導入（structured result）、CLI 出力互換維持 |

---

## Status

| 観点 | 判定 |
|---|---|
| service layer | **`src/services/itinerary_timeline.rs`** 追加 |
| target command | `itinerary timeline` が service 経由 |
| terminal I/O | service 内に `println!` / `eprintln!` なし |
| CLI output | **不変**（human only、`--json` なし） |
| Ordering | **不変**（Sequence-first: Day → `sort_order` → `id`） |
| Trip not found | **不変**（`Trip not found: <id>`） |
| DB schema / migration | **変更なし** |

---

## Background

目標形:

```text
CLI command
  → service/usecase
  → repository
  → SQLite

Tauri command
  → same service/usecase
  → repository
  → SQLite
```

`itinerary timeline` は GUI の旅程タイムライン表示に直結する。v4.6.9 では表示仕様改善は行わず、既存挙動を保ったまま service boundary のみ導入する。

---

## Target command

- `itinerary timeline <trip_id>`
- `--json` フラグは**なし**（従来どおり human output のみ）

---

## Current flow before split

```text
main.rs (ItineraryAction::Timeline)
  → itinerary::list_itinerary_items(conn, trip_id)
  → trip::get_trip(conn, trip_id)
  → println!("{trip.name} のタイムライン:")
  → itinerary::print_itinerary_timeline(&items)
```

DB 取得と terminal 出力が handler に混在していた。

---

## Service boundary introduced

新規:

```text
src/services/itinerary_timeline.rs
```

API:

```rust
pub struct ItineraryTimelineServiceResult {
    pub trip: Trip,
    pub items: Vec<ItineraryItem>,
}

pub fn get_timeline(conn: &Connection, trip_id: i64) -> Result<ItineraryTimelineServiceResult>
```

service は `itinerary::list_itinerary_items` と `trip::get_trip` を orchestrate し、**structured result のみ**返す。

---

## CLI compatibility

- human 出力は従来どおり:
  - `{trip.name} のタイムライン:` 見出し（handler の `println!`）
  - `print_itinerary_timeline(&items)` による Day グループ表示
- empty state: `日程はまだ登録されていません。`（`print_itinerary_timeline` 内）

---

## JSON compatibility

- `itinerary timeline` に `--json` は存在しない（変更なし）

---

## Ordering compatibility

Sequence-first ordering は `itinerary::list_itinerary_items` の SQL で担保:

```text
ORDER BY d.day_number, i.sort_order, i.id
```

service は順序を変えない。

---

## Sequence-first behavior

- `sort_order` が主ソートキー
- `start_time` は表示ラベル（並び順に使わない）
- 例: `09:00` の「首里城」（sort_order=1）が `10:50` の「国際通り」（sort_order=2）より先

---

## Error compatibility

- Trip 不存在: `Trip not found: <id>`（`list_itinerary_items` 内の `trip::get_trip` 経由）

---

## Test coverage

- service test:
  - Trip not found message 維持
  - Day → sort_order 順（start_time より優先）
  - sort_order 順（start_time 未定の行を含む）
- 既存 `itinerary.rs` / CLI テストを回帰として維持

---

## What changed

- `src/services/itinerary_timeline.rs` 追加
- `src/services/mod.rs` に `itinerary_timeline` module 追加
- `main.rs` の `ItineraryAction::Timeline` が service 経由に

---

## What did not change

- `itinerary list` / `itinerary show`（対象外）
- `trip list` / `show` / `stats` / `day list`（対象外）
- Travel Book export（対象外）
- write command / migration / FK（変更なし）
- timeline 表示仕様（変更なし）

---

## Lessons learned

- `itinerary timeline` は trip 名 + items の両方が必要 — `Trip` + `Vec<ItineraryItem>` を返す
- `--json` がない command でも service boundary は有効（GUI 再利用のため）
- ordering は repository 側に閉じ、service は加工しない
