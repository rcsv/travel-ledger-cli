## 1. 背景と位置づけ

### 1.1 なぜ add 系より慎重か

| 観点 | add_expense / add_reservation | update_itinerary |
|---|---|---|
| DB 操作 | 子テーブルへ **append** | 既存 itinerary 行の **上書き** |
| 失敗時の影響 | 未追加のまま | 部分更新・意図しない上書き |
| preview | append シミュレーション | **before/after diff** が必須 |
| 競合 | 主に target 解決 | target 解決 + **現行 DB 値との競合** |
| 既存 `trip diff` | 追加検出に有効 | title/day/time 変更は removed+added に見える |

[v4.7.20 §10](v4.7.20-p6-post-implementation-review.md) の推奨どおり、**競合検出 + trip export diff + required_decisions** を confirm 前に必須とします。

### 1.2 既存資産

| 層 | 既存資産 |
|---|---|
| DB | `itinerary_items`（1行 = 1 itinerary） |
| export | `days[].itineraries[]` — `ExportItineraryV3` |
| CLI | `itinerary update <id>` — 部分フィールド更新 |
| write path | `itinerary::update_itinerary_item`（**service 層なし**、CLI 直結） |
| fragment apply | **未対応**（`VALID_INTENTS` に `update_itinerary` なし） |
| diff | `trip diff`（export JSON 比較）、`io/diff.rs` |
| target 解決 | `resolve_apply_target` + `resolve_itinerary_id_for_apply_target`（add 系で確立） |

`enrich` / `replace_candidate` / `reorder_hint` は **update_itinerary の代替ではない**（Day summary / warning のみ）。

## 2. 推奨フェーズ分割

### Phase A — v4.7.28 P-6i dry-run（先行）

```text
fragment apply <file> --dry-run --trip <id>
  → fragment validate（update_itinerary 認識）
  → target itinerary 一意解決
  → 現行 DB snapshot（build_trip_export_v3）
  → candidate_content から更新フィールド抽出・validation
  → 競合検出（baseline mismatch / conflicts）
  → in-memory itinerary patch preview
  → gate report（field-level before/after）
  → --output preview Trip JSON
  → trip validate-export PASS
  → DB 非更新
```

| 項目 | 内容 |
|---|---|
| Command | `fragment apply --dry-run --trip <id>` |
| intent | `update_itinerary` |
| target | itinerary のみ |
| DB | read-only |
| `--confirm` | **未対応** |

### Phase B — v4.7.29 P-6i confirm（後続）

```text
fragment apply <file> --confirm --trip <id>
  → Phase A と同型 gate
  → validate_confirm_scope（update_itinerary + itinerary）
  → itinerary_items UPDATE 1行（許可フィールドのみ）
  → gate report inserted_itinerary_id は使わない
  → updated_itinerary_id（新規候補）を返す
```

| 項目 | 内容 |
|---|---|
| DB operation | **`itinerary_items` UPDATE 1行のみ** |
| write path | `itinerary::update_itinerary_item`（将来 `services::itinerary_update` 化は任意） |
| 確認 | `itinerary show <id>` |

## 3. proposed update fields

### 3.1 v4.7.28 MVP（in-place 属性更新）

既存 `ExportItineraryV3` / `update_itinerary_item` / CLI `itinerary update` と整合する **in-place 更新** に限定します。

| Fragment `candidate_content` キー（alias 含む） | export / DB フィールド | 必須 | validation |
|---|---|---|---|
| `title` | `title` | optional（更新指定時のみ） | 非空文字 |
| `note` | `note` | optional | 空文字でクリア可（CLI 同型） |
| `location` | `location` | optional | |
| `category` | `category` | optional | `parse_itinerary_category`；`none` で解除 |
| `start_time` / `time` | `start_time` | optional | `parse_time_hhmm` |
| `duration_minutes` / `duration` | `duration_minutes` | optional | 非負整数 |
| `travel_minutes` / `travel_time_minutes` / `travel` | `travel_minutes` | optional | 非負整数；矛盾 alias は blocking |

**少なくとも 1 つの更新フィールドが必要。** すべて欠落または現行値と同一 → **no-op** として blocking（`valid: false`）。

### 3.2 Phase B 以降の拡張候補（P-6i MVP 外）

| フィールド | 理由 |
|---|---|
| `day` / `day_reference` 移動 | Day 変更は `day_id` 再解決を伴い、identity 変更に近い |
| `sort_order` / `order` | reorder リスク；[v4.7.20](v4.7.20-p6-post-implementation-review.md) で reorder は defer |
| 子エンティティ（expense / reservation / note） | add 系 intent の責務 |
| Trip / Day 直下の変更 | unsupported |

**推奨:** v4.7.28/29 では **day / sort_order は unsupported** とし、P-6j 以降の `update_itinerary` 拡張または `reorder` intent で別途設計します。

### 3.3 既存 intent との関係

| 既存 intent | P-6i との関係 |
|---|---|
| `add` | 新規 itinerary **append** — 別 intent |
| `enrich` | Day `summary` のみ — 別責務 |
| `replace_candidate` | warning のみ — **update_itinerary に統合しない** |
| `reorder_hint` | warning のみ — defer |

## 4. proposed diff / conflict behavior

### 4.1 before / after の出し方

**二層で提供する**（add 系の append preview だけでは不十分）。

#### A. Gate report 内の field-level diff（主）

対象 itinerary 1件について、**変更されたフィールドのみ** before/after を出す。

```json
{
  "action": "update_itinerary",
  "itinerary_field_changes": [
    {"field": "title", "before": "Morning temple", "after": "Morning temple visit"},
    {"field": "category", "before": "activity", "after": "museum"}
  ],
  "itinerary_fields_before": 7,
  "itinerary_fields_after": 7
}
```

| 方針 | 内容 |
|---|---|
| unchanged field | **省略**（gate report では出さない） |
| 変更なし | `itinerary_field_changes: []` → no-op blocking |
| 対象 itinerary | `resolved_target` + 内部で解決した `itinerary_id`（confirm 用 preview 候補） |

`itinerary_fields_before/after` は **Trip 全体の itinerary 件数**（add 系の `expenses_before/after` と同型）。件数は変わらない想定。

#### B. 外部 `trip diff`（補助）

```bash
caglla trip export <id> --output before.json   # または dry-run 前の snapshot
caglla fragment apply fragment.json --dry-run --trip <id> --output preview.json
caglla trip diff before.json preview.json
```

**注意:** 既存 `trip diff` は itinerary を `day + start_time + title` で同一視するため、`title` / `start_time` / `day` 変更は **modified ではなく removed+added** に見えます（`src/io/diff.rs` の `ItineraryKey`）。  
→ **gate report の field-level diff を正本** とし、`trip diff` は補助ツールと位置づけます。

### 4.2 conflict 検出

| 種別 | 条件 | 扱い |
|---|---|---|
| unresolved target | `target_type: unresolved` | blocking |
| ambiguous target | itinerary_reference が複数一致 | blocking |
| unsupported target | trip / day target | blocking |
| `required_decisions` 非空 | adoption_hints | `valid: false`；confirm 不可 |
| **baseline mismatch** | `candidate_content.expected_*` または `adoption_hints.conflicts` が現行 DB と不一致 | blocking |
| **no-op** | 有効な更新フィールドが 0 | blocking（「変更なし」） |
| invalid field | category / time / 負の duration 等 | blocking |
| fragment_valid false | fragment validate 失敗 | blocking |

#### baseline / conflict の具体案

Fragment 側で任意の expected baseline を受け付ける（将来 GUI 連携向け）:

```json
{
  "candidate_content": {
    "title": "Morning temple visit",
    "category": "museum",
    "expected_title": "Morning temple",
    "expected_category": "activity"
  },
  "adoption_hints": {
    "conflicts": ["Category may have been updated since fragment was authored"]
  }
}
```

| ソース | 比較対象 | mismatch 時 |
|---|---|---|
| `expected_<field>` | 現行 DB / export の同フィールド | blocking error + `conflicts` に記録 |
| `adoption_hints.conflicts` | 人間向け decision 候補 | `required_decisions` にマージ；非空なら `valid: false` |
| `adoption_hints.assumptions` | warning のみ（v4.7.22 以降同型） | DB 不変；confirm 阻害しない |

**confirm 時の追加チェック（Phase B）:** dry-run gate 通過後、**再度**現行 DB を読み、`itinerary_field_changes` の before 値がまだ一致することを確認（TOCTOU 対策）。不一致なら DB 更新なしで blocking。

### 4.3 preview シミュレーション

```text
resolve_apply_target → itinerary (day_number + sort_order)
  → find_itinerary_mut_in_export_day(day, sort_order)
  → candidate_content の指定フィールドのみ上書き
  → 子配列（expenses / reservations / estimates）は触らない
```

preview Trip JSON は schema v8 のまま `trip validate-export` で自己検証します。

## 5. safety boundary

P-6g / P-6h と同型の原則を維持します。

```text
dry-run:
  preview / simulation
  read-only DB access
  no Trip / Day / Itinerary / Expense / Reservation / Note domain write

confirm:
  explicit --confirm のみ
  supported scope 内のみ DB write

--dry-run --confirm:
  error（併用不可）
```

| 条件 | dry-run | confirm |
|---|---|---|
| trip / day target | blocking | blocking |
| unresolved / ambiguous | blocking | blocking |
| required_decisions 非空 | `valid: false` | blocking（DB 不変） |
| conflict / baseline mismatch | blocking | blocking（DB 不変） |
| no-op（変更なし） | `valid: false` | blocking（DB 不変） |
| invalid category / time / numeric | blocking | blocking（DB 不変） |

---

## 6. DB mutation boundary

### dry-run（Phase A）

| 操作 | 境界 |
|---|---|
| **やる** | `build_trip_export_v3` による read |
| **やらない** | いかなる INSERT / UPDATE / DELETE |

Trip / Day / Itinerary / Expense / Reservation / Note / Estimate / Receipt など **domain data は一切変更しない**。

### confirm（Phase B）

| 操作 | 境界 |
|---|---|
| **やる** | `itinerary_items` **1行 UPDATE**（`update_itinerary_item` 経由） |
| **やらない** | INSERT、DELETE、他 itinerary 行の変更 |
| **やらない** | expenses / reservations / notes / estimates の変更 |
| **やらない** | Trip / Day 直下の変更 |
| **やらない** | reorder 一括、子テーブル cascade |

`updated_at` の更新は `update_itinerary_item` 既存仕様に従います。

## 7. gate report 拡張案

`FragmentApplyDryRunReport` / `FragmentApplyPreviewSummary` に optional フィールドを追加（schema version は現状 `2` のまま、optional 追加のみ）。

| フィールド | Phase | 意味 |
|---|---|---|
| `action` | A/B | `update_itinerary` |
| `resolved_target` | A/B | 既存 |
| `itinerary_field_changes[]` | A/B | `{field, before, after}` — 変更フィールドのみ |
| `itinerary_fields_before` / `itinerary_fields_after` | A/B | Trip 全体 itinerary 件数（通常同一） |
| `conflicts[]` | A/B | machine-readable conflict codes（将来 i18n 向け） |
| `updated_itinerary_id` | B のみ | confirm 成功時の対象 itinerary ID |

`inserted_*` 系フィールドは使いません（update のため）。
