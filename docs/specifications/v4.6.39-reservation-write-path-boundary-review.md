## 1. Current state — Reservation write path

### read-only vs write（非対称）

| Command | 層 | 備考 |
|---|---|---|
| `reservation list` | `services::reservation_list` → CLI JSON / `print_reservation_list` | trip 時 `ReservationWithContext` |
| `reservation show` | `services::reservation_show` → JSON / `print_reservation_detail_with_context` | context 付き |
| `reservation add` | handler → `reservation.rs` | **未 service 化** |
| `reservation update` | handler → `reservation.rs` | **未 service 化** — clear フラグ解釈は **handler** |
| `reservation delete` | handler → `reservation.rs` | **未 service 化** |

Expense v4.6.33 前 / Note v4.6.37 前と同型。**read-only のみ service 化、write は module 直結**。

### write path フロー（確認済み — `main.rs`）

#### Add

```text
ReservationAction::Add { itinerary, reservation_type, provider, ... }
  handler:
    add_reservation(conn, itinerary, ...)           → reservation_id
    println!("Reservation を追加しました (ID: {id})")
    get_reservation(conn, id)
    print_reservation_detail(&conn, &reservation)   ← adapter（内部 get_itinerary_item）
```

#### Update

```text
ReservationAction::Update { id, ..., clear_* flags }
  handler:
    clear フラグ → Option<Option<&str>> 変換（confirmation / site_url / remark / start_at / end_at）
    update_reservation(conn, id, ...)
    println!("Reservation を更新しました (ID: {id})")
    get_reservation(conn, id)
    print_reservation_detail(&conn, &reservation)   ← adapter
```

#### Delete

```text
ReservationAction::Delete { id }
  handler:
    get_reservation(conn, id)                       ← 削除前スナップショット
    delete_reservation(conn, id)
    println!("Reservation を削除しました (ID: {id})")
    println provider line                           ← provider_name のみ
```

### `reservation.rs` 内の write 関連関数

| 関数 | 責務 |
|---|---|
| `validate_reservation_type` / `validate_provider_name` | 入力正規化 |
| `add_reservation` | `get_itinerary_item` + validation + INSERT |
| `update_reservation` | 部分更新、clear 対応、`get_reservation` 内包 |
| `delete_reservation` | 存在確認 + DELETE |
| `get_reservation` | ID 取得 |
| `import_reservation_v3` | trip import 経路（別 write path） |
| `delete_reservations_for_itinerary` / `_for_trip` | cascade |
| `print_reservation_detail(&conn, ...)` | **adapter** — itinerary context 取得 → `_with_context` 委譲 |
| `print_reservation_detail_with_context` | human 表示本体 |

**repository 層は存在しない** — SQL は `reservation.rs` 内に直書き。

## 2. 責務分布 — handler / reservation.rs / service / DTO / display

### 層別インベントリ

| 責務 | 所在 | 評価 |
|---|---|---|
| CLI 引数（clap） | `ReservationAction` | ○ |
| update clear フラグ解釈 | **handler**（`Option<Option<&str>>` 組み立て） | △ service 化時に **Params へ移動候補** |
| reservation_type / provider validation | `add_reservation` / `update_reservation` | ○ mutation 近傍 |
| itinerary 存在確認 | `add_reservation` 内 `get_itinerary_item` | ○ |
| 永続化 | `add_reservation` / `update_reservation` / `delete_reservation` | ○ |
| 成功メッセージ | handler `println!` | ○ |
| 成功後 detail | handler → `print_reservation_detail(&conn, ...)` | △ read show の `_with_context` と非対称 |
| JSON 出力（write） | **なし** | — |
| JSON 出力（read） | `Reservation` 直接 serialize | ○ wire DTO なし |

### handler に mutation が寄りすぎているか

| 観点 | 判定 |
|---|---|
| mutation ロジック | **handler にはない** — `add_reservation` 等に集約 |
| update clear 解釈 | **handler にある** — 5 フィールド × clear フラグ（**service 化時の主な移動対象**） |
| add/delete | **薄い配線** — Note add/delete と同程度 |
| Expense update と比較 | Expense は `parse_*` が service 内；Reservation は **clear 解釈が handler** |

**結論:** add/delete は Note と同型の薄 handler。update は **clear フラグ解釈が handler に残る**点で Expense/Note より重い。

## 3. itinerary_id validation 境界

### Itinerary 配下 entity としての一貫性

```text
Trip
  └── ItineraryItem
        └── Reservation（itinerary_id — application-level 参照、DB FK なし）
        └── Expense（itinerary_id — 同型）
```

| 観点 | 挙動 |
|---|---|
| `reservation add` | **`--itinerary` 必須**（clap） |
| `add_reservation` | `get_itinerary_item(conn, itinerary_id)?` |
| `reservation list --itinerary` | `list_reservations_for_itinerary` 内でも `get_itinerary_item` |
| `reservation update/delete` | reservation ID のみ — itinerary 変更不可 |
| cascade | `delete_reservations_for_itinerary` / `_for_trip` |

**判定:** Expense と同型。**Itinerary 配下 entity** として write path でも一貫。DB FK は v4.6.31 §15 と同様 defer。

### list との整合

| list スコープ | write スコープ |
|---|---|
| `--trip` or `--itinerary` | add は **`--itinerary` のみ** |

Expense と同型 — trip 単位は集計・一覧、新規 add は itinerary 固定。

## 4. 表示 context と enriched layer

### read show service（既存）

```rust
ReservationShowServiceResult {
    reservation: Reservation,
    day_number: Option<i64>,
    itinerary_title: Option<String>,
}
```

itinerary から **表示用 context** を取得 — participant/beneficiary のような enrich ではない。

### write path 表示（現状）

| 経路 | 表示関数 | context 取得 |
|---|---|---|
| read show | `print_reservation_detail_with_context` | **service 内** |
| write add/update | `print_reservation_detail(&conn, ...)` | **adapter 内** `get_itinerary_item` |

**非対称:** write は service result を持たず、表示時に conn 経由で context 再取得。

### enriched layer 要否

| 観点 | Expense | Note | Reservation |
|---|---|---|---|
| participant 解決 | あり | なし | **なし** |
| 表示 context | enrich 必須 | 不要 | **day + itinerary title のみ** |
| wire DTO | `ExpenseJson` | `Note` 直接 | **`Reservation` 直接** |
| 専用 EnrichedPart | `ExpenseEnrichedPart` | なし | **不要** — show service result 型で十分 |

**結論:** `ReservationEnrichedPart` 新設は **不採用**。read `reservation_show` と同型の **表示 context 付き result** が write add/update に自然。

## 5. Expense / Note write migration との比較

| 観点 | Expense | Note | Reservation |
|---|---|---|---|
| 親参照 | `itinerary_id` | `owner_type` + `owner_id` | **`itinerary_id`** |
| add 親指定 | `--itinerary` | trip/day/itinerary 多態 | **`--itinerary`** |
| enriched / context | `ExpenseEnrichedPart` | なし | **表示 context のみ** |
| write 表示 | `_from_enriched` | `print_note_detail(&Note)` | **`print_reservation_detail(&conn, ...)` adapter** |
| update 複雑度 | `parse_*` + shared options | title/body のみ | **clear フラグ 5 件** |
| delete snapshot | amount/currency | title | **provider_name** |
| adapter cleanup | W-5 で 3 関数削除 | N-5 N/A | **R-5 で adapter 統一見込み** |
| import 別経路 | `import_expense_v3` | `import_export_notes` | **`import_reservation_v3`** |

### 難度見込み

```text
Note < Reservation < Expense
  Note:         owner 多態のみが追加論点、表示は domain 直接
  Reservation:  itinerary 固定 + update clear フラグ + 表示 context 非対称
  Expense:      enrich + parse_* + shared expense
```

## 6. thin write service 化 — 候補 result 型（plan 用）

### 推奨（Option R-B）

```rust
// 表示 context — read reservation_show と同型
pub struct ReservationDisplayContext {
    pub day_number: Option<i64>,
    pub itinerary_title: Option<String>,
}

// reservation_add
pub struct ReservationAddParams { /* clap mirror — itinerary, type, provider, ... */ }
pub struct ReservationAddServiceResult {
    pub id: i64,
    pub reservation: Reservation,
    pub day_number: Option<i64>,
    pub itinerary_title: Option<String>,
}

// reservation_update
pub struct ReservationUpdateParams {
    pub id: i64,
    // + clear フラグ解釈済みフィールド（handler から移動）
    ...
}
pub struct ReservationUpdateServiceResult {
    pub reservation: Reservation,
    pub day_number: Option<i64>,
    pub itinerary_title: Option<String>,
}

// reservation_delete
pub struct ReservationDeleteServiceResult {
    pub id: i64,
    pub provider_name: String,
}
```

handler 接続イメージ:

```text
add/update 後 → print_reservation_detail_with_context(&result.reservation, result.day_number, ...)
delete 後     → println provider（Note delete title と同型）
```

**`print_reservation_detail(&conn, ...)` は write から除去可能** — read show と同一表示経路へ。

### delete 後に返すもの

| 候補 | 評価 |
|---|---|
| `{ id, provider_name }` | **現行 human と一致** — 推奨 |
| 全文 `Reservation` + context | 可能だが delete 後表示に過剰 |
