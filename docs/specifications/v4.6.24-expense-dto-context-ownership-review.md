# v4.6.24 — Expense DTO context ownership review

v4.6.23 で特定した **ExpenseJson / `expense_to_json` 周辺の helper-side DB context** について、context ownership の方針を documentation-only で整理する review です。

本書でいう **DTO** は新規型名ではなく、DB model ではなく CLI JSON / 将来 GUI・Tauri API へ渡す **出力用 structured data** を指す。現状の `ExpenseJson` はこの意味で **実質的な output DTO** である。

| Item | Status |
|---|---|
| Release | v4.6.24（documentation-only） |
| Scope | `Expense` vs `ExpenseJson` 責務、context ownership 候補比較、推奨方針、reservation show 対比 |
| 実装 | **なし** — DTO 新規実装 / service 変更 / JSON shape 変更は行わない |

関連:

| Version | 内容 |
|---|---|
| v4.6.23 | read-only helper context review（Expense が主 gap） |
| **本書** | **v4.6.24** — expense DTO context ownership |
| v4.6.25（候補） | expense output DTO migration plan / itinerary show aggregate review |

## Status

| 観点 | 判定 |
|---|---|
| expense list/show service boundary | **導入済み**（v4.6.14 / v4.6.21） |
| `ExpenseJson` 生成 | **helper `expense_to_json(&conn, ...)` が担当** |
| service result | `Expense` のみ（enriched context なし） |
| 短期方針 | **現状維持**（CLI 互換 gate） |
| 中期方針 | **GUI / Tauri 前に ownership 決定** |
| reservation show 即時統一 | **避ける**（複雑度が異なる） |
| itinerary show aggregate | **別論点**（本 review 対象外） |

**結論（推奨）:**

```text
ExpenseJson は現在 CLI JSON output DTO として機能している。
Expense 単体では作れず、participant / beneficiary / shared context が必要。
現状の expense_to_json(&conn, ...) は互換性維持のため短期的には許容。
GUI / Tauri API 化前に、enriched context の ownership を service / application 側へ寄せる
migration path（Option C 中心）を別途設計する。
reservation show と同じ形へ即時統一するのは避ける。
itinerary show aggregate は別 review として残す。
```

---

## Background

v4.6.14 / v4.6.21 で expense list/show を service 化した際、gate は次の通りだった。

```text
service は Expense 本体を返す
JSON / human output は既存 helper をそのまま使う
JSON shape / human output は一切変えない
```

その結果、`ExpenseListServiceResult` / `ExpenseShowServiceResult` は thin だが、**CLI 互換の `ExpenseJson` は service の外で組み立てられている**。

v4.6.24 では「弁当箱（ExpenseJson）の中身と持ち主」を整理する。

## Expense と ExpenseJson の責務整理

### `Expense`（domain / DB model）

`src/domain/models.rs` — `expenses` テーブル 1 行:

| フィールド | 由来 |
|---|---|
| `id`, `itinerary_id` | DB |
| `title`, `amount`, `currency` | DB |
| `paid_by_name` | DB（free-text payer） |
| `paid_by_participant_id` | DB（participant FK、optional） |
| `expense_date`, `note` | DB |
| `sort_order`, `created_at`, `updated_at` | DB |

**持たないもの:** beneficiary 行、participant 名、`shared` フラグ。

### `ExpenseJson`（output DTO — 現状 CLI `--json` の shape）

`src/expense.rs` — `Expense` のコピー + enriched fields:

| フィールド | 由来 |
|---|---|
| `Expense` と同型の scalar 群 | `Expense` からコピー |
| `paid_by_participant_name` | **`get_participant(conn, paid_by_participant_id)`** |
| `beneficiaries: Vec<ExpenseBeneficiaryJson>` | **`list_beneficiaries_for_expense` + 各件 `get_participant`** |
| `shared: bool` | **`expense_is_shared(beneficiaries.len())`** — beneficiary が 1 件以上で `true` |

`ExpenseBeneficiaryJson`:

```text
participant_id  ← expense_beneficiaries.participant_id
name            ← participants.name（DB lookup）
sort_order      ← expense_beneficiaries.sort_order
```

### `ExpenseListJson`（list wrapper — CLI JSON only）

```text
trip_id: Option<i64>        ← CLI が ExpenseListTarget から組み立て
itinerary_id: Option<i64>   ← 同上
expenses: Vec<ExpenseJson>  ← 各件 expense_to_json
```

service は `target` + `Vec<Expense>` を返す。wrapper の `trip_id` / `itinerary_id` は **CLI が target から付与**。

### `expense_to_json` が解決する追加 context（v4.6.23 確認 + 漏れチェック）

| 追加 context | 解決方法 | v4.6.23 記載 |
|---|---|---|
| `beneficiaries` 行 | `list_beneficiaries_for_expense(conn, expense.id)` | ✓ |
| beneficiary `name` | `get_participant(conn, participant_id)` × N | ✓ |
| `paid_by_participant_name` | `get_participant(conn, paid_by_participant_id)` | ✓ |
| `shared` | `expense_is_shared(beneficiary_count)` — `count > 0` | ✓ |
| `paid_by_name` | **Expense からコピーのみ**（追加 DB なし） | （v4.6.23 暗黙） |

**漏れなし。** `expense_to_json` の DB アクセスは上記 3 系統（beneficiaries 一覧、payer participant、beneficiary participant 名）に限定される。

### human output helper との関係

| Helper | `conn` 使用 | enriched context |
|---|---|---|
| `print_expense_detail` | ✓ | **内部で `expense_to_json` を呼ぶ** → JSON DTO と同一 enriched データ |
| `print_expense_list` | ✓ | **payer 名のみ** `get_participant`（beneficiaries / shared は list では非表示） |

list human は JSON より薄い。detail human は実質 **ExpenseJson 経由**。

### ExpenseJson の位置づけ（論点）

| 観点 | 現状評価 |
|---|---|
| CLI JSON output DTO か | **はい** — `--json` の正典 shape |
| GUI / Tauri API でもそのまま使えるか | **shape としては使える** — ただし生成が helper + conn 依存 |
| application 層 DTO か | **未分離** — `expense.rs` に同居、Serialize 付き |
| 名前 `ExpenseJson` のままでよいか（短期） | **はい** — 互換 gate のため変更しない |
| 将来の rename 候補 | `ExpenseOutput` / `ExpenseView` / `ExpenseDto` 等は **別 layer 分離時**に検討可（今回は実施しない） |

## 現行フロー（read-only）

### expense list

```text
CLI ExpenseAction::List
  → services::expense_list::list_expenses
      → ExpenseListServiceResult { target, expenses: Vec<Expense> }
  → --json:
      CLI が target → trip_id / itinerary_id
      各 Expense → expense_to_json(&conn, e) → ExpenseListJson → print_json
  → human:
      print_expense_list(&conn, target, &expenses)
```

### expense show

```text
CLI ExpenseAction::Show
  → services::expense_show::show_expense
      → ExpenseShowServiceResult { expense }
  → --json:
      expense_to_json(&conn, &result.expense) → print_json
  → human:
      print_expense_detail(&conn, &result.expense)
```

### ownership 現状マップ

```text
┌─────────────────────────────────────────────────────────┐
│ service layer                                           │
│   ExpenseListServiceResult / ExpenseShowServiceResult   │
│   owns: Expense (domain), target (list only)            │
└───────────────────────────┬─────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│ CLI layer (main.rs)                                       │
│   ExpenseListJson wrapper 組み立て                       │
│   expense_to_json 呼び出し                                │
└───────────────────────────┬─────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│ expense module helpers (expense.rs)                     │
│   expense_to_json      — enriched DTO 生成 + DB lookup    │
│   print_expense_detail — human（expense_to_json 経由）    │
│   print_expense_list   — human（payer lookup のみ）       │
└─────────────────────────────────────────────────────────┘
```

## Context ownership 候補比較

### A. 現状維持 — `expense_to_json(&conn, &expense)` が context を解決

| 観点 | 評価 |
|---|---|
| メリット | JSON / human 互換ゼロリスク。実装済み。service は thin のまま |
| デメリット | GUI / Tauri が `ExpenseJson` を得るには conn + helper 依存。service result が API として不完全 |
| JSON 互換 | **影響なし** |
| GUI 向き | **不向き** — API 境界が helper に残る |
| 実装リスク | **低**（何もしない） |

### B. service result が `ExpenseJson` 相当まで作って返す

| 観点 | 評価 |
|---|---|
| メリット | GUI が service result をそのまま serialize 可能。一覧取得の enriched も一括 |
| デメリット | service が output DTO（Serialize shape）に密結合。`ExpenseJson` が service 層に入るか、duplicate 型が必要 |
| JSON 互換 | **維持可能** — 返却型が `ExpenseJson` そのものなら CLI は `print_json` のみ |
| GUI 向き | **向き** — ただし layer 境界の議論あり（output DTO in service） |
| 実装リスク | **中** — list/show service 拡張、テスト追加。terminal I/O なしは維持可能 |

### C. service result が `Expense` + context parts を返し、CLI が `ExpenseJson` を組み立て

例（概念のみ、今回は実装しない）:

```text
ExpenseShowEnrichedResult {
    expense: Expense,
    paid_by_participant_name: Option<String>,
    beneficiaries: Vec<ExpenseBeneficiaryPart>,  // id + name + sort_order
    shared: bool,
}
CLI: ExpenseJson { ... }  // 既存 shape へマッピング
```

| 観点 | 評価 |
|---|---|
| メリット | enriched context の **ownership が service 側**に移る。`ExpenseJson` は CLI wire format として残せる。reservation show の「context in service」に近い |
| デメリット | service result 型の設計が必要。CLI に薄い mapper が残る。list は N 件分の enrichment コスト |
| JSON 互換 | **維持可能** — CLI mapper が既存 `ExpenseJson` を組み立て |
| GUI 向き | **向き** — GUI は enriched result か mapper 経由で同一 shape を得られる |
| 実装リスク | **中** — service 拡張 + mapper。B より layer 分離はきれい |

### D. application DTO layer を別途作り、CLI / GUI が共用

```text
application/expense_view.rs（概念）
  build_expense_view(conn, expense) → ExpenseView（≈ ExpenseJson 同等）
CLI  → serialize ExpenseView as ExpenseJson
Tauri → ExpenseView を API response に
```

| 観点 | 評価 |
|---|---|
| メリット | ownership が最も明確。CLI / GUI / 将来 API が同じ enrichment を共有 |
| デメリット | 新 layer 導入コスト。既存 `expense.rs` helper との関係整理が必要 |
| JSON 互換 | **維持可能** — wire shape は `ExpenseJson` alias または同一 struct 移動 |
| GUI 向き | **最向き**（長期） |
| 実装リスク | **高** — モジュール配置・命名・段階移行計画が要る |

### 比較サマリ

| Option | JSON 互換 | GUI API | layer 明確さ | 短期コスト | 推奨フェーズ |
|---|---|---|---|---|---|
| A 現状維持 | ✓ | ✗ | 低 | なし | **今（v4.6.24）** |
| B service → ExpenseJson | ✓ | ○ | 中 | 中 | 可能だが DTO-in-service 懸念 |
| C service → parts + CLI map | ✓ | ○ | 高 | 中 | **migration 推奨** |
| D application layer | ✓ | ◎ | 最高 | 高 | 長期 |

## 現時点での推奨方針

### 短期（v4.6.24 〜 GUI 着手前）

```text
Option A — 現状維持
ExpenseJson / expense_to_json は変更しない
expense list/show service result は Expense のみのまま
CLI が expense_to_json / print_* を呼ぶ現行フローを維持
```

**理由:** v4.6.14 / v4.6.21 の互換 gate を壊さない。review のみのフェーズ。

### 中期（GUI / Tauri API 化の直前）

```text
ownership を Option C（第一候補）または Option D（長期）で決定する
判断材料:
  - Tauri command が service を直接呼ぶか、application mapper を挟むか
  - ExpenseJson という名前を wire format として残すか
  - list N 件 enrichment を service 一括にするか
```

### migration path（推奨 — 別 ticket で設計・実装）

**推奨: Option C を段階的に導入**

```text
Phase 1（設計）: v4.6.25 expense output DTO migration plan
  - ExpenseEnrichedParts（仮称）のフィールド定義
  - list/show service result 拡張案
  - CLI mapper: enriched parts → ExpenseJson（shape 不変）
  - テスト方針（JSON golden 不変）

Phase 2（実装・将来）:
  - service が beneficiaries + payer name + shared を返す
  - expense_to_json は mapper へ縮小 or deprecated（内部で parts を呼ぶ）
  - GUI / Tauri は enriched service result または application mapper を利用

Phase 3（任意・長期）:
  - Option D へ昇格（application layer 抽出）
```

**Option B は採用しない理由（現時点）:** service 層に Serialize 付き output DTO を直接置くと、v4.6.x で確立した「service = use case result、JSON wrapper = CLI」の分離と緊張する。

**Option A のまま GUI に進まない理由:** Tauri 初号機の expense list/show は enriched data が必要で、毎回 `expense_to_json(&conn, ...)` を GUI 層で呼ぶのは API 境界として不自然。

## reservation show との対比

### service result の形

| | reservation show | expense show |
|---|---|---|
| domain | `Reservation` | `Expense` |
| display context | `day_number`, `itinerary_title` | **なし**（helper が解決） |
| context の DB 源 | `get_itinerary_item`（1 lookup） | beneficiaries 一覧 + participant × (1 + N) |
| JSON 出力 | `Reservation` のみ | `ExpenseJson`（enriched 全体） |
| human | service context → `print_*_with_context` | `expense_to_json` → `print_expense_detail` |

### 即時統一を避ける理由

```text
reservation show の context は itinerary から 2 フィールド — 軽量な display context
expense show の context は Shared Expense モデル全体 — beneficiaries 配列、participant 名、shared 導出
reservation JSON は domain のみだが、expense JSON は enriched DTO が正典
同じ「service に context を足す」パターンでも、expense は parts の設計が要る
v4.6.20 の reservation show を「薄い context」参考例とし、expense は Option C で別設計する
```
