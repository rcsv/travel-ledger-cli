|---|---|

| write service result（add/update） | **read `reservation_show` と同型フィールド** — `Reservation` + `day_number` + `itinerary_title`（いずれも `Option` where applicable） |
| write service result（delete） | **`{ id, provider_name }`** — enrich / context なし |

**結論（推奨）:**

```text

handler は Params 組み立て、成功 println!、print_reservation_detail_with_context のみ。
R-2 のみ先行リリースは避け、R-2+R-3 を v4.6.41 で一括実装する。
R-5 は Expense W-5 相当の adapter 削除（print_reservation_detail(&conn, ...)）— Note N-5 とは異なり実装 cleanup。
```

## 2. write service result 型の決定

### 2.1 候補比較（add / update）

| 候補 | 内容 | 判定 |
|---|---|---|
| `reservation_id` のみ | `{ id: i64 }` | **不採用** — handler が再 fetch + context 取得が必要 |
| `Reservation` のみ | domain 行のみ | **不採用** — Day / Itinerary 行が human から欠落 |
| **read show 同型フィールド** | `Reservation` + `day_number` + `itinerary_title` | **採用** — read/write 対称、新型最小 |
| 専用 `ReservationEnrichedPart` | enrich 風ラッパー | **不採用** — participant なし、show result で十分 |
| `ReservationJson` 新設 | wire DTO | **不採用** — `Reservation` が serde 済み |

### 2.2 表示 context — optional vs 専用型

read `ReservationShowServiceResult`（既存）:

```rust
pub struct ReservationShowServiceResult {
    pub reservation: Reservation,
    pub day_number: Option<i64>,
    pub itinerary_title: Option<String>,
}
```

itinerary 孤児行では `day_number` / `itinerary_title` が **None**（service unit test 済み）。

**判定:** write add/update result も **同一 optional セマンティクス** を採用。専用 context 型の新設は **不要** — フィールド複製で read show と揃える。

### 2.3 確定型（概念・未実装）

```rust
// services/reservation_add.rs
pub struct ReservationAddParams { /* §4.1 */ }
pub struct ReservationAddServiceResult {
    pub id: i64,
    pub reservation: Reservation,
    pub day_number: Option<i64>,
    pub itinerary_title: Option<String>,
}

// services/reservation_update.rs
pub struct ReservationUpdateParams { /* §4.2 */ }
pub struct ReservationUpdateServiceResult {
    pub reservation: Reservation,
    pub day_number: Option<i64>,
    pub itinerary_title: Option<String>,
}

// services/reservation_delete.rs
pub struct ReservationDeleteParams {
    pub id: i64,
}
pub struct ReservationDeleteServiceResult {
    pub id: i64,
    pub provider_name: String,
}
```

**`id` を add result に含める理由:** 成功メッセージ `Reservation を追加しました (ID: {id})` が handler に残るため。`reservation.id` と同値 — Expense/Note W-1/N-1 と同型。

**update result に top-level `id` を含めない理由:** `result.reservation.id` で十分。handler の `println!` は `result.reservation.id` を使用（Note update と同型）。

**delete の `provider_name`:** 現行 human `Provider: {name}` と一致。`String`（必須 — mutation 前 snapshot）。

### 2.4 delete が display context を含まない理由

v4.6.39 / Note delete と同型。削除後 human は provider 1 行のみ — context over-fetch 不要。

## 3. read / write service の対称性

### 3.1 揃えるもの（必須）

| 観点 | read show | write add/update（目標） |
|---|---|---|
| 粒度 | 1 command = 1 module | **同じ** |
| terminal I/O | なし | **なし** |
| domain 型 | `Reservation` | **同じ** |
| display context | `day_number`, `itinerary_title` | **同じ optional セマンティクス** |
| context 解決 | `get_itinerary_item` | **同一 helper ロジック**（§5.5） |
| wire DTO in service | なし | **なし** |

### 3.2 揃えなくてよいもの（許容）

| 観点 | read | write |
|---|---|---|
| 入力 | `id` or list target | **params struct** |
| add result の `id` 明示 | N/A | **あり**（success line） |
| delete result | N/A | **provider snapshot のみ** |
| 副作用 | なし | **mutation** |

### 3.3 モジュール対応表（実装後）

| Command | Read service | Write service |
|---|---|---|
| list | `reservation_list` | — |
| show | `reservation_show` | — |
| add | — | **`reservation_add`** |
| update | — | **`reservation_update`** |
| delete | — | **`reservation_delete`** |

## 4. service API と params 設計

### 4.1 `ReservationAddParams`（handler → service）

clap `ReservationAction::Add` の mirror。**serde 不要**。

```text
itinerary: i64
reservation_type: String
provider: String
confirmation: Option<String>
site_url: Option<String>
remark: Option<String>
start_at: Option<String>
end_at: Option<String>
```

service 内 orchestration:

```text
add_reservation(conn, itinerary, ...)
get_reservation(conn, id)
load_display_context(conn, reservation.itinerary_id)   // §5.5
→ ReservationAddServiceResult { id, reservation, day_number, itinerary_title }
```

**handler から `add_reservation` / `get_reservation` / `print_reservation_detail(&conn, ...)` を除去。**

### 4.2 `ReservationUpdateParams`

clap `ReservationAction::Update` の mirror — **clear フラグをそのまま含める**（handler は解釈しない）。

```text
id: i64
reservation_type: Option<String>
provider: Option<String>
confirmation: Option<String>
site_url: Option<String>
remark: Option<String>
start_at: Option<String>
end_at: Option<String>
clear_confirmation: bool
clear_site_url: bool
clear_remark: bool
clear_start_at: bool
clear_end_at: bool
```

service 内で現行 handler と **同一** の `Option<Option<&str>>` 変換を実行:

```text
confirmation_update = if clear_confirmation { Some(None) }
                    else { confirmation.as_ref().map(|v| Some(v.as_str())) }
// site_url, remark, start_at, end_at も同型
update_reservation(conn, id, ..., confirmation_update, ...)
get_reservation(conn, id)
load_display_context(conn, reservation.itinerary_id)
→ ReservationUpdateServiceResult { reservation, day_number, itinerary_title }
```

**handler から clear 解釈ブロック（main.rs L680–704 相当）を完全除去。**

### 4.3 `ReservationDeleteParams`

```text
id: i64
```

service 内:

```text
get_reservation(conn, id)           // snapshot
delete_reservation(conn, id)
→ { id, provider_name: reservation.provider_name }
```

### 4.4 `add_reservation` / `update_reservation` / `delete_reservation` の配置

**Phase R-2〜R-5:** `reservation.rs` に残置。service は orchestration のみ。

```text
services/reservation_add::add_reservation
  → reservation::add_reservation
  → reservation::get_reservation
  → load_display_context (§5.5)
```

repository 化は v4.7.x defer。

## 5. 表示 context と adapter 寿命

### 5.1 `print_reservation_detail_with_context` — 正式 human 経路

add/update/show 共通:

```rust
print_reservation_detail_with_context(
    &result.reservation,
    result.day_number,
    result.itinerary_title.as_deref(),

```

read show は既に使用。**write add/update も R-3 後はこれのみ。**

### 5.2 `print_reservation_detail(&conn, &Reservation)` — adapter 寿命

| Phase | write path | adapter |
|---|---|---|
| R-0〜R-1 | handler → adapter | **使用中** |
| **R-3** | handler → `_with_context` | write から **不使用** |
| **R-5** | — | **関数削除**（production 呼び出しゼロ後） |

現状 production 呼び出し: **write add/update のみ**（`main.rs` L624, L718）。show は `_with_context` 使用済み。

**R-5 判定: Expense W-5 相当の実装 cleanup** — Note N-5 docs-only とは **異なる**。

### 5.3 adapter 削除 gate（R-5）

```text
□ write handler が print_reservation_detail(&conn, ...) を呼ばない（R-3 完了）
□ grep 上 production 参照ゼロ
□ reservation_cli human 不変
□ 削除後 make check PASS
```

### 5.4 `load_display_context` — 共有 helper（R-2 推奨）

read `reservation_show` と write service で **同一ロジック** を使うため、R-2 実装時に以下いずれか:

| 案 | 内容 | 推奨 |
|---|---|---|
| A | `reservation.rs` に `load_reservation_display_context(conn, itinerary_id)` を追加し show / write から呼ぶ | **推奨** — 小さな DRY |
| B | write service が `show_reservation(conn, id)` を mutation 後に呼ぶ | 非推奨 — show は read use case、mutation 後 ID 取得で間接 |
| C | ロジック複製 | 非推奨 |

```rust
// reservation.rs（概念）
pub(crate) fn load_reservation_display_context(
    conn: &Connection,
    itinerary_id: i64,
) -> (Option<i64>, Option<String>) {
    // reservation_show と同一: get_itinerary_item → (Some(day), Some(title)) or (None, None)
}
```

**R-1 では関数追加しない** — R-2 実装タスクとして plan に記載。

## 6. handler 最終形（Phase R-3 目標）

```text
ReservationAction::Add { ... } => {
    let result = services::reservation_add::add_reservation(&conn, ReservationAddParams { ... })?;
    println!("Reservation を追加しました (ID: {})", result.id);
    reservation::print_reservation_detail_with_context(
        &result.reservation,
        result.day_number,
        result.itinerary_title.as_deref(),
    );
}

ReservationAction::Update { ... } => {
    let result = services::reservation_update::update_reservation(&conn, ReservationUpdateParams { ... })?;
    println!("Reservation を更新しました (ID: {})", result.reservation.id);
    reservation::print_reservation_detail_with_context(
        &result.reservation,
        result.day_number,
        result.itinerary_title.as_deref(),
    );
}

ReservationAction::Delete { id } => {
    let result = services::reservation_delete::delete_reservation(&conn, ReservationDeleteParams { id })?;
    println!("Reservation を削除しました (ID: {})", result.id);
    println!("  Provider: {}", result.provider_name);
}
```

### handler に残す責務（確定）

```text
clap 分解済み引数 → Params struct 組み立て（clear フラグは raw のまま渡す）
成功 / 削除メッセージ println!
print_reservation_detail_with_context（add/update）
delete provider 表示
```

### handler から除去する責務

```text
clear フラグ → Option<Option<&str>> 解釈
add_reservation / update_reservation / delete_reservation 直接呼び出し
get_reservation（add/update/delete すべて）
print_reservation_detail(&conn, ...) adapter
```
