## 1. Definition（`reorder_itinerary` の意味）

`reorder_itinerary` は、**同一 Day 内**の Itinerary の並び順（表示順 / 実行順）を変更する structural operation である。

- **対象**: Itinerary の **順序**（`sort_order` / Day 内並び）
- **非対象**: `start_time`（時間ラベル）。reorder の主対象ではない
- **原則**: Sequence-first ordering を維持（`sort_order` を primary sequence とする）

期待するユーザー体験（CLI / 将来 GUI 共通）:

- 「A を B の前に移動」など、**順序の意図**を表明できる
- 変更前後の順序が dry-run preview で確認できる

---

## 2. Scope split（same-day / cross-day を混ぜない）

本書では reorder を以下に分割して扱う。

### 2.1 same-day reorder（今回の初期 scope）

- **同じ day_number** の中で Itinerary の順序を変更する
- 変更は `itinerary_items.sort_order` の更新（複数行）として実装される可能性が高い

### 2.2 cross-day move（defer）

Day を跨ぐ移動は reorder と day move が混ざるため **別 intent** として扱うのが安全。

- cross-day move（A を Day 2 → Day 3へ）
- day move + reorder（移動先 Day のどこに入れるか）

v4.7.x では **defer**（planning を v4.7.37+ へ）。

### 2.3 sort_order direct update（defer / internal）

`sort_order` を数値として直接編集するのは、ユーザー向け intent ではなく **内部実装**（または管理者向け debug）扱いが望ましい。

---

## 3. Reorder method candidates（候補比較）

reorder は TOCTOU / baseline mismatch の影響を受けやすいため、**「どの並びを前提にするか」**を request に含めるほど安全になる。

ここでは候補を比較する。

### 3.1 `before_itinerary_id` / `after_itinerary_id`

- **Pros**: UI drag-and-drop の意図に近い。payload が小さい
- **Cons**: anchor が消えた/移動した/別 Day になった場合の挙動が難しい
- **Ambiguity**: ID ベースなら曖昧さは低いが、ID 解決に DB が必要
- **TOCTOU risk**: 中（anchor 周辺が変わると意図が変わり得る）

### 3.2 `position: before/after + anchor selector`

anchor を selector（title 等）で指定する案。

- **Pros**: fragment author が ID を知らなくても作れる
- **Cons**: selector が曖昧になりやすく blocking が増える
- **TOCTOU risk**: 高（title 変更 / 重複など）

### 3.3 explicit `expected_order`（推奨候補）

「現在の並び（baseline）として何を期待しているか」を明示し、gate で一致確認する。

- **Pros**: baseline mismatch を確実に検出できる。TOCTOU に強い
- **Cons**: payload が大きい。authoring が難しい（GUI 支援前提）
- **No-op**: expected と after が同じなら blocking にできる

### 3.4 explicit new sequence list（target Day の全 ID リスト）

after の並びを全て列挙する。

- **Pros**: 意図が最大に明確。適用結果が一意
- **Cons**: payload 最大。欠落/重複の validation が必要
- **TOCTOU risk**: 低（baseline mismatch を別途検出できれば）

### 3.5 direct `sort_order`

- **Pros**: 実装は単純に見える
- **Cons**: sparse ordering の維持/再採番の問題を user payload に押し付ける
- **TOCTOU**: sort_order の衝突や並行更新に弱い

## 4. Baseline / conflict policy（blocking 方針）

reorder は structural であり、baseline mismatch を **blocking** とするのが安全。

### 4.1 `expected_order` を必須にするか

**結論（提案）**: v4.7.x の最初の導入では **expected_order を必須**にする。

理由:

- drag-and-drop の再現には、適用対象の order 前提が必要
- reorder は no-op / 別意図に化けやすく、silent apply を避けたい

### 4.2 expected の表現

**候補**:

- **IDs の配列**（推奨）: DB ID が唯一性を担保
- day_number + sort_order + title の組: ID を知らない authoring 向けだが曖昧・TOCTOU に弱い

**結論（提案）**: gate で itinerary を解決できる前提なら **ID 配列**が最も安全。

### 4.3 Blocking rules（抜粋）

- anchor が曖昧 / 未解決 → blocking
- current order が expected と不一致 → blocking
- no-op reorder → blocking
- target と anchor が同一 → blocking
- target / anchor が別 Day → blocking

## 5. Preview shape（dry-run）

dry-run は **before/after の順序差分**を明示し、変更行が追える必要がある。

提案:

```json
{
  "preview": {
    "action": "reorder_itinerary",
    "day_number": 2,
    "itinerary_order_changes": [
      {
        "itinerary_id": 10,
        "title": "A",
        "before_sort_order": 1000,
        "after_sort_order": 1000
      },
      {
        "itinerary_id": 30,
        "title": "C",
        "before_sort_order": 3000,
        "after_sort_order": 2000
      },
      {
        "itinerary_id": 20,
        "title": "B",
        "before_sort_order": 2000,
        "after_sort_order": 3000
      }
    ]
  }
}
```

検討事項:

- **changed rows** だけに絞るか、Day 内全件を列挙するか
  - 初期は **Day 内全件**を列挙（GUI なしでも確認しやすい）
- `unchanged_rows` を明示するか
  - 省略しても差分で読み取れるが、全件列挙なら不要

## 6. DB mutation policy（confirm は未実装だが方針を明文化）

same-day reorder の confirm は、通常 **複数行 UPDATE** が必要になる。

方針案:

- **transaction 必須**
- UPDATE 対象は **同一 Day の itinerary_items のみ**
- **row count 検証**（更新対象の件数が期待と一致）
- **TOCTOU 再検証**（expected_order の一致を再確認）
- child entities（Expense / Estimate / Reservation / Note / Venue）は変更しない
- sparse `sort_order` は維持（`SORT_ORDER_STEP` を活用）し、必要時のみ再採番
  - 再採番は safety/UX の観点から **別フェーズ**で導入検討してもよい

---

## 7. Safety boundary（blocking 方針）

以下は blocking とする（v4.7.x 初期案）。

- Trip target / Day target
- unresolved target
- ambiguous target
- required_decisions 非空
- adoption_hints.conflicts 非空
- target itinerary not found
- anchor itinerary not found
- target / anchor が別 Day
- current order baseline mismatch（expected_order と不一致）
- no-op reorder
- duplicate IDs in expected_order
- missing target in expected_order
- missing anchor in expected_order
- cross-day move attempt
- invalid sort_order proposal（内部計算が破綻する入力）

---

## 8. Relationship with UI drag-and-drop（分離）

UI の drag-and-drop は以下を守ることで CLI fragment と整合できる。

- same-day reorder と cross-day move を混ぜない
- preview で before/after を明確にする
- transaction と TOCTOU を必須にする
- `sort_order` 再採番方針を文書化する（いつ/どこで起き得るか）
