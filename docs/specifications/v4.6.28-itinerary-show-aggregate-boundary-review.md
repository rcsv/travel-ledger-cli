# v4.6.28 — Itinerary show aggregate boundary review

v4.6.10 で `itinerary show` の itinerary 本体は service 化済みだが、**human output の Reservations 表示**は handler 側で追加取得している。本書はこの **aggregate boundary** を documentation-only で整理する。

| Item | Status |
|---|---|
| Release | v4.6.28（documentation-only） |
| Scope | `itinerary show` の service / CLI / human aggregate / JSON 分離 |
| 実装 | **なし** |

関連:

| Version | 内容 |
|---|---|
| v4.6.10 | itinerary show service boundary（`ItineraryItem` のみ） |
| v4.6.13 / v4.6.20 | reservation list / show service |
| v4.6.14 / v4.6.26 | expense list / show enriched service |
| v4.6.12 / v4.6.18 | note list / show service |
| v4.6.27 | expense migration follow-up（adapter 残置妥当） |
| **本書** | **v4.6.28** — itinerary show aggregate review |
| v4.6.29（候補） | aggregate migration plan |

---

## Status

| 観点 | 判定 |
|---|---|
| itinerary 本体 | **service 化済み**（`ItineraryShowServiceResult { item }`） |
| Reservations human aggregate | **handler 側追加取得** |
| JSON output | **`ItineraryItem` のみ** — Reservations なし |
| Notes / Expenses in show | **なし** — 別 list/show 導線 |
| 短期方針 | **現状維持（許容）** |
| 中期 | aggregate ownership 方針整理 → migration plan |

**結論（推奨）:**

```text
itinerary show の itinerary 本体取得は service boundary 化済み。
一方、human-only Reservations 表示は handler 側追加取得として残っている。
これは現時点では互換性維持のため許容。
ただし、GUI / Tauri API 化前に aggregate ownership の方針整理が必要。
Reservations だけを service result に含めるか、Notes / Expenses も含む aggregate detail として扱うかは別途判断する。
既存 itinerary show JSON shape は変更しない。
短期は現状維持。次に進むなら aggregate migration plan を documentation-only で作る。
```

---

## 1. 現行 `itinerary show` の責務整理

### フロー（v4.6.28 確認）

```text
ItineraryAction::Show { id, json }
  → services::itinerary_show::show_itinerary(&conn, id)
      → ItineraryShowServiceResult { item: ItineraryItem }

  --json:
    print_json(&result.item)                    // ItineraryItem のみ

  human:
    print_itinerary_detail(&result.item)        // itinerary 本体
    list_reservations_for_itinerary(&conn, id)  // handler 追加取得
    if non-empty:
      println Reservations (N):
      per reservation: id, type, provider, confirmation, period
```

### 責務分担表

| 層 | 担当 | 内容 |
|---|---|---|
| **service** | `itinerary_show` | `ItineraryItem` 取得、`Itinerary not found: <id>` |
| **CLI JSON** | handler | `ItineraryItem` serialize のみ |
| **CLI human — 本体** | handler | `print_itinerary_detail(&result.item)` |
| **CLI human — aggregate** | handler | `list_reservations_for_itinerary` + `println!` ループ |
| **service** | — | Reservations **含まない** |

### JSON に含まれる / 含まれないもの

| データ | `--json` | human |
|---|---|---|
| `ItineraryItem` 本体 | ✓ | ✓（`print_itinerary_detail`） |
| Reservations | ✗ | ✓（handler 追加） |
| Notes | ✗ | ✗ |
| Expenses | ✗ | ✗ |

### Reservations は human-only aggregate か

| 観点 | 判定 |
|---|---|
| 現行 CLI JSON | **human-only** — JSON 正典は `ItineraryItem` のみ |
| 将来 GUI | itinerary detail 画面で reservation サマリを見せたい可能性は **ある** |
| 将来 JSON/API | 既存 `--json` を壊さず、**別 DTO / 別 endpoint** が自然 |
| `reservation list --itinerary` | **別導線で service 化済み** — 完全な list は既に存在 |

Reservations 表示は **CLI human の便宜機能**として始まり、**正式な aggregate DTO ではない**。

## 2. JSON 互換性

| gate | 状態 |
|---|---|
| `itinerary show --json` shape | **`ItineraryItem` のみ** |
| Reservations in JSON | **含まれない**（v4.6.10 から維持） |
| v4.6.28 | **JSON shape 変更なし** |
| 将来 aggregate | 新フィールド追加 or 別 use case — **既存 JSON 不変が前提** |

migration path（概念）:

```text
既存: itinerary show --json → ItineraryItem
将来案1: 同一 command、JSON は不変、human のみ service aggregate
将来案2: itinerary detail aggregate API（GUI 専用）— CLI JSON とは別
将来案3: --json に wrapper 追加は breaking — 非推奨
```

## 3. human output の確認

### itinerary detail（`print_itinerary_detail`）

`src/itinerary.rs` — conn 不要、`ItineraryItem` のみ:

```text
ID, 旅行 ID, 日目, 時刻, 並び順, 所要時間, 移動時間, タイトル,
Category（あれば）, 場所, メモ, 作成日時, 更新日時
```

### Reservations セクション（handler 内 `println!`）

| 観点 | 挙動 |
|---|---|
| 取得 | `list_reservations_for_itinerary(&conn, id)` |
| 前提 | itinerary 存在確認（関数内で `get_itinerary_item`） |
| 並び順 | **`ORDER BY id ASC`**（`list_reservations_where`） |
| 空の場合 | **セクション全体を出力しない**（見出しもなし） |
| 表示フィールド | `[id] type provider confirmation` + optional period 行 |
| helper | `fmt_optional_text`, `format_period` — **表示のみ** |
| terminal I/O | **handler 内 `println!`** — service 外（v4.6.11 方針どおり human は CLI） |

### handler 追加取得の範囲

```text
Reservations のみ。
Notes / Expenses の追加取得・表示は itinerary show human にない。
```

## 4. aggregate boundary 候補比較

### A. 現状維持 — handler human-only Reservations

| 観点 | 評価 |
|---|---|
| メリット | JSON/human 互換ゼロリスク。実装済み。service は thin |
| デメリット | GUI が itinerary+reservations を一括取得できない。handler に DB 呼び出し残存 |
| JSON 互換 | **影響なし** |
| human | **影響なし** |
| GUI 向き | **不向き** — 2 回呼び出し（show + reservation list）が必要 |
| 実装リスク | **なし** |

### B. service result に Reservations を含める

```text
ItineraryShowServiceResult {
    item: ItineraryItem,
    reservations: Vec<Reservation>,
}
```

| 観点 | 評価 |
|---|---|
| メリット | GUI/Tauri が 1 use case で human 同等データ取得。expense enriched と同型の方向 |
| デメリット | JSON は依然 `ItineraryItem` のみなら CLI で二重構造。reservation list との重複 |
| JSON 互換 | **維持可能** — CLI は `result.item` のみ serialize |
| human | handler を `result.reservations` 表示に変更可能（出力不変 gate） |
| GUI 向き | **向き** |
| 実装リスク | **低〜中** — service 拡張 + handler 整理 |

### C. 別 aggregate detail service

```text
itinerary_detail / itinerary_aggregate_show
  → { item, reservations, notes?, expenses? }
```

| 観点 | 評価 |
|---|---|
| メリット | `itinerary show` を薄く保てる。aggregate 意図が明確 |
| デメリット | 新 command / 新 service。CLI surface 増加 |
| JSON 互換 | `itinerary show` **不変** — 新 API は別 shape |
| human | 既存 show を変えずに新 command 追加可能 |
| GUI 向き | **最向き**（長期） |
| 実装リスク | **中** — 設計・命名・導線整理が要る |

### D. GUI / Tauri 用別 DTO（CLI JSON 互換維持）

| 観点 | 評価 |
|---|---|
| メリット | CLI `--json` 正典を守りつつ GUI に rich DTO |
| デメリット | DTO 層増加。B/C と組み合わせが多い |
| JSON 互換 | **維持** |
| human | 変更なし可能 |
| GUI 向き | **最向き** |
| 実装リスク | **中〜高** — v4.7.x 判断 |

### 比較サマリ

| Option | 短期 | JSON | GUI | 推奨フェーズ |
|---|---|---|---|---|
| A 現状維持 | ✓ | ✓ | △ | **今（v4.6.28）** |
| B service + reservations | | ✓ | ○ | migration plan 後 |
| C 別 aggregate service | | ✓ | ◎ | 長期 |
| D GUI 別 DTO | | ✓ | ◎ | v4.7.x |

## 5. Reservations / Notes / Expenses の扱い

### 現行 read-only 導線（service 化済み）

```text
Itinerary show
  → Note list / Note show        [v4.6.12 / v4.6.18]
  → Reservation list / show      [v4.6.13 / v4.6.20]
  → Expense list / show          [v4.6.14 / v4.6.26 enriched]
```

### itinerary show human の aggregate 範囲

| 子 entity | itinerary show human | 専用 list/show |
|---|---|---|
| Reservations | **サマリ表示あり** | service 済み |
| Notes | **なし** | service 済み |
| Expenses | **なし** | service 済み（enriched） |

### 設計論点

```text
itinerary detail に Notes / Expenses / Reservations を全部詰め込む必要はない。
GUI 初号機は個別 service の組み合わせで十分な可能性が高い。
itinerary show の Reservations サマリは「CLI human の便宜」に留まっている。
aggregate detail に拡張するなら、何を1画面にまとめるかを UX で決めてから。
```

**推奨:** 全面 aggregate より **Reservations のみ service 化（Option B）** または **GUI 専用 aggregate（C/D）** を段階的に検討。Notes/Expenses の一括 inclusion は **慎重に defer**。

## 6. service boundary 方針との整合

| 方針 | itinerary show 現状 |
|---|---|
| service は structured result | ✓ `ItineraryItem` |
| service に terminal I/O なし | ✓（`rg` 確認済み） |
| JSON wrapper は CLI | ✓ `--json` は handler |
| human-only は CLI | ✓ Reservations `println!` は handler |
| JSON shape 不変 | ✓ |
| human output 不変 | ✓（本 review では変更しない） |
| write 未着手 | ✓ `ItineraryAction::Update` 等は未 service 化 |
| Tauri 未着手 | ✓ |

**Reservations handler 取得は「human-only 表示を CLI に残す」方針の延長** — v4.6.11 以来の意図的判断と一致。

expense migration（v4.6.26）とは対照的:

| | expense show | itinerary show |
|---|---|---|
| JSON enriched | `ExpenseJson`（CLI mapper） | `ItineraryItem` のみ |
| human enriched | service `ExpenseEnrichedPart` | handler Reservations fetch |
| 理由 | JSON 正典が enriched | JSON は domain のみ、human のみ aggregate |

## 7. handler-side Reservations 取得の扱い（結論）

| 項目 | 内容 |
|---|---|
| 所在 | `main.rs` `ItineraryAction::Show` human branch |
| 関数 | `reservation::list_reservations_for_itinerary(&conn, id)` |
| 性質 | **human-only aggregate 補完** |
| 現時点 | **許容** — v4.6.10 からの互換維持 |
| GUI 前 | ownership 決定が必要（B / C / D） |
| 即 service 移行 | **非推奨** — migration plan なしでの変更は gate リスク |
