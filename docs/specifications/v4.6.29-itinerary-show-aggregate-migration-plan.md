## Status

| 観点 | 判定 |
|---|---|
| 短期 | **Option A** 維持 — handler human-only aggregate **許容** |
| 中期有力候補 | **Option D**（GUI/Tauri 専用 aggregate DTO）+ **Option C**（別 aggregate detail service） |
| 慎重扱い | **Option B** — `ItineraryShowServiceResult` への reservations 単独追加 |
| JSON gate | `itinerary show --json` = **`ItineraryItem` のみ、不変** |
| human gate | Reservations セクション含め **出力テキスト不変** |
| GUI 初号機 | **個別 read-only service の組み合わせで十分**な可能性が高い |

**結論（推奨）:**

```text
短期は Option A を維持する。handler human-only Reservations は許容。
itinerary show --json の ItineraryItem shape は変更しない。
Reservations だけを急いで ItineraryShowServiceResult に混ぜない。
GUI / Tauri 初号機は個別 service 組み合わせを優先する。
aggregate detail が必要になった段階で Option C または Option D を検討する。
CLI human の内部実装寄せ（Phase 4）は任意・後回し可。
```

## 1. Current state

### read-only フロー（現状・確認済み）

```text
ItineraryAction::Show { id, json }
  → services::itinerary_show::show_itinerary(&conn, id)
      → ItineraryShowServiceResult { item: ItineraryItem }   // reservations なし

  --json:
    print_json(&result.item)                                 // ItineraryItem のみ

  human:
    print_itinerary_detail(&result.item)                     // itinerary 本体
    list_reservations_for_itinerary(&conn, id)               // handler 追加取得
    if non-empty → Reservations セクション（println!）
    if empty     → セクション出力なし
    並び順: ORDER BY id ASC
```

### 現状サマリ

| 項目 | 状態 |
|---|---|
| `ItineraryShowServiceResult` | **`item` のみ** |
| handler 追加取得 | **Reservations のみ**（human output） |
| `itinerary show --json` | **`ItineraryItem` のみ** |
| Notes / Expenses in human show | **表示なし** |
| 子 entity read-only 導線 | **別 list/show service 済み** |

### 既存 read-only ナビゲーション

```text
Itinerary show
  → Note list / Note show        [v4.6.12 / v4.6.18]
  → Reservation list / show      [v4.6.13 / v4.6.20]
  → Expense list / show          [v4.6.14 / v4.6.26 enriched]
```

### expense migration との対比（参考）

| | expense show (v4.6.26) | itinerary show (現状) |
|---|---|---|
| JSON 正典 | enriched → `ExpenseJson`（CLI mapper） | `ItineraryItem` のみ（domain） |
| human enriched | service `ExpenseEnrichedPart` | handler Reservations fetch |
| 理由 | JSON 正典が enriched shape | JSON は domain のみ；human のみ partial aggregate |

itinerary は **JSON が domain のみ** のため、expense と同型に「service enriched → CLI mapper」へ寄せる必然性は **低い**。aggregate は **別 use case / 別 DTO** として切り出す方が自然。

## 2. Boundary problem

### 何が曖昧か

```text
handler が human-only aggregate（itinerary + reservations サマリ）を組み立てている
service result（ItineraryItem のみ）と human output の情報量が一致していない
itinerary show が「単体表示」なのか「Itinerary detail aggregate」なのか名前と責務がずれている
GUI / Tauri API 化時に ownership（誰が reservations を取るか）が曖昧になる
Reservations だけ先に ItineraryShowServiceResult に混ぜると Notes / Expenses との整合性が崩れる
```

### 急いで直す不具合ではない

| 観点 | 判定 |
|---|---|
| CLI 互換 | **問題なし** — v4.6.10 から意図的 |
| human 表示 | **動作している** — handler aggregate は許容範囲 |
| service purity | **改善余地あり** — 中長期課題 |
| 緊急度 | **低** — migration plan 後に段階実施 |

### Reservations だけ service に混ぜない理由

```text
Notes / Expenses は itinerary show human に出ていないが、同じ親子関係を持つ
Reservations だけ ItineraryShowServiceResult に入れると「show = 本体 + reservations」という中途半端な aggregate になる
reservation list service と責務が重複する（list は既に itinerary フィルタ対応）
将来 Notes / Expenses を入れるべきかという未解決論点が残る
itinerary show の名前・CLI 契約（JSON = ItineraryItem）と aggregate の境界がさらに曖昧になる
```

**推奨:** Option B は **最後の手段または Phase 4 の内部実装寄せ** に留め、第一選択にしない。

## 3. Migration options

### 比較マトリクス

| Option | CLI互換 | JSON互換 | human互換 | GUI/Tauri | service責務 | Notes/Expenses整合 | 実装リスク |
|---|---|---|---|---|---|---|---|
| **A** 現状維持 | ◎ | ◎ | ◎ | △ | △（handler に DB） | ○（未混在） | **なし** |
| **B** show result + reservations | ○ | ○（item のみ serialize） | ○（可能） | ○ | △（中途半端 aggregate） | **×**（偏り） | 低〜中 |
| **C** 別 aggregate detail service | ◎ | ◎（show 不変） | ◎ | ◎ | ◎ | ◎ | **中** |
| **D** GUI/Tauri 専用 DTO | ◎ | ◎ | ◎ | **◎** | ◎ | ◎ | 中〜高 |

---

### Option A — handler human-only aggregate 維持

```text
ItineraryShowServiceResult { item }
human: handler が list_reservations_for_itinerary + println!
```

| 観点 | 内容 |
|---|---|
| CLI互換性 | **完全維持** |
| JSON互換性 | `ItineraryItem` のみ — **不変** |
| human output互換性 | **不変** |
| GUI/Tauri適性 | itinerary show + reservation list の **2 呼び出し** |
| service責務 | itinerary 本体のみ — **明確だが human と不一致** |
| Notes/Expenses整合 | aggregate に混ぜない — **整合** |
| 実装リスク | **なし** |

**フェーズ:** Phase 0〜1（短期デフォルト）

---

### Option B — `ItineraryShowServiceResult` に reservations 追加

```text
ItineraryShowServiceResult {
    item: ItineraryItem,
    reservations: Vec<Reservation>,
}
CLI --json: 依然 result.item のみ
CLI human: result.reservations から既存 println! と同等表示
```

| 観点 | 内容 |
|---|---|
| CLI互換性 | JSON path は `item` のみなら維持可能 |
| JSON互換性 | **維持可能** — serialize 対象を `item` に限定 |
| human output互換性 | **維持可能** — 表示ロジックを service result 参照に変更 |
| GUI/Tauri適性 | 1 use case で human 同等 — **向き** |
| service責務 | **曖昧化** — show が partial aggregate になる |
| Notes/Expenses整合 | **崩れやすい** — reservations だけ特別扱い |
| 実装リスク | 低〜中（ただし設計負債リスク **中**） |

**判定:** **慎重扱い** — expense enriched とは異なり、JSON 正典が domain のみのため show service を膨らませる動機が弱い。

---

### Option C — 別 aggregate detail service

```text
itinerary_detail_show / itinerary_aggregate_show（名称は実装時に確定）
  → ItineraryDetailServiceResult {
        item: ItineraryItem,
        reservations: Vec<Reservation>,   // 初期はこれのみでも可
        notes: Option<Vec<Note>>,           // 将来・任意
        expenses: Option<Vec<ExpenseEnrichedPart>>,  // 将来・任意
    }
既存 itinerary show は変更しない
```

| 観点 | 内容 |
|---|---|
| CLI互換性 | **既存 show 不変** — 新 command は追加のみ |
| JSON互換性 | `itinerary show --json` **不変** — 新 API は別 shape |
| human output互換性 | 既存 show **不変** — 新 command または GUI のみ |
| GUI/Tauri適性 | **最向き** — aggregate 意図が明示的 |
| service責務 | **明確** — single item show vs detail aggregate |
| Notes/Expenses整合 | **段階追加可能** — 全 inclusion は UX 判断後 |
| 実装リスク | **中** — naming / CLI surface / 重複 list との関係整理 |

**フェーズ:** Phase 3 第一候補（GUI 要求確定後）

---

### Option D — GUI / Tauri 専用 aggregate DTO

```text
CLI:
  itinerary show --json → ItineraryItem（不変）
  human → 現状または Phase 4 で内部寄せ

GUI / Tauri:
  ItineraryDetailDto（または同等）— wire shape は GUI 要件で設計
  内部で itinerary_show + reservation_list + note_list + expense_list を
    1 command にまとめるか、application layer で compose するかは実装時判断
```

| 観点 | 内容 |
|---|---|
| CLI互換性 | **完全維持** — CLI DTO と GUI DTO 分離 |
| JSON互換性 | CLI `--json` **不変** |
| human output互換性 | **不変**（CLI human は CLI 層のまま） |
| GUI/Tauri適性 | **最向き** — API 契約を GUI に最適化 |
| service責務 | service は structured result、**DTO 組み立ては application/CLI** |
| Notes/Expenses整合 | **柔軟** — 画面ごとに必要な子 entity のみ |
| 実装リスク | **中〜高** — DTO 層・Tauri command 設計（v4.7.x） |

**フェーズ:** Phase 2〜3 — GUI 初号機は **compose（個別 service）** から開始し、必要なら DTO 化

### Option B / C / D の採用順序（推奨）

```text
1. Option A（維持）                    ← 今
2. Option D の軽量版（compose のみ）    ← GUI 初号機（個別 service 呼び出し）
3. Option C または Option D（DTO 固定） ← 画面要件が固まったら
4. Option B（任意）                    ← CLI human 内部実装寄せのみ検討（Phase 4）
```

Option B を Phase 3 より前に採用しない。

### Phase 1 — CLI 現状維持（v4.6.30 以降のデフォルト）

| 項目 | 内容 |
|---|---|
| 目的 | Option A 継続 |
| `itinerary show --json` | **`ItineraryItem` shape 維持** |
| human output | **変更しない** |
| handler Reservations | **そのまま許容** |
| 実装 | **なしでも可** — 他 track（expense write、migration runner）と並行 |

**gate:**

```text
□ itinerary show --json が ItineraryItem のみ
□ human Reservations セクションの有無・並び・フィールド不変
□ JSON golden 変更なし
```

---

### Phase 2 — GUI / Tauri 初号機：個別 service compose

| 項目 | 内容 |
|---|---|
| 目的 | aggregate DTO を **急がず**、read-only service を組み合わせる |
| パターン | `itinerary_show` + `reservation_list`（itinerary filter）+ 必要なら `note_list` / `expense_list` |
| aggregate DTO | **作らない**（初号機） |
| CLI | **触らない** |

**判断:**

```text
多くの itinerary detail 画面は、親 itinerary + 子リストの lazy load で足りる
1 round-trip にこだわる前に UX とキャッシュ方針を決める
```

**v4.6.x でやるべきか:** GUI 未着手なら **Phase 2 は計画のみ**。実装は v4.7.x。

---

### Phase 3 — GUI 要求確定後：Option C または Option D

| 項目 | 内容 |
|---|---|
| トリガー | itinerary detail 画面で **1 回の fetch** が必要、または子 entity の **固定セット** が決まった |
| Option C | 新 aggregate detail service +（任意）新 CLI command |
| Option D | `ItineraryDetailDto` を Tauri / application 層に定義 |
| Notes / Expenses inclusion | **UX で判断** — 全部入りを前提にしない |
| CLI `itinerary show` | **JSON / human 不変** |

**aggregate detail に含めるかの判断タイミング:**

```text
□ GUI で itinerary detail が 1 画面に何を表示するか wireframe 確定
□ 子 entity のうち常時表示 vs タブ / 遅延読み込みを決定
□ reservation list service の返却 shape で足りるか検証
□ note / expense list が itinerary スコープで十分か検証
□ 上記で不足が証明されたときだけ aggregate detail を設計
```

## 5. JSON / human 互換性の守り方

### 不変 gate（全 Phase 共通）

```text
itinerary show --json の root object は ItineraryItem のまま
Reservations / Notes / Expenses を itinerary show JSON に含めない（既存契約）
serde / field name / optional field の既存挙動を維持
JSON export golden は変更なし
human: print_itinerary_detail の出力不変
human: Reservations セクションの有無・見出し・フィールド・並び（id ASC）不変
```

### 将来 aggregate を追加する場合

```text
既存: itinerary show --json → ItineraryItem
新規: itinerary detail API / Tauri command → 別 DTO（wrapper でも可）
非推奨: itinerary show --json の root を wrapper 化（breaking）
推奨: --include-reservations 等の flag 追加も v4.7.x 以降の明示的判断
```

## 6. GUI / Tauri 初号機の推奨スタンス

```text
itinerary_show(id)           → ヘッダ / 本体
reservation_list(itinerary_id) → 予約一覧（既存 service）
note_list(itinerary_id)      → 必要なら
expense_list(itinerary_id)   → 必要なら（enriched）

→ 4 本の read-only service で初号機は構成可能
→ aggregate DTO は「性能・UX・型安全」が必要になってから
```

CLI human の Reservations サマリは **簡易プレビュー** に相当。GUI では reservation list の方が **正** のデータ源。
