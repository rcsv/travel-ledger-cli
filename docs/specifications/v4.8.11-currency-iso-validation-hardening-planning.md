## 2. Validation policy（提案）

### 2.1 Registry scope

**採用方針（v4.8.11 planning）:**

| Policy | Decision |
|---|---|
| ISO 4217 alpha-3 official code | **registry 照合で許容** — active + withdrawn（historical）code を含む |
| 非存在 code（`JPN`, `ABC`, `ZZZ`） | **reject**（write path） |
| 現行「英字3文字」 | registry 導入後は **第1段階（format）+ 第2段階（registry）** の二段構成 |
| lowercase input | **uppercase normalize してから registry 照合**（現行維持） |
| 空文字 / whitespace-only | **reject**（現行維持） |
| null（Receipt optional） | amount なしなら currency も null 可；amount ありなら currency 必須（現行ペア validation 維持） |
| currency-only update（Estimate） | amount 必須ルール維持 — ISO check は normalize 後 |

### 2.2 Pseudo / special codes

| Code | ISO status | Write path（提案） | Rationale |
|---|---|---|---|
| `JPY`, `USD`, `EUR` 等 | active | **allow** | 主 travel currency |
| `XXX` | ISO 4217 "no currency" | **reject** | 旅行費用ドメインに意味なし（[roadmap memo](../future-roadmap-planning-memo.md)） |
| `XTS` | testing code | **reject** | 本番 travel ledger に不要 |
| `XAU`, `XAG`, `XPT`, `XPD` | precious metals | **reject**（v4.8.12+） | expense/estimate 対象外；将来 commodity 拡張時に再検討 |
| `XBC`, `XBA` 等 | bond market units | **reject** | 同上 |
| Obsolete（例: `HRK`, `LTL`） | withdrawn | **allow** | 既存旅行記録・historical receipt 互換 |

**拒否リスト（denylist）** を registry とは別に持つ — ISO に存在しても travel money として不適切な code を明示排除。

### 2.3 Historical / non-current code

- **allow** — withdrawn code は registry に残る ISO dataset で判定
- read/export/import format gate では **warning のみ**（validate-export phase）
- 新規 create でも historical code を許す（ユーザーが旧通貨記録を入力するケース）

### 2.4 Minor units との境界

| Layer | Scope | v4.8.11 |
|---|---|---|
| `validate_currency_code` | alpha-3 registry + denylist | **in scope（planning）** |
| `currency_minor_unit_digits` | decimal places for amount parse | **out of scope** — 現行 hardcoded 表維持 |
| amount decimal validation | `parse_amount_for_currency` | 現行 minor digits 表に依存 — **変更しない** |

**分離理由:** minor unit を ISO-backed にすると、既知 code の decimal 扱いが変わり **amount parse 互換** に影響しうる（例: 表に無い code が 2→0 桁に変わる）。currency code validation とは独立 phase とする。

### 2.5 Validator API（実装 preview — 本 release では未実装）

```text
// 提案 internal API（v4.8.12）
enum CurrencyValidationMode {
    FormatOnly,           // 現行互換（import transitional）
    IsoStrict,            // registry + denylist（write path）
    IsoPermissiveRead,    // format only — read path 用（実質 no-op）
}

fn validate_currency_code(code: &str, mode: CurrencyValidationMode) -> Result<String>
```

正本関数は `src/money.rs` に集約。Expense / Estimate / Receipt / Fragment apply はすべてここを経由。

Registry 実装候補: embedded static list または `iso_currency` crate — v4.8.12 で選定。

## 3. Backward compatibility policy

### 3.1 原則

```text
Breaking:     既存 DB の read/list/export/export-md が unknown currency で失敗すること
Non-breaking: 新規 write path で unknown/non-ISO を reject すること
Non-breaking: validate-export が unknown currency を warning すること
Breaking:     trip import が legacy export を一括 reject すること（採用しない）
```

### 3.2 既存 DB に unknown currency がある場合

| Operation | Policy |
|---|---|
| `expense list/show`, `estimate list/show` | **表示可能** — registry 検証しない |
| `trip export` | **値そのまま出力** |
| `trip export-md` | **値そのまま表示** |
| `expense update`（currency 変更時） | **IsoStrict** — 新値は registry 必須 |
| `expense update`（currency 変更なし） | 既存値触らない — OK |

**DB migration / cleanup:** 行わない。unknown currency は data quality 問題として validate-export / 将来 doctor で surface。

### 3.3 Import policy

| Phase | `trip import` | Rationale |
|---|---|---|
| v4.8.15 以前（現行） | format validation のみ | round-trip 維持 |
| v4.8.15（提案） | **default: format only**（現行同等） | 破壊的変更回避 |
| v4.8.15+ optional | `--strict-currency` flag（将来検討） | opt-in ISO strict import |

Import 時に unknown ISO code があっても **import 自体は成功** — quality signal は validate-export へ委譲。

### 3.4 validate-export policy

| Condition | Severity（提案） |
|---|---|
| empty / non-alpha-3 format | **error**（現行維持） |
| alpha-3 format OK, registry unknown | **warning** `currency code is not a known ISO 4217 code: {code}` |
| alpha-3 format OK, denylist（XXX 等） | **warning** `currency code is not allowed for travel expenses: {code}` |
| valid ISO active/withdrawn | pass |

validate-export は **quality gate** — import blocker ではない。

### 3.5 CLI create/update strict 化

| Command | v4.8.13 以降（提案） |
|---|---|
| `expense add/update --currency` | IsoStrict |
| `estimate add/update --currency` | IsoStrict |
| `receipt` metadata update | IsoStrict（currency 指定時） |
| Fragment apply write intents | v4.8.14 — gate + confirm で IsoStrict |

### 3.6 破壊的 migration 回避

- DB schema 変更なし
- export JSON schema 変更なし
- 既存 golden（JPY のみ）影響なし
- `XXX` unit test のみ behavioral change — test update は v4.8.12+ implementation release

## 4. Proposal Fragment apply との関係

### 4.1 Blocking phase

| Phase | ISO strict（提案） |
|---|---|
| Fragment validation（JSON schema / field presence） | **しない** — presence のみ |
| Dry-run simulate（parse） | **する** — v4.8.14 |
| Confirm transaction | **する** — parse 再実行で自然にカバー |

gate 入口で fragment JSON 自体に `currency: "XXX"` があっても、simulate parse 段階で blocking — DB write 前。

### 4.2 Structured error 接続

**v4.8.11 判定: 新規 code 不要（初期）**

| 項目 | 提案 |
|---|---|
| code | **`APPLY_FIELD_INVALID`**（既存 registry） |
| kind | `validation` |
| phase | `simulate` / `confirm` |
| field_path | `candidate_content.currency`（hint — v4.8.7 unstable grammar 維持） |
| message | `currency code is not a valid ISO 4217 code` 等（unstable） |
| legacy `errors[]` | 従来 message も **維持**（dedupe 対象） |

`APPLY_CURRENCY_INVALID` は **optional** — v4.8.14 実装時に専用分岐需要があれば additive 追加可（forward-compatible）。

### 4.3 `schema_version: 2` への影響

- **なし** — structured_errors[] additive のみ
- registry code 追加は v4.8.5 model の forward-compatible unknown code 方針に従う

### 4.4 Legacy `errors[]` 互換

```text
errors[]: "candidate_content.currency が不正です: ..."
structured_errors[]: { code: "APPLY_FIELD_INVALID", field_path: "candidate_content.currency", ... }
```

v4.8.10 limited exposure 方針 — wiring 後も `errors[]` は残す。
