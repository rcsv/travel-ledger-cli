# v4.6.31 — Expense write path migration plan

[v4.6.30](v4.6.30-expense-write-path-boundary-review.md) で棚卸しした expense **write path** を、**Option W-B（thin write services）** へ段階移行するための documentation-only migration plan です。

| Item | Status |
|---|---|
| Release | v4.6.31（documentation-only migration plan） |
| Scope | add/update/delete の service 化 Phase、責務境界、adapter 方針、golden gate |
| 実装 | **なし** — 本 release は **Phase W-0（計画書）のみ** |

関連:

| Version | 内容 |
|---|---|
| v4.6.25〜v4.6.27 | read-only expense DTO migration（完了） |
| v4.6.30 | write path boundary review |
| **本書** | **v4.6.31** — write migration plan（Phase W-0） |
| v4.6.32（候補） | Phase W-2+W-3 実装 |

---

## Status

| 観点 | 判定 |
|---|---|
| 基本方針 | **Option W-B** — thin write services（1 command = 1 service module） |
| 短期 | handler + `expense.rs` **維持**（Phase W-1） |
| repository 層 | **本 plan 対象外** — 別ストリーム（v4.7.x 候補） |
| parse_* | **service 内へ移動**（Phase W-2〜W-3、挙動不変） |
| add/update 後 enrich | **service result に含める**（二重 fetch 回避） |
| delete 後表示 | **amount のみ維持** — enrich 不要 |
| `print_expense_detail` | Phase W-3 まで残置 → W-5 で write 呼び出し除去 |
| Note / Reservation | **同型パターンを参照** — 一括同時 service 化はしない |
| JSON golden | write は JSON 出力なし — **全 Phase で不変** |

**結論（推奨）:**

```text
Expense write path は thin write services（expense_add / expense_update / expense_delete）へ段階移行する。
最小単位は read path と同型の「1 command = 1 service module」。
handler は clap 引数・成功メッセージ・human 表示のみ残す。
mutation・participant 解決・トランザクションは service が orchestrate し、初期実装は既存 add_expense 等を内部呼び出し。
add/update の service result には ExpenseEnrichedPart を含め、handler は print_expense_detail_from_enriched を使う。
delete は削除前スナップショット（amount/currency）のみ返す — enrich しない。
repository 層抽出は本 plan に含めない。
v4.6.31 は plan のみ。実装は v4.6.32 候補（Phase W-2+W-3 同一リリース推奨）。
```

## 1. 現行構造の再整理（write path）

### read vs write（v4.6.30 確認）

```text
READ（完了）:
  expense list  → services::expense_list   → ExpenseListServiceResult { enriched[] }
  expense show  → services::expense_show    → ExpenseShowServiceResult { enriched }

WRITE（未 service 化）:
  expense add    → parse_* → add_expense → get_expense → print_expense_detail
  expense update → get_expense → parse_* → update_expense → get_expense → print_expense_detail
  expense delete → get_expense → delete_expense → println amount
```

### 責務ブロック

| ブロック | 現所在 | 将来 |
|---|---|---|
| clap 引数 | command 定義 / handler | **handler（維持）** |
| `parse_expense_shared_options_*` | `expense.rs`、handler から呼び出し | **service 内** |
| `add_expense` / `update_expense` / `delete_expense` | `expense.rs` | **service から呼び出し**（初期） |
| `enrich_expense` | `expense.rs`、read service 使用 | **write service も使用** |
| 成功メッセージ `println!` | handler | **handler（維持）** |
| 成功後 detail | `print_expense_detail` adapter | **`print_expense_detail_from_enriched`** |
| delete 後表示 | handler `println!` amount | **handler（維持）** |
| `import_expense_v3` | `expense.rs` | **別 Phase**（W-6 以降） |

## 2. thin write service の最小単位

### 推奨: 1 command = 1 service module

read path（`expense_list` / `expense_show`）と **同型**:

| Service module | Command | ServiceResult（推奨形） |
|---|---|---|
| `services::expense_add` | `expense add` | `ExpenseAddServiceResult { id: i64, expense: ExpenseEnrichedPart }` |
| `services::expense_update` | `expense update` | `ExpenseUpdateServiceResult { expense: ExpenseEnrichedPart }` |
| `services::expense_delete` | `expense delete` | `ExpenseDeleteServiceResult { id, amount, currency }` |

**1 つの mega `expense_write` service にまとめない** — テスト・責務・GUI API の粒度が read と揃う。

### delete が enrich を含まない理由

```text
現行 delete 後表示は amount のみ
削除後行が存在しないため enrich の材料がない
ExpenseEnrichedPart は不要 — 削除前 domain スナップショットで十分
```

### add/update が enrich を含む理由

```text
現行は mutation 後 get_expense + print_expense_detail（内部 enrich）で二重取得
service が mutation 直後に enrich_expense すれば handler の get + adapter を省略可能
human 出力は print_expense_detail_from_enriched で read show と同一ロジック
```

## 3. handler と service の境界

### handler に残すもの

```text
clap からの生引数受け取り
成功 / 削除確認メッセージの println!（日本語固定文言 — 既存互換）
print_expense_detail_from_enriched（add/update 後）
delete 後の amount 表示 println!
terminal I/O 全般
```

### service に移すもの

```text
parse_expense_shared_options_for_add / _for_update の呼び出しと結果の利用
itinerary / participant / amount / currency / date の検証オーケストーション
add_expense / update_expense / delete_expense の呼び出し
mutation 後の enrich_expense（add/update のみ）
structured ServiceResult の組み立て
service 内 unit test（mutation + enrich、not found、validation error）
```

### service に入れないもの

```text
println! / eprintln!（v4.6.11 方針）
ExpenseJson 組み立て（write は JSON 出力なし）
clap 型への依存
```

### 将来の handler 形（概念・Phase W-3）

```text
ExpenseAction::Add { ... } => {
    let result = services::expense_add::add_expense(&conn, ExpenseAddParams { ... })?;
    println!("Expense を追加しました (ID: {})", result.id);
    print_expense_detail_from_enriched(&result.expense)?;
}

ExpenseAction::Update { ... } => {
    let result = services::expense_update::update_expense(&conn, ExpenseUpdateParams { ... })?;
    println!("Expense を更新しました (ID: {})", result.expense.expense.id);
    print_expense_detail_from_enriched(&result.expense)?;
}

ExpenseAction::Delete { id } => {
    let result = services::expense_delete::delete_expense(&conn, id)?;
    println!("Expense を削除しました (ID: {})", result.id);
    println!("  Amount: {}", format_amount_display(result.amount, &result.currency));
}
```

`ExpenseAddParams` / `ExpenseUpdateParams` は clap 引数の mirror（`String` / `Option`）— **wire DTO ではない**。

## 4. `parse_expense_shared_options_*` の扱い

### 現状

handler が `parse_expense_shared_options_for_add/update` を呼び、`ExpenseSharedOptions` を `add_expense` / `update_expense` に渡す。

update では handler が先に `get_expense` して `itinerary_id` を得ている。

### 推奨（Phase W-2〜W-3）

| 項目 | 方針 |
|---|---|
| 関数本体 | **当面 `expense.rs` に残置** — service から呼ぶ |
| 呼び出し元 | **handler → service** に移動 |
| update の itinerary_id | **service 内**で `get_expense` してから parse（handler の二重 get 解消） |
| リネーム | 任意 — `build_expense_shared_options_for_add` 等（挙動不変なら Phase W-5） |

**Phase W-4（任意）:** `ExpenseAddParams` 構築時に parse を inline 化するか、input builder に集約。benefit が小さいため **W-2+W-3 と同リリースで十分**。

### update handler の二重 `get_expense`

現行 `update_expense` 内でも `get_expense` がある。service 化時に **1 回の load + mutate** に整理可能 — ただし **human / error message 不変**が gate。

## 5. `add_expense` / `update_expense` / `delete_expense` の配置

### Phase W-2〜W-5（推奨）

```text
expense.rs:
  add_expense / update_expense / delete_expense  ← 実装の正（SQL・transaction）
  enrich_expense / parse_*                       ← shared helpers

services/expense_add.rs:
  pub fn add_expense(conn, params) -> Result<ExpenseAddServiceResult>
    → parse_* → add_expense(...) → enrich_expense(...)

services/expense_update.rs:
  pub fn update_expense(conn, params) -> Result<ExpenseUpdateServiceResult>
    → get_expense → parse_* → update_expense(...) → enrich_expense(...)

services/expense_delete.rs:
  pub fn delete_expense(conn, id) -> Result<ExpenseDeleteServiceResult>
    → get_expense（snapshot）→ delete_expense(...) → return snapshot fields
```

**read migration（v4.6.26）と同型:** service は orchestration + structured result、重いロジックは既存 module 関数を再利用。

### `import_expense_v3`

trip import 経路 — **write service Phase とは別**。共通化するなら W-6 以降で `add_expense` core 抽出を検討。**v4.6.31 では defer**。

### cascade helpers

`delete_expenses_for_itinerary` / `_for_trip` — itinerary/trip delete から呼ばれる。**write service 化の対象外**（内部 API）。

## 6. `print_expense_detail` adapter の寿命

| Phase | 方針 |
|---|---|
| W-0〜W-2 | **現状維持** — write は adapter 使用 |
| W-3 | handler が `print_expense_detail_from_enriched` に切替 — **human 不変** |
| W-4 | read path は既に `_from_enriched` — 変更なし |
| W-5 | write が adapter を呼ばなくなった後、`print_expense_detail` は tests / 外部互換のみ — `#[allow(dead_code)]` 継続可 |
| W-6+ | tests を `enrich + _from_enriched` に寄せたら adapter 削除検討 |

**v4.6.27 判断の延長:** adapter 即削除は **非推奨**。write service 接続（W-3）後に縮小。

### add/update 後の enrich を service に寄せるか

**はい（推奨）** — `ExpenseAddServiceResult` / `ExpenseUpdateServiceResult` に `ExpenseEnrichedPart` を含める。

| 観点 | 内容 |
|---|---|
| メリット | handler から `get_expense` + `print_expense_detail` 除去、read/show と対称 |
| gate | `tests/expense_cli.rs` の add/update 出力 **byte-level 一致** |
| リスク | enrich タイミング（commit 後）— 現行と同じく commit 後 enrich でよい |

## 8. repository 層 — 本 plan に含めるか

**含めない（defer）**

| 理由 | 内容 |
|---|---|
| スコープ | SQL 移動だけでは service boundary の benefit が read と重複しない |
| リスク | expense.rs 全体の repository 化は write + import + cascade を巻き込む |
| タイミング | write service が安定した **v4.7.x** または dedicated refactor |
| 本 plan | `add_expense` 等は **module 内関数のまま** service から呼ぶ |

別ストリーム名（概念）: **expense persistence layer extraction** — v4.6.31 とは独立 milestone。

## 9. Note / Reservation との同型性

### 現状横並び

| Entity | read service | write |
|---|---|---|
| Note | ✓ | module 直結 |
| Reservation | ✓ | module 直結 |
| Expense | ✓ enriched | module 直結 + enrich adapter |

### 方針

```text
Expense write を先に thin service 化する（structured options・beneficiary が最複雑）
Note / Reservation write service 化は別 milestone — パターンは踏襲
v4.6.32 で Expense だけ先行しても、Note/Reservation との「同時リリース」は要求しない
将来 note_add service 等は expense_add と同型の Result パターンを使う
```

**同型性は architectural pattern として維持、実装タイミングは entity ごと。**
