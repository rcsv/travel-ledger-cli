## Executive summary（結論）

| 質問 | 判定 |
|---|---|
| P-6k `reorder_itinerary` は完了か | **はい** — same-day dry-run + confirm の計画 scope は達成 |
| P-6l `move_itinerary` は完了か | **はい** — cross-day dry-run + confirm の計画 scope は達成 |
| GUI drag-and-drop の backend 基盤として妥当か | **はい** — itinerary ID 配列 + baseline 契約は再利用可能（推奨事項あり） |
| P0 / P1 問題 | **なし** — データ整合性を脅かす既知欠陥は未検出 |
| v4.7.40+ で必須の follow-up | **なし（blocking）** — 任意改善は P2/P3 |
| 次に Proposal を続けるか / cleanup を挟むか | **Proposal Fragment 継続を推奨** — `apply.rs` 整理は P2 で段階的に |

設計境界はコード・テスト・リリースノートで一貫している:

```text
reorder_itinerary = same-day ordering change（day target）
move_itinerary    = cross-day ownership + ordering change（itinerary target）
```

意図的な重複はなく、same-day を `move_itinerary` で処理したり、cross-day を `reorder_itinerary` で処理したりする経路はない。

## 1. Design boundary verification

### 1.1 Intent split

| 観点 | `reorder_itinerary` | `move_itinerary` |
|---|---|---|
| target | `day` | `itinerary` |
| Day 変更 | なし | あり（`day_id` + `day`） |
| order 配列 | `expected_order` / `after_order` | 4 配列（source / destination × expected / after） |
| same-day | **必須** | **拒否**（`reorder_itinerary` を案内） |
| cross-day | **拒否**（title ヒント付き） | **必須**（`from_day != to_day`） |
| Trip 拘束 | `--trip` + Day 内 items のみ | `--trip` + `find_day_by_trip_and_day_number` |

### 1.2 Overlap / duplication

- **same-day move:** `move_itinerary` は `from_day == to_day` で明示 reject（dry-run / confirm 共通、`compute_move_itinerary_plan`）。
- **cross-day reorder:** `reorder_itinerary` は Day 内 `list_itinerary_items_for_day` に拘束。title が別 Day に存在する場合は cross-day ヒント付きで reject（`resolve_reorder_order_in_day`）。
- **ID ベース cross-day 誤用:** reorder で別 Day の itinerary ID を列挙しても、その ID は当該 Day の items に存在せず not-found で reject。

 accidental overlap は見当たらない。

## 2. GUI drag-and-drop readiness

### 2.1 Contract assessment

**Same-day reorder（推奨 GUI 形状）**

```json
{
  "expected_order": [101, 102, 103],
  "after_order": [101, 103, 102]
}
```

**Cross-day move（推奨 GUI 形状）**

```json
{
  "from_day": 1,
  "to_day": 2,
  "expected_source_order": [101, 102],
  "expected_destination_order": [201, 202],
  "after_source_order": [101],
  "after_destination_order": [201, 102, 202]
}
```

| 観点 | 判定 | コメント |
|---|---|---|
| itinerary ID 配列 | **適切** | ID selector confirm テストあり。GUI は **常に ID** を使うべき |
| title / sort_order fallback | **CLI 向け** | 数値は id 優先 → sort_order fallback。GUI では不要な複雑性 |
| ambiguous numeric reject | **十分** | id と sort_order 両一致時は reject。専用テストあり |
| full-order baseline | **適切** | stale UI を `expected_*` mismatch で検出。payload は大きいが安全 |
| Tauri / API 再利用 | **可能** | Fragment JSON をそのまま生成可能。薄い DTO ラッパで十分 |

### 2.2 GUI authoring recommendations（契約変更なし）

1. **order 配列は itinerary ID のみ** — title / sort_order selector は CLI・手動編集用と位置づける。
2. **drag 操作ごとに `expected_*` を現行 DB から再取得** — optimistic UI は dry-run gate で検証してから confirm。
3. **same-day drag → `reorder_itinerary`**、**cross-day drag → `move_itinerary`** — intent を UI 層で分岐。
4. **move の target** は移動対象 itinerary を `target.itinerary_reference`（ID 推奨）で指定。現行実装は sort_order 解決も使用するが、GUI は ID 解決済み target を渡せる。

### 2.3 Concerns（非 blocking）

- move target 解決が `itinerary_sort_order` 依存のため、GUI 統合時は target 解決 API の明文化が有用（P2 docs）。
- 4 配列すべて必須のため、GUI は「現在順序スナップショット + 操作後順序」を毎回組み立てる必要がある（意図的な安全設計）。

## 4. Transaction / TOCTOU safety

### 4.1 `reorder_itinerary --confirm`

`execute_confirm_reorder_itinerary`（`src/proposal/apply.rs`）:

1. トランザクション内で Day items 再取得
2. `expected_order` / `after_order` 再解決
3. baseline（expected ≡ 現行順）再検証
4. sparse slot 再計算
5. gate `reorder_preview` と `after_sort_order` 一致確認
6. 変更必要行のみ UPDATE（`sort_order`, `updated_at`）
7. 各行 `changed == 1` を検証

**懸念（P2）:** dry-run（`apply_reorder_itinerary_preview`）と confirm で **同一の plan 関数を共有していない**。move は `compute_move_itinerary_plan` で共有済み。reorder も将来 `compute_reorder_itinerary_plan` 抽出を推奨（dry-run / confirm 乖離リスクの低減）。

### 4.2 `move_itinerary --confirm`

`execute_confirm_move_itinerary`:

1. トランザクション内で `compute_move_itinerary_plan` 再実行
2. `verify_move_plan_matches_preview` で gate preview と完全一致要求
3. 変更計画に含まれる行のみ UPDATE
4. moved row: `day_id`, `day`, `sort_order`, `updated_at`（`WHERE day = from_day` で TOCTOU）
5. 各行 `changed == 1`

**評価:** TOCTOU 境界は十分。dry-run と confirm の sort_order 戦略は **同一関数** で一致。

### 4.3 Partial commit

いずれも `with_transaction` + row count mismatch で rollback。部分 move / 部分 reorder は commit されない。

### 4.4 Forced row-count mismatch test

本番コードにテストフックなし。**意図的に未テスト**。トランザクション内の `changed != 1` 分岐はコードレビューで確認。invasive hook は不要（P3）。

## 5. Mutation boundaries

### 5.1 Confirmed boundaries

| 操作 | 許可更新 |
|---|---|
| same-day reorder | `sort_order`, `updated_at` |
| cross-day move（移動行） | `day_id`, `day`, `sort_order`, `updated_at` |
| cross-day move（順序影響行） | `sort_order`, `updated_at` |

**不変:** title, start_time, category, location, note, Trip, Day, child rows, Venue / Place, schema。

### 5.2 Child entities

移動は `itinerary_items` の Day 所属のみ変更。expense / reservation / note / estimate は `itinerary_id` FK で同一行に残る（設計どおり）。

**テストギャップ（P2）:** move confirm 後に expense が同一 itinerary に残る integration test は未追加。mutation boundary テストは title / day / sort_order / count のみ。

### 5.3 `day` denormalized column

**現行契約:**

```text
day_id = normalized parent FK（days.id）
day    = denormalized day_number（export / legacy 互換）
```

cross-day move は **両方を同時更新**（`day swap`, `itinerary update --day`, insert と同じ write contract）。

| 観点 | 状態 |
|---|---|
| 直接 `i.day` 読み取り | `reservation.rs` 等に残存 |
| export 主経路 | `INNER JOIN days` 経由（`d.day_number`） |
| write 同期 | add / update / swap / move confirm で `day_id`+`day` ペア |
| raw DB テスト | v4.7.39 `query_itinerary_day_columns` で確認 |
| 片方のみ更新する経路 | reorder confirm では触らない。move / update / swap は同期 |

**将来候補（P3 — 緊急ではない）:**

```text
itinerary_items.day の deprecation / removal は、
compatibility・migration・export・direct-read・schema・golden 影響を計画した上で評価する。
現行 feature stream を中断する必要はない。
```

## 6. sort_order strategy

### 6.1 Same-day reorder

- 現行 Day の sort_order スロットを昇順ソートし、`after_order` に割当。
- 件数不変。global renumber なし。
- `normalize_day_sort_order`（`itinerary.rs`）は **別操作** — apply では自動呼び出ししない。

### 6.2 Cross-day move

- **source:** 残存 item に既存スロットの先頭 N 個を割当。
- **destination:** midpoint → head（`first - SORT_ORDER_STEP`）→ tail（`last + SORT_ORDER_STEP`）。安全 slot なしは blocking。
- global compaction なし。

### 6.3 Safety assessment

| 観点 | 判定 |
|---|---|
| midpoint overflow | **実用上安全** — `prev + 1 < next` ガード。極端な i64 近傍は理論的 P3 |
| head ≤ 0 | **reject** 明示 |
| tail overflow | 理論的 P3（通常 sparse 1000 刻みでは未到達） |
| 繰り返し挿入で gap 枯渇 | **想定内** — blocking が正しい。自動 normalize はしない |
| エラーメッセージ | 日本語で原因区分（no gap / collision / head）あり |

### 6.4 Normalization policy（将来 spec 候補）

**推奨:** 明示的な **user-visible normalization operation**（例: Day 内 `normalize_day_sort_order` の Fragment 化、または admin CLI）を **別 intent / 別 milestone** で設計する。

- apply path での **自動** global renumber は現行方針と矛盾するため採用しない。
- v4.7.40+ で normalization **planning spec** を書く価値はある（P2 optional）が、P-6k/l 完了の blocker ではない。
