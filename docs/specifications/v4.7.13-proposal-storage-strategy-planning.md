## Status（結論サマリ）

| 観点 | v4.7.13 判定 |
|---|---|
| すぐ DB storage に進むべきか | **いいえ** — まだ file-only / validation-only を続ける |
| 保存対象 | **Envelope も Fragment も** — ただし schema v8 Trip とは別系統 |
| storage model | **分離** — 同一 Trip テーブルに混ぜない；型・command・validator を分ける |
| 最小 storage 単位 | **Envelope 1 件 / Fragment 1 件** — 「小さい Trip」にしない |
| 状態管理 | **必要（軽量）** — pending / adopted / rejected / expired（soft） |
| AI 一時提案 vs ユーザー採用判断 | **metadata + 明示 save + gate** で区別；自動 DB 取り込みしない |
| file-only 許容範囲 | **P-5 gate spec 具体化まで継続**；optional file inbox は DB 前の中間案 |
| materialize / apply | **v4.7.13 scope 外** — P-5 planning → P-6 implementation |
| CLI / GUI / public schema | **責務境界を固定** — 本書 §8 |

**固定ルール（v4.7.0〜v4.7.12 から継承）:**

```text
schema v8 Trip          = trip validate-export の対象（採用済み正式データ）
Trip Proposal Envelope  = proposal validate / show / inspect の対象
Proposal Fragment       = fragment validate の対象（既存 Trip への部分提案）
materialize / apply     = 今回対象外 — 人間 gate 通過後のみ正式 Trip へ
```

## 2. 保存対象 — Envelope と Fragment

### 2.1 両方とも保存対象にする

| 種別 | 保存するか | 理由 |
|---|---|---|
| **Trip Proposal Envelope** | **はい** | 旅行全体の未採用案 — 再 inspect・採用判断・監査に必要 |
| **Proposal Fragment** | **はい** | 既存 Trip への部分提案 — target / conflict / intent の履歴に必要 |
| **schema v8 Trip** | 既存 DB | Proposal とは **別責務** — 混ぜない |

Envelope と Fragment は **どちらも「採用前候補」** だが、lifecycle と gate が異なる（materialize vs apply）。保存はするが **同じ意味のエンティティではない**。

### 2.2 採用しない方針

| 方針 | 却下理由 |
|---|---|
| Fragment を「小さい Trip」として `trips` に保存 | schema v8 境界崩れ；validate-export / import 混同 |
| Envelope を Trip export JSON の一部として保存 | 未採用案が正式データ化される |
| AI 出力を validation なしで DB itinerary に直接 insert | v4.7.3 / v4.7.4 で却下済み — 候補と正式データの混在 |
| Proposal 専用 JSON を `trip import` 経路で取り込む | import は schema v8 Trip 専用を維持 |

## 3. 既存 Trip / Day / Itinerary との紐づけ

### 3.1 Envelope

```text
保存時点:
  既存 Trip への FK は不要（新規 Trip 生成前の案）

materialize 成功後:
  新規 schema v8 Trip が生成される
  Envelope 側は adopted + 生成 Trip への参照（監査用）を持てる

DB 設計候補:
  proposal_envelopes テーブル
  materialized_trip_id — nullable、採用後のみ設定
  trips テーブルへの FK は「採用結果の参照」であり、Envelope が Trip そのものではない
```

### 3.2 Fragment

```text
保存時点:
  target が本質 — trip / day / itinerary への「提案先」

file 段階:
  trip_reference / day_reference — 文字列・番号参照（soft link）
  unresolved_target_hints — 未確定を許容

DB 段階（将来）:
  target_trip_id — 既存 Trip への参照（提案対象）
  target_day_number / target_itinerary_id — nullable
  FK は「提案先」であり、Fragment body を itinerary 行として格納しない

採用しない:
  Fragment.candidate_content を itineraries テーブルに先行 insert
```

### 3.3 紐づけの強度

| 段階 | Envelope | Fragment |
|---|---|---|
| file-only | 紐づけなし | trip_reference 等（soft） |
| file inbox | manifest に trip_id 任意 | manifest + target 解決状態 |
| DB storage | materialized_trip_id（採用後） | target_trip_id + optional day/itinerary |

**原則:** 採用前は **弱い参照（soft link）** を許容。gate 通過時に強い参照へ解決する。

## 4. 同一 storage model か、分離か

### 4.1 推奨: **分離（別 type / 別 table / 別 command）**

| 観点 | Envelope | Fragment |
|---|---|---|
| 粒度 | 旅行全体案 | 部分提案 |
| gate | materialize → **新規** Trip | apply → **既存** Trip |
| target | なし（destination / date_policy） | **必須**（target_type / references） |
| validator | `proposal validate` | `fragment validate` |
| show / inspect | 実装済み（Envelope） | 未着手（Fragment） |

```text
推奨:
  proposal_envelopes（または file: envelope/*.json）
  proposal_fragments（または file: fragments/*.json）
  共通: metadata / status / source / valid_until の shape は揃えてもよい
  共通テーブル 1 本に type 列だけ — 将来検討可だが初期は分離を推奨
```

### 4.2 分離を推奨する理由

1. validator / CLI / docs が既に分離されている（v4.7.9〜v4.7.11）
2. gate 操作（materialize vs apply）が根本的に異なる
3. 早期に polymorphic `proposals` 1 テーブルにすると、Trip 混同バグの温床
4. Fragment の target / conflict フィールドは Envelope に不要

### 4.3 共通化してよい層（内部のみ）

```text
OK（内部 helper / 共有 struct 候補）:
  metadata: created_at, source, provider, valid_until
  status enum
  validation report の warnings 集約

NG（公開境界の共通化）:
  trip validate-export への統合
  trip import への統合
  単一 `proposal validate` で Fragment も受ける
```

## 5. 状態管理 — pending / adopted / rejected / expired

### 5.1 推奨: **軽量な lifecycle state を持つ**

| State | 意味 | blocking |
|---|---|---|
| **pending** | 保存済み・未判断 | — |
| **adopted** | gate 通過・正式 Trip へ反映済み | materialize / apply 完了後 |
| **rejected** | ユーザーが却下 | 再採用は明示操作 |
| **expired** | valid_until 超過等（**soft**） | 自動削除しない；warning のみ |

```text
expired:
  v4.7.4 / v4.7.9 と同様 — 警告・inspect 時の目印
  自動 import block や自動削除はしない
  pending から expired への遷移は non-blocking
```

### 5.2 状態を持たない案（却下）

| 案 | 却下理由 |
|---|---|
| 状態なし（ファイルの有無のみ） | list / 採用履歴 / GUI inbox が組めない |
| Trip.status に統合 | schema v8 Trip と Proposal lifecycle の混同 |

### 5.3 遷移（概念）

```text
[AI / file 入力] → pending（明示 save 後）
pending → adopted   （人間 gate + materialize / apply 成功）
pending → rejected  （明示却下）
pending → expired   （時間経過 — soft、判断は人間）
adopted → （原則不可逆）— 正式 Trip は別管理
```

## 6. AI authoring とユーザー採用判断の区別

### 6.1 三層の区別

| 層 | 保管 | 説明 |
|---|---|---|
| **Ephemeral** | なし（stdout / 一時 file） | AI が生成した直後 — save するまで storage 対象外 |
| **Saved proposal** | file inbox または DB | ユーザーが「保留・検討」として明示 save |
| **Adopted trip** | schema v8 Trip（既存 DB） | gate 通過後のみ |

```text
区別手段:
  metadata.source / metadata.provider — 出自
  明示 save 操作 — ephemeral → saved への唯一の入口（自動 DB insert 禁止）
  adoption gate — saved → adopted への唯一の入口
```

### 6.2 推奨ルール

1. AI CLI / assistant 連携は **file 出力または validate まで** — v4.7.9〜v4.7.11 と同型
2. `proposal import` / `fragment import` は **人間の明示操作** として設計（P-4 実装後）
3. provider 名だけでは adopted にしない
4. GUI inbox（将来）も saved layer — gate なしでは Trip DB に触れない

## 7. materialize / apply 前に保持すべき最低限の情報

P-5（gate spec 具体化）と P-6（実装）の入力として、storage は以下を **失わない** こと。

### 7.1 Envelope

| 情報 | 用途 |
|---|---|
| proposal body（title, destination, date_policy, candidates） | inspect / materialize 入力 |
| materialize_hints（missing_fields, assumptions, warnings） | gate warnings |
| metadata（created_at, valid_until, source, provider） | 鮮度・監査 |
| validation report snapshot（optional） | 保存時点の blocking 解消記録 |
| status + adopted_trip_id（採用後） | 監査 |

### 7.2 Fragment

| 情報 | 用途 |
|---|---|
| target（target_type, references, unresolved hints） | apply 先の解決 |
| fragment.intent + candidate_content | apply 内容 |
| adoption_hints（conflicts, missing_fields, required_decisions） | gate / 人間判断 |
| metadata | 鮮度・監査 |
| target_trip_id 解決結果（DB 段階） | apply 実行 |

### 7.3 storage 前に必須ではないもの

```text
確定した schema v8 Trip JSON 全体 — materialize / apply の出力
Receipt / Expense の正式行 — 採用後のみ
JSON schema の最終確定版 — 引き続き non-normative example ベース
```

## 8. file-only 運用 — どこまで許容するか

### 8.1 v4.7.13 判断: **P-5 まで file-only を主経路とする**

| 段階 | 運用 | v4.7.13 |
|---|---|---|
| **現行（v4.7.9〜v4.7.12）** | 任意 path の JSON + validate / show / inspect | **継続** |
| **次の optional 段階** | file inbox（`proposals/inbox/` + manifest TOML/JSON） | **計画のみ** — 実装は P-4b 候補 |
| **DB storage** | proposal_envelopes / proposal_fragments テーブル | **P-5 後に再評価** |
| **import / list** | CLI inbox 管理 | DB または file inbox 設計後 |

```text
file-only で足りる期間:
  fragment show / inspect 実装（file read）
  P-5 materialize / apply planning spec
  gate blocking rules の具体化と dry-run 設計

DB が欲しくなるトリガー:
  同一 Trip への Fragment が多数 pend
  list / filter / status 変更の UX 要求
  GUI inbox 連携
  採用履歴の永続監査
```

### 8.2 file inbox 中間案（採用検討・未実装）

[v4.7.8 §3](v4.7.8-proposal-implementation-planning.md) の `proposals/` inbox を具体化:

```text
~/travel-ledger/proposals/inbox/
  envelopes/
    prop-2026-001.json
  fragments/
    frag-2026-001.json
  manifest.json   — id, path, kind, status, saved_at, target_trip_id?
```

| 利点 | DB 不要；git 管理可；validate guard と相性良 |
| 欠点 | list は manifest 依存；並行編集は弱い |

**推奨:** DB 前の **optional ステップ** として inbox を P-4b 候補に置く。v4.7.13 では spec のみ。

### 8.3 却下: いきなり Trip DB 拡張

```text
trips テーブルに is_proposal 列
itineraries に proposal_source 列
→ schema v8 / export / validate-export 境界が曖昧になる
```

## 9. 将来 CLI / GUI / public schema / AI assistant の境界

| コンポーネント | 責務 | storage との関係 |
|---|---|---|
| **CLI validate** | file 読み取り・report | DB 不要（現行維持） |
| **CLI import / list** | saved layer 管理 | file inbox または DB — gate 前 |

| **public schema** | schema v8 Trip 確定版 | Proposal JSON schema は **別 track**（未確定継続） |

```text

  生成（AI）→ validate（CLI）→ save（inbox/DB）→ inspect → gate → schema v8 Trip
  各段階で command / validator が異なる
  trip validate-export は最終 Trip のみ
```

## 11. P-5 / P-6 への前提（storage 観点）

P-5（materialize / apply planning）に進む前に、本書で固定する前提:

```text
1. Envelope / Fragment は別 storage type
2. 採用前は schema v8 外 — trip import 経路を使わない
3. status lifecycle（pending / adopted / rejected / expired）を spec に含める
4. Fragment target は soft link → gate 時に解決
5. AI 出力の自動 adopt 禁止
6. file-only 経路が dry-run / inspect を支え続ける
7. materialize 出力のみ trip validate-export 対象
```

P-6 実装時に必要になる storage API（概念）:

```text
load_envelope(id | path) → validated view
load_fragment(id | path) → validated view
mark_adopted(id, trip_id)
mark_rejected(id)
list_pending(kind, trip_id?)
```
