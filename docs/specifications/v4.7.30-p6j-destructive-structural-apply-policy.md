## 1. Classification — Proposal Fragment apply operations

Fragment apply の intent は、DB への影響の性質で 4 類型に分類する。

### 1.1 Additive operations（追加系）

既存 Trip 構造に **新規行を追加** する。失敗時は未追加のまま。成功時の主リスクは重複追加・target 誤解決。

| 例 | DB 操作 | 典型 target |
|---|---|---|
| `add_itinerary` | `itinerary_items` INSERT | day |
| `add_note` | `notes` INSERT | trip / day / itinerary |
| `add_expense` | `expenses` INSERT | itinerary |
| `add_reservation` | `reservations` INSERT | itinerary |

**v4.7.x 実装済み:** P-6d〜P-6h（add 系）、P-6f〜P-6h（note / expense / reservation）

### 1.2 In-place update operations（上書き系）

既存行の **許可フィールドのみ** を更新する。失敗時は不変または transaction 内でロールバック。成功時の主リスクは意図しない上書き・TOCTOU。

| 例 | DB 操作 | 典型 target |
|---|---|---|
| `update_itinerary` | `itinerary_items` UPDATE 1行 | itinerary |

**v4.7.x 実装済み:** P-6i（dry-run v4.7.28 / confirm v4.7.29）

### 1.3 Destructive operations（破壊系）

既存行（および場合により子行）を **削除** する。成功後の復元は別メカニズムが必要。preview・confirm・利用者への説明責務が重い。

| 例 | DB 操作 | 典型 target |
|---|---|---|
| `delete_itinerary` | `itinerary_items` DELETE（＋子 CASCADE の可能性） | itinerary |
| `delete_expense`（将来） | `expenses` DELETE | itinerary / expense ref |
| `delete_note`（将来） | `notes` DELETE | trip / day / itinerary |

**v4.7.x:** 未実装。CLI には `itinerary delete` が存在するが、Fragment apply とは別責務。

### 1.4 Structural operations（構造変更系）

並び順・所属 Day など **Trip 内の構造** を変える。1 意図で複数行の `sort_order` / `day_id` 更新が発生しうる。

| 例 | DB 操作 | 典型 target |
|---|---|---|
| `reorder_itinerary` | 複数 `itinerary_items` の `sort_order` UPDATE | day 内 itinerary 列 |
| day move（将来） | `day_id` + `sort_order` の組合せ UPDATE | itinerary |
| `day swap` 相当（将来） | 複数 day / itinerary の再配置 | trip |

**v4.7.x:** 未実装。CLI には `itinerary move` / `normalize` / `day swap` が存在するが、Fragment apply とは別責務。

### 1.5 分類の使い方

```text
additive      → append-only、rollback 不要（失敗時不変で足りる）
in-place      → 1行 PATCH、TOCTOU + expected_<field> が中心
destructive   → DELETE、成功後 undo 方針が必須
structural    → 複数行・順序、baseline order + TOCTOU が中心
```

P-6g〜i で確立した **dry-run 先行 → confirm 後続** パターンは引き続き踏襲するが、destructive / structural では **preview 内容・blocking 条件・利用者向け警告** をより厳格にする。

## 2. Delete policy — `delete_itinerary`

### 2.1 基本方針

| 観点 | 推奨（v4.7.x 初期） |
|---|---|
| intent | `delete_itinerary`（新規） |
| target | **itinerary のみ** |
| Trip / Day target | **unsupported / blocking** |
| delete 方式 | **hard delete**（`DELETE FROM itinerary_items`） |
| child あり itinerary | **blocking** — confirm しない |
| childless itinerary | **のみ** delete 許可 |
| successful confirm 後 undo | **保証しない** — gate report / CLI 出力で明示 |
| failed confirm | **DB 不変** — transaction で保証 |

### 2.2 なぜ childless に限定するか

現行 CLI の `itinerary delete` は transaction 内で以下を **CASCADE hard delete** する（[itinerary.rs](../../src/itinerary.rs) `delete_itinerary_item`）:

- Itinerary 配下 **Note**（owner_type = itinerary）
- **Estimate**
- **Expense**
- **Reservation**
- 対象 **itinerary_items** 行

Fragment apply から無制限にこの経路を使うと、Proposal 1 操作で **複数子エンティティが消失** し、利用者が dry-run preview だけでは損失規模を把握しにくい。

**推奨:** v4.7.x の Fragment apply `delete_itinerary` は **子エンティティゼロの itinerary のみ** を対象とし、子が 1 件でもあれば blocking とする。

| 子種別 | 判定 | 備考 |
|---|---|---|
| Itinerary フィールド `note` | **子ではない** — 1 行の列 | childless 判定に含めない |
| Note（entity） | **子** — blocking | `notes` テーブル |
| Expense | **子** — blocking | |
| Reservation | **子** — blocking | |
| Estimate | **子** — blocking | |
| Attachment / Photo（将来） | **子** — blocking | 未実装だが方針上は同扱い |

### 2.3 Hard delete vs soft delete

| 方式 | v4.7.x Fragment apply | 備考 |
|---|---|---|
| **Hard delete** | **採用（childless のみ）** | 既存 CLI 整合、migration 不要 |
| **Soft delete** | **v4.7.x では採用しない** | `deleted_at` / `is_deleted` は migration + 全読取経路の変更が必要（§3 参照） |

### 2.4 Target 制約

| target | 判定 |
|---|---|
| `itinerary`（一意解決） | 許可候補 |
| `trip` | **blocking** |
| `day` | **blocking** |
| `unresolved` | **blocking** |
| `ambiguous` | **blocking** |

P-6i と同型: Trip / Day 直下の bulk delete は意図が曖昧で、誤削除リスクが高い。

### 2.5 Rollback / undo の前提（重要）

```text
failed confirm の DB 不変     → transaction で保証できる
successful confirm 後の undo → soft delete / journal / backup なしには保証できない
```

したがって v4.7.x で delete を進める場合:

1. **scope を強く制限**（childless のみ）
2. **成功後 undo を保証しない** ことを gate report・CLI 出力・仕様で明示
3. 将来の undo 保証は **v4.8+** の別設計（§3）に委ねる

### 2.6 Dry-run preview で見せるべき内容

| 項目 | 内容 |
|---|---|
| `preview.action` | `delete_itinerary` |
| `preview.deleted_itinerary` | 削除対象の要約（id 解決後: title, day, sort_order） |
| `preview.child_counts` | Note / Expense / Reservation / Estimate 件数（**0 であることの確認**） |
| `preview.trip_export_diff` 相当 | 削除後の export から当該 itinerary が消えることの preview |
| warnings | childless でも「復元不可」を明示 |
| blocking | child > 0 → `required_decisions` または errors で blocking |

JSON golden / schema v8 の構造変更は dry-run 実装フェーズまで不要。gate report schema v2 内の preview 拡張で足りる想定。

### 2.7 Confirm と TOCTOU 再検証

confirm 直前（write 前）に再確認:

| 再検証項目 | 目的 |
|---|---|
| target itinerary が依然として一意に解決できる | reference ずれ防止 |
| `expected_<field>` / `expected_itinerary_id`（要設計） | baseline mismatch 検出 |
| **child counts が依然として 0** | gate 後に子が追加された場合の TOCTOU |
| preview に記録した itinerary 属性（title / day / sort_order）が現行 DB と一致 | P-6i TOCTOU 同型 |

いずれか不一致 → **DELETE 前に blocking**、DB 不変。

### 2.8 Gate report（confirm 成功時）

| フィールド | 意味 |
|---|---|
| `deleted_itinerary_id` | 削除した itinerary ID（新規フィールド候補） |
| `preview` | dry-run 同型を保持 |
| `inserted_*_id` / `updated_*_id` | **設定しない** |

**利用者向け明示文言（案）:**

```text
This operation permanently deletes the itinerary. Undo is not available in v4.7.x.
```

## 3. Rollback / undo options — 成功後復元の将来案

successful confirm 後の undo を将来保証するための候補比較。

### 3.1 Soft delete

| 項目 | 内容 |
|---|---|
| 概要 | `itinerary_items.deleted_at` / `is_deleted` 等で論理削除 |
| メリット | restore が比較的単純；誤削除の安全網 |
| デメリット | **DB migration 必須**；`list` / `show` / `export` / `stats` / `validate-export` / `trip diff` / doctor / advisor 全経路で「削除済み除外」が必要 |
| 子エンティティ | soft delete 時に子も論理削除するか、孤児にするか要設計 |
| v4.7.x 採用 | **No** — スコープ过大 |
| v4.8+ | **候補** — Fragment apply undo の基盤になりうる |

### 3.2 Apply journal / undo log

| 項目 | 内容 |
|---|---|
| 概要 | confirm 前に before snapshot（itinerary + child JSON）を journal テーブルまたはローカルファイルに保存；`fragment undo` 等で restore |
| メリット | schema 変更を最小化できる可能性；additive / update / delete 横断で統一しうる |
| デメリット | **ID collision**（restore 時の新 ID vs 旧 ID）；restore 順序；journal 肥大化；child snapshot の整合 |
| v4.7.x 採用 | **No** |
| v4.8+ | **候補** — 「undo 保証」の本命候補の一つ |

### 3.3 Automatic export backup before destructive confirm

| 項目 | 内容 |
|---|---|
| 概要 | destructive confirm 直前に `trip export` を自動保存（ファイルパスを gate report に記録） |
| メリット | migration 不要；既存 import で **手動復元** 可能 |
| デメリット | **strict undo ではない**；import のマージ / ID 再割当；バックアップ先の運用；CI 非対話環境 |
| v4.7.x 採用 | **Optional 将来** — undo 保証ではなく **recovery hint** として検討可 |
| v4.8+ | **候補（非 strict）** — `export backup path` を report に出す程度から開始可能 |

### 3.4 比較まとめ

| 方式 | migration | strict undo | v4.7.x | v4.8+ 推奨度 |
|---|---|---|---|---|
| なし（現状） | 不要 | 不可 | **採用** | — |
| export backup | 不要 | 不可（手動 recovery） | 任意 hint | 中 |
| apply journal | 小〜中 | 可能（設計次第） | No | **高** |
| soft delete | **大** | 可能 | No | 中（横断影響大） |

**v4.7.x の立場:** undo 保証は求めず、**childless hard delete + 明示警告** で進める。undo 基盤は v4.8+ で journal または soft delete を比較検討。

## 4. Reorder policy — `reorder_itinerary`（planning only）

今回 **実装しない**。delete より複雑なため **別フェーズ** で設計する。

### 4.1 なぜ structural か

| 観点 | update_itinerary | reorder |
|---|---|---|
| 影響行数 | 1 行 | **1 行以上**（同一 day 内の複数 itinerary） |
| 変更列 | 属性フィールド | **`sort_order`**（＋ day move 時は `day_id`） |
| preview | field before/after | **order before/after**（配列） |
| CLI 既存 | `itinerary update` | `itinerary move` / `normalize` / sparse ordering |

[ordering-model.md](ordering-model.md): 現行は **Sequence-first**、**sparse sort_order（1000 刻み）**。

### 4.2 Same-day reorder vs cross-day move

**分離を推奨:**

| 操作 | intent 候補 | 内容 |
|---|---|---|
| Same-day reorder | `reorder_itinerary` | 同一 `day` 内の `sort_order` のみ変更 |
| Cross-day move | `move_itinerary` または `update_itinerary` 拡張（**別 intent 推奨**） | `day_id` + `sort_order` の組合せ |

cross-day drag & drop は **day move と reorder が混ざる** ため、1 intent にまとめない。

P-6i では `day` / `sort_order` を unsupported とした経緯（[v4.7.28-p6i plan](v4.7.28-p6i-update-itinerary-implementation-plan.md)）と整合。

### 4.3 Sparse vs dense sort_order

| 方式 | メリット | デメリット |
|---|---|---|
| **Sparse（現行）** | 挿入時に隣接行の再採番が少ない | reorder preview で「中間値」計算が必要 |
| **Dense（1,2,3…）** | 人間可読 | 挿入のたびに複数行 UPDATE しやすい |

Fragment apply reorder は **既存 CLI の sparse 規約を維持** し、`itinerary normalize` 相当の再採番は confirm 成功後の内部処理または別コマンドに委ねる案が安全。

### 4.4 Anchor-based reorder の曖昧性

Fragment が「X の **後** に移動」のみ示し、現行 order baseline が無い場合、解釈が複数化する。

**必要と考える baseline:**

| フィールド | 用途 |
|---|---|
| `expected_order[]` または `expected_sort_orders` | day 内の削除前 order スナップショット |
| anchor reference | `after_itinerary` / `before_itinerary`（title または sort_order ref） |

曖昧な anchor → **blocking**（P-6i ambiguous target と同型）。

### 4.5 Dry-run preview

| 項目 | 内容 |
|---|---|
| `preview.action` | `reorder_itinerary` |
| `preview.day_number` | 対象 day |
| `preview.order_before[]` | `{ itinerary_id or ref, title, sort_order }` |
| `preview.order_after[]` | 適用後の同一形式 |
| `preview.moved_itinerary_id` | 動かす 1 行（**1 移動 per fragment** を初期推奨） |

### 4.6 Confirm TOCTOU

| 再検証 | 内容 |
|---|---|
| order baseline | `expected_order` と現行 DB order の一致 |
| moved itinerary の存在 | 削除・day 変更されていないこと |
| anchor の存在 | after/before 参照 itinerary が有効 |
| sparse 衝突 | 計算後 sort_order が既存と衝突しないこと |

reorder confirm は **複数行 UPDATE** になりうるため、**単一 transaction** 必須。失敗時は全行ロールバック。

### 4.7 初期 scope 案（将来）

```text
intent:     reorder_itinerary
target:     day（day 内 order 全体）または itinerary（1 件移動）
same-day:   のみ（v4.7.x 初期）
cross-day:  別 intent / 後続
child:      reorder 自体は子を変更しないが、order 変更は export / timeline に影響
```

## 5. Recommended adoption path

### 5.1 提案ロードマップ

| Version | 内容 | 種別 |
|---|---|---|
| **v4.7.30** | **P-6j destructive / structural apply policy（本書）** | documentation-only |
| **v4.7.31** | `delete_itinerary` dry-run | implementation |
| **v4.7.32** | `delete_itinerary --confirm` | implementation |
| **v4.7.33** | reorder policy / planning spec（`reorder_itinerary` 詳細設計） | documentation-only |
| **v4.7.34+** | `reorder_itinerary` dry-run → confirm | implementation（段階分割） |

### 5.2 分割案の理由

1. **delete と reorder を同時に実装しない** — 失敗モードと preview 形状が異なる
2. **delete を先** — 1 行 DELETE + childless 制限で scope を閉じやすい
3. **reorder は planning 先行** — 複数行 UPDATE・anchor 曖昧性・sparse order の設計が必要
4. **各 intent で dry-run → confirm** — P-6a〜i の実績パターン踏襲

### 5.3 代替案（より細分化）

delete で child 検出ロジックが重い場合:

```text
v4.7.31 — delete_itinerary dry-run（preview + child count のみ）
v4.7.32 — delete_itinerary dry-run hardening（baseline / conflict tests）
v4.7.33 — delete_itinerary --confirm
v4.7.34 — reorder planning
```

現時点では **5.1 の 4 段階** で十分と判断。テスト負荷が見えた時点で 5.3 に分割可能。

## 6. Initial recommendation（推奨方針）

現時点の推奨を明文化する。

### 6.1 P-6j（v4.7.30）

- **実装しない** — policy の確定のみ
- **delete と reorder を同じリリースで実装しない**
- Venue model implementation を混ぜない

### 6.2 Delete（v4.7.31–32 想定）

- intent: `delete_itinerary`
- target: **itinerary のみ**
- Trip / Day target: **blocking**
- **childless itinerary の hard delete のみ**
- Note / Expense / Reservation / Estimate が 1 件でもあれば **blocking**
- successful confirm 後 **undo 保証なし** — 仕様・report・CLI で明示
- failed confirm: **transaction で DB 不変**
- dry-run: 削除対象要約 + child counts + export preview
- confirm: child count TOCTOU + itinerary baseline TOCTOU

### 6.3 Reorder（v4.7.33+ 想定）

- delete 完了後に **別 planning spec** を作成
- same-day reorder と cross-day move を **分離**
- 初期は **1 itinerary 移動 per fragment** を推奨
- sparse sort_order 維持
- `expected_order` baseline 必須
- confirm は **複数行 UPDATE を単一 transaction** で実行

### 6.4 Undo / rollback（v4.8+）

- v4.7.x では **undo 保証しない**
- v4.8+ で **apply journal** または **soft delete** を比較検討
- 短期の緩和策として **pre-delete export backup hint** は任意

### 6.5 既存 CLI との関係

| 経路 | 関係 |
|---|---|
| `itinerary delete` | CASCADE delete あり — Fragment apply とは **別責務** |
| `itinerary move` / `normalize` | reorder 設計の参照実装 |
| `fragment apply` | gate + adoption_hints + TOCTOU を必須とする **安全な入口** |

Fragment apply delete は、当面 **既存 CLI より狭い scope**（childless のみ）で導入し、安全性を優先する。
