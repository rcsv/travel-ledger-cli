## 1. Release の流れ（確定手順）

### 1.1 tag push から assets まで

```text
1. release commit を master に push（または create-tag.sh で commit + push）
2. annotated tag を push
   → GitHub Actions「Release」workflow が起動（on: push tags v*）
3. matrix 3 job（linux / macos / windows）が並列 build
4. 各 job が softprops/action-gh-release@v3 で同一 tag_name へ asset upload
5. 最初の upload で GitHub Release が自動作成される（title は tag 名のみのことが多い）
```

### 1.2 title / notes 更新タイミング

**推奨: 3 OS assets が揃ってから title / notes を編集する。**

| タイミング | 判定 |
|---|---|
| workflow 実行中に `gh release edit` | **非推奨** — v4.6.41 で macOS asset 欠落の一因になり得る |
| workflow 完了後に `gh release edit` | **推奨** — v4.6.42 で成功 |
| `release.sh`（v4.6.43 以降） | **assets 待機後に edit** — 上記推奨をスクリプト化 |

### 1.3 推奨コマンド順（手動 release 時）

```bash
# 1. commit + tag + push
git push origin master
git tag -a v4.6.43 -m "v4.6.43 ..."
git push origin v4.6.43

# 2. Release workflow 完了を待つ
gh run list --repo rcsv/travel-ledger-cli --workflow Release --limit 1
gh run watch --repo rcsv/travel-ledger-cli <run-id> --exit-status

# 3. assets 確認 → title/notes 更新
tools/release/release.sh v4.6.43 "release workflow asset upload follow-up"
# または
gh release edit v4.6.43 --title "..." --notes-file docs/releases/v4.6.43-notes.md

# 4. 最終確認
gh release view v4.6.43 --repo rcsv/travel-ledger-cli --json tagName,name,assets
```

`full-release.sh` 利用時も、内部の `release.sh` が v4.6.43 以降は assets-first 順序を使う。

## 2. 3 OS assets 確認手順

期待する asset 名（`tools/release/env.sh` の `expected_asset_names` と一致）:

```text

travel-ledger-cli-<version>-macos-arm64.tar.gz
```

確認:


```bash

```

tag 付き Release に 3 assets が揃っていれば OK。

## 3. macOS asset 欠落時の recovery

workflow が success でも tag 付き Release に macOS asset だけ無い場合:

```bash
version=4.6.43
asset="travel-ledger-cli-${version}-macos-arm64.tar.gz"

cargo build --release
tar -czf "/tmp/$asset" -C target/release travel-ledger-cli

gh release upload "v${version}" "/tmp/$asset" \
  --repo rcsv/travel-ledger-cli --clobber

gh release view "v${version}" --repo rcsv/travel-ledger-cli --json assets
```

注意:

- upload 先は **tag 付き Release**（`v4.6.xx`）であること
- `--clobber` は同名 asset の上書き用
- ローカル build は **tag が指す commit** で行う（`git checkout v4.6.xx` 後に build が安全）
- title/notes は assets 揃い後に `gh release edit`

## 4. `docs/current-work.md` follow-up commit

正式リリース後、**別 commit** で current-work を更新する（v4.6.39〜v4.6.42 と同型）:

```text
release commit   → tag push → workflow → GitHub Release
follow-up commit → docs/current-work.md を released / next planning に更新
```

release commit に current-work の「released」状態を含めなくてもよい。tag は release commit を指す。
