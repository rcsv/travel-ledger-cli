# v4.8.14 — Proposal Fragment apply strict currency integration (implementation)

| Release | v4.8.14（released — 本書） |

v4.8.12 で導入した `CurrencyValidationMode::IsoStrict` を、**Proposal Fragment apply の金額系 intent** に限定して適用する implementation release です。

v4.8.13 で CLI create/update write path は strict 化済み。本 release は Fragment apply dry-run / confirm gate の parse 層を対象とします。

## 1. IsoStrict 適用 surfaces

### 1.1 Fragment apply（v4.8.14）

| Intent | Field | Validation | Phase |
|---|---|---|---|
| `add_expense` | `candidate_content.currency` | 常に IsoStrict | simulate（gate）+ confirm 再 parse |
| `add_estimate` | `candidate_content.currency` | 常に IsoStrict | 同上 |
| `update_estimate` | `candidate_content.currency` | **currency 明示時のみ** IsoStrict | 同上 |
| `update_estimate` | 既存 DB currency | currency 未指定 update では **再 strict 検証しない** | — |

共通 helper:

```rust
fn validate_fragment_candidate_currency(
    currency_text: &str,
    report: &mut Option<&mut FragmentApplyDryRunReport>,
    intent: &str,
    phase: ApplyErrorPhase,
) -> Result<String, String>
// 内部: validate_cli_write_currency_code() → IsoStrict
```

dry-run と confirm transaction 内の再 parse は同一 helper / 同一 strict ロジックを通します。confirm は gate（simulate）で先に検証し、`report.valid = false` なら DB write しません。

### 1.2 format-only 維持 surfaces

| Surface | 理由 |
|---|---|
| `validate_currency_code()` wrapper | 既定 FormatOnly — domain / import 互換 |
| `expense::add_expense` / `update_expense` | domain 層 |
| `estimate::add_estimate` / `update_estimate` | domain 層 — import / internal tests |
| `trip import` / `validate-export` | v4.8.15 まで format-only |
| Receipt / inbox add/update | workflow 複合のため defer |
| read / list / show / export / export-md | legacy unknown currency 非破壊 |
| CLI write path（v4.8.13） | 本 release では変更なし（既に strict） |

## 2. Strict reject 方針

| Input | Result |
|---|---|
| `JPY`, `USD`, `EUR`, withdrawn（`HRK`, `LTL` 等） | OK（uppercase normalize） |
| `jpy`, `usd` | OK → normalize |
| `ZZZ`, `ABC`, `JPN` | blocking — `currency は有効な ISO 4217 コードではありません` |
| `XXX`, `XTS`, `XAU`, `XAG` 等 denylist | blocking — `currency は旅行費用として許可されていません: {code}` |
| `JP`, `JPYY`, 空文字 | blocking — 既存 format message |

## 3. Structured error contract

新規 structured error code は追加しません（v4.8.11 planning 方針）。

| 属性 | 値 |
|---|---|
| `code` | `APPLY_FIELD_INVALID` |
| `kind` | `blocking` |
| `phase` | `simulate`（dry-run / confirm gate） |
| `field_path` | `candidate_content.currency` |
| `intent` | `add_expense` / `add_estimate` / `update_estimate` |
| legacy `errors[]` | 同一 `message` を mirror（`push_structured_error`） |
| `message` | **unstable** — client 分岐正本は `code` + `field_path` |

success report は空 `structured_errors` キーを JSON から省略します（v4.8.6 契約維持）。

---

## 4. Update estimate contract

- `currency` **未指定** update（title / note / sort_order のみ等）→ 既存 DB currency を strict 再検証しない（legacy `ZZZ` 等 OK）
- `currency` **明示** → IsoStrict + `amount` 必須（既存 contract 維持）
