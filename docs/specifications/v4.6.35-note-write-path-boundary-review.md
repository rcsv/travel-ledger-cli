## 1. Current state — Note write path

### read-only vs write（非対称）

| Command | 層 | 備考 |
|---|---|---|
| `note list` | `services::note_list` → CLI `NoteListJson` / `print_note_list` | owner 解決は service 内 |
| `note show` | `services::note_show` → JSON / `print_note_detail` | |
| `note add` | handler → `note.rs` | **未 service 化** |
| `note update` | handler → `note.rs` | **未 service 化** |
| `note delete` | handler → `note.rs` | **未 service 化** |

Expense v4.6.33 前と同型。**read-only のみ service 化、write は module 直結**。

### write path フロー（確認済み — `main.rs`）

#### Add

```text
NoteAction::Add { trip, day, itinerary, title, body }
  handler:
    resolve_note_owner_for_add(conn, trip, day, itinerary)  → ResolvedNoteOwner
    add_note(conn, owner, title, body)                      → note_id
    println!("Note を追加しました (ID: {id})")
    get_note(conn, id)
    print_note_detail(&note)                                ← domain Note 直接
```

#### Update

```text
NoteAction::Update { id, title, body }
  handler:
    update_note(conn, id, title, body)                       ← 内部で get_note
    println!("Note を更新しました (ID: {id})")
    get_note(conn, id)                                      ← handler で再取得
    print_note_detail(&note)
```

#### Delete

```text
NoteAction::Delete { id }
  handler:
    get_note(conn, id)                                      ← 削除前スナップショット
    delete_note(conn, id)                                   ← 内部でも get_note
    println!("Note を削除しました (ID: {id})")
    println title line                                      ← enrich なし・title のみ
```

### `note.rs` 内の write 関連関数

| 関数 | 責務 |
|---|---|
| `resolve_note_owner_for_add` | CLI フラグ → `ResolvedNoteOwner` + owner 存在確認 |
| `resolve_note_owner_for_list` | add と **同一実装**（委譲） |
| `add_note` | body 検証、owner 再確認、INSERT、`last_insert_rowid()` |
| `update_note` | 部分更新、body 検証、`get_note` 内包 |
| `delete_note` | 存在確認 + DELETE |
| `get_note` | ID 取得 |
| `delete_notes_for_trip` / `_day` / `_itinerary` | cascade（trip/day/itinerary delete から） |
| `swap_day_note_owners` | day swap 時の owner_id 交換 |
| `import_export_notes` / `build_export_notes` | trip import/export 経路（別 write path） |

**repository 層は存在しない** — SQL は `note.rs` 内に直書き。

## 2. 責務分布 — handler / note.rs / service / DTO / display

### 層別インベントリ

| 責務 | 所在 | 評価 |
|---|---|---|
| CLI 引数（clap） | `NoteAction` — `trip` / `day` / `itinerary` / `title` / `body` | ○ |
| owner 解決（add/list） | `resolve_note_owner_for_*` | ○ `note.rs` に集約 |
| owner 存在確認 | resolve + `add_note` 内の再確認 | △ 二重だが防御的 |
| body 検証 | `validate_body` — add/update | ○ mutation 近傍 |
| 永続化 | `add_note` / `update_note` / `delete_note` | ○ 単一モジュール |
| 成功メッセージ | handler `println!` | ○ human は CLI |
| 成功後 detail | handler → `print_note_detail(&Note)` | ○ read show と **同一** |
| JSON 出力（write） | **なし** | — |
| JSON 出力（read list） | `NoteListJson { owner_type, owner_id, notes: Vec<Note> }` | ○ `Note` をそのまま serialize |
| JSON 出力（read show） | `Note` 直接 | ○ 別 wire DTO なし |

### handler に mutation / validation が寄りすぎていないか

| 観点 | 判定 |
|---|---|
| mutation ロジック | **handler にはない** |
| owner validation | **handler にはない** — `resolve_*` / `add_note` |
| handler の行数 | add は resolve → mutate → display の **薄い配線** |
| update/delete | ID のみ — owner 変更なし |
| Expense write（v4.6.33 後）と比較 | **同型の薄 handler**；Note は parse_* 相当が owner resolve のみで軽い |

**結論:** handler はオーケストレーション過多ではない。主塊は `note.rs`。

### DTO / 表示 / 永続化の分離

| 経路 | 型 | 永続化 | 表示 |
|---|---|---|---|
| read list | `Note` in `NoteListJson` | service → `list_notes_for_owner` | `print_note_list` |
| read show | `Note` | service → `get_note` | `print_note_detail` |
| write add/update | `Note` | `note.rs` | `print_note_detail`（**read と同一**） |
| write delete | なし | `note.rs` | title 1 行 |
| trip import | `ExportNote` | `import_export_notes` | なし |

Expense と異なり **enriched part / wire DTO adapter 層がない**。`Note` が単一の正。

## 3. owner_type + owner_id — validation 境界

### polymorphic owner モデル

```text
Trip Note:       owner_type = trip       owner_id = trips.id
Day Note:        owner_type = day        owner_id = days.id
Itinerary Note:  owner_type = itinerary  owner_id = itinerary_items.id
```

DB: `CHECK (owner_type IN ('trip','day','itinerary'))` + **FK なし**（[note-model.md](note-model.md) 案 C）。

### `resolve_note_owner_for_add` の分岐（確認済み）

| CLI 指定 | 解決 | 存在確認 |
|---|---|---|
| `--trip T` のみ | `ResolvedNoteOwner::Trip(T)` | `get_trip` |
| `--trip T --day N` | `Day(day_id)` via `find_day_id_by_trip_and_day_number` | trip + day 番号の整合 |
| `--itinerary I` のみ | `ResolvedNoteOwner::Itinerary(I)` | `get_itinerary_item` |
| 複数 owner フラグ | **拒否** | 明確なエラーメッセージ |
| 未指定 | **拒否** | `owner を指定してください` |

`resolve_note_owner_for_list` は **同一関数** — list/add で owner 解釈がずれない。✓

### `add_note` 内の再確認

| Resolved owner | 再確認 |
|---|---|
| Trip | `get_trip` |
| Day | `find_day_by_id` |
| Itinerary | `get_itinerary_item` |

resolve 直後の二重確認。**意図的防御**（直接 `add_note` 呼び出し・将来 import 経路）。service 化時も `add_note` を mutation 本体として残すなら維持可。

### update / delete の owner 扱い

| Action | owner validation |
|---|---|
| update | **なし** — note ID のみ。owner は変更不可 |
| delete | **なし** — note ID のみ |

update/delete 時に owner 行が孤児化しているケースは、通常 cascade 削除前提では発生しにくい。FK 導入前は **application-level cascade**（`delete_notes_for_*`）に依存 — Expense `itinerary_id` と同型の integrity モデル。

### Trip / Day / Itinerary 存在確認の一貫性

| Owner | add（resolve） | add（mutation 再確認） | list（resolve） |
|---|---|---|---|
| Trip | `get_trip` | `get_trip` | 同左 |
| Day | `find_day_id_by_trip_and_day_number` | `find_day_by_id` | 同 resolve |
| Itinerary | `get_itinerary_item` | `get_itinerary_item` | 同左 |

**判定:** 三 owner とも存在確認パスは一貫。Day だけ CLI は `(trip, day_number)` → `day_id` の二段解決。

### service 化時の owner 複雑さ

| リスク | 評価 |
|---|---|
| owner 解決を handler に残す | **非推奨** — list service と非対称 |
| owner 解決を write service に集約 | **推奨** — `note_list` と同パターン |
| `ResolvedNoteOwner` の service 公開 | 内部型のまま service 内で完結可 |
| update/delete の owner | ID ベースのため **Expense より単純** |

**最重要境界:** add の `resolve_note_owner_for_add` + `add_note` の責務分割を service plan で明示する。update/delete は owner 非関与。

## 4. Expense write migration との比較

### 共通点

| 観点 | Expense | Note |
|---|---|---|
| read list/show | thin read service | thin read service ✓ |
| write | handler → module mutation | 同左 ✓ |
| repository 層 | なし | なし ✓ |
| DB FK | なし（app validation） | なし（app validation） ✓ |
| write JSON 出力 | なし | なし ✓ |
| success `println!` | CLI | CLI ✓ |
| delete snapshot | amount/currency | title ✓ |
| `last_insert_rowid()` | add（W-H defer） | add（同様 defer 可） |
| import 経路 | `import_expense_v3` 別 path | `import_export_notes` 別 path |

### 相違点（Note の追加論点）

| 観点 | Expense | Note |
|---|---|---|
| 親参照 | `itinerary_id` 固定 | `owner_type` + `owner_id` 多態 |

| enriched result | `ExpenseEnrichedPart` 必要 | **不要** — `Note` 十分 |
| wire DTO | `ExpenseJson`（CLI mapper） | **なし** — `Note` = JSON |
| parse_* 層 | participant / shared options | **なし** |
| 成功後表示 | v4.6.33 前は adapter 経由 enrich | **最初から `print_note_detail` 直** |
| adapter cleanup Phase | W-5 で 3 関数削除 | **軽い or 省略可**（dead adapter なし） |
| update の二重 get | handler + mutation 内 | **同型**（`get_note` 2〜3 回） |

### Expense 型紙の再利用度

| Expense Phase | Note への適用 |
|---|---|
| W-0 boundary review | **本書（v4.6.35）** |
| W-1 migration plan | v4.6.36 候補 |

## 5. enriched result の要否

| 観点 | 判定 |
|---|---|
| participant / beneficiary 解決 | **なし** |
| owner 名の human 表示 | `owner_type` + `owner_id` をそのまま表示（trip/day/itinerary 名は出さない） |
| JSON shape | `Note` struct が serde 済み |
| list/show service result | 既に `Note` / `Vec<Note>` |

**結論:** `NoteEnrichedPart` 相当は **不要**。write service result は domain `Note` を返せば read `note_show` と同型にできる。

## 6. thin write service 化 — 候補 result 型

### 推奨（Option N-B — Expense W-2+W-3 踏襲）

```rust
// note_add
pub struct NoteAddParams {
    pub trip: Option<i64>,
    pub day: Option<i64>,
    pub itinerary: Option<i64>,
    pub title: Option<String>,
    pub body: String,
}
pub struct NoteAddServiceResult {
    pub id: i64,
    pub note: Note,
}

// note_update
pub struct NoteUpdateParams {
    pub id: i64,
    pub title: Option<String>,
    pub body: Option<String>,
}
pub struct NoteUpdateServiceResult {
    pub note: Note,
}

// note_delete
pub struct NoteDeleteServiceResult {
    pub id: i64,
    pub title: Option<String>,  // 削除前スナップショット（human 用）
}
```

### handler 接続イメージ（将来）

```text
note add:
  service → resolve_note_owner_for_add → add_note → get_note
  handler → println! + print_note_detail(&result.note)

note update:
  service → update_note → get_note
  handler → println! + print_note_detail(&result.note)

note delete:
  service → get_note (snapshot) → delete_note
  handler → println! + title line
```

read `note_show` と add/update 後の `Note` 型が一致 — **Expense v4.6.33 の ExpenseEnrichedPart 統一と同趣旨だが層が薄い**。

### delete 後に返すもの

| 候補 | 評価 |
|---|---|
| `{ id, title }` のみ | **現行 human と一致** — 推奨 |
| 全文 `Note` | 可能だが delete 後表示に body は不要 |
| 空 result | human 互換を壊す |

Expense delete の `{ id, amount, currency }` と同型の **最小スナップショット** が妥当。

## 7. read/write 非対称性の評価

| 観点 | 現状 | service 化後 |
|---|---|---|
| list | service | 変更なし |
