# v4.8.17 — Currency hardening follow-up review (documentation-only)

| Release | v4.8.17（released — 本書） |

v4.8.13〜v4.8.16 で段階的に実施した currency hardening の**現在地**を棚卸しし、残 scope / 方針を明文化する review です。

本書は documentation-first（実装を急がない）であり、**挙動変更・DB schema 変更・JSON schema breaking change は行いません**。

---

## 0. 用語

- **FormatOnly**: `validate_currency_code()` による **英字3文字 + uppercase normalize** のみ（unknown / denylist を許容）
- **IsoStrict**: ISO 4217 registry + travel-expense denylist を enforce（unknown / denylist を reject）
- **Denylist**: travel expense として禁止の ISO code（例: `XXX`, `XTS`, `XAU`, `XAG` 等）

正本実装:

- `src/money/mod.rs` — `CurrencyValidationMode`, `validate_currency_code()`, `validate_cli_write_currency_code()`
- `src/money/currency_registry.rs` — ISO registry + denylist

## 1. Strict 化済み surfaces（IsoStrict）

### 1.1 CLI write path（v4.8.13）

| Command | Currency surface | Policy |
|---|---|---|
| `expense add` | `--currency` | IsoStrict |
| `expense update` | `--currency` 明示時のみ | IsoStrict（未指定 update は既存値 untouched） |
| `estimate add` | `--currency` | IsoStrict |
| `estimate update` | `--currency` 明示時のみ | IsoStrict（未指定 update は既存値 untouched） |

validator:

- `validate_cli_write_currency_code(code)`（IsoStrict）

### 1.2 Proposal Fragment apply（v4.8.14）

| Intent | Field | Policy |
|---|---|---|
| `add_expense` | `candidate_content.currency` | IsoStrict |
| `add_estimate` | `candidate_content.currency` | IsoStrict |
| `update_estimate` | `candidate_content.currency`（明示時のみ） | IsoStrict（未指定 update は既存値 untouched） |

dry-run / confirm の共通 helper:

- `validate_fragment_candidate_currency()` → `validate_cli_write_currency_code()`

Structured errors:

- `APPLY_FIELD_INVALID` + `field_path: candidate_content.currency`

### 1.3 Receipt / inbox CLI（v4.8.16）

| Command | Currency surface | Policy |
|---|---|---|
| `receipt add` | `--currency`（`--amount` とセット） | IsoStrict |
| `receipt update` | `--currency` 明示時のみ | IsoStrict |
| `receipt assign` | CLI `--currency` 明示時のみ | IsoStrict |

重要:

- memo-only update 等で currency 未指定の場合、既存値を再 strict 検証しない（non-breaking）
- assign で stored receipt currency をそのまま使う場合も再 strict 検証しない（non-breaking）

## 2. Warning-only surfaces（quality signal）

### 2.1 `trip validate-export` currency warnings（v4.8.15）

| Condition | Severity |
|---|---|
| format invalid（空/英字3文字でない） | error（従来どおり） |
| registry unknown（例: `ZZZ`, `ABC`） | warning |
| denylist（例: `XXX`, `XTS`, `XAU`） | warning |
| valid ISO | pass（warning なし） |

helper:

- `validate_export_currency_quality_warning(code, field_path) -> Option<String>`

保証:

- warning-only は `valid: true` を維持（import 可能）

## 3. Format-only を意図的に維持している surfaces

### 3.1 trip import（互換最優先）

- import は round-trip 互換を優先し、unknown currency を blanket reject しない
- Receipt import（`import_receipt_v7`）も含め、currency は format-only / 未検証の path が残る

### 3.2 read/list/show/export/export-md（互換最優先）

- 既存 DB に unknown / denylist currency が存在しても、表示・エクスポートで失敗させない
- export は値を保持し、品質 signal は validate-export に寄せる

### 3.3 domain layer（互換と分離）

domain 層は既存互換のため `validate_currency_code()`（FormatOnly）既定を維持。

---

## 4. 既存 DB / import 互換のため strict 化していない surfaces（理由付き）

- **trip import**: 旧 export や既存データを壊す risk が高い（breaking になりやすい）
- **read/export 系**: 表示・持ち出しで失敗すると復旧不能になりうる
- **domain layer**: 多数の呼び出し元（import、内部機能、tests）があり、一括 strict はスコープ逸脱 + breaking risk

---

## 5. 次に strict 化すべき候補（提案）

### P1: Receipt / inbox の内部 write path 集約（任意）

現状は CLI entry で strict。将来 API/GUI 入口が増える場合に備えて、

- `receipt.rs` 内の write パラメータ解釈層（入力文字列 → 正規化 currency）を service 層で共有

ただし v4.8.16 で CLI strict は達成済みのため、直近の必須ではありません。

### P1: validate-export の receipt currency quality warning

v4.8.15 は expenses / estimates に限定。Receipt は export 形状（schema v7+）で独立配列 `receipts[]` を持つため、

- `receipts[].currency` の unknown / denylist を warning 化

は data quality として妥当です（ただし scope としては別リリースに分離推奨）。

---

## 6. まだ strict 化すべきでない候補（提案）

### trip import blanket strict reject

breaking risk が高く、opt-in / warning-first が妥当。

推奨方針:

- default: import は成功（format-only）
- validate-export で warning を出す
- 将来: `trip import --strict-currency` の opt-in で reject（段階導入）

---

## 7. minor units validation をいつ扱うべきか

現状:

- `currency_minor_unit_digits` は subset table で、unknown は 2 桁 fallback

提案:

- **ISO-backed minor unit lookup** は amount parse の意味論に影響しうるため、currency strict とは別リリースで扱う
- 前提として validate-export / doctor で quality signal を拡充し、移行リスクを可視化してから着手

---

## 8. domain layer / wrapper 整理をいつ行うべきか

現状:

- `validate_currency_code()` は FormatOnly の互換 wrapper
- `validate_cli_write_currency_code()` が strict surfaces の入口

提案:

- 入口が増えた段階（CLI / Fragment / Receipt 以外が増える）で service 層に shared helper を置く
- それまでは wrapper 置換や大規模整理は避ける（scope / risk の観点）

---

## 9. Consistency checklist（v4.8.13〜v4.8.16 整合）

- strict: CLI write / Fragment apply / Receipt CLI
- warning: validate-export（expenses / estimates）
- format-only: import / read / export / domain layer
- ISO registry + denylist: `currency_registry.rs` が正本
- tests / release notes: v4.8.13〜v4.8.16 で段階導入を明記
