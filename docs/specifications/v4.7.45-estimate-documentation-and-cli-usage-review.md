## 1. 利用者向け契約

### 1.1 Estimate と Expense

```text
Estimate = Planned Money（予定金額）
Expense  = Actual Money（実際に発生した金額）
```

| ルール | 説明 |
|---|---|
| Estimate の役割 | 旅行前・旅行中の **見込み金額** を Itinerary 配下に記録する |
| Expense の役割 | 旅行中・旅行後の **実際の支払い** を記録する |
| 独立性 | `add_estimate` を実行しても **Expense は作成されない** |
| 逆方向も同様 | `add_expense` を実行しても **Estimate は自動更新されない** |
| 変換 | **Estimate-to-Expense conversion は未対応** |
| 共存 | 同一 Itinerary に Estimate と Expense は **独立した行**として共存しうる |

例:

| 状況 | 使うもの |
|---|---|
| 「入場料はだいたい JPY 10,000 くらい」（旅行前） | Estimate（`add_estimate` または `estimate add`） |
| 「入場料に JPY 9,800 を支払った」（旅行後） | Expense（`add_expense` または `expense add`） |

### 1.2 supported target

`add_estimate` が **現在対応する target** は **Itinerary のみ** です。

```text
target_type: itinerary
```

| target | Fragment apply `add_estimate` |
|---|---|
| **Itinerary** | **対応** — `day_reference` + `itinerary_reference` で既存 1 件に解決 |
| **Trip** | **未対応** — blocking |
| **Day** | **未対応** — blocking |
| その他 entity | **未対応** |

CLI では **`--trip <id>`** が操作対象 Trip の正本です。解決された Itinerary はその Trip に属している必要があります。

### 1.3 dry-run

```bash
caglla fragment apply <fragment.json> --dry-run --trip <id> [--output preview.json] [--json]
```

| 動作 | 説明 |
|---|---|
| candidate 解析 | `candidate_content` の amount / currency / title / note / sort_order を検証 |
| target 解決 | itinerary target を一意に解決（§3） |
| validation 表示 | gate report（`--json`）または人間可読メッセージ |
| DB | **書き込まない**（read-only） |
| 目的 | 成功可能性を確認する **preview** |
| 結果 | dry-run だけでは **Estimate は作成されない** |
| `inserted_estimate_id` | **含まれない**（confirm 専用） |

`--output` を指定すると、追加予定 Estimate を含む **preview Trip JSON** を書き出します（DB は不変）。invalid gate では preview ファイルは抑制されます（既存 Fragment apply 慣行）。

### 1.4 confirm

```bash
caglla fragment apply <fragment.json> --confirm --trip <id> [--json]
```

| 動作 | 説明 |
|---|---|
| 前提 | 同一 Fragment JSON で dry-run が通ること（推奨） |
| preview 一致 | 成功時は gate の `estimate_preview` と candidate fields の一致を transaction 内で再確認 |
| 成功時 DB | **`estimates` に 1 行 INSERT** のみ |
| 成功時 report | **`inserted_estimate_id`** を返す（新規 Estimate の DB ID） |
| Expense | **書き込まない** |
| 失敗時 | **DB を変更しない**（rollback） |
| `--dry-run` 併用 | **不可** |

`--dry-run` と `--confirm` は同時に指定できません。

### 1.5 amount / currency（minor units 契約）

DB の `estimates.amount` は **常に最小通貨単位の整数** です（[estimate-entity-design.md §3](estimate-entity-design.md)）。

#### JSON number（integer）— Fragment / Proposal 向け

```json
{
  "amount": 1250,
  "currency": "USD"
}
```

| 入力 | DB 保存 | 補足 |
|---|---|---|
| `amount: 1250` + `USD` | **`1250`** | JSON **number** は **minor units としてそのまま** 解釈する |
| confirm 時 | 100 倍などの再変換 **しない** | v4.7.43 で二重変換バグを回避 |

「12.50 USD」を **minor units の 1250** として表したい場合は、上記のように **`1250` を number で渡す**（回帰テストで確認済み: `apply-add-estimate-valid-minor-units.json`）。

#### JSON string — CLI 互換の小数テキスト

```json
{
  "amount": "14000",
  "currency": "JPY"
}
```

| 入力 | DB 保存 | 補足 |
|---|---|---|
| `"amount": "14000"` + `JPY` | **`14000`** | 円単位の整数テキスト（fixture `apply-add-estimate-valid.json` で確認） |
| `"amount": "12.50"` + `USD` | **`1250`** | 共有 `parse_amount_for_currency` により小数テキストを minor units に変換（[estimate-entity-design.md §3](estimate-entity-design.md) の CLI 契約と同型；**add_estimate 専用 fixture は未整備**） |

#### 通常 CLI（参考 — Fragment とは別入口）

```bash
caglla estimate add --itinerary 12 --amount 12.50 --currency USD
```

`--amount` は **文字列**として受け取り、通貨精度に応じて minor units に変換します（USD `12.50` → DB `1250`）。

#### 共通ルール

| ルール | 内容 |
|---|---|
| `currency` | 非空 3 文字コード；正規化あり（例: `jpy` → `JPY`） |
| 負数 | **拒否** |
| 欠落 / 非数値 | **blocking** |
| 小数桁過多 | **拒否**（通貨精度に基づく） |

**注意:** JSON number と JSON string は **別の解釈規則** です。`1250`（number）と `"12.50"`（string）はどちらも USD で DB `1250` になりますが、`"1250"`（string）を USD に渡すと **1250 ドル（minor units 125000）** として解釈される点に注意してください（string は常に `parse_amount_for_currency` 経由）。

### 1.6 Proposal Fragment `candidate_content` フィールド

| Field | Required | 意味 |
|---|---|---|
| `amount` | **yes** | string（小数テキスト可）または number（minor units） |
| `currency` | **yes** | 3 文字通貨コード |
| `title` | no | 見積行タイトル（alias: `description`, `label`） |
| `note` | no | Estimate 専用メモ（alias: `memo`, `fragment.notes`） |
| `sort_order` | no | 同一 Itinerary 内の並び；省略時 **`0`** |

## 2. target selector の安全規則

Itinerary target は `day_reference` と `itinerary_reference` で指定します。`itinerary_reference` は **文字列（title）** または **数値** です。

### 2.1 数値 selector の解決順（安全規則）

意図しない Itinerary への追加を防ぐため、数値は次の順で解決します。

```text
1. Itinerary ID として解決を試みる
2. ID で解決しない場合のみ sort_order として解決
3. 曖昧な場合は reject（DB 不変）
```

| ケース | 結果 |
|---|---|
| 数値が **Itinerary ID** に一致 | その Itinerary を採用（**推奨**） |
| ID に一致せず **sort_order** にのみ一致 | sort_order で解決 |
| ID と sort_order が **別 Itinerary** を指す | **拒否**（曖昧） |
| どちらにも一致しない | **拒否**（未解決） |

title 文字列 selector は、同一 Day 内で **タイトルが一意** のときのみ解決します。重複タイトルは **拒否** されます（テストで「曖昧」メッセージを確認）。

### 2.2 例: 明示的な Itinerary ID（推奨）

Trip 1・Day 1 の Itinerary ID が `42` のとき:

```json
{
  "target": {
    "target_type": "itinerary",
    "day_reference": 1,
    "itinerary_reference": 42
  },
  "fragment": {
    "intent": "add_estimate",
    "candidate_content": {
      "amount": "2500",
      "currency": "JPY"
    }
  },
  "adoption_hints": { "required_decisions": [] }
}
```

テスト `cli_fragment_apply_add_estimate_itinerary_id_selector_confirm_succeeds` で confirm 成功を確認。

### 2.3 例: title selector（手書き Fragment の便宜）

```json
{
  "target": {
    "target_type": "itinerary",
    "day_reference": 1,
    "itinerary_reference": "Morning temple"
  },
  "fragment": {
    "intent": "add_estimate",
    "candidate_content": {
      "amount": "14000",
      "currency": "JPY",
      "title": "Aquarium tickets"
    }
  },
  "adoption_hints": { "required_decisions": [] }
}
```

fixture `apply-add-estimate-valid.json` / dry-run・confirm テストで使用。

### 2.4 例: 曖昧で拒否されるケース

**重複 title:** 同一 Day に `"Morning temple"` が 2 件ある状態で上記 §2.3 と同じ `itinerary_reference` を指定 → **拒否**（`cli_fragment_apply_add_estimate_ambiguous_title_target_blocks_without_db_write`）。

**数値曖昧性:** 数値 `2` が Itinerary ID=2 と sort_order=2 の **別行**に同時に一致する状態では、共通 target resolver が **拒否** します（`reorder_itinerary` の `cli_fragment_apply_reorder_itinerary_confirm_ambiguous_number_selector_is_rejected` で同 resolver を検証）。`add_estimate` でも **同一ルール** が適用されます。
