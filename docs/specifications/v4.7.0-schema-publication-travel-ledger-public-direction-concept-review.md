## 1. Travel Ledger public direction

### 1.1 何を解決する形式か

旅行計画データは、予約サイト・地図アプリ・メモ・メール・チャット・スプレッドシートに分散しがちである。サービスをまたぐと履歴が消え、同行者と共有しづらく、AI の提案をそのまま信用できない。

**Travel Ledger** は、採用済みの旅行計画・実績・予約・費用・メモを **旅行者自身がローカルで保持できる構造化形式** として定義する。

```text
解決する問題:
  - 旅行データの分散とロックイン
  - AI / 業者提案の無批判取り込み
  - 計画と実績の混同
  - 将来 GUI / 自動化 consumer の共通土台不足
```

### 1.2 レイヤー構成（概念）

```text
Travel Ledger schema (canonical trip data)
  └─ export schema v8（現行）— Trip / Day / Itinerary / … の正式データ

travel-ledger-cli (reference implementation)
  ├─ validator      trip validate-export
  ├─ converter      import / export / diff
  ├─ doctor/advisor trip doctor / trip advisor
  └─ materializer   （将来）Proposal → Trip

Proposal layer（schema 外）
  ├─ Trip Proposal      旅行全体の候補案
  └─ Proposal Fragment    既存 Trip への差し込み候補

future GUI (schema consumer)
  └─ local-first planner — mutation は service 境界経由
```

### 1.3 CLI の位置づけ

| 役割 | 内容 |
|---|---|
| **正本の実装例** | export/import が schema の living reference |
| **検証器** | `trip validate-export` が public contract の gate |
| **変換器** | v1〜v8 import 互換が evolution の証拠 |
| **診断** | doctor / advisor が data quality を人間向けに説明 |
| **正本ではないもの** | 予約サイト、地図、SNS、業務帳票ツール |

v4.6.x で整えた **thin service layer + handler 責務分離** は、将来 GUI が同じ service result を消費する前提と整合する。

### 1.4 GUI の位置づけ

GUI（将来 `travel-ledger-app`）は **schema consumer** として設計する。

```text
GUI は SQLite を直接いじらず、service API / structured result を経由
human output は CLI 固有 — GUI は別 presentation
read path は v4.6.x で確立済みの service 境界を再利用可能
write path も Expense / Note / Reservation 型紙を横展開可能
```

v4.7.0 では GUI 実装は scope 外。Tauri spike は v4.7.x 後半以降の候補。

### 1.5 AI / provider 連携の位置づけ

```text
AI / 旅行業者 / ブログ / 手入力
  → Proposal Envelope（候補）
  → 人間が比較・確認
  → materialize（採用 gate）
  → schema v8 Trip（正式データ）
  → export / Travel Book / 将来 Calendar 等
```

**Trip ≠ Proposal** の分離を public narrative の中核とする。

## 3. Proposal Envelope / Proposal Fragment

### 3.1 前提（固定）

```text
schema v8 の Trip は実日付（start_date / end_date）を持つ正式データ
日付未定・未採用の旅行提案を Trip に無理に入れない
提案は外側の Proposal Envelope として扱う
採用時に materialize して Trip / Itinerary を生成する
```

### 3.2 概念モデル

```text
Trip Proposal（旅行全体の提案）
  - 候補目的地・大まかな日程感・概算・注意事項
  - date_mode: fixed | flexible | undated
  - 採用前は Trip DB 行を作らない（または proposal 専用 store）

Proposal Fragment（部品提案）
  - 既存 Trip / Day への差し込み候補
  - 例: 「Day 2 にこのレストランを追加」
  - materialize 時に Itinerary / Reservation 等へ変換

Materialize（採用 gate）
  - 人間が明示採用
  - start_date / day mapping を確定
  - schema v8 準拠の Trip JSON または DB mutation を生成
  - import とは別コマンド（将来: proposal materialize）
```

### 3.3 Envelope metadata 候補

| フィールド | 用途 |
|---|---|
| `generated_by` | 生成主体 ID（human / org / model） |
| `generator_type` | `ai` / `provider` / `manual` / `import` |
| `provider_name` | 旅行業者・サービス名 |
| `created_at` | RFC3339 |
| `valid_until` | 提案の明示有効期限 |
| `no_expiration` | `true` で期限なしを明示 |
| `assumptions` | 前提条件の配列（文字列） |
| `warnings` | 注意事項 |
| `date_mode` | `fixed` / `flexible` / `undated` |
| `commercial_offer` | 商業提案か（価格・予約リンク含むか） |

### 3.4 有効期限セマンティクス（候補）

```text
valid_until がある → その日時まで有効
valid_until がない → created_at + 1 year を既定の soft expiry（warning 用）
no_expiration: true → 期限なしを明示（既定 1 year を適用しない）
期限切れ → import 禁止ではなく warning（proposal show / materialize 時）
```

即時 import 禁止にしない理由: 過去の提案をアーカイブ参照する用途を残す。

### 3.5 Trip Proposal vs Proposal Fragment

| 観点 | Trip Proposal | Proposal Fragment |
|---|---|---|
| スコープ | 旅行全体 | 既存 Trip の一部 |
| 日付 | 未定可 | 既存 Day にアンカー or 未定 |
| materialize 先 | 新 Trip | 既存 Trip へ merge |
| 典型 source | AI 旅行プラン | レストラン提案・オプションツアー |

v4.7.2 / v4.7.3 で個別 concept spec に分割予定。

### 3.6 materialize gate（概念）

```text
入力: Proposal Envelope + 人間の採用パラメータ（start_date, day mapping, 選択肢）
処理: Envelope → schema v8 Trip JSON 生成 → trip import 相当
出力: Trip ID + materialize report（何を採用/却下したか）
```

v4.7.4 で validation rules を詳細化。v4.7.0 では gate の存在のみ確定。
