# v4.9.0 — Desktop transition and Trip metadata foundation (documentation-only)

| Release | v4.9.0（released — 本書） |

v4.8.x currency hardening を v4.8.18 で一区切りとし、**GUI / desktop app に向けた方針と Trip optional metadata の設計基礎**を明文化する documentation-first release です。

本書は **挙動変更・DB schema 変更・CLI / export schema 変更を行いません**。次の実装リリース（Trip metadata DB / service facade / desktop vertical slice）へ進めるための設計正本です。

## 1. Why now（背景）

これまで export / import / validate / diff / proposal apply を強化してきた。デスクトップアプリを目指すなら、これ以上データ交換機能を深掘りし続けるより、**GUI が使える公開境界**と **Trip 作成 UX** に進む。

| これまでの強み | GUI に足りないもの |
|---|---|
| 堅牢な CLI とデータ完全性 | Trip 新規作成フォーム向けの補助情報 |
| export / import / validate | service layer を UI から直接呼べる境界 |
| Proposal / Fragment apply | 日常操作（Trip / Day / Itinerary）の読み取り vertical slice |

v4.8.x（currency hardening）は完了扱い。minor unit / import strict は **defer**（GUI 移行より後でもよい）。

## 2. GUI transition policy

### 2.1 Priority shift

| 優先度 | 方針 |
|---|---|
| 下げる | export / import の追加強化（当面の主戦場にしない） |
| 上げる | Trip / Day / Itinerary の日常操作（list / detail / timeline） |
| 上げる | CLI と Desktop が同じ service layer を使う構成 |

### 2.2 Architecture principles

```text
Avoid:
  GUI → shell out → CLI command string → parse stdout

Prefer:
  GUI / CLI → services/* (use case facade) → domain / storage
```

- GUI から CLI コマンド文字列を叩く構成は避ける
- `src/services/mod.rs` を **use case facade** として整える方向を future work に置く
- 既存の `trip_list` / `trip_show` / `day_list` / `day_show` / `itinerary_*` は既に「terminal I/O なし」の雛形がある（これを拡張する）

### 2.3 First desktop vertical slice（方針のみ）

最初の desktop vertical slice は **read-only** から始める。

1. Trip list
2. Trip detail
3. Day timeline

書き込み（add / update / delete）・Tauri scaffold・cloud / login / sync は後続。

### 2.4 What GUI should optimize first

Trip 新規作成フォームでユーザーが自然に入れたくなる補助情報:

- 代表目的地（表示・検索・list 補助）
- 旅行先国（統計・初期入力補助）
- 既定通貨候補（Expense 入力の初期候補）

これらは **Venue / Expense / Currency の正本ではない**（§4）。

## 3. Trip optional metadata design

### 3.1 Fields（設計）

| Field | Type (想定) | Example | Purpose |
|---|---|---|---|
| `main_destination` | optional text | `Okinawa`, `Hawaii`, `New York` | 代表目的地。表示・検索・パーソナライズ・Trip list 補助 |
| `main_destination_country_code` | optional ISO 3166-1 alpha-2 | `JP`, `US`, `DE` | 国別統計、初期入力補助、default currency **suggestion** |
| `default_currency` | optional ISO 4217 | `JPY`, `USD`, `EUR` | Expense / Estimate / Receipt 入力時の初期候補 |

現状の `Trip`（`id`, `name`, `start_date`, `end_date`, `summary`, timestamps）にはこれらの列は存在しない。実装は後続リリース（§6.1）。

### 3.2 Optionality

- 3 フィールドとも **optional**
- 既存 Trip は metadata なしで **valid**
- NULL / 未設定を許容し、migration は既存行を壊さない（将来実装時）

### 3.3 Validation policy（将来実装時の方針）

| Field | 将来の検証方針（本書は方針のみ） |
|---|---|
| `main_destination` | 空文字は NULL 扱い。長さ上限は実装時に決める。Venue FK にはしない |
| `main_destination_country_code` | 設定時は ISO 3166-1 alpha-2（uppercase normalize）を想定。**形式検証（英字2文字）か ISO registry strict かは v4.9.0 では未確定**。Exact registry validation, reserved or historical code handling, and the registry source are decided in the implementation release（v4.9.1） |
| `default_currency` | 設定時は IsoStrict（CLI write path と整合）を想定 |

country → currency の **自動確定はしない**（suggestion UI のみ許可）。

### 3.4 Multi-country / multi-currency trips

- `main_destination` は **代表 1 件**のラベルであり、旅行中の全地点を列挙しない
- `main_destination_country_code` は代表国であり、複数国旅行を禁止・正規化しない
- `default_currency` は入力補助であり、Day / Expense ごとに別通貨を持つことを妨げない

### 3.5 Relation to existing `summary`

| Field | Role |
|---|---|
| `summary` | 旅行の短い説明文（既存） |
| `main_destination` | 代表目的地ラベル（新規・補助） |

`summary` に目的地を埋め込む運用は可能だが、GUI list / filter / suggestion には分離フィールドが向く。重複があっても矛盾解決の正本は置かない（ユーザー編集のまま）。

## 4. Boundaries（混同禁止）

### 4.1 Main Destination ≠ Venue

| Concept | Meaning |
|---|---|
| **Main Destination** | Trip レベルの代表目的地 **ラベル**（自由記述寄り） |
| **Venue** | 地理的な場所エンティティ（施設・POI）。[venue-model-introduction-policy.md](venue-model-introduction-policy.md) |

- Main Destination は Venue FK / Venue 正本ではない
- Venue model 導入を本 release で前提にしない
- Itinerary is not a venue 原則を維持する

### 4.2 Default Currency ≠ Currency ownership / Expense currency

| Concept | Meaning |
|---|---|
| **Default Currency** | Trip の入力補助（初期候補） |
| **Expense / Estimate / Receipt currency** | 各レコードの金額通貨（正本） |

- `default_currency` は Expense currency を **上書きしない**
- 明示された Expense / Estimate / Receipt currency が常に優先
- Currency ownership（誰がどの通貨を「持つ」か）とは無関係
- v4.8.x IsoStrict は CLI write path のまま。本 metadata はそれを置き換えない

### 4.3 Country code ≠ Currency

- 旅行先国と通貨は同一視しない（例: `DE` と `EUR`、`US` と `USD` はよくあるが固定ではない）
- country code から currency は **推定候補**にできるが、**自動確定しない**
- ユーザーが `default_currency` を明示するまで Expense 入力は従来どおり明示が必要（自動適用は out of scope）

### 4.4 Not a completed DB normalization

本 metadata は **GUI の初期入力補助**のための Trip 補助情報であり、国・都市・Venue・通貨マスタの正規化完成形ではない。

## 6. Future implementation candidates

### 6.1 Trip optional metadata DB implementation（v4.9.1 — 次の実装リリース）

```text
trips.main_destination
trips.main_destination_country_code
trips.default_currency
```

想定 surface:

- migration（NULL 許容、既存行 valid）
- `trip add` / `trip update` / `trip show` / JSON output
- 必要なら export schema 拡張は **別判断**（本 foundation では必須としない）

### 6.2 Desktop readiness service facade

`src/services/` を Desktop / CLI 共用の use case facade として整える。

候補 DTO / use cases:

| Candidate | Role |
|---|---|
| `TripSummary` | list 用の軽い要約（metadata 含む可） |
| `TripDetail` | 詳細表示 |
| `DayDetail` | Day 単位の詳細 |
| `ItineraryDetail` | Itinerary 単位の詳細 |
| structured errors | GUI が機械可読に扱えるエラー |
| DB path handling | `db use` / `--db` と整合した接続境界 |

既存: `trip_list`, `trip_show`, `day_list`, `day_show`, `itinerary_timeline` 等。

### 6.3 Desktop vertical slice

1. Tauri scaffold（または同等の desktop shell）
2. Trip list（read-only）
3. Trip detail（read-only）
4. Day timeline（read-only）
5. その後に書き込み UX / Trip 作成フォーム（metadata 入力）
