## Status（結論サマリ）

| 観点 | 判定 |
|---|---|
| **Adoption gate** | 人間が確認・決定したうえでだけ正式データ化する **門番** |
| **Materialize** | Envelope → **新規** schema v8 Trip（広い概念） |
| **Apply** | Fragment → **既存** schema v8 Trip へ反映（操作名候補） |
| **Blocking validation** | 必須情報欠如など — **採用を止める** |

**固定ルール（v4.7.0〜v4.7.3 から継承）:**

```text
schema v8 Trip = 採用済みの正式データ（実日付を持つ）
Proposal       = 候補案 — schema v8 の外側
Materialize    = 人間が採用したときだけ正式 Trip へ変換・反映する gate
```

## 1. Adoption gate とは何か

### 1.1 定義

**Adoption gate**（採用 gate）は、Trip Proposal Envelope または Proposal Fragment を **schema v8 の正式データにしてよいか** を人間が判断する境界です。

Materialize は「AI 提案を自動で旅程に流し込む仕組み」**ではない**。

```text
採用前:
  Proposal / Fragment
  schema v8 の外側
  trip export 対象外

Gate:
  人間の確認
  required decisions の確定
  blocking validation
  warnings / conflicts の確認

採用後:
  新規 schema v8 Trip
  または既存 schema v8 Trip の正式要素
```

### 1.2 Gate の責務

| 責務 | 説明 |
|---|---|
| **人間の明示採用** | 誤タップ・バッチ自動化による無批判取り込みを防ぐ |
| **必須情報の確定** | schema v8 に必要な日付・title 等を gate 前に揃える |
| **Validation** | blocking で止め、non-blocking で警告 |
| **Conflict 確認** | Fragment 特に — 既存 Trip との整合 |
| **監査可能性** | 何を採用/却下/保留したか（将来 report 概念） |
| **schema v8 の保護** | `trip validate-export` の意味を曖昧にしない |

### 1.3 用語（概念）

| 用語 | 意味 |
|---|---|
| **adoption gate** | 採用判断の境界の **総称** |
| **materialize** | Proposal を正式 Trip にする **広い概念**；Envelope → 新 Trip |
| **apply** | Fragment を既存 Trip に反映する **操作名候補** |
| **reject** | 採用しない（Proposal は外側に残す or 破棄は将来実装） |
| **defer** | あとで見る（未採用のまま保持） |

コマンド名・JSON schema は v4.7.4 では確定しない。

## 2. 二つの採用ルート

Envelope と Fragment は **同じ adoption gate の思想** を持つ **別ルート** である。一つの変換処理に潰し込まない。

### 2.1 Trip Proposal Envelope → materialize

```text
入力: Trip Proposal Envelope + 人間の採用パラメータ
処理: required decisions 確定 → blocking validation → warnings 確認
出力: 新しい schema v8 Trip（ledger に追加）
```

| 観点 | 内容 |
|---|---|
| 既存 Trip | **不要** |
| 主な確定事項 | `start_date` / `end_date`、title、destination、Day 構成、Itinerary 採否 |
| 日付 | `date_policy` が undated / flexible でも、gate 通過時は **実日付必須** |

### 2.2 Proposal Fragment → apply / materialize

```text
入力: Proposal Fragment + target Trip + 人間の採用パラメータ
処理: target 確定 → intent 確認 → conflict / warning 確認 → blocking validation
出力: 更新された既存 schema v8 Trip
```

| 観点 | 内容 |
|---|---|
| 既存 Trip | **必須**（参照） |
| 主な確定事項 | target Trip / Day / Itinerary、intent、反映内容、既存データ変更の是非 |
| Fragment | **小さい Trip ではない** — 部分反映のみ |

### 2.3 比較表

| 観点 | Trip Proposal Envelope | Proposal Fragment |
|---|---|---|
| Gate 操作（候補名） | materialize | apply |
| 出力 | **新規** Trip | **既存** Trip 更新 |
| target 確定 | Day 構成・日付 | Trip / Day / Itinerary |
| conflict | 比較的少ない | **重要** |
| required decisions | 日付・全体構成 | target・intent・placement |

## 3. Gate の入出力（概念）

```text
Input:
  Trip Proposal Envelope
  Proposal Fragment
  （将来）人間の採用パラメータ — 日付、target、採否リスト等

Gate:
  human review
  required decisions
  blocking validation
  non-blocking warnings
  conflict confirmation
  warning acknowledgment（明示確認の概念）

Output（いずれか）:
  new schema v8 Trip
  updated existing schema v8 Trip
  rejected proposal（採用しない）
  deferred proposal（保留 — 未採用のまま）
```

**reject / defer** は概念として存在する。実装（コマンド・DB 状態）は v4.7.4 scope 外。

## 4. Required decisions（採用前に人が決めること）

Gate は「正式データ化に足りない情報」を人間に決定させる。

### 4.1 Trip Proposal Envelope

| 決定事項 | 説明 | 典型の blocking |
|---|---|---|
| `start_date` / `end_date` | 実日付の確定 | 未確定なら **blocking** |
| `title` | Trip 名 | 欠如なら **blocking** |
| `destination` | 目的地 | 欠如なら **blocking**（推奨必須） |
| Day 構成 | 何日目に何をするか | 採用する Day / Itinerary の選択 |
| Itinerary 採否 | 候補行程の取捨選択 | 人間が明示 |
| `sort_order` / 時間帯 | 順序・時刻 | flexible 時は人間が確定 |
| 不足情報 | reservation / expense / note | 正式化に必須なら blocking、否则 warning |

`date_policy` が `undated` / `flexible_dates` でも、**gate 通過後の schema v8 Trip には実日付が要る**。

### 4.2 Proposal Fragment

| 決定事項 | 説明 | 典型の blocking |
|---|---|---|
| target Trip | どの Trip へ | 存在しない → **blocking** |
| target Day / Itinerary | 差し込み先 | unresolved から人間が確定 |
| `intent` | add / enrich / replace 等の確認 | 誤解防止 |
| 反映内容 | 何を正式要素にするか | 必須フィールド欠如 → blocking |
| 既存データ変更 | enrich / replace で上書きするか | 人間が明示 |
| conflict 対応 | overlap / duplicate 等 | 通常 **non-blocking** — 人間が判断 |

### 4.3 materialize hints / adoption hints との関係

v4.7.2 / v4.7.3 で定義した hints は、gate 入力の **事前計算候補** として使える。

```text
Envelope.materialize hints.required_decisions
Fragment.adoption hints.required_decisions
  → gate UI / CLI が「まだ決めること」を表示
```

## 5. Validation rules

### 5.1 二層モデル

```text
blocking validation:
  正式データ化できない — 採用を止める

non-blocking warning:
  古い・曖昧・衝突可能性 — 人間が理解すれば採用可能
```

**原則:**

- 期限切れ・古さ → **warning のみ**（自動破棄しない）
- schema v8 に必要な必須情報欠如 → **blocking**
- 衝突の「可能性」→ 通常 **warning**（人間が確認）

### 5.2 Blocking validation（例）

#### Trip Proposal Envelope → 新規 Trip

| ルール | 理由 |
|---|---|
| `start_date` / `end_date` 未確定 | schema v8 Trip に実日付必須 |
| `title` 欠如 | 識別不能な Trip |
| `destination` 欠如 | 公開契約上の最小情報（推奨 blocking） |
| 採用 Day / Itinerary が空 | 中身のない正式 Trip |

#### Proposal Fragment → 既存 Trip

| ルール | 理由 |
|---|---|
| target Trip が存在しない | 反映先なし |
| target Day / Itinerary が削除済み | `target_missing` — blocking |
| 正式要素に必要な必須フィールド欠如 | export 契約違反のまま反映不可 |
| `intent: warning` のみで反映内容なし | 注意喚起だけなら Trip 変更不要 — 採用パス設計は将来 |

### 5.3 Non-blocking warning（例）

[v4.7.2](v4.7.2-trip-proposal-envelope-concept-spec.md) / [v4.7.3](v4.7.3-proposal-fragment-concept-spec.md) の conflict 概念を gate で **確認対象** として扱う。

| Warning | 意味 |
|---|---|
| `valid_until` 超過 / `created_at + 1 year` 超過 | 提案が古い可能性 |
| `stale_source` | 料金・営業時間・空席等が古い可能性 |
| `time_overlap` | 時間帯の重なり可能性 |
| `duplicate_candidate` | 似た予定が既にある可能性 |
| `sort_order_conflict` | 差し込み位置が曖昧 |
| `date_mismatch` | 候補日と Trip 日程のずれ |
| `reservation_ambiguity` | 予約として未確定 |
| `expense_ambiguity` | 金額・通貨が未確定 |

Gate ではこれらを **表示し、人間の acknowledgment（理解した）を促す** 概念とする。自動却下しない。

### 5.4 Blocking vs warning の判定フロー（概念）

```text
1. required decisions がすべて埋まっているか
   → いいえ: blocking（gate 通過不可）

2. blocking validation ルール
   → 失敗: 採用停止、不足を表示

3. non-blocking warnings を列挙
   → 人間が確認（将来: explicit ack）

4. 人間が明示採用
   → schema v8 Trip 生成 or 既存 Trip 更新
```

## 6. Expiry / stale source と gate

[v4.7.2](v4.7.2-trip-proposal-envelope-concept-spec.md) / [v4.7.3](v4.7.3-proposal-fragment-concept-spec.md) の Expiry 方針を継承。

```text
valid_until がある     → soft expiry として warning
valid_until がない     → created_at + 1 year を既定 warning
no_expiration: true    → 既定 1 year を適用しない
期限切れ               → warning のみ — 自動破棄・import 禁止にしない
```

### 6.1 Gate での見せ方（概念）

| 段階 | 扱い |
|---|---|
| Proposal 一覧 | expiry 近い / 超過に **badge または warning 表示** |
| Gate 画面 | `stale_source` を **non-blocking** で列挙 |
| 採用時 | 「内容が古い可能性があります」— 人間が続行可否を判断 |
| 採用後 | 正式 Trip として保持 — expiry は Proposal 側の metadata |

期限切れを理由に **blocking にしない**（必須日付欠如など別ルールは blocking のまま）。

## 7. schema v8 との境界

### 7.1 変更しないもの

```text
schema v8:
  採用済み Trip の正本 — 変更なし

trip export:
  schema v8 Trip のみ出力

trip validate-export:
  schema v8 契約の conformance gate

trip import:
  既に schema v8 準拠の JSON 用 — Proposal の代替ではない
```

### 7.2 Gate 出力の契約

Gate 通過後に生成・更新されるデータは **schema v8 に準拠** している必要がある。

```text
Gate 出力 → trip validate-export で通る状態
           （実装時の目標 — v4.7.4 では概念のみ）
```

Proposal / Fragment 自体は validate-export の対象外。

### 7.3 混同を避ける

```text
誤: gate 通過前の候補を trip export に含める
誤: placeholder Trip として DB に候補を入れ、後で gate
正: 外側で保持 → gate → 正式 Trip のみ ledger / export
```

## 8. 人間の採用判断が必要な理由

| 理由 | 説明 |
|---|---|
| **AI/provider は誤りうる** |  hallucination、古い情報、商業バイアス |
| **旅行は個人の選択** | 採用は旅行者の意思決定 |
| **正式データの信頼** | export / 共有 / 将来 GUI は採用済みを前提 |
| **conflict** | Fragment は既存計画と衝突しうる |
| **日付・target 未確定** | 機械では推測しすぎると危険 |

```text
Your travel data should belong to you.
AIが提案し、人が選び、旅の記録として残る。
```

Gate はこの narrative の **技術的な門番** である。
