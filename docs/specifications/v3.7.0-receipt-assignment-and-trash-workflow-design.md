# v3.7.0 Receipt Assignment and Trash Workflow Design

v3.6.0 で導入した **Receipt Inbox（metadata-only）** の次段階として、**Receipt assignment / trash workflow** を整理する **workflow design** です。

位置づけ:

```text
v3.7.0 candidate
workflow design
documentation-only
implementation deferred
```

重要な前提（v3.6.0 方針の継続）:

```text
Receipt Inbox = Expense 化待ちの未整理支払い候補
Receipt は Actual ではない
Receipt は Planned vs Actual / trip stats / export-md に含めない
Receipt image / Evidence / Attachment は future work
```

---

## 1. Summary

v3.7.0 候補では、Receipt Inbox の user workflow を次のように整理する。

```text
Receipt Inbox は Expense 化待ちの未整理支払い候補である。
Receipt は Itinerary に link するのではなく、Itinerary に assign されることで Expense になる。
不要な Receipt は ignored ではなく trash に送る方向で検討する。
```

狙いは、status を増やすことではなく、GUI / CUI での判断と操作を **「必要 / 不要 / 保留」** に寄せて単純化すること。

---

## 2. Current v3.6.0 model

v3.6.0 の現状（実装済み）:

```text
Receipt status:
  unreviewed
  ignored
```

```text
Implemented:
  receipt add/list/show/update/ignore/delete
  Trip-level receipts[]
  export schema v7
  v6 import compatibility
```

```text
Not implemented:
  receipt link
  receipt convert / promote
  image_path
  Evidence / Attachment
```

---

## 3. Problem with `ignored`

`ignored` は v3.6.0 の lifecycle 単純化（`unreviewed` / `ignored`）の中で有効だったが、将来 `trash` 概念を導入すると役割が重複しやすい。

論点:

- `ignored`: 対象外として **残す**
- `trash`: 不要なので **Inbox から退避**（見えなくする / 後で復元できる余地）
- GUI / CUI では、ユーザーが `ignored` と `trashed` を使い分ける負担が高い
- 家族旅行用途では、監査的に「対象外として残す」必要性は高くない
- Receipt Inbox の主目的は「未整理支払いを Expense 化する」か「不要なら取り除く」こと
- `ignored` を残すと Inbox lifecycle が複雑になり、UX が悪化する

結論候補:

```text
If trash is introduced, ignored status should be deprecated or removed from the future Receipt lifecycle.
```

---

## 4. Proposed future Receipt lifecycle

将来の lifecycle は status よりも **「Inbox にある / Trash にある」** で整理する案を優先する。

概念:

```text
Receipt Inbox:
  active receipt   = 未整理（未割り当て）
  trashed receipt  = 不要としてゴミ箱にある
```

DB 的な表現候補:

```text
receipts
  ...
  trashed_at optional
```

意味:

```text
trashed_at IS NULL     -> Inbox に残る active Receipt（未整理）
trashed_at IS NOT NULL -> Trash にある Receipt
```

この方向では:

- `status` 自体を削る（または最小化する）余地がある
- 少なくとも `ignored` は廃止候補として扱える

---

## 5. Core user operations

Receipt 側の user operation を以下に整理する。

```text
assign  = Receipt を Itinerary に割り当てて Expense 化する
trash   = Receipt を不要としてゴミ箱へ移動する
update  = 未整理 Receipt の情報を修正する
delete/purge = ゴミ箱から完全削除する（future）
```

GUI イメージ:

```text
Receipt Inbox row:
  [Expense にする] [ゴミ箱へ] [編集]
```

CLI naming candidate（実装は今回しない）:

```bash
caglla receipt assign <receipt_id> --itinerary <itinerary_id>
caglla receipt trash <receipt_id>
caglla receipt restore <receipt_id>
caglla receipt purge <receipt_id>
```

---

## 6. Receipt assign semantics

`assign` の意味を明確にする。

```text
Receipt を Itinerary に link するのではない。
Receipt を Itinerary に assign することで Expense を作成する。
```

処理イメージ（transaction 前提）:

```text
BEGIN
  create Expense from Receipt
  delete Receipt from Inbox, or mark it as resolved by removal
COMMIT
```

初期方針（推奨）:

```text
metadata-only Receipt は assign 後に Receipt レコードを削除する（Inbox item は一時的）
```

理由:

- Receipt は一時的な Inbox item
- Actual は Expense が担う
- converted status を持つと lifecycle が再び複雑になる
- 画像証憑がない metadata-only Receipt では、Receipt を残す価値が限定的

補足（将来）:

- Evidence / Attachment が導入された場合、Receipt を残す意味論は再検討が必要（証憑が Receipt に紐づく可能性）

---

## 7. Expense correction routes（future work）

Expense 側の修正ルートは将来の設計論点として整理する（本書では実装しない）。

将来候補:

```text
expense reassign = Expense の Itinerary 割り当て先を変更する
expense unassign = Expense を Receipt Inbox に戻す
expense trash    = 誤登録・不要な Expense をゴミ箱へ移動する
```

意味:

| Operation | Meaning |
|---|---|
| `expense reassign` | 支払いは Expense として正しいが、Itinerary が違った |
| `expense unassign` | Expense として確定できなくなったため Inbox に戻す |
| `expense trash` | そもそも不要・誤登録なので捨てる |

---

## 8. Trash vs delete vs ignore

3つの違い:

```text
ignore       = 対象外として残す
trash        = 不要なのでゴミ箱へ移す
delete/purge = 完全削除
```

今回の方向性:

```text
Receipt Inbox においては、ignore より trash の方が自然。
ignored status は将来的に廃止候補。
```

ただし v3.6.0 互換を壊さないため、移行方針は implementation plan で決定する（本書は workflow design のみに留める）。

---

## 9. Receipt Inbox summary / pending sum（future work）

Receipt Inbox では、未整理 Receipt の合計表示（pending sum）が有用である。

目的:

```text
まだ Expense 化されていない支払い候補がいくら残っているかを確認する
```

表示候補:

```text
Unassigned Receipts:
  count
  total by currency
  without amount count
```

重要:

```text
Receipt sum は Actual ではない。
Receipt sum は Planned vs Actual には含めない。
```

概念整理:

```text
Planned = Estimate
Actual  = Expense
Pending = active Receipt Inbox total
```

例（補助表示の候補）:

```text
Confirmed Actual: JPY 48,300
Pending Receipts: JPY 6,700
Potential Actual: JPY 55,000
```

この表示は future work とする（v3.7.0 で実装するかは別途 implementation plan で判断）。

## 10. Settlement / Balance relationship（deferred）

将来 Balance / Settlement を行う場合、未整理 Receipt が残っていると精算結果が変わる可能性がある。

将来の警告例:

```text
There are pending receipts totaling JPY 6,700.
Settlement may be incomplete until they are assigned or trashed.
```

ただし **Balance / Settlement は引き続き deferred**（本書では workflow design の論点整理のみ）。

## 11. Evidence / Attachment relationship（future work）

画像証憑は引き続き future work。

重要な方針:

```text
Receipt にだけ image_path を持たせない
Expense にだけ image_source を持たせない
画像証憑は将来 Evidence / Attachment 共通レイヤーで検討する
```

将来案:

```text
Evidence / Attachment
  can belong to Receipt
  can belong to Expense
```

## 12. Migration considerations（future work）

v3.6.0 では `ignored` status が存在するため、将来 `trash` に移行する場合の方針を整理する（実際の migration は implementation plan で決定）。

候補:

```text
ignored -> trashed
```

または:

```text
ignored -> active receipt, but excluded from default list
```

推奨候補:

```text
ignored -> trashed
```

理由:

- v3.6.0 の ignored は「対象外として処理済み」という意味
- 新設計では対象外 Receipt は trash に送るのが自然
- Inbox summary（pending sum）からも除外しやすい

## 14. Recommended direction

結論:

```text
v3.7.0 should focus on workflow design before implementation.

Receipt Inbox should move toward a simpler lifecycle:
- active pending Receipt
- trashed Receipt

Receipt actions should be:
- assign to Expense
- trash
- update

The v3.6.0 `ignored` status should be treated as a migration target rather than a long-term lifecycle state.
```
