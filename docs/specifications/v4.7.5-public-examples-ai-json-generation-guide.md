## 1. 目的

v4.7.2〜v4.7.4 で概念整理した以下を、**実装者・provider・生成 AI** が参照できる形に落とす。

```text
schema v8 Trip           — 採用済み正式データ
Trip Proposal Envelope   — 旅行全体の未採用案
Proposal Fragment        — 既存 Trip への部分提案
Materialize / Apply      — 人間の採用 gate
```

v4.7.5 の核心は「AI にきれいな JSON を作らせる」ことではなく、**未確定情報を正式データとして混ぜない** こと。

---

## 2. 三層モデルの再掲

```text
Layer A — Candidates (schema 外)
  Trip Proposal Envelope
  Proposal Fragment

Layer B — Adoption gate
  human review / validation / warnings

Layer C — Canonical (schema v8)
  trip export / trip validate-export
```

---

## 3. schema v8 Trip を直接生成してよい条件

| 条件 | 必須 |
|---|---|
| `start_date` / `end_date` 確定 | ✓ |
| `title` / `destination` 明確 | ✓（destination は推奨 blocking） |
| Day 構成確定 | ✓ |
| Itinerary 採用済み | ✓ |
| 予約・費用・メモ | 載せるなら **事実確認済み** |

### 3.1 直接生成してはいけない場合

- 日付未定 → **Trip Proposal Envelope**
- 既存 Trip への部分案 → **Proposal Fragment**
- 予約・料金未確認 → Envelope / Fragment + `missing_fields`
- 複数案比較段階 → Envelope（並列保持）

---

## 4. Trip Proposal Envelope を生成すべき条件

[v4.7.2](v4.7.2-trip-proposal-envelope-concept-spec.md) 準拠。

```text
旅行全体の未採用案
date_policy: fixed_dates | flexible_dates | undated
materialize_hints.required_decisions / missing_fields / warnings
```

AI 出力は **validate-export 非対象**。

---

## 5. Proposal Fragment を生成すべき条件

[v4.7.3](v4.7.3-proposal-fragment-concept-spec.md) 準拠。

```text
既存 Trip 参照（target）
intent: add | enrich | replace_candidate | reorder_hint | warning
adoption_hints.conflicts / warnings
target_type: unresolved 可
```

Fragment ≠ 小さい Trip。

---

## 6. 捏造禁止と不明値

### 6.1 捏造してはいけない例

```text
confirmation_code（予約番号）
amount / currency（未確認の料金）
営業時間・定休日
空席・在庫
便名・ゲート（未確認）
```

### 6.2 不明値の逃がし先

| フィールド（概念） | 用途 |
|---|---|
| `missing_fields` | 正式化に必要だが未入手の情報 |
| `assumptions` | AI/provider が明示する前提・推測 |
| `warnings` | 古さ・衝突可能性・要確認 |

```text
推測で schema v8 フィールドを埋めない
推測は assumptions に隔離
```

---

## 7. Materialize gate 前提の出力ルール

[v4.7.4](v4.7.4-materialize-gate-concept-validation-rules.md) 準拠。

```text
AI 出力 alone ≠ 採用済み
gate 通過後のみ schema v8 ledger / export

blocking validation 失敗 → 採用停止
non-blocking warning → 人間が判断（自動却下しない）
```

### 7.1 AI 出力後の想定フロー

```text
1. AI → Envelope / Fragment / （条件を満たす場合のみ）schema v8 草案
2. 人間 → review
3. gate → materialize / apply / reject / defer
4. schema v8 Trip → validate-export
```

## 8. validate-export narrative

| JSON | validate-export |
|---|---|
| schema v8 Trip | **対象** — public conformance gate |
| Trip Proposal Envelope | 対象外 |
| Proposal Fragment | 対象外 |
| gate 後の Trip | 対象 |

```bash
trip validate-export export.json
```

Proposal をこのコマンドに通すよう AI / provider に指示しない。

## 11. プロンプト設計原則

| # | 原則 |
|---|---|
| 1 | 採用済み事実と候補案を混ぜない |
| 2 | 日付未定 → Envelope、部分案 → Fragment |
| 3 | 不明は missing_fields — 推測で埋めない |
| 4 | 予約・料金・営業時間を invent しない |
| 5 | gate bypass 禁止 |
| 6 | validate-export は schema v8 のみ |
| 7 | warnings / assumptions を積極的に使う |
