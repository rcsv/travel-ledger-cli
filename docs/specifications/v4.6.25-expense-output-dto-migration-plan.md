## 1. 現行構造の再整理

### 関係図

```text
Expense                    = DB model に近い domain data（expenses テーブル 1 行）
ExpenseJson                = CLI JSON output DTO（--json の正典 shape）
expense_to_json(&conn, &expense)
                           = Expense + DB context から ExpenseJson を作る helper
print_expense_detail       = expense_to_json を使う human show helper
print_expense_list         = payer name を DB lookup する human list helper
ExpenseListJson            = CLI JSON wrapper（trip_id / itinerary_id + expenses[]）
```

### read-only フロー（現状）

```text
expense list:
  service → ExpenseListServiceResult { target, expenses: Vec<Expense> }
  --json  → CLI: 各件 expense_to_json → ExpenseListJson → print_json
  human   → print_expense_list(&conn, target, &expenses)

expense show:
  service → ExpenseShowServiceResult { expense }
  --json  → expense_to_json → print_json
  human   → print_expense_detail → expense_to_json 経由
```

### `Expense` 単体で `ExpenseJson` を作れない理由

`Expense`（`src/domain/models.rs`）は `expenses` 行のみ。次は **DB 追加 lookup または導出** が必要:

| `ExpenseJson` フィールド | `Expense` にあるか | 追加 context |
|---|---|---|
| scalar 群（id, amount, …） | ✓ コピー | — |
| `paid_by_participant_name` | ✗ | `get_participant(conn, paid_by_participant_id)` |
| `beneficiaries[]` | ✗ | `list_beneficiaries_for_expense` + 各 `get_participant` |
| beneficiary `.name` | ✗ | participant lookup |
| `shared` | ✗ | `expense_is_shared(beneficiaries.len())` — `count > 0` |

`paid_by_name` は `Expense` にあり、追加 DB なし。

### `expense_to_json` の DB アクセス（確認済み）

```text
list_beneficiaries_for_expense(conn, expense.id)
get_participant(conn, paid_by_participant_id)     // payer name
get_participant(conn, b.participant_id)           // per beneficiary
expense_is_shared(beneficiary_count)              // 導出のみ
```

## 2. Option C の具体化

### 設計原則

```text
service layer:
  - Expense 本体
  - payer context（participant 名）
  - beneficiaries context（participant_id + name + sort_order）
  - shared 判定に必要な情報（beneficiary count または shared: bool）
  - list / show 両方で再利用可能な enriched parts
  - ExpenseJson は返さない（output DTO は service の正典にしない）
  - terminal I/O なし

CLI layer:
  - service enriched result から既存 ExpenseJson を組み立てる mapper
  - ExpenseListJson wrapper 組み立ては現状どおり CLI
  - JSON shape 完全維持
  - human: print_* は enriched parts から組み立てるか、mapper 経由で互換維持
```

### 推奨 enriched parts 構造（概念・未実装）

**単一 expense 用（show / list 共通要素）:**

```text
ExpenseEnrichedPart {          // 仮称 — 本 release では型を作らない
    expense: Expense,
    paid_by_participant_name: Option<String>,
    beneficiaries: Vec<ExpenseBeneficiaryPart>,
    shared: bool,
}

ExpenseBeneficiaryPart {       // 仮称
    participant_id: i64,
    name: String,
    sort_order: i32,
}
```

**list service result（Phase 2 想定）:**

```text
ExpenseListEnrichedServiceResult {   // 仮称
    target: ExpenseListTarget,
    items: Vec<ExpenseEnrichedPart>,
}
```

**show service result（Phase 2 想定）:**

```text
ExpenseShowEnrichedServiceResult {   // 仮称
    item: ExpenseEnrichedPart,
}
```

### CLI mapper（Phase 3 想定）

```text
fn expense_enriched_part_to_json(part: &ExpenseEnrichedPart) -> ExpenseJson
  // conn 不要 — part に既に name / shared が入っている
  // フィールド対応は現行 expense_to_json と 1:1
```

`ExpenseListJson` 組み立て:

```text
trip_id / itinerary_id  ← target（現状どおり CLI）
expenses                ← items.iter().map(expense_enriched_part_to_json)
```

### enrichment の実装場所（Phase 2）

service 内で private helper を呼ぶ想定:

```text
fn enrich_expense(conn, expense: &Expense) -> Result<ExpenseEnrichedPart>
  - list_beneficiaries_for_expense
  - get_participant（payer + beneficiaries）
  - expense_is_shared
  - terminal I/O なし
```

現行 `expense_to_json` の DB ロジックを **service 側 enrichment に移す**のが Option C の本質。`ExpenseJson` 組み立ては CLI mapper に残す。

### list の N 件 enrichment

```text
list_expenses:
  expenses を load（現状どおり）
  各 expense を enrich_expense — O(N × (1 + B)) lookup
  ordering は現状維持（itinerary_id, sort_order, id）
```

パフォーマンスは初号機規模では許容。将来 batch lookup は Phase 2 以降の最適化候補（本 plan では必須にしない）。

## 3. 新しい構造名の候補（比較のみ・今回は作らない）

| 候補名 | 意味 | 向き | 不向き |
|---|---|---|---|
| `ExpenseOutputContext` | 出力用 context 全体 | JSON/human 両方の upstream と明示 | 「output」が wire DTO と混同しやすい |
| `ExpenseDisplayContext` | 表示用 context | human 寄りの意図が伝わる | JSON 正典にも使うので狭い |
| `ExpenseResolvedContext` | DB 解決済み context | lookup 完了のニュアンスが明確 | やや長い |
| `ExpenseEnrichment` | enrichment 結果 | service 層の動詞と一致 | 単数/複数の命名要検討 |
| `ExpenseWithContext` | Expense + 付帯情報 | 直感的 | 「context」の範囲が曖昧 |
| `ExpenseBeneficiaryPart` | beneficiary 1 件分 | 配列要素として自然 | payer 用の対になる名前も要る |
| `ExpensePayerContext` | payer 部分のみ | 分割設計時 | 単体では list/show に不足 |

**推奨（将来実装時）:**

```text
親: ExpenseEnrichedPart または ExpenseResolvedContext
子: ExpenseBeneficiaryPart（ExpenseBeneficiaryJson とフィールド対応を維持）
service result: ExpenseListEnrichedServiceResult / ExpenseShowEnrichedServiceResult
```

`ExpenseJson` / `ExpenseBeneficiaryJson` は **wire format 名として維持**。enriched parts は service–CLI 間の内部構造。

## 5. JSON 互換性の守り方

### 不変 gate（全 Phase 共通）

```text
ExpenseJson の field name / nesting / optional field / shared 判定は変えない
ExpenseListJson の wrapper shape（trip_id, itinerary_id, expenses）は変えない
expense show --json と expense list --json の出力 shape は変えない
serde skip_serializing_if 等の既存挙動を維持
JSON export golden は変更なしを維持する
```

### mapper 実装時のチェックリスト（将来）

```text
□ ExpenseJson 全フィールド 1:1 対応
□ paid_by_participant_id が None のとき paid_by_participant_name も None
□ beneficiaries 空のとき shared == false
□ beneficiaries 非空のとき shared == true
□ beneficiary sort_order 維持
□ list ordering 不変
□ ExpenseListJson の trip_id / itinerary_id の排他（Trip vs Itinerary target）
```

### 移行前後の検証方針

```text
同一 DB に対し expense list/show --json を Phase 3 前後で比較
差分ゼロを merge gate とする
既存 expense.rs 内 JSON roundtrip tests を mapper にも適用
```

## 6. human output 互換性の守り方

### 不変 gate

```text
print_expense_detail の表示ラベル・順序・内容を変えない
print_expense_list の表形式・payer 列を変えない
payer: paid_by_participant_name 優先、なければ paid_by_name
shared 行: beneficiaries 名の comma join（detail のみ）
list では beneficiaries / shared 非表示（現状どおり）
format_amount_display / fmt_optional_text は原則そのまま
```

### Phase 3 での human 接続案

**案 A（推奨）:** `print_expense_detail_from_enriched(part)` を新設し、内部は enriched → ExpenseJson 相当を組み立ててから現行 println ロジックを再利用。

**案 B:** `print_expense_detail` が enriched を受け取るオーバーロードを追加。旧 `&conn` 版は Phase 4 まで write path 用に残す。

**plan では human output の改善・再設計には進まない。**

## 7. reservation show との違い

| 観点 | reservation show | expense（Option C） |
|---|---|---|
| service context | `day_number`, `itinerary_title`（2 フィールド） | `paid_by_participant_name`, `beneficiaries[]`, `shared` |
| JSON との関係 | JSON は `Reservation` のみ（context は human） | **JSON 正典が enriched**（`ExpenseJson`） |
| lookup 複雑度 | 1 itinerary item | 1 + N beneficiaries + payer |
| human helper | conn 不要（`print_*_with_context`） | 現状 conn 必要 → Phase 3 で enriched 化 |
| 統一 | 参考にできるのは「context in service」の方向性のみ | **別 migration plan 必須**（本書） |

```text
reservation show の context は表示補助として単純。
expense output context は JSON shape と human output の両方に関係する。
expense は shared expense model と participant lookup を含む。
reservation show と同じフィールド追加では済まない。
```
