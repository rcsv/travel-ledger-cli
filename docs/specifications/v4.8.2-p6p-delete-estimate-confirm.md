## 1. Confirm transaction

```text
with_transaction("delete_estimate confirm")
  1. gate report から preview / estimate_delete_preview を取得
  2. Fragment 再 parse（intent == delete_estimate）
  3. required_decisions 非空チェック
  4. parse_delete_estimate_fields（dry-run 共有）
  5. Itinerary target 再 resolve
  6. Estimate 再取得 + 親 itinerary_id 検証
  7. build_estimate_delete_preview で recomputed snapshot
  8. gate preview agreement 検証
  9. TOCTOU 再検証（expected_* + snapshot field 比較）
 10. exactly one DELETE（WHERE id + itinerary_id）
 11. post-delete verification（row not found）
 12. deleted_estimate_id 設定
 13. commit
```

失敗時は **rollback**、DB 不変。

## 2. Parser / dry-run 契約の再利用

confirm 専用の別解釈は作らない。v4.8.1 dry-run と共有:

* `parse_delete_estimate_fields`
* Itinerary target resolver（ID-first → sort_order fallback → ambiguity reject）
* `detect_delete_estimate_baseline_conflicts`
* `build_estimate_delete_preview`
* mutation field 拒否 / missing estimate blocking 契約

---

## 3. Preview agreement

transaction 内で gate preview と recomputed snapshot を比較:

* `estimate_delete_preview.target_itinerary_id`
* `estimate_delete_preview.target_estimate_id`
* snapshot fields: `title` / `amount` / `currency` / `note` / `sort_order` / `updated_at`

`target_itinerary_title` は identity 正本ではない（agreement 対象外）。

---

## 4. DELETE 契約

```sql
DELETE FROM estimates WHERE id = ? AND itinerary_id = ?
```

| affected rows | 扱い |
|---|---|
| 0 | blocking / rollback |
| 1 | post-delete verification へ |
| 2+ | exactly one contract 違反 / rollback |

`estimate::delete_estimate_row_scoped` を使用。

---

## 5. Success report

成功時のみ:

```json
{ "deleted_estimate_id": <i64> }
```

* dry-run / failure 時は unset
* preview 対象 Estimate と実際に削除した Estimate が一致
* 同一 Itinerary の sibling Estimate / 他 Itinerary の Estimate / Expense 等は不変
* `estimates_before` / `estimates_after` は v4.8.1 dry-run preview 契約と整合（after = before − 1）

## 7. Confirm safety tests（v4.8.2）

### Unit（`apply.rs`）

| Contract | Test |
|---|---|
| Success DELETE | `confirm_delete_estimate_writes_db` |
| Candidate / preview mismatch | `confirm_delete_estimate_candidate_preview_mismatch_blocks_db_write` |
| TOCTOU / gate snapshot drift | `confirm_delete_estimate_toctou_blocks_db_write` |
| Scoped DELETE 0 rows | `delete_estimate_row_scoped_rejects_zero_rows_for_missing_estimate` / `_wrong_itinerary_id` |

CLI からの直接再現が困難な candidate/preview mismatch は production fault injection なしの unit test で検証（`update_estimate` / `delete_itinerary` と同型）。

### Integration（`fragment_apply_cli`）

| Category | Tests |
|---|---|
| Success | `basic_confirm_deletes_estimate`, `sibling_and_foreign_estimates_unchanged_on_confirm`, `itinerary_title_change_confirm_still_succeeds` |
| Blocking / rollback | `confirm_second_attempt_blocks_as_not_found`, `not_found_confirm`, `baseline_mismatch_confirm`, `stale_row_confirm_blocks_without_db_write`, `mutation_field_confirm`, `required_decisions_confirm`, `cross_itinerary_confirm` |

dry-run blocking cases は v4.8.1 CLI tests を共有。confirm テストの重点: transaction、scoped DELETE、preview agreement、TOCTOU、rollback、対象外データ不変性。

`stale_row_confirm`: gate 後に Estimate を更新し、fragment の `expected_*` を旧値のまま指定 → baseline mismatch blocking（DB 不変）。
