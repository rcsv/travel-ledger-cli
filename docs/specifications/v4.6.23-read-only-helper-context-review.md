# v4.6.23 — Read-only helper context review

v4.6.22 で read-only 主軸 command の service boundary 完了を確認した後、**service の外側に残る helper 側 DB context 取得**を棚卸しし、GUI / Tauri API 化前の DTO / context ownership 方針を整理する **documentation-only review** です。

| Item | Status |
|---|---|
| Release | v4.6.23（documentation-only） |
| Scope | read-only 主軸導線の helper-side `&Connection` 使用、JSON/human 分類、許容状態、reservation vs expense 対比、itinerary aggregate 関係 |
| 実装 | **なし** — helper 変更 / DTO 実装 / service 追加は行わない |

関連:

| Version | 内容 |
|---|---|
| v4.6.22 | read-only service boundary completion review（16 modules 完了） |
| **本書** | **v4.6.23** — helper context 棚卸し |
| v4.6.24（候補） | expense DTO context ownership review / itinerary show aggregate boundary review |

## Status

| 観点 | 判定 |
|---|---|
| read-only 主軸 service boundary | **完了**（v4.6.22 確認済み） |
| helper 側 DB context | **一部残存** — 主に Expense 系 |
| 現時点の許容性 | **CLI 互換性維持のため許容** |
| GUI / Tauri API 化前 | **DTO / context ownership の方針整理が必要** |
| 最大の設計論点 | **ExpenseJson 系**、**itinerary show aggregate** |

**結論（推奨）:**

```text
read-only 主軸 command の service boundary は v4.6.22 で完了している。
一方で、一部 helper は DB connection を受け取り、service result だけでは揃わない JSON / human output を補完している。
これは CLI 互換性維持のため現時点では許容される。
ただし GUI / Tauri API 化の前に、特に ExpenseJson 系と itinerary show aggregate について
context ownership（service result vs helper vs CLI）の方針整理が必要。
今回は直さず、補助線を引く。
```

## Background

v4.6.4〜v4.6.21 で 16 read-only service module を追加した際、多くの command は次のパターンを踏襲した。

```text
service → structured ServiceResult（domain model 中心）
CLI     → JSON wrapper 組み立て or 既存型 serialize
CLI     → human display helper 呼び出し
```

ただし **Expense** は Shared Expense（participant / beneficiary）導入以降、JSON shape が `Expense` 単体を超える。v4.6.14 / v4.6.21 で service 化しても、**`expense_to_json(&conn, ...)` が CLI 層に残った**。

v4.6.23 では、この「service の外側の補助線」を棚卸しする。

## JSON / human helper 分類

### 分類軸

| 分類 | 意味 |
|---|---|
| **A. JSON shape helper** | `--json` 出力のフィールドを決める |
| **B. human display helper** | terminal 表示のみ |
| **C. 両方** | JSON 生成を human が再利用 |
| **D. format helper** | 日付・金額等の整形のみ（DB なし） |
| **E. DB context helper** | `&Connection` で追加 query する |

### 主軸 read-only の分類表

| Helper / 処理 | A JSON | B human | C 両方 | D format | E DB context | 備考 |
|---|---|---|---|---|---|---|
| `expense_to_json` | ✓ | | ✓ | | ✓ | **GUI API 化の最大影響** |
| `print_expense_detail` | | ✓ | ✓ | | ✓ | `expense_to_json` 経由 |
| `print_expense_list` | | ✓ | | | ✓ | payer 名のみ追加取得 |
| `ExpenseListJson`（CLI） | ✓ | | | | ✓ | wrapper は CLI、中身は helper |
| itinerary show Reservations（handler） | | ✓ | | | ✓ | JSON には含めない |
| `print_day_list_json` | ✓ | | | ✓ | | service `trip` + `days` で十分 |
| `print_day_show_json` | ✓ | | | ✓ | | service result で十分 |
| `NoteListJson`（CLI） | ✓ | | | | | service result で十分 |
| `ReservationListJson`（CLI） | ✓ | | | | | service result で十分 |
| `print_reservation_list` | | ✓ | | ✓ | | Trip human は service `trip_context` |
| `print_reservation_detail_with_context` | | ✓ | | ✓ | | context は service 由来 |

### `ExpenseJson` の構造（JSON shape への影響）

`Expense` domain model 単体では表現できないフィールド:

```text
paid_by_participant_name   ← participant テーブル lookup
beneficiaries[]            ← expense_beneficiaries + participant name
shared                     ← beneficiaries 件数から導出
```

`expense_to_json` は以下を DB から解決する:

```text
list_beneficiaries_for_expense(conn, expense.id)
get_participant(conn, participant_id)  // payer / 各 beneficiary
```

**したがって、`ExpenseShowServiceResult { expense }` だけでは CLI 互換の `ExpenseJson` を作れない。**

## Service boundary としての許容状態

### 許容マトリクス

| 箇所 | 現時点 | GUI 前の再検討 | JSON shape 不変で service へ寄せる |
|---|---|---|---|
| `expense_to_json` / list / show | **許容**（互換維持） | **必要** | **追加設計が必要**（DTO 拡張 or enrichment layer） |
| `print_expense_list` human payer | **許容** | 中（expense と一体） | service が payer 名を返せば可能だが JSON 方針と連動 |
| itinerary show Reservations | **許容**（human-only 既存互換） | **必要**（aggregate 論点） | JSON は現状 `ItineraryItem` のみ — 変更不要なら service 外のまま可 |
| day list/show JSON helpers | **許容** | 低 | service result が既に十分 |
| reservation list human `trip_context` | **許容** | 低 | **service が既に解決**（v4.6.13） |
| reservation show context | **許容** | 低（参考パターン） | service が既に解決（v4.6.20） |

### 今すぐ修正しない理由

```text
1. v4.6.14 / v4.6.21 で expense list/show を service 化する際、JSON / human output は一切変えないことが gate だった
2. expense_to_json は v3 Shared Expense 以来の確立済み JSON 変換であり、移すには DTO ownership の設計判断が先
3. itinerary show Reservations は human-only 互換の意図的 CLI 残置（v4.6.10 から維持）
4. 他の主軸 command は service result だけで CLI が完結するか、service が context を返している
5. いきなり service / DTO に寄せると JSON golden / CLI golden が変わるリスクがある
```

## reservation show と expense show の対比

### service result の形

**reservation show（v4.6.20）:**

```rust
ReservationShowServiceResult {
    reservation: Reservation,
    day_number: Option<i64>,
    itinerary_title: Option<String>,
}
```

- service が itinerary context を **解決して返す**
- JSON: `Reservation` のみ（context は serialize しない）
- human: `print_reservation_detail_with_context` が service context を使用（**conn 不要**）

**expense show（v4.6.21）:**

```rust
ExpenseShowServiceResult {
    expense: Expense,
}
```

- service は `Expense` 本体のみ
- JSON: CLI が `expense_to_json(&conn, &result.expense)` → `ExpenseJson`
- human: CLI が `print_expense_detail(&conn, &result.expense)` → 内部で `expense_to_json`

### 責務設計上の違い

| 観点 | reservation show | expense show |
|---|---|---|
| display context の所在 | **service result** | **helper（CLI 層）** |
| JSON に context を含むか | 含まない（`Reservation` のみ） | **含む**（`ExpenseJson` が participant 名等を含む） |
| human context 源 | service → print helper | helper が DB 再取得 |
| GUI が欲しい DTO | `Reservation` + optional context fields | **`ExpenseJson` 相当**（service からは未提供） |

### 標準パターンについて（今回は決めない）

```text
reservation show 型: 「domain + human 用 context を service が返し、JSON は domain のみ」
expense show 型:     「domain のみを service が返し、JSON 用 enriched DTO は helper が生成」

どちらを今後の標準にするかは、次の論点に依存する:
- GUI API が CLI --json と同一 shape を要求するか
- enriched DTO を service result に含めるか、別 mapper に置くか
- JSON shape 不変を gate に据えるか

v4.6.23 では揃えない。v4.6.24 以降の DTO / aggregate review で判断する。
```

## itinerary show aggregate との関係

### 現状（handler-side fetch）

`ItineraryAction::Show` human path（`src/main.rs`）:

```text
1. services::itinerary_show::show_itinerary → ItineraryItem
2. print_itinerary_detail(&result.item)
3. list_reservations_for_itinerary(&conn, id)  ← handler 直接
4. Reservations (N): ... を println!
```

| 観点 | 状態 |
|---|---|
| service result | `ItineraryItem` のみ |
| JSON | `ItineraryItem` のみ（予約なし） |
| human | itinerary detail + **Reservations サマリ** |
| context 取得場所 | **CLI handler**（helper 関数ではないが同型の「service 外補完」） |

### v4.6.23 でやらないこと

```text
itinerary show aggregate service 化
reservations / expenses / notes を含む itinerary detail DTO 実装
human output の再構成
JSON shape 変更
```

### aggregate との関係（known note）

itinerary show の Reservations は **aggregate boundary** の一例。expense helper context と並ぶ「service の外側の補完」だが、性質が異なる:

| 論点 | expense helper | itinerary show Reservations |
|---|---|---|
| 影響範囲 | JSON + human | **human-only** |
| GUI 初号機 | Expense detail 画面で **必須** | itinerary detail で reservation list 導線が別途ある |
| 次フェーズ | DTO context ownership | **aggregate boundary review**（別 ticket 推奨） |

## expense list / show service との関係

### expense list service

```text
ExpenseListServiceResult { target, expenses: Vec<Expense> }
```

| 出力 | CLI の処理 |
|---|---|
| `--json` | 各 `Expense` を `expense_to_json(&conn, e)` → `ExpenseListJson` |
| human | `print_expense_list(&conn, target, &expenses)` |

service は ordering / target 解決まで。enriched 表示は **すべて CLI/helper**。

### expense show service

```text
ExpenseShowServiceResult { expense: Expense }
```

| 出力 | CLI の処理 |
|---|---|
| `--json` | `expense_to_json(&conn, &result.expense)` |
| human | `print_expense_detail(&conn, &result.expense)` |

v4.6.21 の判断（正しかった点）: service 薄化 + 互換維持。  
v4.6.23 の論点: GUI API 化時に **同じ判断では不足**。

## 優先度付き — 将来 DTO / service result へ寄せる候補

| 優先度 | 対象 | 理由 |
|---|---|---|
| **高** | `expense_to_json` / `ExpenseJson` | GUI expense detail/list で CLI 互換 JSON が必要。service からは未提供 |
| **高** | `print_expense_list` payer 解決 | list human も conn 依存。expense と一体で整理 |
| **中** | itinerary show Reservations handler fetch | human aggregate。別導線で list は service 済み |
| **低** | day JSON helpers | service result 十分。conn 不要 |
| **低** | reservation list/show | service が context を既に返却。参考パターン |
