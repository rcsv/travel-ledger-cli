## CLI / service / repository boundary

### 責務分担（確立済みパターン）

```text
CLI layer (main.rs):
  - clap args 解釈
  - --json / human 選択
  - JSON wrapper 組み立て（NoteListJson, ExpenseListJson 等）
  - expense_to_json 等の変換（participant 名解決含む）
  - human display（print_*_list / print_*_detail）
  - human-only 追加セクション（itinerary show Reservations、checklist 見出し、timeline 見出し）
  - 成功メッセージ println!（write command）

service layer (src/services/):
  - use case orchestration
  - owner / target 解決（list 系）
  - structured ServiceResult
  - repository / domain 関数呼び出し
  - no terminal I/O

repository / domain layer:
  - DB query / mutation
  - domain model
  - ordering / validation
  - not found error message
```

### 意図的に CLI に残した判断（v4.6.11 以降も維持）

| 判断 | 理由 |
|---|---|
| **NoteListJson を CLI に残す** | JSON shape 互換。service は `owner_type` / `owner_id` / `notes` を返す |
| **ReservationListJson を CLI に残す** | JSON shape 互換。`trip_context` は human 専用 |
| **ExpenseListJson / expense_to_json を CLI に残す** | JSON shape 互換。participant 名解決は表示層の関心事 |
| **Checklist list 見出しを CLI に残す** | `旅行 ID {trip_id} のチェックリスト:` は terminal 文言 |
| **itinerary show Reservations** | JSON は `ItineraryItem` のみ。予約一覧は human-only |
| **timeline に --json 非新設** | 既存互換。GUI は service result をそのまま利用 |
| **DayListJson を CLI に残す** | v4.6.7 から維持。service は `Trip` + `Vec<Day>` |

### service が CLI option に寄りすぎていないか

| Module | CLI args を service に渡す | 評価 |
|---|---|---|
| `note_list` | `trip`, `day`, `itinerary` Option | 許容 — owner 解決は use case の一部 |
| `reservation_list` | `trip`, `itinerary` Option | 許容 — target 解決は use case の一部 |
| `expense_list` | `trip`, `itinerary` Option | 許容 — 同上 |
| その他 | `trip_id` / `id` のみ | 問題なし |

service は clap 型を知らず、`Option<i64>` で受け取るのみ。✓

## Lessons learned

1. **v4.6.12〜v4.6.15 の list 系**は v4.6.4〜v4.6.10 と同じ薄い service パターンを踏襲できた
2. **target / owner 解決を service に含める**のは list 系で自然（`note_list`, `reservation_list`, `expense_list`）
3. **trip_context** のような human 専用データは result に含めても JSON には出さない（`reservation_list`）
4. **JSON wrapper は CLI 層** — 11 module すべてで shape 互換を維持
5. **list 導線が揃った**ので、次は **show 系を 1 command ずつ** が安全
