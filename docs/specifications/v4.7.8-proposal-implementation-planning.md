| 観点 | v4.7.8 判定 |
|---|---|
| 実装方針 | **段階分け** — 一気に DB + import + materialize へ入らない |

**固定ルール（v4.7.0〜v4.7.7 から継承）:**

```text
schema v8 Trip          = 採用済み正式データ
Trip Proposal Envelope  = 旅行全体の未採用案（schema 外）
Proposal Fragment       = 既存 Trip への部分提案（schema 外）
Materialize / Apply       = 人間の採用 gate — AI 出力の自動取り込み禁止

```

## 2. 実装フェーズ（Phase P-0〜P-6）

v4.7.8 では **フェーズを決めるだけ**。コード・migration は書かない。

| Phase | 内容 | v4.7.8 後の想定 |
|---|---|---|
| **P-0** | documentation / planning only | **本書（v4.7.8）** |
| **P-1** | Proposal Envelope **file validation only** | 次の実装候補 |
| **P-2** | Proposal Envelope **show / inspect** | P-1 の直後 |
| **P-3** | Proposal Fragment **file validation only** | Envelope の後 |
| **P-4** | Proposal / Fragment **storage strategy** | file → 必要なら DB 拡張 |
| **P-5** | materialize / apply **planning spec**（blocking rules の具体化） | v4.7.4 の実装者向け展開 |
| **P-6** | materialize / apply **implementation** | 最もリスクが高い — 最後 |

```text
P-0 ──► P-1 ──► P-2 ──► P-3 ──► P-4 ──► P-5 ──► P-6
plan    env      env      frag     storage  gate      gate
        validate inspect  validate           spec      impl
```

**一気にやらない理由:**

- Envelope / Fragment / Trip の **境界崩れ** が最も起きやすい
- `trip import` に Proposal を混ぜる誘惑を避ける
- materialize 前に **inspect 可能な file path** を確立する方が安全

## 3. file-based vs DB-backed

### 3.1 比較

| 観点 | file-based（初期推奨） | DB-backed（後段） |
|---|---|---|
| schema v8 Trip との混同 | **低い** — 拡張子・コマンド・validator が分離しやすい | **高い** — Trip テーブル近傍で境界が曖昧になりやすい |
| 実装コスト | **低い** — 読み込み + validate + 表示 | migration・list/show・status・FK 設計が必要 |
| public examples との相性 | **よい** — [examples-non-normative/](../public/examples-non-normative/) と同型 | インポート経路の説明が増える |
| AI / provider 入力確認 | **向く** — ファイルを渡して inspect | inbox 概念は将来有効だが初期は過剰 |
| CLI 内管理 | 弱い（パス指定） | **強い** — list / show / status |
| migration リスク | **なし** | **あり** — Proposal 専用テーブル設計が必要 |

### 3.2 v4.7.8 判断

**まず file-based validation / inspection から始める**（P-1〜P-3）。

```text
入力: ローカル JSON ファイル（Envelope / Fragment）
処理: 読み込み → 概念 validator → human-readable report
出力: exit code + テキスト / --json レポート
DB: 触らない
trip import: 触らない
```

DB-backed は **P-4** で再評価する。必要になったときの候補:

```text
proposals/ ディレクトリ inbox（file + manifest、DB なし中間案）
proposal_envelopes テーブル（専用、trips と FK しない）
proposal_fragments テーブル（trip_id 参照は「対象 Trip」であり正式データ化ではない）
```

**P-4 以前に DB migration を入れない。**

## 4. Command 候補（v4.7.8 では未実装）

コマンド名・責務・入出力の **候補整理**。確定ではない。

### 4.1 命名方針

```text
trip *           — schema v8 Trip 専用（既存）
proposal *       — Trip Proposal Envelope 専用（新規・別系統）
fragment *       — Proposal Fragment 専用（新規・別系統）

trip validate-export  — Proposal / Fragment には使わない
proposal validate     — Envelope 専用 validator（候補）
fragment validate     — Fragment 専用 validator（候補）
```

`trip` サブコマンドに `trip proposal validate` を **入れない** 方が、export/import との混同が少ない（`proposal` トップレベルまたは `caglla proposal` 系を推奨）。

### 4.2 Envelope 候補

| Command（候補） | 責務 | 入力 | 出力 |
|---|---|---|---|
| `proposal validate <file>` | Envelope 構造・blocking 前チェック | JSON ファイル | 成否 + errors/warnings |
| `proposal show <file>` | 人間向け要約 | JSON ファイル | テキスト |
| `proposal inspect <file>` | 詳細 inspect（date_policy, missing_fields, adoption hints） | JSON ファイル | テキスト / `--json` |
| `proposal materialize <file> --output <trip.json>` | **P-6** — gate 通過後 schema v8 Trip 生成 | Envelope + 人間パラメータ | schema v8 JSON ファイル |

P-1〜P-2 では **validate / show / inspect のみ**。

### 4.3 Fragment 候補

| Command（候補） | 責務 | 入力 | 出力 |
|---|---|---|---|
| `fragment validate <file>` | Fragment 構造・target 整合の前チェック | JSON ファイル | 成否 + errors/warnings |
| `fragment show <file>` | 人間向け要約 | JSON ファイル | テキスト |
| `fragment inspect <file>` | target / intent / conflict / warnings 詳細 | JSON ファイル | テキスト / `--json` |
| `fragment apply <file> --trip <trip-id>` | **P-6** — gate 通過後既存 Trip へ反映 | Fragment + Trip + 人間パラメータ | 更新 Trip（DB） |

P-3 では **validate / show / inspect のみ**。

### 4.4 意図的に後回しにする command

```text
proposal import / list     — DB-backed または inbox 設計後（P-4+）
fragment import / list     — 同上
proposal materialize       — P-6（gate spec 具体化後）
fragment apply             — P-6
```

### 4.5 trip export/import との分離

| 既存 | Proposal 系 |
|---|---|
| `trip export` → schema v8 | Envelope / Fragment は **export しない** |
| `trip import` → DB へ Trip | Proposal ファイルは **import しない** |
| `trip validate-export` | **`proposal validate` / `fragment validate` と別実装** |

materialize / apply の **出力** のみ schema v8 Trip となり、その時点で `trip validate-export` 対象になる。

## 5. Envelope と Fragment の実装順序

### 5.1 検討した選択肢

| 順序案 | メリット | デメリット |
|---|---|---|
| Envelope 先 | 新規 Trip 生成ルートが明確；既存 Trip 依存が少ない | Fragment の target 検証は後回し |
| Fragment 先 | 既存 Trip ユーザー向け価値が早い | target / conflict が複雑；Trip DB 前提が増える |
| 共通 validator 先 | DRY | 概念が異なりすぎて早合流は境界を曖昧にしやすい |
| gate 前提のみ先 | 安全原則の固定 | inspect 手段がなく運用に乗らない |

### 5.2 v4.7.8 推奨順序（理由付き）

```text
1. Proposal Envelope file validation / inspect   (P-1, P-2)
2. Proposal Fragment file validation / inspect   (P-3)
3. materialize / apply planning spec 具体化      (P-5)
4. storage 検討（file inbox vs DB）              (P-4 — P-5 と並行可)
5. materialize / apply 実装                      (P-6)
```

**理由:**

1. **Envelope は Trip 非依存** — file read のみで P-1〜P-2 が完結し、境界テストが容易
2. **Fragment は target Trip 参照** — Envelope で validator パターンを確立してから入る方が安全
3. **materialize / apply は最後** — 誤って正式 Trip を作るリスクが最大；inspect 文化を先に作る
4. **共通 validator は内部 helper に留める** — 公開 CLI は `proposal` / `fragment` で分離

Envelope / Fragment を **同時リリース（同じ PR）にしない**。段階的に merge する。

## 6. schema v8 Trip との境界（最重要）

### 6.1 三層モデル（維持）

```text
Layer A — Proposal（schema 外）
  Trip Proposal Envelope
  Proposal Fragment

Layer B — Adoption gate
  materialize（Envelope → 新 Trip）
  apply（Fragment → 既存 Trip）

Layer C — Canonical Trip（schema v8）
  trip export / trip import / trip validate-export
  採用済み・実日付あり
```

### 6.2 禁止パターン

```text
× Proposal JSON を trip import に流す
× Envelope を schema_version: 8 の Trip として validate-export する
× Fragment を「小さい Trip export」として扱う
× AI 出力を DB Trip へ自動 insert する
× materialize / apply を --yes デフォルトで無人実行する
```

### 6.3 validator 分離

| Validator | 対象 | 将来コマンド候補 |
|---|---|---|
| `trip validate-export` | schema v8 Trip のみ | 既存 |
| Envelope validator | Trip Proposal Envelope | `proposal validate` |
| Fragment validator | Proposal Fragment | `fragment validate` |
| Post-gate Trip | materialize / apply **出力** | `trip validate-export` |

gate **出力** は schema v8 Trip であるべき — [v4.7.4](v4.7.4-materialize-gate-concept-validation-rules.md) の blocking rules に従う。

## 7. JSON schema — 未確定・将来論点

v4.7.8 では **JSON schema を確定しない**。P-1 / P-3 の validator 実装前に、以下を spec で整理しておく。

### 7.1 Trip Proposal Envelope（将来）

| 論点 | 内容 |
|---|---|
| required fields | `proposal_id`, `title`, `source`, `_disclaimer`（non-normative 継続） |
| optional fields | `destination`, `date_policy`, `candidate_days`, `adoption_hints` |
| metadata | `created_at`, `valid_until`, `provider`, `model` |
| missing_fields | inspect で人間に提示 — blocking か warning か |
| warnings | 古い価格・未確認の flight 等 |
| assumptions | AI が置いた前提の明示 |
| adoption hints | materialize 前の required decisions |

参照: [trip-proposal-envelope.example.json](../public/examples-non-normative/trip-proposal-envelope.example.json)

### 7.2 Proposal Fragment（将来）

| 論点 | 内容 |
|---|---|
| required fields | `fragment_id`, `intent`, `target`, `_disclaimer` |
| optional fields | `candidate_itinerary`, `placement`, `conflict_notes` |
| target | trip_id / day / itinerary 参照 — **未確定可** |
| intent | add / replace / supplement 等 |
| adoption hints | conflict 解消・予約要否 |
| warnings | 既存 Itinerary との時間衝突 |

参照: [proposal-fragment.example.json](../public/examples-non-normative/proposal-fragment.example.json)

### 7.3 schema 確定のタイミング

```text
P-1 着手前: Envelope draft schema（documentation + validator 用 internal 定義）
P-3 着手前: Fragment draft schema
P-5: gate 出力が schema v8 と整合する mapping 表
公開 JSON schema ファイル: P-6 以降でも可 — export-schema とは別リポジトリ/path 推奨
```

## 8. materialize / apply — 計画のみ

v4.7.8 では **実装しない**。[v4.7.4](v4.7.4-materialize-gate-concept-validation-rules.md) を実装計画へ接続する。

### 8.1 materialize（Envelope → 新 Trip）

```text
入力: Trip Proposal Envelope + 人間の採用パラメータ（日付確定等）
処理: required decisions → blocking validation → warnings 確認 → 人間 confirm
出力: 新規 schema v8 Trip JSON（file）または trip import 可能な形式
原則: 自動採用禁止
```

### 8.2 apply（Fragment → 既存 Trip）

```text
入力: Proposal Fragment + target Trip + 採用パラメータ
処理: target 確定 → intent 確認 → conflict → blocking / warnings → 人間 confirm
出力: 更新された schema v8 Trip（DB）
原則: 自動反映禁止
```

### 8.3 共通 gate 原則（維持）

```text
人間の採用判断
required decisions を gate 前に揃える
blocking validation — 止める
non-blocking warnings — 人間に委ねる
AI / provider 出力を無批判に正式 Trip へ流さない
```

P-5 で [v4.7.4](v4.7.4-materialize-gate-concept-validation-rules.md) の blocking / non-blocking 一覧を **command オプション・exit code** に落とす spec を書く（v4.7.8 では未着手）。
