## 1. Trip Proposal とは何か

### 1.1 定義

**Trip Proposal** は、旅行者がまだ採用していない **旅行全体の候補案** です。

AI、旅行業者、ブログ、手入力、他ツールからの出力など、多様な source があり得ます。いずれも **「これから検討する案」** であり、Travel Ledger の正式 Trip ではありません。

```text
Trip Proposal:
  未採用
  日付未定可
  不完全可
  複数候補を並列保持可
  schema v8 の外側
```

### 1.2 Proposal が Trip ではない理由

schema v8 Trip は **採用済み** の旅行データ契約です。`start_date` / `end_date` を持ち、`trip export` の正本として第三者に説明できる状態を想定します。

一方 Trip Proposal は:

- 日付が未定、または候補のみ
- 人間がまだ採用していない
- 航空券・ホテル料金・営業時間などが **古くなりうる**
- 複数案の比較対象として並列に存在しうる

これらを schema v8 Trip に無理に入れると、**正式データと候補案の区別が消え**、export / validate-export / Travel Book の意味が曖昧になります。

```text
誤ったモデル:
  AI 提案を placeholder Trip として DB に入れる
  → 「正式 Trip」と「検討中案」が混ざる

正しいモデル:
  Proposal Envelope（外側）で保持
  → 人間が採用したときだけ materialize
  → schema v8 Trip として ledger に入る
```

### 1.3 Trip Proposal Envelope の責務

**Trip Proposal Envelope** は、単一の Trip Proposal を包む概念コンテナです。

| 責務 | 説明 |
|---|---|
| **識別** | 提案を一意に参照できる（`proposal_id` 等） |
| **出所の記録** | AI / provider / manual など source を残す |
| **有効期限の表現** | soft expiry 用 metadata |
| **提案本体の保持** | タイトル・目的地・日程方針・候補内容 |
| **採用前ヒント** | materialize 前に人間が決めるべきことを示す |
| **schema v8 との分離** | Trip export JSON ではないことを明示 |

Envelope は **保存形式の実装詳細ではない**。将来 JSON ファイル・SQLite 専用テーブル・inbox ディレクトリなどになりうるが、v4.7.2 では概念のみ確定する。

## 2. schema v8 Trip との境界

### 2.1 二層モデル

```text
Layer A — Proposal（schema 外）
  Trip Proposal Envelope
  Proposal Fragment（v4.7.3）

Layer B — Canonical Trip（schema v8）
  trip export / trip validate-export の対象
  採用済み・実日付あり
```

### 2.2 比較表

| 観点 | Trip Proposal Envelope | schema v8 Trip |
|---|---|---|
| 状態 | 未採用 | 採用済み |
| 日付 | 未定 / 候補 / 確定のいずれも可 | `start_date` / `end_date` 必須 |
| 完全性 | 不完全可 | export 契約に準拠 |
| validate-export | 対象外 | 対象 |
| trip export 出力 | しない | する |
| 複数並列 | 比較用に複数保持可 | 1 Trip = 1 採用済み旅行 |
| 期限切れ | warning（soft） | N/A（記録として保持） |

### 2.3 変換の唯一の gate

Proposal から schema v8 Trip への遷移は **materialize** のみです。

```text
Proposal Envelope  ──materialize（人間採用）──►  schema v8 Trip
                      ▲
                      └── v4.7.4 で詳細化。v4.7.2 では境界のみ確定
```

`trip import` は **既に schema v8 準拠の Trip JSON** を取り込むコマンドであり、Proposal の代替ではない。

## 3. Envelope 概念構造

以下は **概念整理**。フィールド名・JSON shape の確定は v4.7.2 scope 外。

```text
Trip Proposal Envelope
  ├─ metadata
  │   ├─ proposal_id
  │   ├─ created_at
  │   ├─ valid_until
  │   ├─ no_expiration
  │   ├─ source
  │   └─ provider
  │
  ├─ proposal
  │   ├─ title
  │   ├─ destination
  │   ├─ date_policy
  │   ├─ candidate_days
  │   ├─ candidate_itineraries
  │   └─ notes
  │
  └─ materialize hints
      ├─ required_decisions
      ├─ missing_fields
      └─ warnings
```

### 3.1 metadata

| 概念 | 用途 |
|---|---|
| `proposal_id` | ローカル参照用の識別子 |
| `created_at` | 提案が生成・取り込まれた時刻 |
| `valid_until` | 明示的 soft expiry |
| `no_expiration` | 期限なしを明示（既定 1 year を適用しない） |
| `source` | `ai` / `provider` / `manual` / `import` 等 |
| `provider` | 旅行業者名・モデル名・サービス名 |

### 3.2 proposal（本体）

候補案として持つべき **最小情報**（概念）:

| 概念 | 必須度 | 説明 |
|---|---|---|
| `title` | 推奨 | 人間が一覧で識別する名前 |
| `destination` | 推奨 | 地域・国・都市など |
| `date_policy` | 推奨 | 日付の扱い方針（後述） |
| `candidate_days` | 任意 | 日ごとの候補（日付未定でも可） |
| `candidate_itineraries` | 任意 | 行程候補（会場・移動・食事など） |
| `notes` | 任意 | 自由記述・前提・注意 |

**必須にしないもの:** `start_date` / `end_date`（Trip 化前は不要）、完全な Day 列、予約 ID、expense 確定額。

### 3.3 materialize hints

materialize 実装は v4.7.4 だが、Envelope 段階で **人間向けのヒント** を持てる。

| 概念 | 用途 |
|---|---|
| `required_decisions` | 採用前に人が決めること（例: 出発日、何日目に何をするか） |
| `missing_fields` | Trip 化に不足している情報 |
| `warnings` | 古い料金・閉店・季節制限など |

## 4. date_policy（日付方針）

Proposal は日付未定を許容する。schema v8 Trip のように `start_date` / `end_date` を Envelope 段階で必須にしない。

### 4.1 三つの方針

```text
fixed_dates:
  日付が確定している提案
  例: 「2026-10-10 〜 2026-10-14 で沖縄」
  ただし未採用のまま — Trip ではない

flexible_dates:
  日付候補はあるが確定していない
  例: 「11月のどこかの週末」「3泊4日で候補は2つ」

undated:
  日付未定の旅行案
  例: 「いつか行きたい北欧旅行」「夏に東南アジア」
```

### 4.2 materialize 時の前提

いずれの `date_policy` でも、**materialize して schema v8 Trip にする時点では実日付が必要** になる。

```text
fixed_dates     → その日付を採用するか人が確認
flexible_dates  → 候補から1つを選ぶ
undated         → 人が start_date / 行程日を与える
```

v4.7.4 で validation rules を詳細化。v4.7.2 では「Trip 化には実日付が要る」境界のみ確定。

### 4.3 schema v8 との対比

| | Proposal `date_policy` | schema v8 Trip |
|---|---|---|
| `undated` | 許容 | 不可 |
| 日付候補のみ | 許容（flexible） | 不可（確定日が要る） |
| 確定日付 | 許容（fixed）でも未採用 | 必須かつ採用済み |

## 5. Expiry（有効期限）

v4.7.0 / v4.7.1 の方針を継承し、Trip Proposal Envelope に適用する。

### 5.1 ルール

```text
valid_until がある:
  それを soft expiry として使う

valid_until がない:
  created_at + 1 year を既定扱い（warning 用）

no_expiration: true:
  期限なしとして扱う（既定 1 year を適用しない）

期限切れ:
  warning
  ただし即 import 禁止にはしない
```

### 5.2 設計意図

期限切れ Proposal を **危険データとして破棄しない**。

- 航空券・ホテル料金・営業時間・イベント情報は古くなりうる → **warning**
- 過去の提案をアーカイブ参照する用途は残す
- materialize 時に人間が内容を再確認する前提

```text
期限切れ ≠ 削除
期限切れ = 「内容が古い可能性があります」
```

## 6. 入力 source の想定

| Source | 典型入力 | Envelope 上の扱い |
|---|---|---|
| **AI** | チャット・エージェント出力 | `source: ai`, `provider` にモデル名 |
| **Provider** | 旅行業者・OTA・パッケージ案 | `source: provider`, 商業条件は warning 対象 |
| **Manual** | 手入力・メモからの整理 | `source: manual` |
| **Import** | 外部ファイル・ブログ転記 | `source: import` |

いずれも **Envelope に入る前に正規化はしない** 前提でもよい。正規化・検証は将来の `proposal import` 実装の責務。v4.7.2 では「多様な source を同じ Envelope 概念で包める」ことだけ確定。

## 7. materialize との境界（概要）

### 7.1 責務分離

```text
Proposal:
  候補案
  未採用
  日付未定可
  不完全可

Materialize:
  人間が採用する gate
  必須情報を確定する
  schema v8 Trip に変換する

schema v8 Trip:
  採用済み
  実日付あり
  Travel Ledger の正式 Trip
```

### 7.2 materialize 前後

| 段階 | 保持場所 | 人間の役割 |
|---|---|---|
| 取り込み後 | Proposal Envelope | 一覧・比較・却下 |
| 検討中 | Proposal Envelope | 日付・行程の選択 |
| 採用時 | materialize gate | 明示的な「採用」操作 |
| 採用後 | schema v8 Trip（ledger） | 通常の Trip 編集・export |

materialize command・validation rules・report 形式は **v4.7.4** で扱う。

## 8. Proposal Fragment との境界（軽量）

v4.7.3 で詳細化。v4.7.2 では区別のみ。

| 観点 | Trip Proposal | Proposal Fragment |
|---|---|---|
| スコープ | 旅行全体 | 既存 Trip / Day の一部 |
| 典型例 | AI 旅行プラン全体 | 「Day 2 にこのレストラン」 |
| materialize 先 | 新 Trip | 既存 Trip へ merge |
| Envelope | Trip Proposal Envelope | Fragment Envelope（v4.7.3） |

Trip Proposal Envelope は **新規旅行の未採用案** に焦点を当てる。部分提案は別概念として v4.7.3 に defer。
