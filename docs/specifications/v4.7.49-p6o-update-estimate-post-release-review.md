**`required_decisions` に関する注記（誤解防止）:**

Authoritative field-level blocking conditions are re-parsed and revalidated inside the confirm transaction.

The remaining P2 concerns rechecking the opaque `adoption_hints.required_decisions` report container itself; it does not represent a known data-safety or rollback defect.

日本語: confirm transaction 内では、**field 単位の blocking 条件**（更新 field 0 件、currency-only、explicit null、value+clear 競合、invalid amount/currency、wrong parent、stale baseline、preview mismatch 等）は `parse_update_estimate_fields` 等により **再 parse・再検証済み**（v4.7.48）。残る P2 は、opaque な `adoption_hints.required_decisions` **コンテナ自体**を transaction 内で再確認しない点に限られる。**既知の data-safety 欠陥や rollback 不備ではない**。

## 1. planning / dry-run / confirm 契約整合

### 1.1 Identity

| 契約 | planning (v4.7.46) | dry-run (v4.7.47) | confirm (v4.7.48) | 確認根拠 |
|---|---|---|---|---|
| `estimate_id` 正本 | ✓ | ✓ | ✓ | `parse_estimate_id_field` |
| title/sort_order/amount で選択しない | ✓ | ✓ | ✓ | selector 未使用 |
| 親 Itinerary 検証 | ✓ | ✓ | ✓ | dry-run + confirm transaction |
| UPDATE `WHERE id + itinerary_id` | planning | n/a | ✓ | `update_estimate_row_scoped` |

### 1.2 Partial update

更新 field 集合は 3 段階で一致:

```text
amount, currency, title, note, sort_order, clear_title, clear_note
```

- omission → 変更なし: `parse_update_estimate_fields` + `compute_update_estimate_proposed`
- 有効 field ≥ 1: `has_update_field` check
- unknown-only → blocking（更新 field 0 件）: CLI test `unknown_field_only`
- dry-run / confirm **同一 parser**: `parse_update_estimate_fields`（dry-run は `Some(report)` で warning、confirm transaction は `None`）

### 1.3 Confirm reuse（別解釈なし）

confirm が dry-run と共有する helper（`src/proposal/apply.rs`）:

| Helper | dry-run | confirm |
|---|---|---|
| `parse_update_estimate_fields` | ✓ | ✓（transaction 内） |
| `resolve_apply_target` | ✓ | ✓（transaction 内） |
| `compute_update_estimate_proposed` | ✓ | ✓ |
| `build_update_estimate_field_changes` | ✓ | ✓ |
| `parse_update_estimate_amount_value` / `parse_amount_for_currency` 経由 | ✓ | ✓ |
| `reject_explicit_null_*` / clear flags | ✓ | ✓ |
| `detect_update_estimate_baseline_conflicts` | ✓ | ✓（transaction 内 TOCTOU） |

## 4. Preview agreement / TOCTOU

### 4.1 Preview agreement

transaction 内 `verify_update_estimate_gate_preview_matches` が検証:

- `target_itinerary_id` / `target_estimate_id`
- `estimate_field_changes`（field / before / after）
- proposed values（after 一致）

Itinerary title は agreement 対象外（itinerary title 変更後 confirm 成功テストで確認）。

### 4.2 Candidate mismatch

- stale gate report + changed candidate → reject
- unit test: `confirm_update_estimate_candidate_preview_mismatch_blocks_db_write`
- DB / Expense / `updated_at` 不変

### 4.3 Lost-update prevention

- `expected_*` baseline mismatch → blocking（dry-run + confirm CLI）
- `estimate_field_changes[].before` vs 現行 DB → TOCTOU（`revalidate_update_estimate_before_write`）
- last-write-wins **非採用**（planning どおり）

### 4.4 Remaining race window

| 窗口 | 保護 | 評価 |
|---|---|---|
| 同一 confirm 内 gate → transaction | 同一 JSON・同一 connection・短時間 | 実用上安全 |
| dry-run → confirm 間の外部 UPDATE | `expected_*` / field_changes.before（confirm 時） | 利用者が pin 可能 |
| `expected_*` なしの並行 confirm | 後勝ち UPDATE + TOCTOU は gate 再計算で部分保護 | P2-3 — API 前に version/ETag 検討 |
| schema version column | なし | 現行は **妥当**。将来 API で P2 |

## 12. Versioning 方針（documentation-only）

### 候補比較

| 観点 | v4.8.0 / v4.8.1 / …（現行） | v4.8.0-1.plan / … prerelease |
|---|---|---|
| Cargo compatibility | 単調増加、シンプル | prerelease 順序理解が必要 |
| tag / Release workflow | 既存 `tools/release/` と整合 | workflow / asset 命名変更コスト |
| Latest Release 表示 | 明確 | prerelease は non-latest 扱いになりやすい |
| 利用者誤解 | 低（各 patch が formal release） | 「不安定版」誤解リスク |
| 移行コスト | なし | 中〜高 |

**推奨:** 現行の **patch 段階リリース**（v4.8.0 planning → v4.8.1 dry-run → …）を維持。prerelease suffix への移行は **不要**。
