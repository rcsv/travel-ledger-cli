## 1. Executive summary

| 観点 | 判定 |
|---|---|
| Receipt Inbox の現行実装 | **Healthy** — v3.6–v3.7 系列の意図どおり動作 |

| Travel Book との境界 | **明確** — Receipt / Expense / Difference は含めない |

**結論:** Receipt Inbox は **Expense 化前の未整理証憑候補** として健全に位置づけられている。Travel Book から除外する方針は正しい。ledger 側（stats / doctor / advisor）での未整理 receipt 可視化は **設計着手可・実装候補**、Actual 集計への混入は **Do not do**。

## 2. Receipt Inbox の定義

### 2.1 正本定義

```text
Receipt Inbox = Trip スコープの未整理証憑候補の集合
Receipt       = 1 件の未整理支払い候補レコード（Actual ではない）
Expense       = Itinerary に紐づいた確定実績費用（Actual）
```

Receipt Inbox は **確定済み Expense そのものではない**。Expense 化・照合・分類が完了する前の **入力箱** である。

### 2.2 現行実装（v4.4.8 時点）

| 項目 | 内容 |
|---|---|
| スコープ | Trip 直下（`receipts.trip_id`） |
| 粒度 | metadata-only（画像・ファイルなし） |
| 必須コンテンツ | memo または amount+currency のいずれか |
| 状態 | `unreviewed`（Inbox）/ `ignored`（Trash） |
| Inbox / Trash 判別 | `trashed_at IS NULL` / `IS NOT NULL` |
| Itinerary 紐づけ | **なし** — 判明時は `assign` で Expense 化 |
| Expense との関係 | **convert（消費）** — assign 後 Receipt は削除 |

CLI: `receipt add/list/show/update/assign/trash/restore/delete`（`ignore` は trash の deprecated alias）

実装: `src/receipt.rs`、モデル `src/domain/models.rs`

### 2.3 ライフサイクル

```text
receipt add
    → unreviewed（Inbox）
        → receipt assign --itinerary <id>
            → Expense 作成 + Receipt 削除（Actual に入る唯一の経路）
        → receipt trash
            → ignored + trashed_at（Trash）
                → receipt restore → unreviewed
                → receipt delete（物理削除）
```

**除外した中間状態**（v3.6 方針、維持）:

```text
linked    — Itinerary 判明だが Expense ではない中途半端な状態を避ける
converted — Expense 化済み Receipt を Inbox に残さない（assign で削除）
```

**判定:** **Keep** — シンプルで二重計上を防ぎやすい。

## 3. Expense との境界

### 3.1 責務比較

| 観点 | Receipt Inbox | Expense |
|---|---|---|
| 意味 | 未整理の支払い候補 | 確定した実績費用 |
| Actual に含むか | **いいえ** | **はい** |
| itinerary 紐づけ | なし | 必須（`itinerary_id`） |
| amount / currency | 任意（片方だけは不可） | 必須 |
| paid_at / expense_date | `occurred_date`（ヒント） | `expense_date`（確定） |
| title / label | memo のみ | title + note |
| 証憑メタ | memo が主 | note に構造化メモ（費用区分等） |
| participant / 精算 | なし | paid_by / beneficiaries 等 |
| Planned vs Actual | 含めない | Actual 側の正本 |

### 3.2 assign 導線

`receipt assign` は **Expense 化の唯一の正規導線**。

- Receipt 側 amount/currency が欠けていれば assign 時に補完可能
- Expense title = memo（なければ `"Receipt <amount>"` 自動生成）
- `occurred_date` → Expense `expense_date`
- トランザクション内で Expense 作成後 **Receipt を DELETE**（二重保持しない）

コードコメント（`src/receipt.rs`）:

```text
Receipt は Actual ではない。assign により Expense を作ることでのみ Actual に入る。
```

### 3.3 二重計上防止

| リスク | 現行対策 | 判定 |
|---|---|---|
| Receipt と Expense が同時に Actual に数えられる | stats は Expense のみ集計 | **Keep** |
| assign 後も Receipt が残る | assign で DELETE | **Keep** |
| import 時に linked/converted が復活 | `unreviewed` に正規化 | **Keep** |
| 同一支払いを Receipt と Expense に重複登録 | 運用 + assign 消費モデル | **Clarify**（doctor 助言で補強可） |

**判定:** Expense との境界は **Healthy**。assign 消費モデルは **Keep**。

## 4. Estimate との境界

### 4.1 三概念モデル（正本）

```text
Estimate      = Planned cost（旅行前・旅行中の予定）
Receipt Inbox = 未整理の実績候補（post-trip 整理向け）
Expense       = 確定した実績（Actual）
Difference    = Actual(Expense) − Planned(Estimate)
```

### 4.2 混同しやすい点

| 誤解 | 正しい理解 |
|---|---|
| Receipt を「予定の下書き」にする | Estimate が予定。Receipt は actual 側の未整理候補 |
| Receipt amount を Planned に足す | 不可 — Estimate と独立 |
| Receipt を Difference に含める | 不可 — Difference は確定 Expense のみ |
| 「Potential Actual」= Difference | 不可 — 参考表示のみ（§5 参照） |

Estimate は itinerary 必須・amount 必須。Receipt は trip 直下・amount 任意。データモデル上も完全分離。

**判定:** **Keep** — 境界は明確。混同防止の docs 明文化は本書で完了。

## 5. Travel Book との境界

### 5.1 現行方針（v4.2.0 以降）

Travel Book（`trip export-md`）は **旅行前・旅行中に読むしおり**。

章構成:

```text
Cover → Trip overview → Daily schedule → Reservations → Checklist
     → Planned cost → Notes → Colophon
```

**含めないもの:**

```text
Expense
Receipt / Receipt Inbox
Planned vs Actual Difference
```

実装確認: `src/io/markdown.rs` に receipt / expense 参照なし。テストで明示的に除外を検証。

### 5.2 Receipt Inbox を Travel Book に含めない理由

| 理由 | 説明 |
|---|---|
| 読者 | Travel Book は旅行者向け itinerary guide。会計整理情報は対象外 |
| タイミング | Receipt Inbox は **旅行後・post-trip review** 向け（v3.5.0 位置づけ） |
| ノイズ | 「未整理レシート 5 件」はしおり本文の可読性を下げる |
| 責務分離 | 会計確認は `trip stats` / ledger 系 / 将来 advisor で扱う |
| 三層整合 | Travel Book = presentation layer。Receipt は ledger / evidence layer |

Planned cost（Estimate）は Travel Book に含める — これは **旅行前の予定** としてしおりに有用。Receipt は **未確定の actual 候補** であり性質が異なる。

**判定:** **Keep** — Travel Book 除外は正しい。変更不要。

## 6. Trip stats / Ledger / Doctor / Advisor

### 6.1 trip stats（現状）

`TripStats`（`src/analysis/statistics.rs`）:

| 含む | 含まない |
|---|---|
| `expense_count` / `expense_totals` | receipt_count |
| `estimate_count` / `estimate_totals` | pending_receipt_sum |
| `difference_totals`（Estimate + Expense 両方あるとき） | unreviewed_receipt_count |

**Actual total = 確定 Expense のみ。** 未整理 Receipt は含めない。

### 6.2 receipt list pending summary（現状）

`receipt list` のみが `compute_pending_receipt_summary` を表示:

- active（unreviewed, non-trashed）件数
- amount 未設定件数
- currency 別合計（設定済み amount のみ）

**stats とは別経路** — ledger 作業用の補助表示。

### 6.3 doctor / advisor（現状）

`DoctorIssueCode` に Receipt 関連コード **なし**。Itinerary 完全性・Participant・Shared Expense 等が中心。

v3.8.0 で **設計着手可** とされていた拡張例:

```text
未整理 Receipt が多い Trip への INFO / 助言
Estimate あり・Expense なし Itinerary への軽い注意
Pending Receipt sum の表示（stats ではなく advisor 助言のみ）
```

### 6.4 レビュー判断

| 観点 | 推奨 | 分類 |
|---|---|---|
| 未整理 Receipt を actual total に含める | **しない** | **Do not do** |
| 確定 Expense のみ actual total | **維持** | **Keep** |
| unreviewed receipt count を stats に追加 | **しない**（stats は集計正本） | **Defer** |
| pending receipt を advisor で警告 | **実装候補** | **Implementation candidate** |
| doctor で amount 未設定 receipt を INFO | **実装候補** | **Implementation candidate** |
| Potential Actual 表示（stats / export-md） | **しない** | **Do not do** |

Potential Actual（v3.5.0 §4 の参考表示）は **Difference の正本ではない**。trip stats や Travel Book への反映は v3.8.0 以降も **Do not do** として維持。

**判定:** stats 境界は **Healthy**。ledger 側の可視化は **doctor/advisor 活用** で補強するのが自然（Actual 定義は変えない）。

## 7. Export / Import

### 7.1 schema 位置

| 項目 | 内容 |
|---|---|
| schema version | **8**（receipts は v7 追加、`trashed_at` は v8） |
| 階層 | Trip 直下 `receipts: []`（Expense と同階層、別配列） |
| フィールド | `day_ref`, `amount`, `currency`, `occurred_date`, `memo`, `status`, `trashed_at` |
| 含まない | DB `id`, `image_path`, `itinerary_id`, `linked_expense_id` |

### 7.2 roundtrip

- export: **trashed 含む全 receipt**（restore 可能性維持）
- import: legacy `linked`/`converted` → `unreviewed` 正規化
- `ignored` + `trashed_at` 欠落時は import 側で補完
- テスト: `test_export_import_receipt_roundtrip`

### 7.3 Expense との export 関係

```text
receipts[]  — 未整理候補（Inbox + Trash）
days[].itineraries[].expenses[]  — 確定 Actual
```

assign 済み Receipt は export に **現れない**（DB から削除済み）。Expense としてのみ残る。

### 7.4 二重登録防止（import）

| リスク | 対策 |
|---|---|
| import で receipt + 対応 expense が重複 | receipt は独立配列。assign 済みは receipt 側に存在しない設計 |
| legacy linked/converted 復活 | status 正規化 |
| trashed receipt の誤復元 | `trashed_at` を v8 で保持 |

### 7.5 Okinawa canonical sample

`seed.sh`: 6 receipts（5 active + 1 trashed）、assign は行わず Inbox 状態を維持。

`expected-export-v3.json`: 6 entries、`trashed` は jq 正規化フィールド。

**判定:** export / import 方針は **Keep**。schema 変更は v4.5.0 では **Do not do**。

## 8. Future features との境界

### 8.1 概念マップ

```text
Receipt Inbox   — 証憑候補・会計整理の入口（metadata-only 現行）
Attachment      — ファイル添付一般（Evidence 共通レイヤー）
Photo           — 旅行写真（Travel Journal 向け）
Travel Journal  — 旅行後の記録・感想・体験ログ（v5）
```

### 8.2 境界ルール

| 概念 | Receipt Inbox に含めないもの | 接続点 |
|---|---|---|
| **Attachment** | 汎用ファイル管理 | 将来 Evidence 層経由で receipt に evidence をリンク |
| **Photo** | 観光写真・アルバム | Travel Journal 側。Receipt 画像と混同しない |
| **Travel Journal** | 感想・体験 narrative | post-trip だが **会計証憑ではない** |
| **OCR** | 自動パース | Attachment / Evidence の後 |

### 8.3 image_path 方針（維持）

Receipt 専用 `image_path` カラムの先行実装は **採用しない**（v3.5.0 §7、v3.8.0 Do not start）。

将来:

```text
Evidence / Attachment（共通）
    ↓ 参照
Receipt（metadata） / Expense（metadata） / Journal entry
```

### 8.4 Travel Book / TravelBookDocument

Receipt Inbox は Travel Book model にも **含めない**（v4.4.x 結論と整合）。`TravelBookDocument` prototype は UI/Venue 要件まで **Defer**。

**判定:** 将来境界は **Clarify 完了**（本書）。Evidence/Attachment 設計は **Defer**（Travel Journal のブロッカー）。

## 10. Next milestone recommendation

**推奨: A**

```text
A. v4.5.1 — doctor / advisor Receipt utilization（推奨・次）
   未整理 receipt の助言・警告を ledger 側に追加。
   trip stats / Travel Book / Actual 定義は変更しない。

B. Evidence / Attachment concept design（設計のみ）
   Travel Journal（v5）のブロッカー解消。実装は別 milestone。

C. TravelBookDocument prototype
   → Defer（UI / Venue 要件まで — v4.4.7 結論維持）

D. Receipt Inbox 実装拡張（purge / image / OCR）
   → Defer または Do not do（v3.8.0 リスト維持）
```

**理由:**

1. v4.4.x で Travel Book 側の責務は整理完了 — **ledger 側の補完**が自然な次ステップ
2. Receipt Inbox 実装は **Healthy** — 大きな schema / workflow 変更は不要
3. doctor/advisor 活用は v3.8.0 から **設計着手可** — Actual 定義を汚さず未整理 receipt を可視化できる
4. Evidence/Attachment は Travel Journal の前提だが、Receipt Inbox 単体の緊急度は低い
5. `TravelBookDocument` は UI 要件なしでは依然 premature
