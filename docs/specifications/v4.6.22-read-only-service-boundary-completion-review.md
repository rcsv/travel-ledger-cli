## Service boundary 基本方針（再確認）

v4.6.4〜v4.6.21 を通じて以下が守られていることを review として記録する。

| 方針 | 状態 |
|---|---|
| service layer は structured result を返す | ✓ 全 16 module |
| service layer に `println!` / `eprintln!` を入れない | ✓ 確認済み |
| JSON wrapper は CLI 層に残す | ✓ NoteListJson, ReservationListJson, ExpenseListJson, DayListJson 等 |
| human-only 表示は CLI 層に残す | ✓ 見出し、Reservations セクション、timeline |
| JSON shape を変えない | ✓ golden 不変 |
| human output を変えない | ✓ |
| DB schema / migration に触れない | ✓ |
| write command にはまだ踏み込まない | ✓ |
| Tauri / GUI 実装はまだ行わない | ✓ |

### 責務分担（確立済みパターン）

```text
CLI layer (main.rs):
  - clap args 解釈
  - --json / human 選択
  - JSON wrapper 組み立て（NoteListJson, ExpenseListJson 等）
  - expense_to_json 等の変換（participant 名解決含む）
  - human display（print_*_list / print_*_detail）
  - human-only 追加セクション（itinerary show Reservations、checklist 見出し、timeline 見出し）
  - 成功メッセージ println!（write command）

service layer (src/services/):
  - use case orchestration
  - owner / target 解決（list 系）
  - structured ServiceResult
  - repository / domain 関数呼び出し
  - no terminal I/O

repository / domain layer:
  - DB query / mutation
  - domain model
  - ordering / validation
  - not found error message
```

## Known notes（今すぐ直さない）

以下は将来の GUI API 化・aggregate 境界整理の再検討対象。v4.6.22 では **修正しない**。

### expense show / expense list — helper 側 DB context

| 観点 | 現状 |
|---|---|
| service result | `Expense` 本体のみ（`note_show` と同型パターン） |
| JSON | CLI が `expense_to_json(&conn, &expense)` で `ExpenseJson` 生成（participant 名解決含む） |
| human | CLI が `print_expense_detail(&conn, &expense)`（内部でも `expense_to_json`） |
| GUI 影響 | Tauri 初号機で expense detail を出すとき、service result だけでは JSON 互換 DTO が揃わない |

`expense list` も JSON 出力時に各件 `expense_to_json(&conn, e)` を CLI が呼ぶ。participant / beneficiary 解決は **表示層 helper に残存**。

### reservation show — service に display context を含める

| 観点 | 現状 |
|---|---|
| service result | `reservation` + `day_number` + `itinerary_title` |
| JSON | `Reservation` のみ serialize（context は human 専用） |
| human | `print_reservation_detail_with_context` が service context を使用 |

reservation show は **itinerary 文脈を service が解決**するパターン。expense show とは対照的。

### itinerary show — Reservations は handler 側追加取得

| 観点 | 現状 |
|---|---|
| service result | `ItineraryItem` のみ |
| JSON | `ItineraryItem` のみ（予約一覧なし） |
| human | `print_itinerary_detail` の後、handler が `list_reservations_for_itinerary` で追加表示 |

itinerary detail の human 表示に予約サマリが含まれるが、**service / JSON には入っていない**。aggregate boundary の整理候補。
