# v4.7.2 — Trip Proposal Envelope concept specification

## Summary

**v4.7.2** documents the **Trip Proposal Envelope** concept: how unadopted whole-trip suggestions live **outside** schema v8, why they are not Trips, and how expiry / date policy / materialize boundaries work — following [v4.7.1](../specifications/v4.7.1-public-readme-schema-docs-outline.md).

Spec: [v4.7.2-trip-proposal-envelope-concept-spec.md](../specifications/v4.7.2-trip-proposal-envelope-concept-spec.md)

## What changed

- **New:** Trip Proposal Envelope concept specification
- **Updated:** `docs/public/proposals.md` — Envelope overview deepened (entry-level)
- **Not changed:** export schema, CLI behavior, JSON golden

## Headline decisions

- **Trip Proposal** — unadopted whole-trip candidate; **not** a schema v8 Trip
- **Trip Proposal Envelope** — conceptual container (metadata + proposal + materialize hints)
- **date_policy** — `fixed_dates` / `flexible_dates` / `undated` (dates not required before adoption)
- **Expiry** — soft warnings only; expired proposals are not auto-deleted or import-blocked
- **Materialize** — adoption gate to schema v8 Trip (detail deferred to v4.7.4)
- **Proposal Fragment** — partial proposals deferred to v4.7.3
