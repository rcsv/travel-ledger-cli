# v4.7.33 — P-6j safety / UX hardening for delete_itinerary

## Summary

**v4.7.33** hardens the **safety / UX** of `delete_itinerary` (dry-run / confirm) by returning a **structured** `preview.delete_preview.blocking_children` even when deletion is blocked (invalid report), and by adding confirm-focused regression tests (estimate / unresolved / ambiguous / not-found).

This release does **not** change the DB mutation boundary: successful confirm still deletes **exactly one** row from `itinerary_items` via the existing non-cascade path, and never deletes child entities.

## What changed

- **Improved:** when delete is blocked by children, the gate report still includes `preview.delete_preview.blocking_children` (with `valid=false` and `errors` populated)
- **Improved:** invalid delete reports do not write preview export JSON
- **Added:** confirm hardening tests — estimate child blocking, unresolved / ambiguous / not-found confirm blocking, structured blocking counts
- **Unchanged:** schema v8, DB migration, JSON golden, Markdown golden

## Safety / UX details

- Blocking reasons remain in `errors`
- `valid=false` when blocking applies
- For invalid reports, preview Trip JSON is not produced / written
- Venue / Place remains outside blocking child logic (no Venue model work)
