## Reviewed (documented)

### Storage vs display

```text
保存: Itinerary 配下（reservations.itinerary_id）
表示: Trip 単位で集約（list --trip / export-md）
```

Trip-level listing is a **display / aggregation view**, not a separate storage model.

### Cross-cutting responsibilities

- vs **Remark** — structured booking vs inline supplement; migration period from Remark allowed
- vs **Note** — confirmation is Reservation; background context is Note
- vs **Expense** — independent entities
- vs **Summary** — no booking numbers in Summary; no trip overview in Reservation
- vs **Itinerary timing** — itinerary `start_time` vs reservation `start_at`/`end_at` (no forced sync)

### Integration surfaces

| Surface | v1.18.0 verdict |
|---|---|
| Export/import v3 `itineraries[].reservations[]` | Aligned with plan |
| `trip export-md` `## Reservations` | First-cut Travel Book section |
| `validate-export` | Adequate for MVR |
| `trip diff` | added / removed / modified — adequate |
| `trip doctor` | Not implemented — improvement candidate |

### Future relationships

- **Participant** — no direct link in v1.18.0; settlement context is future v2
- **Travel Book** — Summary + Reservation + Itinerary + Checklist forms the pre-trip sharing skeleton
- **Photo / Attachment** — booking PDFs/images are future; URL only today

### Improvement candidates (recorded, not implemented)

- `trip doctor` optional checks (missing confirmation, date inversion)
- Markdown inline Reservation under Itinerary sections
- Deferred MVR fields (`reservation_site_name`, contact fields)
- flight / rental_car structured details
- Remark → Reservation migration suggestions
- Expense diff parity
- Small standalone sample trip for guidebook verification
