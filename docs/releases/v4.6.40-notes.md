# v4.6.40 — Reservation write service migration plan

## Summary

**v4.6.40** is an **implementation-ready documentation-only migration plan** (Phase R-1) building on [v4.6.39](../specifications/v4.6.39-reservation-write-path-boundary-review.md) (Phase R-0). It fixes Reservation write service API contracts, display context handling, clear-flag boundaries, and phase gates following Expense and Note patterns.

Plan: [v4.6.40-reservation-write-service-migration-plan.md](../specifications/v4.6.40-reservation-write-service-migration-plan.md)

## Headline decisions

- **Services:** `reservation_add` / `reservation_update` / `reservation_delete` (1 command = 1 module)
- **Add/update result:** `Reservation` + `day_number: Option<i64>` + `itinerary_title: Option<String>` — same optional semantics as read `reservation_show` (not full enriched layer)
- **Add result includes explicit `id`** — for success message (same as Expense/Note)
- **Delete result:** `{ id, provider_name }` — no display context
- **Clear flags:** interpreted inside `reservation_update` service; handler passes raw clap mirror fields only
- **Human output:** `print_reservation_detail_with_context` for add/update (read show path); remove `print_reservation_detail(&conn, ...)` from write at R-3
- **R-5 (v4.6.42):** **implementation cleanup** — delete `print_reservation_detail(&conn, ...)` adapter (unlike Note N-5 docs-only)
- **`load_reservation_display_context`:** shared helper recommended at R-2 (align show + write)
- **`import_reservation_v3`:** separate path — deferred (R-7+)
- **Next implementation:** v4.6.41 — Phase R-2+R-3 together (recommended)
