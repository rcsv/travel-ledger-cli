# v4.8.8 — Fragment apply structured errors limited wiring expansion

## Summary

**v4.8.8** expands structured error wiring on a **limited, safe subset** of paths identified in the v4.8.7 public contract review. Existing clients remain compatible: legacy `errors[]` is preserved, JSON changes are additive only, and `schema_version` stays `2`.

Specification: [v4.8.8-fragment-apply-structured-errors-limited-wiring-expansion.md](../specifications/v4.8.8-fragment-apply-structured-errors-limited-wiring-expansion.md)

Prior contract review: [v4.8.7 public contract review](../specifications/v4.8.7-fragment-apply-structured-errors-public-contract-review.md)

Cargo version bumped to `4.8.8`.

## What ships

- **`APPLY_REQUIRED_DECISION`** now emits **`kind=decision_required`** (was provisional `blocking`)
- **`add_estimate` unsupported target** → `APPLY_UNSUPPORTED_TARGET` on simulate path
- **`delete_estimate` confirm** representative wiring:
  - gate preview mismatch → `APPLY_PREVIEW_MISMATCH` (`confirm_transaction`)
  - revalidate TOCTOU messages → `APPLY_TOCTOU_DRIFT` (`confirm_transaction`)
- Legacy **`errors[]`** mirror preserved on all wired paths
- **`required_decisions[]`** channel unchanged
- **`schema_version: 2`** maintained
- Success reports still **omit empty `structured_errors`**

## JSON contract notes

- **Client branching:** stable **`code`** is authoritative; **`message`** remains unstable
- **`kind=decision_required`:** aligns with v4.8.4 planning; clients may also use `code=APPLY_REQUIRED_DECISION`
- No new registry codes — existing v4.8.5 registry reused

## Not in scope

- Full confirm transaction wiring (other intents)
- `APPLY_SCOPED_WRITE_*` wiring
- Fragment validation / export validation paths
- `schema_version` bump
- DB schema / migration changes
