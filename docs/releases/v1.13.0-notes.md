## Clarified

### Goals / Non-goals

- **Goals:** structured confirmation codes, reservation site info, check-in/return periods, separation from Itinerary, trip-level aggregate display, future doctor/export support.
- **Non-goals:** Expense, Note, Venue, Routing, GPS, maps, subtype tables, external APIs.

### Minimum Viable Reservation (MVR)

Single `reservations` table with Core fields (`itinerary_id`, `reservation_type`, `provider_name`, `confirmation_code`, site fields, `start_at` / `end_at`, contact fields, `remark`, timestamps). No `sort_order` or `details_json` in the first cut.

### Database

- Start with **one table** (`reservations`) only ÔÇö no `flight_reservations` / `hotel_reservations` subtype tables in v1.x.
- Migration pattern aligned with Expense (CREATE TABLE only, no backfill).

### CLI (future)

```text
reservation add / list / show / update / delete
```

Itinerary-scoped add; `list --itinerary` or `list --trip` for aggregate display. No separate `trip reservation-list` command in the first cut.

### Display integration (future)

- `reservation list --trip` ÔÇö primary trip-level reservation view
- `itinerary show` ÔÇö inline reservations under itinerary detail
- `trip export-md` ÔÇö reservation section (later phase)

### Export / import (future ÔÇö not finalized)

Implementation plan lists **schema v3 extension** (`days[].itineraries[].reservations[]`) as a **first candidate only**.

This is **not a committed decision**. Summary, Reservation, Participant, Photo, Attachment, and other models may still be organized before implementation; **Export Schema v4** at Reservation implementation time remains a realistic option. Re-evaluate in the implementation phase.

### Doctor / Markdown (future)

Optional suggestions only (missing confirmation, inverted dates, heuristic ÔÇťreservation expectedÔÇŁ checks). Not in v1.13.0 scope.

### Canonical sample

`okinawa_sesoko_2026` ÔÇö **no bulk Reservation seed** in the first implementation phase. Representative 2ÔÇô3 items or a small demo sample later.

### Future expansion

`flight` and `rental_car` need richer fields eventually; v1.x does not adopt subtype tables. Extension via `details_json`, export nesting, or GUI remains **future decision**.

### Reservation series pause

With responsibilities, entity design, and implementation plan complete, **Reservation documentation pauses here**. Implementation can proceed when prioritized, with export/doctor/markdown integration revisited at that time.
