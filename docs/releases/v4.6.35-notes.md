# v4.6.35 — Note write path boundary review

## Summary

**v4.6.35** is a **documentation-only boundary review** of Note write commands (`add` / `update` / `delete`) after Expense write migration (v4.6.31–v4.6.34), as the first step toward applying the same thin write service pattern to Note.

Review: [v4.6.35-note-write-path-boundary-review.md](../specifications/v4.6.35-note-write-path-boundary-review.md)

## Headline findings

- **Write path** — not service-backed; handler orchestrates `note.rs` mutations
- **Handler** — thin wiring (resolve → mutate → display); not over-loaded
- **Polymorphic owner** — `owner_type` + `owner_id`; validation centered in `resolve_note_owner_for_add` + `add_note`
- **Owner resolution** — list and add share the same resolver — consistent across Trip / Day / Itinerary
- **Read/write asymmetry** — list/show use services; write uses module functions (same pattern as pre-v4.6.33 Expense)
- **No enriched layer** — domain `Note` is sufficient for JSON, human output, and future write service results
- **Display path** — `print_note_detail` is already shared by read show and write add/update (unlike Expense pre-W-3)
- **Adapter cleanup** — no dead_code adapters today; N-5 phase may shrink or be docs-only
- **DB integrity** — no FK on `notes.owner_*`; application-level validation (same model as Expense `itinerary_id`)
- **JSON golden** — unaffected (write emits no JSON)

## Compatibility

- **JSON golden:** unchanged
- **Markdown golden:** colophon version only (`4.6.35`)
- **Runtime:** no code changes

## Non-goals

- Implementation, service layer, output changes
