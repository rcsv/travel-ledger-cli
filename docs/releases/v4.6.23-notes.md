# v4.6.23 — Read-only helper context review

## Summary

**v4.6.23** is a **documentation-only review** of helper-side DB context resolution that remains outside the v4.6.4〜v4.6.21 read-only service boundaries. It inventories `&Connection`-taking display/JSON helpers, classifies JSON vs human responsibilities, compares reservation show vs expense show context ownership, and records itinerary show Reservations as a related aggregate note.

Review: [v4.6.23-read-only-helper-context-review.md](../specifications/v4.6.23-read-only-helper-context-review.md)

## Headline findings

- **Read-only main-path service boundaries remain complete** (v4.6.22)
- **Expense is the primary gap** — `expense_to_json`, `print_expense_detail`, and `print_expense_list` still resolve participant/beneficiary context via `&Connection` in CLI/helpers
- **`ExpenseShowServiceResult { expense }` alone cannot produce CLI-compatible `ExpenseJson`**
- **reservation show** already returns display context in the service result; **expense show** delegates enrichment to helpers
- **itinerary show** still fetches Reservations in the handler (human-only) — separate aggregate topic
- **Acceptable for now** for CLI compatibility; **revisit before GUI / Tauri API work**

## Helper inventory (main read-only path)

| Area | conn-based helper / fetch | Impact |
|---|---|---|
| expense list/show JSON | `expense_to_json` | High — JSON shape |
| expense list/show human | `print_expense_detail`, `print_expense_list` | High — display |
| itinerary show human | handler `list_reservations_for_itinerary` | Medium — human aggregate |
| day / note / reservation (most) | none extra in CLI | Low — service result sufficient |
