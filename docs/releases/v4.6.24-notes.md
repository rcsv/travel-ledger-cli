# v4.6.24 — Expense DTO context ownership review

## Summary

**v4.6.24** is a **documentation-only review** of who should own enriched expense output context (`ExpenseJson`, `expense_to_json`, participant/beneficiary resolution) after v4.6.14 / v4.6.21 service boundaries. It compares ownership options, recommends short-term preservation and a medium-term Option C migration path, and contrasts expense show with reservation show without implementing changes.

Review: [v4.6.24-expense-dto-context-ownership-review.md](../specifications/v4.6.24-expense-dto-context-ownership-review.md)

## Headline findings

- **`ExpenseJson` is the de facto CLI JSON output DTO** — not just a domain model
- **`Expense` alone is insufficient** — needs `paid_by_participant_name`, `beneficiaries[]` with names, and `shared` (derived)
- **`expense_to_json` resolves:** `list_beneficiaries_for_expense`, `get_participant` (payer + each beneficiary), `expense_is_shared` — v4.6.23 list confirmed, no omissions
- **Short term:** keep Option A (helper owns enrichment) for CLI compatibility
- **Medium term:** move enriched context toward service/application layer (Option C recommended migration path)
- **Do not** immediately align with reservation show — expense context is structurally richer
- **itinerary show aggregate** remains a separate topic

## Ownership recommendation

| Horizon | Policy |
|---|---|
| Now | Option A — no changes to `ExpenseJson` / `expense_to_json` |
| Before GUI / Tauri | Decide Option C vs D |
| Migration path | **Option C** — service returns `Expense` + context parts; CLI maps to unchanged `ExpenseJson` |

## Non-goals

- New DTO types or `expense_to_json` moves
- Service / handler changes
- JSON shape or human output changes
