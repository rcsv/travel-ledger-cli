# v4.6.36 — Note write service migration plan

## Summary

**v4.6.36** is an **implementation-ready documentation-only migration plan** (Phase N-1) building on [v4.6.35](../specifications/v4.6.35-note-write-path-boundary-review.md) (Phase N-0). It fixes Note write service API contracts, owner validation boundaries, result types, and phase gates following the Expense write migration pattern.

Plan: [v4.6.36-note-write-service-migration-plan.md](../specifications/v4.6.36-note-write-service-migration-plan.md)

## Headline decisions

- **Services:** `note_add` / `note_update` / `note_delete` (1 command = 1 module)
- **Add/update result:** `Note` — same as read `note_show` (no enriched layer)
- **Delete result:** `{ id, title: Option<String> }` — title-only display unchanged
- **Owner validation:** `resolve_note_owner_for_add` called from `note_add` service only; `add_note` defensive double-check **retained**
- **Handler keeps:** success `println!`, `print_note_detail` (add/update) — already shared with read show
- **Params:** `NoteAddParams` / `NoteUpdateParams` / `NoteDeleteParams` (clap mirrors)
- **`import_export_notes`:** separate path — commonization deferred (N-7+)
- **`last_insert_rowid` → `RETURNING id`:** Phase N-H optional hardening after N-3
- **N-5 adapter cleanup:** **not needed** — v4.6.38 docs-only closeout recommended
- **Repository / DB FK:** out of scope — defer to v4.7.x / migration runner track
- **Next implementation:** v4.6.37 — Phase N-2+N-3 together (recommended)
