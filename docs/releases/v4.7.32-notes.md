# v4.7.32 — Fragment apply delete_itinerary --confirm (P-6j)

## Summary

**v4.7.32** adds **P-6j Phase 2 confirm path**: `fragment apply --confirm` performs a **row-only delete** of a resolved **itinerary target** (`itinerary_items` exactly 1 row) after TOCTOU revalidation and blocking-child checks, returning `deleted_itinerary_id` in the gate report. Child entities are **not deleted** and `--dry-run` behavior remains read-only.

Policy / planning: [v4.7.30-p6j-destructive-structural-apply-policy.md](../specifications/v4.7.30-p6j-destructive-structural-apply-policy.md) · [v4.7.31-p6j-delete-itinerary-dry-run.md](../specifications/v4.7.31-p6j-delete-itinerary-dry-run.md)

## What changed

- **New:** `fragment apply --confirm` — `delete_itinerary` for **itinerary target only**
- **New:** non-cascade delete path `delete_itinerary_item_row_only` — deletes exactly one row in `itinerary_items`
- **New:** gate report `deleted_itinerary_id`
- **Added:** TOCTOU revalidation before DELETE — identity + blocking children re-count from DB
- **Unchanged:** `fragment apply --dry-run` delete preview behavior, schema v8, JSON golden, DB migration

## Usage

```bash
caglla fragment apply fragment.json --confirm --trip 1 --json
caglla itinerary list 1
```

## Safety

- Confirm deletes **exactly one** `itinerary_items` row (transactional; row count must be 1)
- **No cascade delete** — does not delete Expense / Estimate / Reservation / Note entity
- Trip / Day / unresolved / ambiguous target → blocking
- `required_decisions` / `adoption_hints.conflicts` → blocking
- blocking children (Expense / Estimate / Reservation / Note entity) → blocking; DB unchanged
- inline `itinerary.note` text → **not** a blocking child
- Venue / Place → **not** blocking (no Venue model in v4.7.32)
