# v4.11.0 — Desktop trip and activity creation workflows

## Summary

**v4.11.0** turns the Desktop developer preview into a practical write-enabled travel-planning workflow:

- Create a Trip from the Trip Navigator, including its generated Days
- Add an activity to the selected Day with a required title and optional start time, location, and note
- Read committed writes back from SQLite so the Trip List, Day 1, and timeline reflect the database source of truth
- Keep activity quick add append-only and sequence-first, using the existing sparse Itinerary ordering
- Preserve structured Desktop write errors, duplicate-submit protection, and stale-target protection

The user-facing Desktop vocabulary is **Activity**. The internal domain and storage model remains **Itinerary**.

## Desktop workspace

- Organizes the selected Trip as a Trip Context Header plus Plan workspace
- Keeps Trip metadata, Day selection, and the timeline in one continuous workspace document
- Presents Settings as an independent full-width application view
- Constrains the app shell to the window viewport, with independently scrollable Trip Navigator and Trip Workspace regions

## Write boundary

- Trip creation and activity quick add use public Core write services shared with the CLI where applicable
- Core owns normalization, validation, target resolution, and ordering
- Trip plus generated Days are committed atomically
- Activity quick add appends to the selected Day and refreshes only that Day's timeline after a successful write
- The database schema, migrations, Travel Ledger export/import schema, and CLI command surface are unchanged

## Packaging note

- Desktop remains a **developer preview** that is run from source
- Desktop bundling remains disabled (`bundle.active = false`)
- GitHub Release assets remain the CLI archives for Linux amd64, macOS arm64, and Windows amd64 only
- No Desktop installer is included in this release
