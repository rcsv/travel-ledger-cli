# v4.7.36 — P-6k reorder_itinerary --confirm (same-day)

## Summary

**v4.7.36** adds **`reorder_itinerary --confirm`** (same-day only) for Proposal Fragment apply.
The confirm path applies the same reorder that dry-run previews by updating **only**:

- `itinerary_items.sort_order`
- `itinerary_items.updated_at`

within the target Day.

Safety highlights:

- **Transaction required** (all checks + updates are performed atomically).
- **TOCTOU revalidation inside the transaction**: re-fetch Day rows, re-resolve selectors, and re-check baseline match before writing.
- **Row count verification**: updates must affect exactly the expected rows; otherwise rollback.
- **Same-day only**: cross-day move remains **rejected / deferred** (future intent).
- **Dry-run unchanged**: v4.7.35 dry-run preview behavior remains read-only.
- **ID-based selectors preserved** for future GUI drag-and-drop (`expected_order` / `after_order` arrays).

## What changed

- **Added:** `fragment apply --confirm` support for `reorder_itinerary` (same-day)
- **Added:** confirm integration tests for success + rejection cases (missing fields, baseline mismatch, set mismatch, no-op, cross-day/ambiguous selector)
- **Unchanged:** schema v8, DB migration, JSON golden, Venue / Place

## Compatibility / mutation boundary

- **DB mutation**: `itinerary_items.sort_order` + `itinerary_items.updated_at` only
- **No changes**: `day_id`, `title`, `start_time`, child entities (expenses/estimates/reservations/notes)
- **schema v8**: unchanged
- **DB migration**: none
- **JSON golden**: unchanged
- **Markdown golden**: colophon version only (`4.7.36`)
