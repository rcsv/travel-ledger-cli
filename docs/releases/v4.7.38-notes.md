# v4.7.38 — P-6l move_itinerary dry-run

## Summary

**v4.7.38** adds **dry-run** support for the structural apply intent **`move_itinerary`** (cross-day only).
It previews moving **exactly one** itinerary item from a **source Day** to a **destination Day** within the **same Trip**, including source and destination order changes, **without mutating the DB**.

Design boundary (preserved):

```text
reorder_itinerary = same-day sort_order change
move_itinerary    = cross-day day_id + sort_order change
```

Spec: [v4.7.37-p6l-cross-day-move-planning.md](../specifications/v4.7.37-p6l-cross-day-move-planning.md)

## What changed

- **Added:** `fragment apply --dry-run` preview for `move_itinerary` (cross-day)
- **Added:** structured `preview.move_preview` with source/destination order changes
- **Added:** source/destination baseline validation (`expected_source_order` / `expected_destination_order`)
- **Added:** ID-based order selector support (GUI drag-and-drop friendly)
- **Added:** fixtures + CLI integration tests for valid move, invalid inputs, same-day rejection, ambiguous selectors, and confirm blocking
- **Unchanged:** `reorder_itinerary` behavior, schema v8, DB migration, JSON golden

## Sort order preview strategy

Dry-run follows the v4.7.37 planning recommendation:

```text
minimal updates + deterministic slot generation
```

- **source Day:** remove moved itinerary; reassign existing source slots to remaining items
- **destination Day:** insert moved itinerary; use midpoint or head/tail `SORT_ORDER_STEP` where possible; block if no safe deterministic slot exists
- **no global renumbering / compaction** in v4.7.38

## Compatibility / safety

- **DB mutation:** none (dry-run only)
- **confirm:** blocked / unsupported (`move_itinerary --confirm` not implemented)
- **same-day move:** rejected (use `reorder_itinerary`)
- **schema v8:** unchanged
- **DB migration:** none
- **JSON golden:** unchanged
- **Markdown golden:** colophon version only (`4.7.38`)
