# v4.7.43 — P-6n Fragment apply add_estimate --confirm

## Summary

**v4.7.43** adds confirm support for Proposal Fragment **`add_estimate`** on existing Itinerary targets.

```text
fragment apply <fragment.json> --confirm --trip <id>
intent: add_estimate
target_type: itinerary
```

The operation runs the existing Fragment apply gate, reparses and revalidates the candidate fields, re-resolves the current Itinerary target, verifies agreement with the gate preview (TOCTOU safety), inserts **exactly one** row into `estimates`, verifies the stored row, and returns `inserted_estimate_id`. No other entities are mutated.

Spec: [v4.7.43-fragment-apply-add-estimate-confirm.md](../specifications/v4.7.43-fragment-apply-add-estimate-confirm.md)

## Semantic boundary

```text
Estimate = Planned Money
Expense  = Actual Money
```

```text
add_estimate -> estimates / Planned Money
add_expense  -> expenses  / Actual Money
```

`add_estimate` must never create or update an Expense row. There is no automatic Estimate-to-Expense conversion.

## Mutation boundary

Allowed mutation:

```text
INSERT exactly one estimates row
```

Inserted columns:

```text
itinerary_id
title
amount
currency
note
sort_order
created_at
updated_at
```

Not included:

- Expense INSERT / UPDATE
- Reservation INSERT / UPDATE
- Note entity INSERT
- Itinerary UPDATE
- Trip / Day mutation
- Participant / payer / beneficiary changes
- child deletion
- Estimate-to-Expense conversion
- Venue / Place mutation
- DB migration / schema v8 change / export schema change

## Amount handling (minor units safety)

Candidate `amount` supports:

- string decimal text (e.g. `"12.50"` with `USD`)
- integer minor units (e.g. `1250` with `USD`)

Both store the same DB value:

```text
1250
```

This release introduces a narrow write helper that accepts already-normalized minor units:

```text
estimate::add_estimate_minor_units(...)
```

The existing CLI-facing `estimate add` path remains decimal-text based and now calls the helper after parsing. Fragment confirm uses the already-parsed minor units directly, preventing double-parsing regressions (e.g. `1250` → `125000`).

## Confirm report

The Fragment apply report adds:

```text
inserted_estimate_id
```

The field is present only on successful confirm, and is absent on dry-run and all invalid / rolled-back confirms.

## Transaction and TOCTOU safety

Confirm performs the mutation inside:

```text
with_transaction("add_estimate confirm")
```

Inside the transaction it:

- reparses the Fragment JSON
- confirms `intent == add_estimate`
- reruns `parse_add_estimate_fields`
- reruns target resolution and resolves current itinerary DB ID
- verifies agreement with gate preview for:
  - target itinerary ID
  - amount / currency / title / note / sort_order
- inserts exactly one Estimate
- loads the inserted row and verifies stored fields

Any mismatch returns an error and rolls back the transaction, leaving domain tables unchanged.
