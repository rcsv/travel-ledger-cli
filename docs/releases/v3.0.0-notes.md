# v3.0.0 — Shared Expense

## Summary

First **v3** feature release. Extends **Expense** as a Transaction Record with structured **payer** and **beneficiaries** (equal split recording only), opt-in CLI flags, export **schema v5**, and integration with diff / Markdown export / doctor / advisor.

Requires **v2 Participant Foundation** — structured expense fields reference Trip-scoped participants by name.

### Terminology

| Term | Meaning |
|---|---|
| **Personal expense** | No beneficiaries — same meaning as v1/v2 (`beneficiary` rows = 0) |
| **Shared expense** | One or more beneficiaries — equal split **recording** only (no settlement math) |
| **Payer** | Who paid — optional structured link via `paid_by_participant_id` |
| **Beneficiary** | Who the expense is for — rows in `expense_beneficiaries` |

personal / shared は **明示列なし**。beneficiary 行数で判定します。

## Added

### Database

`expenses` table extension:

| Column | Notes |
|---|---|
| `paid_by_participant_id` | INTEGER NULL — structured payer (no FK; Trip-scoped validation on write) |

New table `expense_beneficiaries`:

| Column | Notes |
|---|---|
| `expense_id` | NOT NULL |
| `participant_id` | NOT NULL |
| `sort_order` | NOT NULL DEFAULT 0 |
| `created_at` / `updated_at` | timestamps |

Constraints / indexes:

- `UNIQUE (expense_id, participant_id)`
- `idx_expenses_paid_by_participant`
- `idx_expense_beneficiaries_expense` / `idx_expense_beneficiaries_participant`

Existing DBs: `migrate_expenses_shared_expense` on `open_db` — idempotent column/table creation. Existing expenses remain personal (`paid_by_participant_id = NULL`, no beneficiary rows).

`paid_by_name` is **retained** for free-text payer and backward compatibility.

### CLI

Existing minimal path **unchanged**:

```bash
expense add --itinerary 12 --amount 1500 --currency JPY
expense add --itinerary 12 --amount 980 --currency JPY --paid-by-name 太郎
```

New **opt-in** options on `expense add` / `expense update`:

| Option | Behavior |
|---|---|
| `--paid-by-participant <id\|name>` | Structured payer; syncs `paid_by_name` to Participant.name |
| `--beneficiary <id\|name>` (repeatable) | Beneficiaries — **full replace** on update |
| `--shared-with all` | All Trip participants as beneficiaries — **full replace** on update |
| `--clear-paid-by` | update only — clear structured payer and `paid_by_name` |
| `--clear-beneficiaries` | update only — remove all beneficiaries → personal |

Rules:

- `--shared-with` and `--beneficiary` are **mutually exclusive** on **add and update**
- Structured options require at least one Participant on the Trip (`--paid-by-name` alone still works with zero participants)
- Duplicate beneficiary on one expense → error
- Ambiguous participant name (duplicate names on Trip) → error

### Export / import (schema v5)

Current `trip export` writes **`schema_version: 5`**. Expense objects gain optional fields:

```json
{
  "schema_version": 5,
  "participants": [
    { "name": "Alex", "sort_order": 0, "is_self": true },
    { "name": "Jordan", "sort_order": 1, "is_self": false }
  ],
  "days": [
    {
      "day_number": 1,
      "itineraries": [
        {
          "title": "Lunch",
          "expenses": [
            {
              "amount": 4000,
              "currency": "JPY",
              "paid_by_participant_ref": "Alex",
              "beneficiaries": [
                { "participant_ref": "Alex" },
                { "participant_ref": "Jordan" }
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

- `participant_ref` = **`participants[].name` exact match** (internal DB id not exported)
- **v4 import** remains supported — omitted v5 fields → personal expense
- **v3 import** remains supported
- Duplicate participant names + ref ambiguity → **import / validate-export error**
- `validate-export`: v5 ref rules; v4 files skip v5-only ref checks (v4 participant rules still apply)

### Diff

`trip diff` reports Expense changes when **both** exports are schema v5+:

- added / removed (existing expense key matching)
- modified: `payer`, `beneficiaries`

### Markdown export

Expense lines in `trip export-md` may include:

```text
- 入館料: 2,500 JPY — Paid by: Alex · Shared: Alex, Jordan
```

### Doctor / advisor

New **warnings** (in addition to existing Participant warnings):

| Code | Condition |
|---|---|
| `DuplicateParticipantNames` | Same Trip has duplicate participant names (import ref risk) |
| `SharedExpenseSingleBeneficiary` | Shared expense with exactly one beneficiary |
| `PaidByNameParticipantMismatch` | Structured payer set but `paid_by_name` ≠ Participant.name |

### Participant delete

Deleting a participant:

1. Sets `paid_by_participant_id = NULL` on affected expenses
2. Deletes their `expense_beneficiaries` rows (shared expense may become personal)

## Migration / compatibility

| Topic | Notes |
|---|---|
| **Existing `caglla.db`** | Opening the DB runs `migrate_expenses_shared_expense` — no manual step |
| **Existing expenses** | Unchanged semantics — personal until you add beneficiaries |
| **Export v4 files** | Import still works; payer/beneficiary fields omitted → personal |
| **Export v5** | Use v3.0.0+ CLI to export/import with shared expense fields |
| **Downgrade** | v2.x CLI cannot read new DB columns/tables; export v5 JSON before downgrade |

## Upgrade

1. Replace binary with v3.0.0 build (or `cargo build --release` from tag `v3.0.0`).
2. Run any command — DB migration applies automatically.
3. Register participants (`participant add --trip <id> ...`) before using structured expense options.
4. Re-export trips to obtain schema v5 JSON with `paid_by_participant_ref` / `beneficiaries[]`.
