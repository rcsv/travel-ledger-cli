# v4.7.39 — P-6l move_itinerary --confirm

## Summary

**v4.7.39** adds **`move_itinerary --confirm`** (cross-day only) for Proposal Fragment apply.
The confirm path applies the same cross-day move that v4.7.38 dry-run previews by moving **exactly one** itinerary from a **source Day** to a **destination Day** within the **same Trip**.

Design boundary (preserved):

```text
reorder_itinerary = same-day sort_order change
move_itinerary    = cross-day day_id + sort_order change
```

Spec: [v4.7.37-p6l-cross-day-move-planning.md](../specifications/v4.7.37-p6l-cross-day-move-planning.md)

## What changed

- **Added:** `fragment apply --confirm` support for `move_itinerary` (cross-day)
- **Added:** shared `compute_move_itinerary_plan` for dry-run and confirm validation / sort_order planning
- **Added:** transaction-protected confirm with TOCTOU revalidation inside the transaction
- **Added:** preview / plan consistency verification before writes
- **Added:** row count verification for every UPDATE (exactly 1 row per planned change)
- **Added:** confirm integration tests for success, rejection, and mutation boundaries
- **Added:** raw DB test reinforcement for denormalized `day_id` / `day` synchronization after confirm
- **Unchanged:** `reorder_itinerary` behavior, schema v8, DB migration, JSON golden

## Safety highlights

- **Transaction required** — all revalidation + updates are atomic
- **TOCTOU revalidation** — re-fetch source/destination Days, re-resolve selectors, re-check baselines inside the transaction
- **Preview guard** — recomputed plan must match gate `move_preview` before any write
- **Same-day move rejected** — use `reorder_itinerary`
- **Cross-trip move** — unsupported / rejected
- **Dry-run unchanged** — v4.7.38 dry-run remains read-only
- **Child entities** — remain attached to the moved itinerary by itinerary ID; not moved, deleted, or rewritten independently

## Mutation boundary

**Moved itinerary row** may update:

```text
itinerary_items.day_id
itinerary_items.day
itinerary_items.sort_order
itinerary_items.updated_at
```

**Affected source / destination ordering rows** may update:

```text
itinerary_items.sort_order
itinerary_items.updated_at
```

`itinerary_items.day` is a required denormalized `day_number` column kept in sync with `day_id` (same write contract as `day swap`, `itinerary update --day`, and insert paths).

**Not updated:** `title`, `start_time`, `category`, `location`, `note`, Trip rows, Day rows, child entities, Venue / Place data, schema / migration data.

## Test reinforcement

`cli_fragment_apply_move_itinerary_confirm_mutation_boundary` includes raw SQLite assertions:

- `itinerary_items.day_id` equals destination Day `days.id`
- `itinerary_items.day` equals destination `day_number`

This verifies denormalized column sync directly, not only JOIN-based CLI output.
