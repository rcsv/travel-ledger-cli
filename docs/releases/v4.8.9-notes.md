# v4.8.9 — Fragment apply confirm transaction structured errors follow-up

## Summary

**v4.8.9** extends `delete_estimate` **confirm transaction** structured error wiring to cover **baseline mismatch** and **scoped write** outcomes. This is a focused follow-up — not full confirm transaction migration. Existing clients remain compatible: legacy `errors[]` is preserved, JSON changes are additive only, and `schema_version` stays `2`.

Specification: [v4.8.9-fragment-apply-confirm-transaction-structured-errors-follow-up.md](../specifications/v4.8.9-fragment-apply-confirm-transaction-structured-errors-follow-up.md)

Prior release: [v4.8.8 limited wiring expansion](../specifications/v4.8.8-fragment-apply-structured-errors-limited-wiring-expansion.md)

Cargo version bumped to `4.8.9`.

## What ships

- **confirm baseline mismatch** → `APPLY_BASELINE_MISMATCH` (`confirm_transaction`)
- **scoped DELETE 0 rows** → `APPLY_SCOPED_WRITE_ZERO_ROWS` (`confirm_transaction`)
- **scoped DELETE multiple rows** → `APPLY_SCOPED_WRITE_MULTIPLE_ROWS` (`confirm_transaction`; classifier-level test)
- Legacy **`errors[]`** mirror preserved
- **`schema_version: 2`** maintained
- Success reports still **omit empty `structured_errors`**
- **No new registry codes**

## JSON contract notes

- **Client branching:** stable **`code`** is authoritative; **`message`** remains unstable
- Optional **`field_path`** on baseline mismatch when derivable (`candidate_content.expected_*`)
- Additive exposure only — existing clients can ignore new `structured_errors` entries

## Not in scope

- Full confirm transaction wiring (other intents)
- Fragment validation / export validation paths
- `schema_version` bump
- DB schema / migration changes
