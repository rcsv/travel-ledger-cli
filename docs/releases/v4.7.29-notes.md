# v4.7.29 — Fragment apply update_itinerary --confirm (P-6i)

## Summary

**v4.7.29** completes **P-6i confirm path**: `fragment apply --confirm` updates exactly one Itinerary row for `update_itinerary` intent on **itinerary target only**, using the same gate as v4.7.28 dry-run plus TOCTOU revalidation before write.

Spec: [v4.7.29-fragment-apply-update-itinerary-confirm.md](../specifications/v4.7.29-fragment-apply-update-itinerary-confirm.md)

## What changed

- **New:** `fragment apply --confirm` — `update_itinerary` for itinerary target
- **New:** gate report `updated_itinerary_id` on confirm success
- **New:** TOCTOU revalidation before `itinerary_items` UPDATE
- **Extended:** `validate_confirm_scope` — supports `update_itinerary` (itinerary only)
- **Reused:** `itinerary::update_itinerary_item` write path (allowed MVP fields only)
- **Unchanged:** `update_itinerary --dry-run`, schema v8, JSON golden

## Usage

```bash
caglla fragment apply fragment.json --dry-run --trip 1 --output preview.json
caglla fragment apply fragment.json --confirm --trip 1 --json
caglla itinerary show <id> --json
caglla trip export 1 --output trip.json
```

## Safety

- Itinerary UPDATE only — one row, allowed MVP fields
- trip / day target → blocking
- unresolved / ambiguous / required_decisions / conflicts / baseline mismatch → blocking
- no-op / invalid category / time / negative numeric → blocking
- TOCTOU mismatch → blocking before UPDATE
- child Expense / Reservation / Note unchanged
- `--dry-run --confirm` → error
