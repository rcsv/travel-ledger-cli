# v4.7.27 — Fragment apply add_reservation --confirm (P-6h)

## Summary

**v4.7.27** completes **P-6h confirm path**: `fragment apply --confirm` inserts one Reservation for `add_reservation` intent on **itinerary target only**, using the same gate as v4.7.26 dry-run.

Spec: [v4.7.27-fragment-apply-add-reservation-confirm.md](../specifications/v4.7.27-fragment-apply-add-reservation-confirm.md)

## What changed

- **New:** `fragment apply --confirm` — `add_reservation` for itinerary target
- **New:** gate report `inserted_reservation_id` on confirm success
- **Extended:** `validate_confirm_scope` — supports `add_reservation` (itinerary only)
- **Reused:** `services::reservation_add::add_reservation` write path
- **Unchanged:** `add_reservation --dry-run`, schema v8

## Usage

```bash
caglla fragment apply fragment.json --dry-run --trip 1 --output preview.json
caglla fragment apply fragment.json --confirm --trip 1 --json
caglla reservation list --itinerary 1 --json
caglla reservation show <id>
```

## Safety

- Reservation INSERT only — no update/delete
- trip / day target → blocking
- invalid `reservation_type` / missing provider → blocking
- required decisions / unresolved / ambiguous → no DB write
- `note` / `memo` map only to Reservation `remark`; no Note entity is created
- `--dry-run --confirm` → error
