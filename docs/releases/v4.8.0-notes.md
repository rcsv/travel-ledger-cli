No Rust code, tests, fixtures, or CLI behavior changes. Cargo version bumped to `4.8.0`.

Planning spec: [v4.8.0-p6p-delete-estimate-planning.md](../specifications/v4.8.0-p6p-delete-estimate-planning.md)

This planning change includes no runtime behavior changes.

## Recommended contract

- intent: `delete_estimate`
- target: one existing Itinerary
- canonical Estimate identity: `candidate_content.estimate_id`
- required ownership check: Estimate belongs to resolved target Itinerary
- missing / already deleted: blocking, not idempotent success
- dry-run: full delete snapshot and export preview, DB unchanged
- confirm: candidate / identity / snapshot agreement inside one transaction
- write: `DELETE ... WHERE id = ? AND itinerary_id = ?`
- affected rows: exactly one; otherwise error and rollback
- success field: `deleted_estimate_id`
- Estimate / Expense boundary: Planned Money row only; Expense unchanged

## Required decisions

- Adopt `estimate_id`; reject title / sort_order / amount selectors.
- Treat already-deleted targets as blocking.
- Defer strict idempotency until retry token / operation journal design exists.
- Require a full gate snapshot plus optional author-supplied `expected_*` baselines.
- Recheck `adoption_hints.required_decisions` and authoritative blocking conditions in the confirm transaction.
- Keep current string errors for the initial implementation; structured codes / field paths remain cross-cutting work.
- Do not introduce public Proposal Fragment schema versioning in this planning release.

## Findings

- **P0:** none known at planning time
- **P1:** identity / ownership, destructive preview, TOCTOU, scoped exactly-one DELETE, rollback, boundary preservation, not-found semantics
- **P2:** strict idempotency / retry token, structured errors, version / ETag, public schema versioning, undo / recovery
- **P3:** shared parser / resolver test layering, unit-level fault injection, informative-only titles, minimal DTO duplication
