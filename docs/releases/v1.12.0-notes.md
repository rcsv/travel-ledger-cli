## Clarified

### Web 版 Caglla.Travel からの教訓

- Reservation type + provider + confirmation + type-specific information の思想は継承。
- CLI / Travel Ledger では **初手から種別専用テーブルには進まない**。

### Reservation Core fields

共通フィールド案を整理（`itinerary_id`, `reservation_type`, `provider_name`, `confirmation_code`, `reservation_site_name` / `url`, `start_at` / `end_at`, contact fields, `remark` など）。

### `reservation_type` 候補

```text
hotel / flight / restaurant / rental_car / activity / parking / ticket / other
```

（enum 候補の整理のみ — 未実装）

### 種別ごとの適合度

| 種別 | v1.12.0 見立て |
|---|---|
| hotel, restaurant, activity, parking, ticket | **Core fields で表現できる可能性が高い** |
| flight, rental_car | **将来拡張が必要になりやすい**（便名・空港・車両クラス等） |

### 専用テーブルに進まない

v1.12.x では `flight_reservations` 等の **種別専用テーブルは作らない**。Core で多くを表現し、schema 肥大化と CLI 入力の複雑化を避ける。

### 拡張実装は将来判断

以下は候補として列挙のみ — **本リリースでは未決定**:

```text
A. Core table + details_json
B. Core table + subtype tables
C. Export only nested details
D. CLI Core only, GUI detail extension
```

### Routing

引き続き **Reservation の対象外**。空間的移動経路は将来の独立モデル。

### 予約サイト taxonomy

Web 版のアイコン用内部分類（楽天・じゃらん等）は優先度低。`reservation_site_name` / `reservation_site_url` で十分。
