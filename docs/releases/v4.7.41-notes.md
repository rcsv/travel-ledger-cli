# v4.7.41 — P-6n add_estimate Proposal Fragment planning (documentation-only)

## Summary

**v4.7.41** adds a **documentation-only planning spec** for the next Proposal Fragment intent, **`add_estimate`**. It defines how a Fragment can propose **Planned Money** on an existing Itinerary without recording an actual expense.

Spec: [v4.7.41-p6n-add-estimate-planning.md](../specifications/v4.7.41-p6n-add-estimate-planning.md)

## Planning conclusions

- **`add_estimate`** is the next Proposal Fragment intent
- Inserts Planned Money into the **existing `estimates` table** — no new DB entity
- Targets exactly **one existing Itinerary** (`target_type = itinerary`)
- **No** DB migration, schema v8, or export schema change required
- Estimate and Expense remain semantically separate

Canonical boundary:

```text
Estimate = Planned Money
Expense  = Actual Money
```

- `add_estimate` must **never** create an Expense row
- `add_expense` must **not** be used for uncertain, approximate, or planned costs

## Intent / target contract

```text
intent:      add_estimate
target_type: itinerary
```

Target resolution: exactly one existing Itinerary; itinerary ID preferred for GUI / API; title / sort_order remain CLI conveniences. Missing / unresolved / ambiguous / cross-Trip targets block. Trip and Day targets are unsupported.

## Candidate content contract

```json
{
  "intent": "add_estimate",
  "candidate_content": {
    "amount": "14000",
    "currency": "JPY",
    "title": "Aquarium tickets",
    "note": "Estimated total for five people",
    "sort_order": 0
  }
}
```

| Field | Required | JSON type |
|---|---:|---|
| `amount` | yes | string decimal text or integer minor units (`add_expense` conventions) |
| `currency` | yes | string |
| `title` | no | string; empty normalizes to null |
| `note` | no | string; remains an Estimate note |
| `sort_order` | no | integer; defaults to `0` |

## Planned dry-run direction (v4.7.42)

Resolve the target Itinerary, validate amount / currency / optional fields, produce a structured `estimate_preview`, expose `estimates_before` / `estimates_after`, append one candidate Estimate to preview export under `days[].itineraries[].estimates[]`, keep DB unchanged, and suppress preview export on invalid reports. **Not implemented in v4.7.41.**

## Planned confirm direction (v4.7.43)

```text
INSERT exactly one estimates row
```

Allowed future inserted columns: `itinerary_id`, `title`, `amount`, `currency`, `note`, `sort_order`, `created_at`, `updated_at`. Planned result: `inserted_estimate_id`. **Not implemented in v4.7.41.**

## Amount / currency policy

Reuse the shared money contract: existing amount parsing and currency validation, DB amount as minor-unit INTEGER, reject negatives, preserve decimal precision and unknown three-letter currency behavior. No Proposal-specific money parsing, no FX, no tax, no unit × quantity, no payer / beneficiaries.

## Relationship to add_expense

```text
add_estimate → estimates
add_expense  → expenses
```

Later reuse may cover target resolution, amount parsing, currency validation, gate reports, and integration-test patterns — **without** a broad money or generic financial-intent refactor.

## Relationship to delete_itinerary

```text
An Itinerary with Estimate children remains blocking for delete_itinerary.
```

The `add_estimate` plan does not weaken or alter delete safety behavior.
