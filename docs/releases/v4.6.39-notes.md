# v4.6.39 — Reservation write path boundary review

## Summary

**v4.6.39** is a **documentation-only boundary review** of Reservation write commands (`add` / `update` / `delete`) after Expense and Note write migrations, as the first step toward applying the same thin write service pattern.

Review: [v4.6.39-reservation-write-path-boundary-review.md](../specifications/v4.6.39-reservation-write-path-boundary-review.md)

## Headline findings

- **Write path** — not service-backed; handler orchestrates `reservation.rs` mutations
- **Parent reference** — `itinerary_id` fixed (same model as Expense); app-level validation via `get_itinerary_item`
- **Read/write asymmetry** — list/show use services; write uses module functions
- **Display context** — read `reservation_show` returns `{ reservation, day_number, itinerary_title }`; write add/update use `print_reservation_detail(&conn, ...)` adapter to re-fetch context
- **No full enriched layer** — no participant resolution; display context only (lighter than Expense, richer than Note)
- **Update complexity** — handler maps five `clear_*` flags to `Option<Option<&str>>` (main service migration target)
- **Delete snapshot** — `{ id, provider_name }` (like Note delete title)
- **R-5 outlook** — one adapter to remove after R-3 (`print_reservation_detail(&conn, ...)`)
- **Difficulty** — between Note and Expense
- **JSON golden** — unaffected (write emits no JSON)

## Compatibility

- **JSON golden:** unchanged
- **Markdown golden:** colophon version only (`4.6.39`)
- **Runtime:** no code changes

## Non-goals

- Implementation, service layer, output changes
