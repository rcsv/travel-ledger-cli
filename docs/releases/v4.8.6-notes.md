# v4.8.6 — Fragment apply JSON structured_errors[] exposure

## Summary

**v4.8.6** exposes v4.8.5's internal structured errors as an **additive** `structured_errors[]` field on `FragmentApplyDryRunReport` JSON output.

Existing `errors: Vec<String>` and human CLI output remain unchanged. `schema_version` stays `2`.

Specification: [v4.8.6-fragment-apply-json-structured-errors-exposure.md](../specifications/v4.8.6-fragment-apply-json-structured-errors-exposure.md)

Internal model basis: [v4.8.5 internal structured error model](../specifications/v4.8.5-fragment-apply-internal-structured-error-model.md)

## What shipped

- **`structured_errors[]`** on `--json` report (additive)
- Empty array omitted on success reports (`skip_serializing_if`)
- Each element: `code`, `kind`, `message`, `phase` + optional `field_path` / `intent` / `target_type`
- **`errors[]`** mirror preserved for backward compatibility
- **`schema_version: 2`** unchanged
- Exposure limited to v4.8.5 wired blocking paths only

## JSON contract notes

- **`code` / `kind` / `phase`** — stable contract candidates for API/GUI
- **`message`** — human-readable / unstable (same as legacy `errors[]` text)
- Old clients can ignore unknown `structured_errors` field

## Not in scope

- CLI behavior / human output changes
- `errors[]` removal or rename
- Full error path migration
- confirm transaction wiring (preview mismatch, TOCTOU, scoped write)
- DB schema / migration
- retry token / ETag / optimistic concurrency
- public schema versioning hardening (v4.8.7 candidate)
