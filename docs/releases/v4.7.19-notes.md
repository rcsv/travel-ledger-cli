# v4.7.19 — Fragment apply --confirm (P-6d)

## Summary

**v4.7.19** implements **P-6d**: `fragment apply <file> --confirm --trip <id>` — adopt a Proposal Fragment into an **existing Trip Day** with a single new itinerary insert, after the same gate as dry-run.

Spec: [v4.7.19-fragment-apply-confirm.md](../specifications/v4.7.19-fragment-apply-confirm.md)

## What changed

- **New:** `fragment apply --confirm --trip <id>` — Day target + `add_itinerary` only
- **New:** gate report schema v2 — `confirm`, `inserted_itinerary_id` fields
- **Unchanged:** `--dry-run` — read-only DB access, no Trip domain data mutation
- **DB write:** `add_itinerary_item` insert only — no update/delete/reorder
- **Not changed:** fragment validate (file-only), proposal materialize, schema v8

## Usage

```bash
# Recommended flow
caglla fragment apply fragment.json --dry-run --trip 1 --output preview.json
caglla trip validate-export preview.json
caglla fragment apply fragment.json --confirm --trip 1

caglla itinerary list 1
caglla day show 1 1
```

## Safety

- No Trip domain data mutation without explicit `--confirm`
- `--dry-run` — read-only DB access, in-memory simulation only
- `--dry-run` and `--confirm` **cannot be combined** — `dry-run means no Trip domain data side effects`
- `--confirm` blocked when: unsupported intent/target, unresolved target, unresolved required decisions, gate errors
- Typical flow: `--dry-run` → review → separate `--confirm` run

## Compatibility

- **JSON golden:** unchanged
- **Markdown golden:** colophon version only (`4.7.19`)
- **Gate report JSON:** schema version 2 when using `--json`
