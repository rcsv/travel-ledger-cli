# v4.8.2 — P-6p delete_estimate --confirm

## Summary

**v4.8.2** implements Proposal Fragment **`delete_estimate --confirm`** (P-6p).

This release adds the scoped DELETE path for existing Estimates via `fragment apply --confirm`, reusing the v4.8.1 dry-run parser and preview contracts.

Implementation spec: [v4.8.2-p6p-delete-estimate-confirm.md](../specifications/v4.8.2-p6p-delete-estimate-confirm.md)

Dry-run spec: [v4.8.1-p6p-delete-estimate-dry-run.md](../specifications/v4.8.1-p6p-delete-estimate-dry-run.md)

Planning spec: [v4.8.0-p6p-delete-estimate-planning.md](../specifications/v4.8.0-p6p-delete-estimate-planning.md)

## What shipped (confirm only)

- **`fragment apply --confirm --trip <id>`** supports intent `delete_estimate`
- **Itinerary target only** — reuses existing numeric selector resolver
- **Estimate identity:** `candidate_content.estimate_id` with parent Itinerary verification
- **Single transaction:** `with_transaction("delete_estimate confirm")`
- Transaction内で再 parse / blocking 再確認 / target 再 resolve / Estimate 再取得 / 親子検証
- **Preview agreement** + **TOCTOU** (`expected_*` + `estimate_delete_preview` snapshot)
- **Scoped DELETE:** `WHERE id AND itinerary_id` — exactly one row
- **Post-delete verification** — re-fetch confirms row not found
- **Success field:** `deleted_estimate_id` (success-only)
- **missing / already deleted** → blocking (not idempotent success)
- **mutation fields rejected**; optional `expected_*` baseline mismatch → blocking
- **Expense rows unchanged**; sibling / foreign Estimates unchanged; no schema / migration changes
- Transaction rollback on failure — not user-facing undo

## Confirm safety tests (v4.8.2)

| Contract | Test | Notes |
|---|---|---|
| Success DELETE | `confirm_delete_estimate_writes_db` | Target removed; Expense unchanged |
| Candidate / preview mismatch | `confirm_delete_estimate_candidate_preview_mismatch_blocks_db_write` | Stale gate report + changed confirm candidate rejected; DB unchanged. Unit test only (same pattern as `update_estimate` / `delete_itinerary`). |
| TOCTOU / gate snapshot drift | `confirm_delete_estimate_toctou_blocks_db_write` | Gate snapshot vs current row mismatch rejected |
| Scoped DELETE 0 rows | `delete_estimate_row_scoped_rejects_zero_rows_for_missing_estimate` / `_wrong_itinerary_id` | Missing estimate ID or wrong `itinerary_id` → error; DB unchanged |
| CLI success + isolation | `basic_confirm_deletes_estimate`, `sibling_and_foreign_estimates_unchanged_on_confirm` | Trip snapshot / raw column comparison |
| Second confirm | `confirm_second_attempt_blocks_as_not_found` | Already-deleted blocking |
| Stale row after gate | `stale_row_confirm_blocks_without_db_write` | Update row + stale `expected_*` → baseline mismatch |

**Test layering:** dry-run and confirm share the same parser / resolver; not every blocking case is duplicated as a confirm-only CLI test.

## Not in scope

- user-facing undo / restore, soft delete
- Expense mutation, Estimate create, Estimate-to-Expense conversion
- Trip / Day target, bulk / upsert
- structured error 全面導入, idempotency key
- DB schema / migration changes
