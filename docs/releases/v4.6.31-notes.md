# v4.6.31 — Expense write path migration plan

## Summary

**v4.6.31** is a **documentation-only migration plan** after [v4.6.30](../specifications/v4.6.30-expense-write-path-boundary-review.md). It defines phased migration of expense add/update/delete to thin write services (Option W-B), and clarifies the DB-level integrity vs application-level validation boundary raised by external review (§15).

Plan: [v4.6.31-expense-write-path-migration-plan.md](../specifications/v4.6.31-expense-write-path-migration-plan.md)

## Headline decisions

- **Pattern:** 1 command = 1 service module (`expense_add` / `expense_update` / `expense_delete`)
- **Handler keeps:** clap args, success `println!`, human display
- **Service owns:** parse shared options, mutation orchestration, enriched result (add/update)
- **Delete result:** snapshot `{ id, amount, currency }` — no enrich; amount-only display unchanged
- **Repository layer:** out of scope — defer to v4.7.x
- **Adapter:** `print_expense_detail` until W-3; then `_from_enriched`; cleanup at W-5
- **Next implementation:** v4.6.32 — Phase W-2+W-3 together (recommended)

## DB integrity vs application validation (§15)

External security/integrity review cross-checked against current code:

- **False positives (not applicable):** `f64` amount (actually `INTEGER`), unvalidated `expense_date` (validated), `PRAGMA foreign_keys` off (set on), missing itinerary add (exists)
- **Valid concern (adopted):** `expenses` / `reservations` / `estimates` `itinerary_id` has **no DB-level FOREIGN KEY**; referential integrity relies on application-level validation (`get_itinerary_item` + app cascade)
- **Docs corrected:** v4.6.30 "itinerary_id FK" wording withdrawn → "application-level validated logical reference; no DB FK"
- **Backlog:** DB FK introduction deferred to migration runner track (orphan detection → user_version → table rebuild + FK), per v4.6.1/v4.6.2 ordering

No schema/migration/FK change in v4.6.31.
