# v1.9.0 вҖ” Sequence-first ordering unification

## Summary

v1.9.0 unifies Itinerary ordering across CLI list views and export paths to **Sequence-first**: within each Day, items sort by **`sort_order` вҶ’ `id`**. `start_time` remains an optional time label; its value is **not** changed by this release.

## What changed

| Output / operation | Before (v1.8.1) | After (v1.9.0) |
|---|---|---|
| `itinerary list` / `timeline` | Time-first | **Sequence-first** |
| `day show` | Time-first | **Sequence-first** |
| `trip export` v3 (`days[].itineraries[]`) | Time-first (Day еҶ…еҶҚ sort) | **Sequence-first** |
| `trip export-md` | Sequence-first | Sequence-first (unchanged) |

Shared ordering constant: `ITINERARY_LIST_ORDER_BY` in `src/itinerary.rs`:

```text
ORDER BY day_number, sort_order, id
```

Markdown export delegates to `list_itinerary_items` (no duplicate SQL).

## Behavioral breaking change (not a DB breaking change)

| Topic | v1.9.0 behavior |
|---|---|
| **DB schema / migration** | **Unchanged** (`schema_version` in DB remains `3`) |
| **Export JSON schema** | Still `schema_version: 3` вҖ” field names unchanged |
| **`start_time` values** | **Unchanged** |
| **Export JSON array order** | **Changed** вҖ” `days[].itineraries[]` now follows `sort_order`, not `start_time` |

This is **not** a database destructive change. Existing `caglla.db` files work without migration.

It **is** a **behavioral breaking change** for tools that treat `days[].itineraries[]` array order as chronological order by `start_time`. After v1.9.0, consumers must use **`sort_order` as the primary sequence key** within a Day. `start_time` may disagree with display order (e.g. time-less rows at `--order 10` appear mid-day, not at the end).

Example (canonical sample Day 1): `иҰіе…ү:йҰ–йҮҢеҹҺ` (`sort_order: 10`, no `start_time`) moves **before** later timed rows such as `еӨ•йЈҹгҒ®иІ·гҒ„еҮәгҒ—` (`sort_order: 12`), instead of appearing after all timed rows.
