# v4.6.13 — Reservation list service boundary

## Summary

**v4.6.13** introduces a minimal **service layer** for the read-only `reservation list` command, keeping existing CLI human output (Trip context vs Itinerary table), JSON shape (`ReservationListJson`), ordering, and not-found behavior unchanged while making the use case reusable for future Tauri GUI screens.

Specification: [v4.6.13-reservation-list-service-boundary.md](../specifications/v4.6.13-reservation-list-service-boundary.md)

## What changed

- New `src/services/reservation_list.rs` — `list_reservations()` returns `ReservationListServiceResult` (no terminal I/O)
- `reservation list` CLI handler calls service, then routes to existing human/JSON output

## Compatibility

- `reservation list` human output unchanged (Trip with Day/Itinerary context, Itinerary table)
- `reservation list --json` unchanged (`ReservationListJson`)
- `itinerary show` Reservations section unchanged (separate handler)
- No DB schema or migration changes
- Golden updates: Okinawa Markdown Colophon `Version:` only

## Non-goals

- No `reservation show` / write command changes
- No `expense list` / `checklist list` service split
- No Tauri / GUI implementation
