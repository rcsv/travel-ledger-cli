# v4.7.48 — P-6o update_estimate --confirm

## Summary

**v4.7.48** implements Proposal Fragment **`update_estimate --confirm`** (P-6o).

This release adds the DB UPDATE path for existing Estimates via `fragment apply --confirm`, reusing the v4.7.47 dry-run parser and preview contracts.

## What shipped (confirm only)

- **`fragment apply --confirm --trip <id>`** supports intent `update_estimate`
- **Itinerary target only** — ID-first selector → sort_order fallback → ambiguity reject
- **Estimate identity:** `candidate_content.estimate_id` with parent Itinerary verification
- **Single transaction:** `with_transaction("update_estimate confirm")`
- Transaction内で再 parse / blocking 再確認 / target 再 resolve / Estimate 再取得 / 親子検証
- **Preview agreement** + **TOCTOU** (`expected_*` + `estimate_field_changes[].before`)
- **Scoped UPDATE:** `WHERE id AND itinerary_id` — exactly one row
- **Post-update verification** — re-fetch row and compare to proposed values
- **Success field:** `updated_estimate_id` (success-only)
- **no-op** (same proposed values) → valid success + UPDATE (`updated_at` refreshed)
- **amount / currency:** amount-only allowed; currency-only rejected; numeric JSON amount = minor units (no double conversion)
- **null / clear:** explicit JSON `null` blocking; clear flags only for title/note; value + clear conflict blocking; empty string → NULL normalization
- **Expense rows unchanged**; no schema / migration changes

## Confirm safety tests (v4.7.48)

The following contracts are covered by unit tests added in this release:

| Contract | Test | Notes |
|---|---|---|
| Candidate / preview mismatch | `confirm_update_estimate_candidate_preview_mismatch_blocks_db_write` | Stale gate report (dry-run amount `1250`) + changed confirm candidate (`1300`) is rejected; `updated_estimate_id` absent; Estimate / `updated_at` / Expense unchanged. CLI cannot reproduce this in one invocation (gate and transaction share the same JSON read); verified without production fault injection via `apply.rs` unit test (same pattern as `delete_itinerary` TOCTOU test). |
| Scoped UPDATE 0 rows | `update_estimate_row_scoped_rejects_zero_rows_for_missing_estimate` / `_wrong_itinerary_id` | Missing estimate ID or wrong `itinerary_id` → error; DB unchanged |
| Post-update verification mismatch | `verify_updated_estimate_matches_proposed_rejects_amount_mismatch` | Proposed vs re-fetched row mismatch detected; no success result |

**Test layering:** dry-run and confirm share the same parser / resolver; not every blocking case is duplicated as a confirm-only CLI test. Confirm tests focus on transaction, preview agreement, TOCTOU, scoped UPDATE, post-update verification, and rollback.

**Duplicate `#[test]` fix:** `fragment_apply_cli` `update_estimate` targeted count is **43** (was 44) after removing a duplicate attribute on `cli_fragment_apply_update_estimate_trip_target_blocks_without_db_write` — expected reduction.
