# v4.7.42 — P-6n Fragment apply add_estimate dry-run

## Summary

**v4.7.42** adds dry-run preview support for Proposal Fragment **`add_estimate`** on existing Itinerary targets.

```text
fragment apply <fragment.json> --dry-run --trip <id>
intent: add_estimate
target_type: itinerary
```

The operation validates Estimate candidate fields, resolves one existing Itinerary, produces a structured gate report, and can write a schema v8 preview Trip JSON with exactly one additional Estimate under `days[].itineraries[].estimates[]`. The database remains unchanged.

Spec: [v4.7.42-fragment-apply-add-estimate-dry-run.md](../specifications/v4.7.42-fragment-apply-add-estimate-dry-run.md)

## Semantic boundary

```text
Estimate = Planned Money
Expense  = Actual Money
```

```text
add_estimate -> preview Estimate only
add_expense  -> Expense / Actual Money
```

`add_estimate` never previews or creates an Expense row. There is no automatic Estimate-to-Expense conversion.

## Target behavior

- Supported target: `target_type = itinerary`
- Numeric selector rule: itinerary ID first, then `sort_order` fallback, reject ambiguity
- Itinerary ID selectors now work for `add_estimate`
- CLI `sort_order` and title selectors remain available
- Missing / unresolved / ambiguous targets block
- Trip and Day targets block
- Resolution remains scoped to selected `--trip`

## Candidate fields

Required:

```text
amount
currency
```

Optional:

```text
title
note
sort_order
```

Amount accepts string decimal input through the shared money parser or integer JSON minor units. Currency uses the existing validation and normalization contract. Optional text aliases remain `title -> description -> label` and `note -> memo -> fragment.notes`; empty optional text normalizes to null. `sort_order` is an optional integer and defaults to `0`.

## Preview report

The gate report adds:

```text
estimates_before
estimates_after
estimate_preview
```

`estimate_preview` includes `target_itinerary_id`, `target_itinerary_title`, normalized minor-unit `amount`, `currency`, optional `title`, optional `note`, and `sort_order`. The before / after counts are Trip-wide Estimate counts.

## Preview export behavior

A valid dry-run appends exactly one in-memory Estimate under:

```text
days[].itineraries[].estimates[]
```

The preview export remains schema v8, passes `trip validate-export`, and does not change expenses, reservations, notes, Itinerary fields, or the database.

Invalid reports return `valid=false`, include blocking errors, do not emit preview output under the current safety policy, and leave the database unchanged. Non-empty `required_decisions` follows the same invalid-output policy.

## Confirm boundary

`add_estimate --confirm` remains unsupported in v4.7.42.

Not included:

- Estimate INSERT from Fragment apply
- `inserted_estimate_id`
- `ConfirmInsertResult::Estimate`
- `execute_confirm_add_estimate`
- confirm dispatch for `add_estimate`
