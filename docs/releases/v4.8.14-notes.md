# v4.8.14 — Proposal Fragment apply strict currency integration

## Summary

**v4.8.14** applies `CurrencyValidationMode::IsoStrict` to **Proposal Fragment apply money intents**:

- `add_expense` — `candidate_content.currency`（常に strict）
- `add_estimate` — 同上
- `update_estimate` — currency 明示時のみ strict；未指定 update では既存 currency を再検証しない

対象 phase: dry-run / simulate gate + confirm 再 parse（同一 validation helper）。

Cargo version bumped to `4.8.14`. Markdown colophon synced. Domain layer and non-Fragment-apply paths remain on `FormatOnly`.

Implementation spec: [v4.8.14-currency-fragment-apply-strict-integration.md](../specifications/v4.8.14-currency-fragment-apply-strict-integration.md)

## Fragment apply behavior change

- Valid ISO codes succeed with uppercase normalization (`jpy` → `JPY`)
- Unknown codes (`ZZZ`, `ABC`, `JPN`) blocked on Fragment apply money intents
- Denylist codes (`XXX`, `XTS`, `XAU`, etc.) blocked
- Format-invalid input blocked (unchanged messages)
- Blocking reports emit `structured_errors[]` with reused `APPLY_FIELD_INVALID` + legacy `errors[]` mirror
- Confirm path uses the same strict validation as dry-run; blocking → no DB write / rollback

## Structured errors

- **No new error codes** — `APPLY_FIELD_INVALID` reused
- `field_path`: `candidate_content.currency`
- `phase`: `simulate`
- `message` is unstable; clients branch on `code` + `field_path`

## Compatibility

- Legacy unknown currency in existing DB: read/list/show/export still work
- `update_estimate` without explicit `currency`: does not re-validate existing currency
- CLI write path (v4.8.13): unchanged
- import / validate-export / domain layer: format-only (unchanged)
- Receipt/inbox CLI: deferred to v4.8.15+
- DB schema / JSON report `schema_version`: unchanged
- JSON golden: unchanged

## Format-only surfaces (unchanged)

- `validate_currency_code()` / domain layer
- `trip import`, `validate-export`
- Receipt / inbox CLI
- read / list / show / export / export-md
